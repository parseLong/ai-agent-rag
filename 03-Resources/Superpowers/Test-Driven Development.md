# Test-Driven Development

> 在实现任何功能或修复 Bug 之前使用，在编写实现代码之前。

---

## 核心原则

**铁律：**
```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

先写代码？删除它。重新开始。

**违反规则的字面意义就是违反规则的精神。**

## 红-绿-重构循环

### RED — 编写失败的测试

- 编写一个最小测试，展示应该发生什么
- 一次只测试一个行为
- 清晰的名称
- 真实代码（除非不可避免，否则不使用模拟）

### Verify RED — 观察它失败

**强制。永不跳过。**

```bash
npm test path/to/test.test.ts
```

确认：
- 测试失败（不是错误）
- 失败消息是预期的
- 失败是因为功能缺失（不是拼写错误）

**测试通过？** 你在测试现有行为。修复测试。
**测试出错？** 修复错误，重新运行直到正确失败。

### GREEN — 最小代码

- 编写最简单的代码使测试通过
- 不要添加功能、重构其他代码或"改进"超出测试范围的内容

### Verify GREEN — 观察它通过

**强制。**

确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（无错误、警告）

### REFACTOR — 清理

仅在绿色后：
- 移除重复
- 改进名称
- 提取辅助函数

保持测试绿色。不要添加行为。

## 好测试的特征

| 质量 | 好的 | 坏的 |
|------|------|------|
| **最小** | 一件事。名称中有"and"？拆分。 | `test('validates email and domain and whitespace')` |
| **清晰** | 名称描述行为 | `test('test1')` |
| **展示意图** | 展示期望的 API | 掩盖代码应该做什么 |

## 常见辩解和真相

| 借口 | 真相 |
|------|------|
| "太简单了，不需要测试" | 简单代码也会坏。测试只需要 30 秒。 |
| "稍后我会测试" | 测试立即通过证明不了什么。 |
| "测试后写能达到同样的目标" | 测试后 = "这做什么？" 测试前 = "这应该做什么？" |
| "我已经手动测试了所有边界情况" | 手动 ≠ 系统化。没有记录，无法重新运行。 |
| "删除 X 小时的工作是浪费" | 沉没成本谬误。无法验证的代码是技术债务。 |
| "TDD 是教条，我在务实" | TDD 就是务实的：提前发现 bug，防止回归。 |

## 验证清单

- [ ] 每个新函数/方法都有测试
- [ ] 实现前观察每个测试失败
- [ ] 每个测试因预期原因失败（功能缺失，不是拼写错误）
- [ ] 编写最小代码使每个测试通过
- [ ] 所有测试通过
- [ ] 输出干净（无错误、警告）
- [ ] 测试使用真实代码（除非不可避免，否则不使用模拟）
- [ ] 覆盖边界情况和错误情况
