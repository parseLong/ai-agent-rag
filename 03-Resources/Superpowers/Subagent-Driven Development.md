# Subagent-Driven Development

> 在会话中执行实施计划时使用，每个任务派遣全新子代理，进行两阶段审查（先规格合规性，再代码质量）。

---

## 核心原则

**子代理驱动开发** = 每个任务一个全新子代理 + 两阶段审查 = 高质量、快速迭代

**为什么使用子代理：**
- 委托给具有隔离上下文的专门代理
- 精确构建指令和上下文，确保他们保持专注
- 他们不会继承会话上下文或历史
- 保留你自己的上下文用于协调工作

## 模型选择

| 任务类型 | 推荐模型 |
|---------|---------|
| **机械实现**（孤立函数，1-2 个文件） | 快速、便宜的模型 |
| **集成和判断**（多文件协调、模式匹配、调试） | 标准模型 |
| **架构、设计和审查** | 最强可用模型 |

**复杂度信号：**
- 触及 1-2 个文件，完整规格 → 便宜模型
- 触及多个文件，有集成问题 → 标准模型
- 需要设计判断或广泛的代码库理解 → 最强模型

## 工作流程

```
读取计划 → 提取所有任务 → 创建 TodoWrite → 循环执行任务 → 最终审查
```

### 每个任务的流程

1. **派遣实现者子代理** — 提供完整任务文本 + 上下文
2. **实现者提问？** — 回答问题，提供上下文
3. **实现者实现、测试、提交、自审** — 完成任务
4. **派遣规格审查者子代理** — 确认代码符合规格
5. **派遣代码质量审查者子代理** — 确认代码质量
6. **标记任务完成** — 在 TodoWrite 中更新状态

### 实现者状态处理

| 状态 | 处理 |
|------|------|
| **DONE** | 继续规格合规性审查 |
| **DONE_WITH_CONCERNS** | 阅读担忧，关于正确性或范围的问题在审查前解决 |
| **NEEDS_CONTEXT** | 提供缺失的上下文，重新派遣 |
| **BLOCKED** | 评估阻碍：上下文问题→提供更多上下文；需要更多推理→更强模型；任务太大→拆分；计划本身错误→升级给人类 |

## 红线和禁止事项

**永远不要：**
- 在未经用户明确同意的情况下在主分支上开始实现
- 跳过审查（规格合规性 OR 代码质量）
- 继续执行未修复的问题
- 并行派遣多个实现者子代理（会产生冲突）
- 让子代理读取计划文件（提供完整文本代替）
- 跳过场景设置上下文
- 忽略子代理的问题
- 在规格合规性通过前开始代码质量审查
- 在审查有未解决问题时转到下一个任务

## 优势

| 对比 | 优势 |
|------|------|
| **vs. 手动执行** | 子代理自然遵循 TDD；每个任务有新鲜上下文；并行安全；可以提问 |
| **vs. Executing Plans** | 同一会话（无需交接）；连续进展（无需等待）；审查检查点自动 |
