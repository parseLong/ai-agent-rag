# Brainstorming

> 在写任何代码之前使用。通过苏格拉底式对话将想法精炼为完整的设计和规格说明。

---

## 核心原则

**HARD-GATE**: 在没有呈现设计并获得用户批准之前，不要调用任何实现技能、写任何代码、搭建任何项目或采取任何实现行动。

## 检查清单

必须按顺序完成以下步骤：

1. **探索项目上下文** — 检查文件、文档、最近的提交
2. **提供视觉伴侣** (如果需要) — 单独的消息，不与其他内容混合
3. **提出澄清问题** — 一次一个问题
4. **提出 2-3 种方案** — 附带优劣势和推荐
5. **呈现设计** — 按复杂度分节展示，每节后获得批准
6. **编写设计文档** — 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
7. **规格自审** — 检查占位符、矛盾、歧义、范围
8. **用户审查规格** — 要求用户在继续前审查规格文件
9. **过渡到实现** — 调用 `writing-plans` 技能创建实施计划

## 关键原则

- **一次一个问题** — 不要一次提出多个问题
- **优先选择题** — 如果可能，使用多选题而非开放式问题
- **YAGNI** — 从设计中删除不必要的功能
- **探索替代方案** — 在确定前总是提出 2-3 种方案
- **增量验证** — 先呈现设计，获得批准后再继续
- **保持灵活** — 当某些内容不清楚时，返回澄清

## 视觉伴侣

当预计问题涉及视觉内容（模型、布局、图表）时，提供浏览器伴侣：

> "Some of what we're working on might be easier to explain if I can show it to you in a web browser. I can put together mockups, diagrams, comparisons, and other visuals as we go. Want to try it?"

**注意：** 即使接受，也要对每个问题判断是否使用浏览器：
- **使用浏览器**：模型、线框图、布局比较、架构图
- **使用终端**：需求问题、概念选择、权衡列表

## 设计文档保存

验证后的设计保存到：
```
docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
```
