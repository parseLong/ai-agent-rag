# Systematic Debugging

> 在遇到任何 bug、测试失败或意外行为时使用，在提出修复之前。

---

## 核心原则

**铁律：**
```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果没有完成第 1 阶段，你不能提出修复。

## 四个阶段

### Phase 1: 根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误消息** — 不要跳过错误或警告，它们通常包含确切的解决方案
2. **稳定复现** — 能可靠触发吗？确切的步骤是什么？
3. **检查最近更改** — Git diff、最近提交、新依赖、配置更改
4. **在多组件系统中收集证据** — 在每个组件边界添加诊断日志
5. **追踪数据流** — 坏值从哪里开始？向上追溯直到找到源头

### Phase 2: 模式分析

**修复前找到模式：**

1. **找到工作示例** — 在同一代码库中找到类似的工作代码
2. **与参考对比** — 如果是实现模式，完整阅读参考实现
3. **识别差异** — 工作和损坏之间有什么不同？列出每一个差异
4. **理解依赖** — 需要哪些其他组件？什么设置、配置、环境？

### Phase 3: 假设和测试

**科学方法：**

1. **形成单一假设** — 清晰陈述："我认为 X 是根因，因为 Y"
2. **最小化测试** — 做最小的可能改变来测试假设
3. **验证后继续** — 有效？→ Phase 4。无效？→ 形成新假设
4. **不知道时** — 说"我不懂 X"。不要假装知道

### Phase 4: 实现

**修复根因，不是症状：**

1. **创建失败测试用例** — 最简单的复现
2. **实现单一修复** — 一次只改一处
3. **验证修复** — 测试通过了吗？其他测试没坏吗？
4. **如果修复无效** — 停下来。如果 < 3 次：返回 Phase 1。如果 ≥ 3 次：**质疑架构**

## 红旗 — STOP 并遵循流程

- "先快速修复，稍后调查"
- "试试改 X 看看是否有效"
- "添加多个更改，运行测试"
- "跳过测试，我会手动验证"
- "很可能是 X，让我修复它"
- "我不太理解但这可能有效"
- "再试一次修复"（已经试了 2+ 次）

**所有这些意味着：STOP。返回 Phase 1。**

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 阅读错误、复现、检查更改、收集证据 | 理解 WHAT 和 WHY |
| **2. 模式** | 找到工作示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug 解决，测试通过 |
