# Verification Before Completion

> 在声称工作完成、修复通过或测试通过之前使用，在提交或创建 PR 之前。

---

## 核心原则

**没有新鲜验证证据，不能声称完成。**

声称完成而没有验证是不诚实，不是效率。

## 铁律

```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
```

如果当前消息中没有运行验证命令，你不能声称它通过。

## 门控函数

```
BEFORE 声称任何状态或表达满意：

1. IDENTIFY: 什么命令能证明这个声称？
2. RUN: 执行完整命令（新鲜、完整）
3. READ: 完整输出，检查退出码，统计失败数
4. VERIFY: 输出确认声称吗？
   - 如果否：用证据陈述实际状态
   - 如果是：用证据陈述声称
5. ONLY THEN: 做出声称

跳过任何步骤 = 撒谎，不是验证
```

## 常见失败

| 声称 | 需要 | 不足够 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 失败 | 上次运行、"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查、外推 |
| 构建成功 | 构建命令：exit 0 | Linter 通过，日志看起来好 |
| Bug 修复 | 测试原始症状：通过 | 代码改了，假设修复了 |
| 回归测试有效 | 红绿循环验证 | 测试通过一次 |
| Agent 完成 | VCS diff 显示更改 | Agent 报告"成功" |
| 要求满足 | 逐行检查清单 | 测试通过 |

## 红旗 — STOP

- 使用"应该"、"可能"、"似乎"
- 验证前表达满意（"Great!"、"Perfect!"、"Done!"）
- 未验证就提交/推送/PR
- 信任 Agent 成功报告
- 依赖部分验证
- 想"就这一次"
- 累了，想结束工作
- **任何暗示成功但没有运行验证的措辞**

## 辩解预防

| 借口 | 真相 |
|------|------|
| "现在应该有效了" | 运行验证 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 无例外 |
| "Linter 通过了" | Linter ≠ 编译器 |
| "Agent 说成功" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查就够了" | 部分证明不了什么 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到: 34/34 通过] "所有测试通过"
❌ "现在应该通过了" / "看起来正确"
```

**回归测试（TDD 红绿）：**
```
✅ 编写 → 运行（通过）→ 恢复修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了回归测试"（没有红绿验证）
```

**构建：**
```
✅ [运行构建] [看到: exit 0] "构建通过"
❌ "Linter 通过了"（linter 不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建检查清单 → 验证每个 → 报告缺口或完成
❌ "测试通过，阶段完成"
```

**代理委托：**
```
✅ Agent 报告成功 → 检查 VCS diff → 验证更改 → 报告实际状态
❌ 信任 Agent 报告
```
