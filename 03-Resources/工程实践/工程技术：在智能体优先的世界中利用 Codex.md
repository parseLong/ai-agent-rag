# 工程技术：在智能体优先的世界中利用 Codex

> [!quote] 核心论题
> ==人类掌舵。智能体执行。== 工程师的工作不再是编写代码，而是设计环境、明确意图和构建反馈回路，使智能体能够可靠地工作。

> [!info] 实验背景
> OpenAI 团队用 5 个月时间，从空的 Git 仓库出发，构建了一款百万行代码的产品。**没有一行代码是人工编写的**——从应用逻辑、测试、CI 配置、文档到内部工具，全由 Codex 生成。3 名工程师推动约 1,500 个 PR，平均每人每天 3.5 个 PR。

---

## 📌 核心概念

> 智能体优先世界中的工程，不再是"写代码"，而是"设计让智能体能可靠工作的环境"。

---

## 🧠 关键知识点

### 1. 重新定义工程师的角色

- **定义**：工程师工作重点转向 ==系统、架构和杠杆作用==
- **理解**：当人类不再写代码，主要任务变成协助智能体完成有用的工作——拆解目标、提示智能体构建模块、追问"还需要什么能力"
- **类比**：就像给新入职工程师提供引导（产品原则、工程规范、团队文化），将这些信息提供给智能体会带来更一致的输出

> [!important] 关键洞察
> 当事情不顺利时，解决方案不是"再努力一点"，而是追问：**"究竟还需要什么样的能力，又如何让它对智能体既清晰可读又可强制执行？"**

**交互方式**：
- 人类几乎完全通过提示与系统交互
- 描述任务 → 运行智能体 → 打开 PR
- 指示 Codex 审核自身更改 → 循环至所有智能体审核员满意（Ralph Wiggum 循环）
- 人类可以审核 PR，但非必须——**几乎所有审核已调整为智能体对智能体方式**

### 2. 提高应用程序的可读性

- **定义**：令 UI、日志、指标等内容对 Codex ==直接可读==，突破人类 QA 能力的瓶颈
- **理解**：人类时间和注意力是固定限制因素，为智能体增加更多功能是破局之道
- **类比**：就像给 QA 工程师配备完整的调试工具链

**三大可读性增强**：

| 能力 | 做法 | 效果 |
|------|------|------|
| UI 验证 | 应用根据 git worktree 启动 + Chrome DevTools 协议接入 | Codex 复现错误、验证修复、推理 UI 行为 |
| 日志查询 | 本地可观测性堆栈（LogQL/PromQL/TraceQL） | "服务启动 800ms 内完成"等量化提示变得可行 |
| 独立环境 | 每次任务运行在完全独立版本上，完成后删除 | 智能体无干扰工作，单次运行可持续 6+ 小时 |

### 3. 代码仓库作为记录系统——给地图而非说明书

- **定义**：==AGENTS.md 是内容目录，不是百科全书==；代码仓库的知识库位于结构化 `docs/` 目录
- **理解**："一个大型的 AGENTS.md"方案失败的原因：
  - **情境是稀缺资源**——巨大指令文件挤掉任务、代码和相关文档
  - **过多指导反而无效**——当一切都"重要"，一切都不重要
  - **立即腐烂**——庞杂手册变成陈旧规则的坟场
  - **难以核实**——单个 blob 不适合机械检查

**渐进式披露架构**：

```
AGENTS.md          ← ≈100行，地图/索引
ARCHITECTURE.md    ← 域和包分层顶层地图
docs/
├── design-docs/   ← 设计文档（含验证状态+核心理念）
├── exec-plans/    ← 执行计划（active/completed/tech-debt）
├── generated/     ← 自动生成（如 db-schema.md）
├── product-specs/ ← 产品规格
├── references/    ← 外部参考（llms.txt 格式）
├── DESIGN.md      ← 各领域顶层文档
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

> [!tip] 关键原则
> 智能体从一个小而稳定的切入点开始，被指导下一步该去哪里查看，而不是一开始就被淹没。

**执行保障**：
- 专职 linter 和 CI 作业验证知识库更新状况、交叉链接和结构
- "doc-gardening" 智能体定期扫描过时文档，发起修复 PR

### 4. 智能体可读性——Codex 看不到的东西就不存在

- **定义**：智能体运行时无法在情境中访问的任何内容 ==都不存在==
- **理解**：Google Docs、聊天记录或人们头脑中的知识无法被系统访问——必须以 Markdown 形式编码到代码库中
- **类比**：就像迟了三个月入职的新员工，对之前的 Slack 讨论一无所知

**框架带来的取舍**：
- 倾向于可完全内化于仓库中推理的依赖项和抽象
- "枯燥"技术更受青睐（可组合性、API 稳定性、训练集表现）
- 有时让智能体重新实现功能子集比绕过不透明的上游行为更便宜
  - 例：没有用 `p-limit` 包，而是自实现带并发的 map 辅助函数——与 OTeL 仪表紧密集成、100% 测试覆盖率、行为完全可控

### 5. 规范架构与品味——强制不变量而非微观管理

- **定义**：==通过强制执行不变量，而非对实施过程进行微观管理==，令智能体快速交付且不削弱基础
- **理解**：智能体在严格边界和可预测结构的环境中最为高效
- **类比**：类似领导大型工程平台组织——中央强制边界，本地允许自主

**分层领域架构**：

每个业务域划分为固定层，依赖方向严格验证：

```
Types → Config → Repo → [Providers] → Service → Runtime → UI
                        ↑
              横切关注点（认证/连接器/遥测/功能标志）通过单一显式接口进入
```

**品味不变式**：
- 结构化日志记录（自定义 lint 静态强制）
- 模式和类型的命名约定
- 文件大小限制
- 特定平台的可靠性要求
- 错误信息中注入修复指令（帮助智能体自我修复）

> [!tip] 关键取舍
> 生成的代码不总是符合人类风格偏好，这没关系。只要输出正确、可维护、对未来智能体运行清晰易读，就达标。

**品味反馈循环**：审查评论 → 文档更新 / 工具编码 → 全量应用

### 6. 吞吐量改变合并理念

- 高吞吐量下，**纠错成本低，等待成本高**
- Pull Request 生命周期很短
- 测试偶发失败通过后续重跑解决，不无限期阻碍进展
- 低吞吐量环境中不负责任的做法，在高吞吐量中是正确选择

### 7. 智能体生成的完整含义

"智能体生成"意味着**整个代码库**——不仅限于产品代码：

| 产出类型 | 示例 |
|----------|------|
| 产品代码与测试 | 应用逻辑、单元/集成测试 |
| CI 配置和发布工具 | GitHub Actions、部署脚本 |
| 内部开发者工具 | 辅助开发脚本 |
| 文档和设计历史 | 设计文档、AGENTS.md |
| 评估框架 | 测试覆盖率、质量评分 |
| 审阅评论和回复 | PR review comments |
| 管理代码仓库的脚本 | lint、doc-gardening |
| 生产仪表板定义 | Grafana/监控配置 |

### 8. 端到端自主能力

Codex 已跨过重要门槛，可端到端驱动新功能：

```
验证代码库当前状态 → 复现已报告漏洞 → 录制故障演示视频
→ 实施修复 → 运行应用验证修复 → 录制解决演示视频
→ 打开 PR → 回应反馈 → 修复构建故障 → 仅需判断时交由人工 → 合并
```

> [!caution] 注意
> 此行为高度依赖特定代码仓库结构和工具，不应在没有类似投入的情况下假定可以泛化。

### 9. 熵与垃圾收集——对抗智能体漂移

- **定义**：完全自主的智能体会复现代码库中的模式——包括不均衡或不够理想的模式，导致漂移
- **理解**：最初团队每周五花 20% 时间清理"AI 残渣"，不可持续
- **类比**：类似垃圾回收——技术债务像高息贷款，持续小额偿还比累积后痛苦一次解决更好

**"黄金原则"编码**：
1. 倾向共享实用程序包而非手工辅助工具——集中不变式
2. 不"YOLO 式"探测数据——验证边界或依赖类型化 SDK

**循环清理流程**：定期运行后台 Codex 任务 → 扫描偏差 → 更新质量等级 → 发起重构 PR → 大多数一分钟内审查自动合并

> [!important] 核心机制
> 人类品味一旦被捕捉，就**持续应用于每一行代码**。每天发现并解决不良模式，不让其在代码库中传播数天或数周。

### 10. 仍在学习的挑战

- 架构连贯性如何在完全由智能体生成的系统中随时间演变？
- 人类判断力在哪些方面发挥最大作用？如何编码使其发挥更大作用？
- 模型功能不断增强，系统将如何演变？

> [!quote] 核心挑战
> 当前最棘手的挑战集中在 ==设计环境、反馈回路和控制系统== 方面——帮助智能体实现目标：大规模构建和维护复杂、可靠的软件。

## 💡 核心洞察提炼

### 智能体优先工程的六大原则

| # | 原则 | 一句话 |
|---|------|--------|
| 1 | **人类掌舵** | 工程师设计环境而非写代码 |
| 2 | **给地图而非说明书** | AGENTS.md 是索引，知识库在 docs/ |
| 3 | **不可见即不存在** | 所有知识必须编码到仓库 |
| 4 | **强制不变量而非微观管理** | lint + 结构测试机械执行 |
| 5 | **纠错低等待高** | 高吞吐量下快速迭代优于完美阻塞 |
| 6 | **品味即代码** | 人类偏好编码到 linter/工具中持续应用 |

## 📝 学习总结

1. **智能体优先世界中，工程师的角色从"写代码"变为"设计让智能体可靠工作的环境"**——拆解目标、提供情境、构建约束、编码品味。
2. **情境管理的核心是"给地图而非说明书"**——渐进式披露让智能体从小而稳定的切入点开始，不被淹没；不可见即不存在，所有知识必须编码到仓库。
3. **对抗智能体漂移的关键是"品味即代码"**——将人类判断力编码为 linter/工具/黄金原则，持续应用于每一行代码，像垃圾回收一样小额偿还技术债务。
