# AI不缺智商缺纪律：我的Harness工程化实践

> 核心观点：AI Coding 的瓶颈正从"模型能力"转移到"流程工程"——模型已经足够聪明，但不稳定，而稳定性必须由外部框架供给。

> 对付 AI 的不确定性，堆 prompt 是负债，做框架才是资产。

---

## 核心摘要

杜学友两个月 Harness 演进复盘，提供了一套可抄的实践方案：

1. **五层架构**：常驻入口层 → 原子规则层 → 角色 Agent 层 → 按需上下文层 → 执行支撑层 + 稳定性支点 → 详见 [[Harness 五层架构实践]]
2. **遗忘根因**：压缩丢失、检索失败、指令遵循失败 → harness 三层设计逐一堵漏 → 详见 [[Agent 遗忘三重根因]]
3. **四阶段演进**：拿来主义 → 重 prompt → 减负分层 → Agent 编排 → 详见 [[Harness 四阶段演进实践]]
4. **经验进化**：lesson → pattern → instinct 三级晋升 → 详见 [[经验三级进化]]
5. **编排选择**：Dispatcher 状态机 + 文件交接 vs Workflow vs Agent Team → 详见 [[Dispatcher 状态机与文件交接]]
6. **七维评分**：100% Python 确定性逻辑、零 LLM 调用 → 详见 [[七维确定性评分]]

## 核心判断

- 主会话的上下文是最贵的资源，不是免费的草稿纸
- 主会话应退化为纯执行器，只听 dispatcher 指令
- 流程强制执行必须从 LLM 推理中外置到确定性基础设施
- 评测平台是评估者，不是执行者
- 宁要可复现的"粗糙分"，不要会漂移的"精准分"
- 失败必须响亮，绝不静默兜底

## 核心金句

> 每条规则都是一次事故的墓志铭。
> 真正需要警惕的不是"agent 多"，而是"agent 间耦合多"。
> prompt 约束是说服，不是强制——你无法用更多的字来对抗概率性的遗忘。
> 真正推动架构演进的从来不是"想要更好"，而是"现在的做法已经崩了"。
> 这套系统最大的风险不是"不够准"，而是"假装它覆盖了一切"。

## 关键研究引用

| 引用 | 核心发现 |
|------|---------|
| VILA-Lab Claude Code 逆向 | 记忆完全基于文件系统，上下文管理靠 5 层渐进式压缩管线 |
| Devin CPO (Latent Space) | 记忆检索到正确记忆——"尚未解决" |
| Stanford "Lost in the Middle" (TACL 2024) | LLM 注意力呈 U 型分布，中部准确率显著下降 |
| NVIDIA RULER | 声称支持 32K+ 的模型仅半数能在该长度保持可靠性能 |
| arxiv 2605.29682 | 验证反馈质量 R²=0.94～0.99，远超 token 消耗 R²=0.33～0.42 |
| VikingMem (VLDB 2026) | 更少 Token 留存 + 更智能组织 > 全量保留（16.82% Token 留存得分 75.80） |

## 欠账与边界

- D2 产物质量只做结构检查，判不了方案设计优劣 → 拟引 LLM 评委 + 多次投票取中位数
- eval 跑批单机单进程 → 需容器化 + 任务持久化
- 评测集仅 5 个 case → 长尾意图覆盖不足
- 流程止步预发，G8 生产上线不强制 → 下一步目标"AI 执行 + 人工兜底"
