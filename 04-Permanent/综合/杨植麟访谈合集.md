# 杨植麟访谈合集

> 杨植麟（月之暗面创始人，Kimi/K2 模型开发者）2025年7月与张小珺的深度访谈，覆盖了AI产品范式、Agent核心能力、泛化与训练、多智能体与AGI分级、推理模型与测试时扩展五大核心主题。本合集将五篇概念卡片与访谈索引整合为完整的知识脉络。

杨植麟在K2模型发布之际接受访谈，核心讨论围绕以下脉络展开：

```
AI产品范式 → Agent定义与核心能力 → Agent泛化瓶颈与训练 → 多智能体与AGI → 推理模型与Test-Time Scaling

主线：从"缸中之脑"到"与世界交互"的范式跃迁
  ├─ 产品层面：一方产品 vs 三方产品
  ├─ 能力层面：多轮 + 工具 = Agent
  ├─ 瓶颈层面：泛化是最大挑战
  ├─ 系统层面：L1-L5 非严格串行
  └─ 技术层面：Test-Time Scaling 统一框架
```

## AI产品范式与开源生态

### "一方产品" vs "三方产品"

在AI Agent产品的开发方式上，存在两种截然不同的范式：

**三方产品（逆向工程）**：基于基础模型，在上面搭建脚手架，通过逆向工程拟合模型训练过程的开发方式。

> 一开始，很多产品是基于基础模型，你在上面搭一些脚手架，或者设计一些工具去更好地让模型使用。它本质做的是逆向工程这个模型的训练过程。

典型代表：[[Manus]]。

**一方产品（正向设计）**：模型公司自己下场做产品，自己控制上下文环境、工具接口、prompt结构，端到端训练模型的开发方式。

> 如果模型公司去做"一方的产品"，逻辑完全不一样。你不再需要这个逆向的过程，更多是正向的做法。我先把这些工具设计好，我的Context Engineering的方法都设计好，我就在这个环境里训练这个模型，所以模型天然在你的环境里表现更好。

典型代表：[[Claude Code]]、ChatGPT Agent。

| 维度 | 三方产品 | 一方产品 |
|------|----------|----------|
| 开发方式 | 逆向工程，拟合模型训练分布 | 正向设计，端到端训练 |
| 上下文控制 | 通过试探找到最优System Prompt | 在训练中内置Context Engineering |
| 工具集成 | 事后适配 | 训前设计，训中深度融合 |
| 上限 | 受限于模型开放的接口和能力 | 垂直整合，上限更高 |

### 模型即产品（Model as Product）

在AI时代，产品开发的逻辑发生了根本性转变：模型训练完成，产品也基本完成。

> 你做一个Agent产品，需要把模型跟工具和Context结合起来。但你会发现，训练模型的时候，基本得把这一整套系统搭好，才能训练这个模型。模型训练完成，产品也基本完成了。

这意味着：
- 模型训练和产品构建是同步进行的
- 工具设计、Context Engineering在训练阶段就已嵌入
- 交互改进是锦上添花，而非产品构建的核心

这与[[AI范式转变]]密切相关——AI开发正从"应用层调用API"转变为"训练即产品"的一体化范式。

### 开源 vs 闭源

当前大模型时代的开源与软件开源有本质不同：

> 如果要贡献到模型本身、让模型变得更强，目前只有原厂能做。

开源的核心价值在于**赋能下游应用**（如基于开源模型训练专用Agent），而非社区贡献反哺基础模型本身。

长期看，开源和闭源之间是动态平衡。杨植麟希望跟社区分享技术know-how，开源协作推动整体技术水位上升——**Race to the top**，而非逐底竞争。

市场趋势：全球大模型市场正在从百花齐放走向少数几家主导的格局。

### Scaling Law与数据飞轮

> Scaling Law遇到数据墙了，这是客观事实。你要突破数据墙，就需要提高token efficiency。但我们现在观察，模型变好的速度并没有减少，甚至在加速。

虽然预训练的Scaling Law撞到了数据墙，但模型的整体提升速度并未放缓——因为强化学习（RL）的scaling效率远高于预训练：RL是on-policy且带负梯度的训练，scaling效率更高。

数据飞轮尚未形成两大原因：
1. **算力驱动的scaling效率极高**：直接scale FLOPs带来的提升远大于基于用户反馈的数据飞轮
2. **大模型对噪声敏感**：数据飞轮依赖外部环境反馈，而这种反馈噪声很大

> 这个平衡什么时候会发生变化？——也有可能你通过新的交互，让你收集到的信号的噪声能够减少。

这暗示了[[AI范式转变]]的另一维度：交互设计本身可能成为解锁数据飞轮的关键。

### 新交互范式

AI时代的交互方式不再固定，而是**随模型能力变化而持续演化**：

| 交互范式        | 特点                    | 代表产品           |
| ----------- | --------------------- | -------------- |
| Chat        | 单轮/多轮对话，即时响应          | ChatGPT        |
| Copilot     | 实时辅助，逐句建议             | GitHub Copilot |
| Cursor      | AI原生的IDE交互            | Cursor         |
| Claude Code | 多步Agentic Coding，异步执行 | Claude Code    |

> 当你有新一代模型，能力提升很多，就会发现交互可以改了。当然，今天Claude Code的交互也不是终极形态，因为模型还会继续提升。

交互范式的代际跃迁直接由模型能力提升驱动。

### 基座模型公司与应用公司的边界

应用公司的生存空间在于：
- 工具的专有性：掌握独特的线下服务入口、专业领域know-how
- 当工具过于通用时，会和一方产品高度重叠，此时垂直整合优势更大

通用Agent的核心逻辑：**通用不是为了技术炫技，而是因为整体商业化效率更高**。

### 通用模型的系统复杂性

通用模型面临"既简单又复杂"的系统挑战：

简单在于：只需要维护一个模型，不需要路由策略。
复杂在于：希望模型在各种场景下都能工作。

具体复杂性：任务间的"打架"问题、Agent泛化的瓶颈、多模态不损伤"脑子"的问题。

## Agent定义与核心能力

### Agent的定义：多轮 + 工具

> 它可能是一个从"缸中之脑"变成可以跟世界交互，因为所谓Agent最重要的特征，就是它可以**多轮地使用工具**。有两个关键点：一个是**多轮**，一个是**工具**。

- **多轮（Multi-turn）**：能执行多次操作，是 [[test-time scaling]] 的一种实现方式——通过增加轮数来规模化扩展推理时的计算量
- **工具（Tools）**：连接模型的"脑"与外部世界的方式。参见 [[工具系统]]

### "缸中之脑"的隐喻

- **"缸中之脑"型模型**：纯粹的推理模型（如o1系列），通过内部的长链式思维进行猜想与验证的循环，但不与外界交互。核心能力是"反思"——不断提出新猜想并自我验证，将 Pass@k 变成 Pass@1。
- **Agentic模型**：从"缸中之脑"升级为"跟世界交互的系统"。一边思考一边操作——调用搜索、使用浏览器、写代码、访问API。

这一转变的核心：从"封闭推理"走向"开放交互"，与 [[智能体循环]] 中的感知-思考-行动循环高度一致。

### test-time scaling：两种路径的共同归宿

> 不管是把轮数打得更多，还是在每一轮有更多思考的token，都是一种规模化token的方法，让你能完成更复杂的任务。

核心洞见：**Agent的多轮工具调用和推理模型的长链式思维，本质上都是在推理阶段规模化消耗token，以换取更高质量的结果输出。**

### Agent与对话模型的区别

| 维度      | 对话模型（Chat） | Agent模型           |
| ------- | ---------- | ----------------- |
| 交互方式    | 单轮输出       | 多轮工具使用            |
| token规模 | 数百token    | 数万至数百万token       |
| 任务复杂度   | 简单问答、文本生成  | 端到端复杂任务           |
| 外部交互    | 无          | 搜索、浏览器、代码执行、API调用 |
| 核心范式    | SFT / RLHF | Agent强化学习         |

### Agentic模型的特征

1. **多轮交互能力**：通过多轮操作逐步逼近目标，将人的工作流拟合到模型中
2. **工具使用与泛化**：模型不仅使用常见工具，还应能泛化到未见过的长尾工具上
3. **端到端训练**："一方产品"思路——在设计的工具和环境里端到端训练模型
4. **泛化是最大挑战**：当前基准测试过于单点，缺乏足够的评估手段衡量OOD场景表现

> 对于一个Agentic模型来讲，现在最大挑战是在模型的泛化上。

### Agent的分类

**Coding Agent**：数字世界的手。有三个层面的特殊价值：
1. 好验证、好学习：代码任务有明确的测试用例，reward信号清晰
2. 数字世界的自动化基础：代码是"数字世界的手"
3. 工具创建能力：Agent的工具本身也可以用代码动态创建

但仅做Coding Agent是不够的——很多非程序员也会用Claude Code完成任务，因为模型在一定程度上有泛化能力。

**通用Agent vs 垂直Agent**：

> 如果泛化能力强，大家讨论的各种垂直Agent就没那么必要了。因为通用Agent泛化到长尾工具上，很多领域专有问题都能通过接入不同工具解决。

这意味着通用Agent的泛化能力是核心护城河，垂直Agent可能是泛化不足时期的过渡方案。

### Agent的工具生态

| 工具类别 | 连接的外部世界 | 示例 |
|----------|--------------|------|
| 搜索引擎 | 互联网信息 | Web搜索、文档检索 |
| 代码执行 | 数字世界的自动化 | 编程、调试、测试 |
| API调用 | 业务系统 | 退票、下单、数据库查询 |
| 个人数据 | 私有知识 | 个人文档、公司内部数据库 |

工具呈现长尾分布。关键挑战：Agent不仅要熟练使用高频工具，还要能泛化到完全未见过（OOD）的长尾工具上。

### Agent与L1-L5分级体系

杨植麟的两个关键修正：
1. **L2和L3不一定是串行依赖关系**：先做出来Agent能力，再去做狭义的Reasoning，可能也是成立的
2. **L4和L5之间也不是严格线性关系**：多智能体系统和创新能力可以同时发展

> 为什么说下一阶段（L4）是Innovation？这里面最关键的是：模型什么时候能参与模型本身的开发？只有当模型参与到开发过程，才能解锁真正的Innovator阶段。

## Agent泛化、评估与训练方法

### Agent泛化性的核心挑战

> 对于一个Agentic模型来讲，现在最大挑战是在模型的泛化上。这个问题在Agent训练更严重。

泛化问题的本质：用户实际使用的工具和环境往往与训练时不同，模型在OOD场景中表现不佳。仅仅在几个已知task上刷高分，用户体感并不会显著提升。

### Benchmark过拟合问题

> 今天Agent的泛化有一个风险，可能会陷入某些Benchmark过拟合，但现在又缺少很好的Benchmark。

[[SWE-bench]]是最主流的Coding Agent评测基准，但正在快速接近饱和。Benchmark分数的提升不等于泛化能力的提升——能观测的指标不能真实反映能力，真实的用户体验又难以量化。

### 当前RL训练的局限性："种瓜得瓜，种豆得豆"

> 现在的RL技术，局限性在于，不管是训练任务还是评价指标，很多时候都是单点。比如你就训SWE-bench同分布的数据，它就提升SWE-bench。但你的指标提升上去之后，并不意味着模型的泛化会变得更好。

即使数学和代码这两个被认为"相对容易泛化"的领域，同样受分布制约。

[[RL_Scaling]]的scaling效率高于[[预训练]]（on-policy采样 + 负梯度），但泛化性依然是根本瓶颈。

### AI训练AI：AI Native训练方式

> 有一种潜在思路，我们需要用更AI native的方式去训练AI。我们希望让模型参与到更多训练过程。

这对应着从L3（Agent）向L4（Innovator）跨越——用L4的技术去解决L3的问题：

> 你要用一些Innovation方式去解决Agent问题。因为Agent泛化不够，你得用创新去解决——用L4的技术去解决L3的问题。

### 课程学习（Curriculum Learning）

> 强化学习如果搭配好的采样策略，本质是隐式的课程学习机制，希望模型从合适的难度开始学习，逐步提升难度。

这与[[强化学习训练环境设计]]的理念一致：环境难度需要与模型当前能力匹配。

### 正向设计 vs 逆向工程

> 关键是让这个设计有更好的通用性。它能做很多任务，不应该为了某些特定任务专门设计工具和环境。当设计足够通用时，模型能在这其中学习，而不是反过来拟合模型。

正向设计（通用工具和环境 → 模型在其中学习）优于逆向拟合（给定模型 → 设计环境适配模型），因为正向设计天然更具通用性。

### 多模态与"脑子"的关系

> 你希望在多模态模式下，跟文本模式下，共用一个"脑子"；你希望它在多模态的模式下，也能把文本那部分的智商激发出来，而不是进入另外一部分参数。

在[[MTP架构]]中，不能让某些expert专门处理多模态，而应该让多模态能力与文本能力共享参数。否则学出来的多模态是"傻的多模态"。

### Long Context架构对智商的影响

> 现在很多任务，128K或256K这种Context完全解决不了，你需要百万级甚至更多。而挑战在于，你不仅要能处理这么长的Context，还要保证"脑子好用"，智商要非常高。

不同[[长上下文]]架构方案会影响模型的"智商"。纯粹的Linear Attention可能影响智商，因为架构会有bias。核心矛盾：压缩率足够高 vs 上下文足够长。

### Multi-Agent RL训练的Reward设计

> 在训练复杂的多智能体系统时，如果只使用端到端的reward，很可能不够。中间的reward如何产生？是否能摆脱一些人工设计？

### 评估瓶颈

> 现在整体的评估，是阻碍Agent模型变得更泛化的重要瓶颈。

缺少好的Benchmark是阻碍Agent泛化的关键瓶颈。需要在以下维度加强：覆盖更多OOD场景、反映真实用户体验而非仅刷榜、评估多步多工具协作能力、量化泛化性而不仅是单点能力。

## 多智能体系统与AGI分级

### OpenAI L1-L5 分级体系

| 层级 | 名称 | 核心能力 |
|------|------|----------|
| L1 | Chatbot | 单轮对话，输出少量token |
| L2 | Reasoner | 长链式思维，猜想与验证，反思能力 |
| L3 | Agent | 与外界交互，多轮操作，端到端完成任务 |
| L4 | Innovator | 模型参与自身开发，提出新想法、做实验 |
| L5 | Organizer | Multi-Agent系统，分工协作 |

### L1-L5 不是严格串行关系

> Agent的（L3）上限取决于你有很强的Reasoning（L2、推理）能力，但并不是必须先有Reasoning。

关键论点：
- **先Agent后Reasoning是可行的**：Claude的路线就是在Reasoning上投入较少，但在Agent上做得非常出色
- **L4和L5可能同时发生**：从一个Agent fork出多个Agent，有些串行有些并行
- **但L2和L3可能是L4和L5的前提**

### 用L4解决L2/L3问题

> 如果你真的要解决开放性推理问题，就需要做很强的Innovation，提出新的模型架构。本质上，是用L4的方法去解决L2的问题。

杨植麟当前最关键的bet：**泛化的Agent，用Innovation（L4）做L3**。

这揭示了分级之间不仅是递进关系，还存在**反向赋能**关系。

### Multi-Agent 系统（L5）

Organizer（L5）的核心特征是 [[多智能体协作|Multi-Agent系统]]：
- Fork出多个Agent，分别承担不同角色
- 串行 + 并行混合
- 分工协作：写测试、写文档、设计软件框架
- 合并与再拆分

核心挑战：

> 你怎么把Multi-Agent系统很好地做端到端训练，不要过拟合到某几种Agent类型，让它有更好的泛化性，比较有挑战。

### L5 不是终点

> 也不是，可能它真的就没有顶。

AI能力的发展是一个持续向上的方向，而非有终点的台阶。

### AGI 的重新定义

> AGI 不是某一级台阶，你爬到了这级台阶，突然一夜之间达到 AGI。而是说：它是一个方向。

今天在很多领域，AI已经达到了AGI级别的表现。与蒸汽机的类比：社会变化需要几十、几百年去消化。一些工作变得不再必要，但会产生新的工作。每个人变成"超人"。

### AI 作为人类文明的放大器

## 推理模型与测试时扩展（Test-Time Scaling）

### 长思考推理模型（Reasoning Model）

以OpenAI o1为代表，核心机制是让模型在推理过程中不断尝试和反思。

#### 反思的两种能力

1. **提出新的猜想**：在解决问题的过程中不断生成新假设
2. **验证猜想**：对提出的猜想进行自我验证，判断对错

> 反思是两种能力：一种是提出新的猜想，一种是验证猜想。模型在解决问题的过程中，会不断提出新的猜想，这个猜想会得到自我验证。

这个过程与人类做科研或解题高度相似。

#### Pass@k 到 Pass@1

通过多次猜想-验证的串行过程，模型等价于尝试了多次，将 **Pass@k** 转化为 **Pass@1**。

- **Pass@k**：模型在一次任务中生成k个候选答案时，至少有一个正确的概率
- **Pass@1**：模型只生成一个答案即正确的概率
- 长思考通过内部串行反思，在一次推理中实现了多次尝试的效果

#### 串行采样与并行采样

推理过程的有效工作方式偏向线性（串行），但可以与并行策略配合。相关研究表明**串行采样上限更高**。

### 缸中之脑：纯推理 vs 与外界交互

长思考推理模型本质上是一个"缸中之脑"——不需要跟外界交互，仅在内部进行纯推理。

Agent范式打破缸中之脑：基于多轮的Agent强化学习范式，模型需要与外界进行多轮交互。

### Test-Time Scaling：推理时的Token规模化扩展

长思考RL和AgentRL共同指向同一个核心概念：

| 路径 | 机制 | 特点 |
|------|------|------|
| **长思考RL** | 在单轮内增加思考Token | 缸中之脑，纯内部推理 |
| **AgentRL** | 增加交互轮数，每轮产生Token | 与外部环境交互 |

Test-Time Scaling使模型能够完成耗时数小时的端到端复杂任务，无需人工参与。

### 与Pre-Training Scaling的关系

传统Pre-Training面临数据墙问题，而Test-Time Scaling（通过RL）提供了新的规模化维度。

> Scaling Law遇到数据墙了，这是客观事实。你要突破数据墙，就需要提高token efficiency。同时你要scale更多算力到各种RL任务上。但我们现在观察，模型变好的速度并没有减少，甚至在加速。

### Next Token Prediction的本质缺陷与RL Scaling

单纯依赖Next Token Prediction进行Pre-Training，推理能力会呈现**先上升后下降**的趋势：

> 随着模型规模扩大，对话能力、知识量和情商都在变强，但推理能力尤其是数据表现是先上升后平缓，再扩大反而下降。

解决方案：搭配RL Scaling。更大的基础模型提供更高的推理上限，但需要RL scaling来"激活"推理能力。

### RL Scaling效率高于Pre-Training Scaling

> RL的scaling效率又比Pre-Training高很多。因为它是on-policy且带负梯度的训练，所以scaling效率更高。

关键机制：
1. **On-Policy采样**：模型从自身当前策略采样中学习
2. **负梯度**：不仅学习"什么是好的"，还能学习"什么是坏的"

### RL vs SFT 的泛化性

> 强化学习本身的泛化性，比做SFT要好，因为过程中有更多的on-policy sample，模型从自身采样中学习，泛化看起来更好，而且有负梯度。

但RL的泛化目前也是有限的，仍然存在"种瓜得瓜，种豆得豆"的分布制约。

### 两句话的哲学底色

David Deutsch 在 *The Beginning of Infinity* 中提出的两句话：
1. **问题是不可避免的**
2. **问题是可以解决的**

每解决一个问题（如长思考RL），就会展开新的问题（如Agent泛化、评估、验证），技术就在这个循环中不断攀登。

## 访谈核心观点汇总

### 杨植麟的八个核心观点

1. **Agent = 多轮 + 工具**：两个必要条件缺一不可
2. **从"缸中之脑"到"与世界交互"**：这是Agent区别于纯粹推理模型的本质跃迁
3. **test-time scaling是统一框架**：长思考和Agent多轮交互都是其不同实现方式
4. **泛化是Agent当前最大瓶颈**：基准测试不够，OOD场景体感不好
5. **通用Agent泛化足够时，垂直Agent不再必要**：一个通用模型 + 定制工具/数据 = 任意垂直Agent
6. **Coding Agent = 数字世界的手**：重要子集，但非终点
7. **AI训练AI是Agent泛化的潜在解法**：用L4的技术解决L3的问题
8. **L1-L5不是严格串行**：不同能力可以并行发展，且存在交叉依赖和反向赋能

### 其他关键洞察

- **一方产品 vs 三方产品**：模型公司自己做产品有垂直整合的结构性优势
- **模型即产品**：模型训练完成，产品也基本完成
- **Harness衰变定律**：模型能力越强，所需Harness越简单
- **RL scaling效率高于Pre-Training**：on-policy采样 + 负梯度
- **数据飞轮尚未形成**：算力驱动scaling效率太高 + 大模型对噪声敏感
- **AGI不是台阶，是方向**：持续向上，没有终点
- **AI是人类文明的放大器**：Meta Science，一夜学会前沿知识
- **人的三个意义**：创造（AI可承担）、体验（以人为中心）、爱（以人为中心）
