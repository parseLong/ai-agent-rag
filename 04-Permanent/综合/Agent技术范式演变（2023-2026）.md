# Agent技术范式演变（2023-2026）

> Agent 从"被动响应"到"自进化"经历了四个显著阶段，六个核心技术维度的运行逻辑、数据流转方式及工程实现范式都发生了翻天覆地的变化。"形"未变，但"神"已大不同——这标志 Agent 正从"魔法调优"走向"系统工程"。

---

## 📖 核心洞察

> [!quote] 形未变，神已大不同
> 从宏观架构上看，今天的 Agent 依然由 Prompt、Planning、Memory、Tools 等经典模块组成，这与 Lilian Weng 早期提出的理论框架并无二致。但每一个模块背后的运行逻辑、数据流转方式以及工程实现范式，都发生了根本性重构——这不是简单的技术升级，而是一场**深刻的内核重构**。

**核心驱动力**：底层基座模型推理能力的持续升级——从"简单交互"到"复杂执行"再到"智能成长"的技术进阶之路。

**核心思想**："通过工程化手段构建确定性，以承载模型不确定性"——这将是很长一段时间内构建高质量 Agent 的基石。

> [!warning] 需要避免的两个误区
> 1. **"为了升级新架构而升级架构"**：不理解演进逻辑就盲目采用新范式，反而可能降低效果
> 2. **"盲目追求最新技术概念"**：新旧技术并非替代关系，而是并存且可组合——关键是为特定场景选型

> [!note] 现实困境："穿越感"
> 当前的 Agent 学习者常遇到"穿越感"——新旧技术概念混杂在一起，加上营销号仍在转载早期文章，导致越了解越迷糊。理解演进逻辑正是消除这种"穿越感"的方法。

---

## 🔄 四阶段演进模型

> [!important] 并存与互补
> 这四个阶段并非替代关系，而是并存且互补的。实际落地中需根据业务复杂度、稳定性要求和成本预算，选择合适范式或将多种范式组合使用。

### 阶段一：被动式 ReAct（2023）

- **定位**：Agent概念的启蒙期，LLM爆发元年
- **理论基础**：[[Lilian Weng]] 的 [[LLM Powered Autonomous Agents]] — Agent = LLM + Planning + Tools + Memory
- **架构本质**：单步 "Reasoning → Observe → Response" 循环
- **交互形态**：增强版 Chatbot，"一问一答"或"指令-执行"
- **能力边界**：依赖用户明确指令，仅能完成单点、短链路小任务
- **量化局限**：当时能做好==3轮以上 Reasoning==的模型都不多
- **局限性**：缺乏长期规划能力，超出上下文窗口或逻辑链条过长时易偏离/中断
- **代表项目**：AgentGPT、[[AutoGen]]、[[MetaGPT]]
- **关联范式**：[[ReAct 范式]]、[[CoT]]

### 阶段二：工作流 Agent（2024）

- **定位**：工程化约束弥补模型不确定性的阶段
- **核心理念**：用工程化约束弥补模型的不确定性——早期的 [[驾驭工程]]（Harness）
- **架构特征**：固定 Workflow + 关键节点嵌入 LLM，或 LLM 中枢调用预定义子 Workflow——==一套比较重的 Harness==
- **价值体现**：对非长尾、非极度复杂场景，==性价比最高、落地最稳定==的方案
- **关键判断**：很多企业服务并不需要真正的"智能决策"，只需按步骤1、2、3按时保质地完成即可——Workflow ==能确保效果的下限==
- **应用场景**：To B 领域极受欢迎，重复性工作只需按步骤完成
- **代表框架**：[[LangGraph]]、[[Dify]] 等提供 Workflow 流程编排
- **关联概念**：[[驾驭工程]]、[[Prompt与Context工程（三次进化）]]

### 阶段三：自主 Agent（2025）

- **定位**：Agent迈向"自主性"的关键转折点
- **核心变化**：具备复杂的 [[任务分解|Planning]] 能力，自行拆解任务、规划路径、调用工具、多轮迭代
- **长程任务能力**：只要用户清晰描述需求并==设定好开发规范（Specs）==，Agent 可连续运行很长时间
- **自我校验**：配合==轻量级 Harness==或自我校验机制，长程运行中不断修正错误（注意：这里的 Harness 从阶段二的"重Harness"变为"轻量级"）
- **角色转变**：从==辅助者→执行者==的根本转变——Agent不再帮你做事，而是替你做事
- **代表产品**：[[Manus]]、Claude Code、Codex；[[本地Agent（Claw）]] 在2026年初进一步巩固了这一趋势
- **关联概念**：[[Plan-and-Solve]]、[[Plan-and-Execute]]、[[驾驭工程]]

### 阶段四：自进化 Agent（2026）

- **定位**：解决"静态模型"与"动态世界"之间的矛盾
- **核心本质**：Agent 不仅完成任务，更在完成任务过程中沉淀经验
- **机制原理**：通过 [[记忆系统]]、反馈循环和自我反思机制，将任务教训转化为新知识或策略
- **最终目标**：实现==越用越好用==——自动优化提示词、工具选择策略、甚至==通过 RL 训练来提升模型能力==，实现自我升级和进化
- **价值重构**：Agent 从==一次性消耗品→可积累资产==——为构建真正具备长期生命力的数字员工奠定基础
- **代表框架**：Hermes Agent + LLM-Wiki 等开源项目
- **关联概念**：[[记忆到技能转化]]、[[Skills技能]]、[[Reflection 范式]]

## 🔧 六维度范式重构

| 维度 | 早期范式 | 当前范式 | 核心变化逻辑 |
|------|----------|----------|-------------|
| [[系统提示词工程\|Prompt]] | 深耦合单体大 System Prompt | System Prompt 固化 + 渐进式加载上下文文件 | 动静分离，降低维护复杂度 |
| [[Plan-and-Solve\|Planning]] | 线性 [[CoT]] 思维链 | 结构化任务拆解 + 长程推理 + 子Agent动态构建 | 基座模型推理能力升级驱动 |
| [[记忆系统\|Memory]] | 纯向量检索（RAG） | 文件系统化沉淀 + 向量检索混合管理 | 从纯检索走向可读、可控、可组织 |
| [[MCP\|Tools]] | [[MCP\|Function Call]] 高成本 API 封装 | CLI 原生 + Script 脚本化 | 利用模型预训练原生知识而非人为适配 |
| [[驾驭工程\|Workflow]] | 刚性外部编排（状态机） | Skill 内聚封装 + Workflow 兜底的混合架构 | 从"硬约束"到"灵活+确定性平衡" |
| [[运行时环境\|Environment]] | 无状态调用 | 有状态的隔离运行时 Runtime 系统 | Agent 从"问答机器"到"数字员工" |

### Prompt：深耦合 → 渐进式加载

早期为每个任务创建独立 Agent，每段 System Prompt 独立存在、包含全部人设/任务/约束/示例——维护成本极高、管理混乱。

当前演进方向：**动静分离**
- **固化层**：System Prompt 只保留底层通用系统级指令和基本行为规范（极度稳定）
- **动态层**：具体任务要求、领域知识等通过渐进式披露（Progressive Disclosure）加载
  - Skill 沉淀：方法论/步骤/约束 → 独立 `SKILL.md` 文件 → 动态加载
  - 配置存储：人设/偏好/规则 → `USER.md` / `SOUL.md` / `CLAUDE.md` → 渐进式加载

> [!tip] 核心变化
> 不是"内容变了"，而是"上下文的组织形式变了"——从单体大 Prompt 到模块化挂载。

详见：[[系统提示词工程]]、[[Skills技能]]、[[Prompt与Context工程（三次进化）]]

### Planning：思维链 → 复杂长程任务

| 早期 | 当前 |
|------|------|
| 依赖模型原生 [[CoT]]，线性串行推理 | 结构化分解 + 多步协同 + 子Agent动态构建 |
| "Let's think step by step" 式提示 | 自动生成 Todo List + 动态调整计划 |
| 单步思考，易陷入死循环 | 长程推理，保持逻辑连贯性 |

**驱动力**：底层基座模型推理能力（Reasoning）的显著增强——Planning 从"提示词技巧"演变为"智能决策中枢"。

详见：[[CoT]]、[[Plan-and-Solve]]、[[Plan-and-Execute]]、[[LLM Compiler]]、[[任务分解]]

### Memory：检索增强 → 文件系统化

**短期记忆**：核心挑战从"存储"转向"管理与压缩"
- 阈值控制：基于固定 token 数或动态语义密度阈值触发压缩
- 结构化摘要：中间过程对话提炼 Summary，保留头尾关键指令和最终结论
- 重点提取：从冗长对话流中提取关键事实，剔除噪音

**长期记忆**：从"向量数据库主导"向"文件系统主导"==回归==

> [!note] "回归"而非"转向"
> 这里用的是"回归"而非"转向"——因为文件系统作为知识存储方式早于向量数据库。向量数据库曾是 RAG 时代的"新范式"，如今 Agent 时代重新认识到文件系统的优势（可控、可读、可组织），形成了一个==历史循环==：文件系统 → 向量数据库 → 文件系统+向量检索混合。

| 记忆子类型 | 早期方式 | 当前方式 | 代表 |
|-----------|---------|---------|------|
| **事项型记忆**（Episodic） | 向量检索 | 文件系统记录（MEMORY.md、日志文件） | [[本地Agent（Claw）]]、Hermes |
| **知识型记忆**（Semantic） | 纯 [[RAG]] | 本地文件系统 + Obsidian + 轻量向量检索 | LLM-Wiki、[[记忆到技能转化]] |

> [!important] 演进本质
> 从纯向量文本检索走向"文件系统化的沉淀+向量检索混合管理"——追求更高的记忆效果、可读性和效率的均衡。

详见：[[记忆系统]]、[[上下文压缩]]、[[RAG]]、[[记忆到技能转化]]

### Tools：Function Call → CLI / Script

| 早期 | 当前 |
|------|------|
| 为每个操作封装专用 API（Function Call） | 利用模型预训练的 CLI 原生知识 |
| 极高的开发维护成本 | 零样本学习优势（CLI 是模型"先天知识"） |
| API Schema 管理复杂 | CLI 可扩展性 + 自解释性（--help） |
| [[MCP]] 仍停留在接口标准化 | CLI + Script 从根本上改变调用逻辑 |

> [!note] MCP 的定位：接口标准化过渡
> [[MCP]] 在工具演进中扮演的是"接口标准化的过渡"角色——它在协议层面优化了工具注册与发现，实现了"一次注册、自动暴露"，但==本质上仍停留在接口标准化的层面，并未从根本上改变工具调用的底层逻辑==。真正的范式转移发生在执行方式上（CLI + Script）。

**CLI 原生化的核心优势**：
1. **零样本学习**：grep/cat/vim 等命令是模型预训练数据的"先天知识"
2. **自解释性**：第三方 CLI 工具遵循标准规范，模型可 --help 即时学习——"按需查询、即时学习"完美契合渐进式加载理念
3. **Skill 集成**：新 CLI 工具通过 Skill 包装，SKILL.md 提供安装指南和使用示例

> [!important] 交互友好性的倒转
> CLI 对==人类==来说是高门槛的、枯燥的交互方式，但对==机器（LLM）==反而足够友好——因为这些命令是其预训练数据的"先天知识"。这种"交互友好性倒转"是 Agent 工具范式转移的底层洞察：**对人类不友好的交互方式，可能是对 Agent 最友好的交互方式**。

**Script 脚本化**：
- 本地与远程统一：脚本封装 API 鉴权、参数拼接
- 协议黑盒化：Agent 只需关注"调用哪个脚本"+"传入什么参数"

> [!tip] 核心变化
> 从"人为适配模型"转向"利用模型原生能力"——不再为每个操作写专用 API，而是利用模型预训练积累的通用计算机操作知识。

详见：[[MCP]]、[[Skills技能]]

### Workflow：刚性编排 → 动态混合封装

| 早期 | 当前 |
|------|------|
| 硬编码的"第一步、第二步、第三步" | Skill 内聚化 + 执行脚本化 |
| 类似传统状态机/Pipeline，==运行过程机械化==——无论外部环境如何变化，仍死板严格遵循固定步骤 | 逻辑写入 SKILL.md，精确控制通过 Script |
| 牺牲灵活性换可控性 | Skill 为主，Workflow 兜底的混合架构 |

**两种重构机制**：
- **逻辑内聚化**：原本分散在 Workflow 引擎中的步骤定义、约束条件、核心判断逻辑 → 写入 SKILL.md，模型通过阅读 Skills 文档即可理解完整链路
- **执行脚本化**：不再依赖外部工作流引擎的状态跳转 → 通过 Skill 关联的 Script 进行代码级编排

> [!important] 可控性与稳定性的博弈
> 纯 Skill 驱动赋予 Agent 更高自主性，但在极端复杂或容错率极低的场景下，模型仍可能出现理解偏差或执行跳跃，导致结果不可控。刚性 Workflow 虽然笨重，却提供了==确定性的边界==。当前正处于新旧技术交叉融合的过渡期。

**混合架构最佳实践**：
- 成熟标准化子任务 → 封装为 Skills（灵活+易用）
- 关键稳定性高的主干流程 → 保留 Workflow 或将 Workflow 封装为特殊 Tool
- Script可读性有时不如Workflow方便——==代码可读性==是保留部分Workflow的一个实际考量

> [!tip] 最佳实践总结
> ==Skill 为主，Workflow 为辅/兜底==——既利用新技术红利，又保留确定性，是当前平衡开发效率与运行稳定性的最佳实践。

详见：[[驾驭工程]]、[[Skills技能]]

## 💡 元学习方法论

> [!quote] 理解演进逻辑比掌握具体工具更重要
> "对于每一位从事 Agent 应用和落地的同学而言，理解这些演进背后的逻辑，比掌握具体的某个工具更为重要。因为 Agent 还在持续发展，模型会继续升级、工具会继续变化、框架会持续更新，但这种'通过工程化手段构建确定性，以承载模型不确定性'的核心思想，将是未来很长一段时间内构建高质量 Agent 的基石。"

**核心启示**：工具和框架是==可变的==，但演进逻辑和核心思想是==稳定的==。掌握"为什么这样演变"比"怎么用这个工具"更有长期价值。
