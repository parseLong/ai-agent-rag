# 工程的三次进化：关系、衰变定律与新范式

> 三次进化·下篇：聚焦 Prompt Engineering、Context Engineering、Harness Engineering 三者的嵌套关系、Harness 衰变定律，以及新范式下工程师的职责重构与实践路线图。

---

## 三者的嵌套关系：不是替代，是嵌套

### 最大的误解

讨论到 Harness Engineering 时，一个自然的反应是："所以 Harness Engineering 是最高级的，前两个都过时了？"

这是一个根本性的误解。三者之间的关系是**层层包裹、相互依存**的嵌套关系，而非线性替代。

### 三个核心问题

三者分别回答不同的核心问题：

| 工程层级 | 核心问题 |
|----------|----------|
| [[提示工程|Prompt Engineering]] | "我该跟模型说什么？" |
| Context Engineering | "模型在回答时该知道什么？" |
| [[驾驭工程|Harness Engineering]] | "整个 AI 系统该如何可靠地运转？" |

三个问题，三个维度，缺一不可。

### 相互依赖关系

> 没有好的 Prompt，Context Engineering 注入的信息无法被模型正确理解。

即使把最相关的文档精准注入了[[上下文窗口管理|上下文]]，如果指令本身模糊不清，模型依然会产生偏差。

> 没有好的 Context，Harness Engineering 的 Agent 在信息真空中瞎跑。

即使设计了完美的[[多智能体协作]]机制、完善的验证回路，如果 Agent 根本不知道业务规则和代码规范是什么，它依然会生成垃圾。

> 没有好的 Harness，再好的 Prompt 和 Context 只是沙滩上的城堡。

即使单次对话的输出质量很高，没有系统级的约束和反馈，在复杂任务中 Agent 依然会累积错误，最终崩溃。

这是一个层层递进的制约链：Prompt 决定了指令的清晰度，Context 决定了信息的完整度，Harness 决定了系统的可靠度。上层失效，下层投入全部白费。

## Harness 衰变定律

### 核心发现：一个反直觉的规律

Anthropic 的研究者在对比不同版本模型的表现时，发现了一个深刻规律：

> [!quote] Harness 衰变定律
> **模型能力越强，所需的 Harness 越简单。**

在 Claude 3.0 时代，为了保证 [[智能体（Agent）|Agent]] 不在复杂任务中途崩溃，需要强制实施极严格的 Harness 约束：逐个功能点执行、频繁重置上下文、大量硬编码的检查规则。

但当模型升级到 Claude 3.5，其全局统筹能力、长上下文处理能力和自我校验能力大幅提升——原本不可或缺的许多 Harness 规则，自然变得不再必要。

### 两层深意

**第一层：Harness 是当下的现实答案。**

在模型能力尚未完美的今天，[[04-Permanent/# 驾驭工程（Harness Engineering）|Harness]] 是让 AI 系统在生产环境可靠运行的必要条件。不做 Harness，就是让野马在生产环境横冲直撞。

**第二层：Harness 可能是一项过渡性技术。**

随着模型能力持续提升，今天需要精心设计的许多 Harness 规则，未来会被模型能力自然吸收。大语言模型正在逐渐"内化"这些系统规则：自动识别[[任务分解|任务优先级]]、自动验证输出、自动处理边界情况。

### 实践建议

> [!quote] 衰变定律的实践指导
> 不要过度设计那些模型未来能自我解决的问题。
>
> 把精力集中在两类场景：
>
> 1. **模型短期内无法通过自身能力解决的业务逻辑边界**——行业特定规则、合规要求、复杂系统协同
> 2. **即使模型能力再强也无法自行建立的外部环境接口**——[[工具系统|工具调用]]、API 集成、[[权限系统|权限控制]]

### 核心能力

谁能根据模型能力的边界，**动态调整 Harness 的"厚度"**，谁就能在工程效率上获得最高回报。

这个能力需要持续校准两个判断：哪些约束是因为模型能力不足而存在的（会被模型进化淘汰），哪些约束是因为业务逻辑本身需要它（长期有效）。

## 新范式下的工程师：Human Steer, Agents Execute

### 一次职责的根本性重构

OpenAI 在实验总结中提出了这个时代最重要的工程哲学：

> [!quote] 核心哲学
> **"Human steer, agents execute."**
> 人类掌舵，Agent 执行。

这不是在说工程师会被取代。恰恰相反，它是在说工程师的价值正在**向上迁移**到一个更高的维度。

在传统开发模式下，工程师是"体力劳动者"：大部分时间耗费在编写具体逻辑、处理报错、维护测试、更新文档上。在 [[04-Permanent/# 驾驭工程（Harness Engineering）|Harness Engineering]] 的范式下，这一切已经可以交给 Agent 执行。

工程师的核心职责重构为三件事：

1. **定方向（Steering）**：清楚地知道要建什么、为什么建、最终形态是什么。这需要产品思维、系统思维和业务洞察——这些是模型目前最难替代的。

2. **搭架子（Harnessing）**：为 Agent 构建可靠的运行支架——制定规则、提供可信的上下文来源、设计自动化验证和反馈回路。这是 [[04-Permanent/# 驾驭工程（Harness Engineering）|Harness Engineering]] 的核心技能，涵盖[[护栏与钩子系统]]、[[重试与恢复机制]]、[[分层安全防御]]等关键模块。

3. **做判别（Decision Making）**：在关键的架构决策点、技术路线选择点、风险边界节点进行人工干预和确认。AI 不应该独立做所有决策，工程师需要知道在哪些节点必须介入。

### 衡量标准的切换

| 过去的衡量标准 | 新的衡量标准 |
|---------------|-------------|
| 每天能写多少行代码 | Harness 能支撑多高的代码产出率（系统杠杆率） |
| 能实现多复杂的业务逻辑 | 能设计多健壮的 Agent 系统 |
| 能处理多难的 Bug | 能构建多完善的自动闭环机制 |
| 对某语言/框架的熟悉程度 | 对 AI 系统边界和失效模式的理解深度 |
| 个人产出（Individual Output） | 系统杠杆（System Leverage） |

一个能设计出高效 Harness 的工程师，他的杠杆率是惊人的：他搭建的系统能让 AI 持续产出远超他个人极限的代码量和质量。这正呼应了 [[构建者信条]] 中的理念——工程师的价值不在于手工劳作，而在于构建能放大自身能力的系统。

## 实践路线图

### 第一步：打牢 Prompt 基础，但不执着于它

> 详细实操指南见 [[成为 Harness Engineering 时代的工程师#第一步：打牢 Prompt 基础，但不要执着于它]]

[[提示工程|Prompt Engineering]] 的底层逻辑是如何清晰、精准地表达意图，是你与 AI 协作的基础语言能力。不必穷尽所有技巧，但要掌握核心：

- 知道如何用思维链引导模型推理
- 知道如何通过角色设定提升输出专业性
- 知道如何设计结构化输出格式以便后续处理

**不要执着于"最佳提示词"的追求。** 随着模型进化，今天的最优提示明天可能不再必要。

### 第二步：系统学习 Context Engineering

> 详细实操指南见 [[成为 Harness Engineering 时代的工程师#第二步：系统学习 Context Engineering]]

这是当下最具差异化的技能之一。需要掌握：

- [[RAG]] 系统的设计与调优：如何分块、如何选择 Embedding 模型、如何优化检索精度
- [[上下文窗口管理]]：如何在有限 Token 中做最优的信息选择
- 记忆系统设计：短期/长期记忆的分层管理，对话历史的压缩策略
- 知识库治理：如何维护一个 Agent 可信赖的单一事实来源（[[代码-文档同步管道]]是这一原则的工程实践）

### 第三步：从系统视角思考 Agent 设计

> 详细实操指南见 [[成为 Harness Engineering 时代的工程师#第三步：从系统视角思考 Agent 设计]]

开始用 Harness 的视角审视你的 AI 项目：

- 我的 Agent 在哪些地方可能跑偏？如何设计约束防止它？（参考[[护栏与钩子系统]]）
- 如何建立验证机制，让 Agent 的"声称完成"变成"验证完成"？
- 任务的哪些部分适合单 Agent？哪些需要[[多智能体协作]]？（参考[[子代理系统]]）
- 什么样的监控和可观测性工具能让我及时发现 Agent 的异常？

### 第四步：培养"动态 Harness 思维"

> 详细实操指南见 [[成为 Harness Engineering 时代的工程师#第四步：培养"动态 Harness 思维"]]

这是最难培养、也是最有价值的能力。随时问自己两个问题：

1. **"这个约束/规则是因为模型能力不足而存在的，还是因为业务逻辑本身需要它？"**
2. **"如果下一版模型变强了 20%，我的 Harness 里哪些部分可以被简化？"**

能清晰回答这两个问题的工程师，能在 Harness 的设计上保持恰当的"薄厚"：不过度设计，不遗漏关键约束。

---

## 结语：三次进化，一个目标

这三次进化，服务于同一个目标：**让大语言模型的能力，真正转化为可靠的生产力。**

- [[提示工程|Prompt Engineering]] 解决了"说清楚"的问题
- Context Engineering 解决了"给够信息"的问题
- [[驾驭工程（Harness Engineering）]] 解决了"系统可靠"的问题

三者缺一不可，层层递进。但最终，它们都在回答同一个问题：如何在人类的掌舵下，让 AI 这匹野马跑得又快又稳？

软件工程没有消失，它在进化。

> 从"写代码的人"，进化为"设计让 AI 把代码写好的系统的人"。

这，才是这个时代工程师最该掌握的技能。
