# 智能体评估

智能体评估是衡量 AI Agent 性能与能力的系统性方法。一个完善的评估体系需要同时关注**量化指标**（如准确率、召回率、F1 分数）与**质性指标**（如用户满意度），并通过标准化的**基准测试**（如 AgentBench、WebArena、SWE-bench）进行多维度验证。

本文件将评估知识分为两大板块：**核心评估指标**与**主流基准测试**，帮助你从指标理解到基准实践形成完整的评估认知链路。

---

## 核心评估指标

评估指标是量化智能体表现的基础工具。不同指标侧重不同维度，需要组合使用才能形成全面判断。

### 准确率

> 准确率（Accuracy）是评估智能体输出正确性的核心指标，定义为预测结果与标准答案一致的样本比例。计算公式为：
>
> $$\text{Accuracy} = \frac{1}{N} \sum_{i=1}^{N} \text{Match}(P_i, G_i)$$
>
> 其中 $N$ 是总样本数，$P_i$ 是第 $i$ 个样本的预测结果，$G_i$ 是第 $i$ 个样本的标准答案，$\text{Match}(\cdot, \cdot)$ 是匹配函数。
>
> 在 [[BFCL]] 评估中，准确率的具体计算公式为：
>
> $$\text{Accuracy} = \frac{1}{N} \sum_{i=1}^{N} \text{AST\_Match}(P_i, G_i)$$
>
> 其中 $\text{AST\_Match}(P_i, G_i) \in \{0, 1\}$ 是 AST 匹配函数。

准确率是最直观、最常用的评估指标。在智能体评估中，匹配函数根据任务类型有所不同：

- **精确匹配**：预测结果与标准答案完全一致
- **AST 匹配**：通过抽象语法树比较函数调用的等价性（如 [[BFCL]] 评估）
- **准精确匹配**：对答案进行归一化后的匹配（如 [[GAIA]] 评估）
- **Quasi Exact Match**：GAIA 基准使用的官方评估算法

准确率虽然直观，但在类别不平衡的数据集上可能产生误导。例如，在 90% 样本都是负类的情况下，即使模型总是预测负类，准确率也有 90%。因此需要结合 [[召回率]]、[[F1 分数]] 等指标综合评估。

**BFCL 中的准确率应用**：

- **分类准确率**：对每个类别 $c \in \{\text{simple}, \text{multiple}, \text{parallel}, \ldots\}$，计算该类别的准确率
- **加权准确率**：考虑不同类别的难度权重进行加权计算
- **错误率**：$\text{Error Rate} = 1 - \text{Accuracy}$

**准确率的重要性**：

- **标准化比较**：提供统一的量化标准，便于横向比较不同智能体的性能
- **快速诊断**：通过分类准确率可定位智能体的薄弱环节（如简单任务 vs 复杂任务）
- **工程指导**：指导模型优化方向，如 BFCL 通过 AST 匹配准确率优化工具调用能力
- **性能监控**：在 [[持续评估系统]] 中，准确率是监控智能体性能变化的核心指标

**实例：BFCL 评估中的准确率**

BFCL 使用 AST（抽象语法树）匹配算法计算准确率：

- `calculate(x=2+3)` 与 `calculate(x=5)` → AST 匹配成功 ✅
- `get_weather(city="Beijing")` 与 `get_temperature(city="Beijing")` → AST 匹配失败 ❌

| 类别 | 正确数/总数 | 准确率 |
|------|-----------|--------|
| Simple | 95/100 | 95% |
| Multiple | 82/100 | 82% |
| Parallel | 68/100 | 68% |
| **加权平均** | — | **81.7%** |

**实例：GAIA 评估中的分级准确率**

$$\text{Accuracy}_\ell = \frac{1}{|D_\ell|} \sum_{i \in D_\ell} \text{Quasi\_Exact\_Match}(A_{\text{pred},i}, A_{\text{true},i})$$

其中 $D_\ell$ 是难度级别 $\ell$ 的样本集合。

> **使用建议**：在使用准确率时，务必关注数据分布，避免被高准确率误导；结合分类准确率可发现智能体在不同难度任务上的表现差异；选择合适的匹配函数至关重要；在 [[持续评估系统]] 中，建议设置准确率阈值，防止性能退化。

### 召回率

> 召回率（Recall）是评估智能体找出所有相关结果能力的指标，定义为正确预测的正例数占实际正例总数的比例。计算公式为：
>
> $$\text{Recall} = \frac{TP}{TP + FN}$$
>
> 其中 $TP$（True Positive）是真正例数，$FN$（False Negative）是假负例数。$TP + FN$ 表示实际正例的总数。

召回率关注的是"不漏"，即模型能够找出多少实际存在的正例。与 [[准确率]] 关注"预测对了多少"不同，召回率关注"找全了多少"。

在智能体评估场景中：

- **信息检索**：召回率衡量智能体从大量信息中检索出相关内容的完整性
- **多步推理**：在多步任务中，召回率反映智能体是否遗漏了关键步骤
- **工具调用**：评估智能体是否遗漏了应该调用的工具

召回率与 [[准确率]] 往往存在权衡（Trade-off）：提高召回率可能导致准确率下降，反之亦然。因此通常需要结合 [[F1 分数]] 进行综合评估。

**召回率的重要性**（高风险场景尤为关键）：

- **医疗诊断**：漏诊（假负例）的后果可能比误诊（假正例）更严重，需要高召回率
- **安全监控**：需要确保所有潜在威胁都被检测到，不能遗漏
- **智能客服**：需要确保用户的问题意图都被识别到，避免遗漏用户需求
- **知识问答**：确保从知识库中检索到所有相关的信息片段

**实例：智能客服意图识别**

假设有 100 条用户咨询，其中 30 条属于"退款"意图：

- 模型正确识别出 25 条"退款"意图（TP = 25）
- 模型将 5 条"退款"意图误判为其他意图（FN = 5）
- 召回率 = 25 / (25 + 5) = 83.3%

**实例：RAG 检索评估**

对于一个多文档问答任务，标准答案需要从 3 个文档段落中获取：

- 智能体检索到了 2 个相关段落，遗漏了 1 个
- 召回率 = 2/3 ≈ 66.7%

> **使用建议**：召回率单独使用时价值有限，应结合精确率和 F1 分数综合判断；在智能体系统中，召回率常用于评估 RAG 检索阶段的效果；可以通过增加检索数量、优化检索策略来提高召回率；过高的召回率可能带来大量无关信息，影响后续处理效率。

在 [[BFCL]] 评估中，召回率概念可扩展为**参数准确率**（Parameter Accuracy），用于评估智能体构造的函数调用中参数命中的比例。在 [[数据生成质量评估]] 中，召回率可用于衡量生成数据对真实数据分布的覆盖程度。

### F1 分数

> F1 分数（F1 Score）是综合考虑 [[准确率]]（Precision）和 [[召回率]]（Recall）的调和平均指标。计算公式为：
>
> $$F1 = 2 \times \frac{\text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}$$
>
> 当 Precision 或 Recall 中有一个为 0 时，F1 分数也为 0。F1 分数在 0 到 1 之间取值，越接近 1 表示模型性能越好。

F1 分数的核心价值在于它是一个综合性的平衡指标：

- **调和平均的特性**：F1 分数对 Precision 和 Recall 的偏差非常敏感。当其中一个指标较低时，F1 分数会显著下降，从而避免"偏科"
- **与算术平均的区别**：如果使用算术平均 $(P+R)/2$，即使 Recall 为 0，只要 Precision 够高，平均值仍会很高。而 F1 分数能更好地惩罚这种不平衡

在智能体评估中，F1 分数特别适用于：

- **信息抽取任务**：需要同时关注抽取结果的准确性（Precision）和完整性（Recall）
- **实体识别**：评估智能体识别命名实体的综合能力
- **分类任务**：尤其是类别不平衡的分类场景

**F1 分数的重要性**：

- **平衡视角**：避免只关注单一指标导致的评估偏差
- **类别不平衡场景**：在正例较少的场景中，F1 分数比 [[准确率]] 更可靠
- **模型选择**：在多个模型间进行选择时，F1 分数提供了简洁的综合判断依据
- **调参指导**：帮助在 Precision 和 Recall 之间找到最佳平衡点

**实例：模型性能对比**

| 模型 | Precision | Recall | F1 分数 |
|------|-----------|--------|---------|
| 模型 A | 90% | 50% | **64.3%** |
| 模型 B | 70% | 70% | **70.0%** |

模型 A 虽然 Precision 很高，但 Recall 较低，导致 F1 分数反而低于模型 B。模型 B 在 Precision 和 Recall 之间取得了更好的平衡。

通过调整分类阈值，可以绘制 Precision-Recall 曲线，找到 F1 分数最大的点作为最佳阈值。

> **补充说明**：F1 分数是最常用的综合评估指标之一，但在某些场景下可能需要调整权重。**F-beta 分数**通过调整 beta 值，可以赋予 Precision 或 Recall 不同的权重：$\beta > 1$ 时更重视 Recall，$\beta < 1$ 时更重视 Precision。F1 分数假设 Precision 和 Recall 同等重要，实际应用中需根据业务需求判断。

在 [[BFCL]] 评估中，F1 分数的计算方式有所不同。`BFCLMetrics` 类会同时计算准确率、AST 匹配率、参数准确率和 F1 分数，综合评估智能体的工具调用能力。F1 分数在智能体评估中的特殊应用：

- **数据生成质量评估**：在 [[LLM Judge]] 评估中，F1 分数可用于综合衡量生成数据在正确性、清晰度、难度匹配、完整性四个维度上的表现
- **多智能体协作评估**：在评估多智能体系统时，F1 分数可以平衡各智能体之间的通信效率与任务完成度
- **类别不平衡场景**：在 [[AgentBench]] 等多领域评估中，F1 分数比单纯的准确率更能反映智能体在少数类别上的真实表现

### 用户满意度

> 用户满意度（User Satisfaction）是从用户视角评估智能体服务质量的综合性指标。它衡量用户对智能体提供的回答、服务质量和交互体验的整体满意程度。常见的量化方式包括：
>
> - **评分制**：1-5 分或 1-10 分的打分方式
> - **净推荐值（NPS）**：用户推荐意愿的评分（0-10 分）
> - **满意度调查问卷**：多维度综合评估
> - **用户留存率**：用户是否持续使用智能体服务

用户满意度与其他技术指标的最大区别在于：**它关注的是"人"的感受，而非"机器"的表现**。

即使一个智能体在 [[准确率]]、[[召回率]]、[[F1 分数]] 等技术指标上表现优异，用户满意度也可能不高。原因可能包括：

- 回答过于机械，缺乏人性化的表达
- 交互流程繁琐，用户体验差
- 未能理解用户的真实意图（尤其是隐含意图）
- 响应速度慢，用户等待时间过长

用户满意度是连接技术指标与商业价值的桥梁。高满意度通常意味着更好的用户留存和口碑传播。

**用户满意度的重要性**：

- **商业价值**：直接关联用户留存、转化和收入增长
- **产品迭代**：指导智能体在功能、交互、表达等方面的优化方向
- **差异化竞争**：在技术同质化的情况下，用户体验是重要的竞争壁垒
- **反馈闭环**：用户满意度数据可用于持续优化智能体模型和策略

**实例：智能客服场景的用户满意度评估**

| 维度 | 权重 | 评估内容 |
|------|------|---------|
| 问题解决度 | 30% | 是否成功解决了用户的问题 |
| 回答准确性 | 25% | 提供的信息是否正确无误 |
| 响应及时性 | 20% | 从提问到获得回答的时间 |
| 表达友好度 | 15% | 回答是否礼貌、易懂、有同理心 |
| 交互流畅度 | 10% | 对话过程是否自然、无需反复确认 |

**LLM Judge 评估用户满意度**

可以使用 LLM 作为评判者，模拟用户视角对智能体的回答进行评分：

```python
# 示例：使用 LLM Judge 评估用户满意度
def evaluate_satisfaction(user_query, agent_response):
    prompt = f"""请从用户角度评估以下智能体回答的满意度（1-5分）：

    用户问题：{user_query}
    智能体回答：{agent_response}

    评分标准：
    - 5分：完全满足需求，超出预期
    - 4分：满足需求，体验良好
    - 3分：基本满足，但有改进空间
    - 2分：部分满足，体验较差
    - 1分：完全未满足需求

    请给出评分并说明理由。"""

    return llm_judge(prompt)
```

> **使用建议**：用户满意度是最贴近商业目标的评估指标，但也是最主观的；建议结合定量指标（如评分）和定性反馈（如用户评价）进行综合评估；定期进行用户满意度调研，建立持续反馈机制；不同用户群体的满意度标准可能存在差异，需要进行用户分层分析。

用户满意度在智能体评估中的特殊应用：

- **LLM Judge 作为用户满意度的自动化代理**：在 [[数据生成质量评估]] 中，[[LLM Judge]] 从正确性、清晰度、难度匹配、完整性四个维度评分，本质上模拟了人类用户对智能体输出的满意度评估
- **Win Rate 作为相对满意度指标**：通过成对对比（生成数据 vs 参考数据），Win Rate 可以衡量用户对智能体输出的相对满意程度。理想情况下，Win Rate ≈ 50% 说明生成质量接近人类水平
- **人工验证作为金标准**：在 [[AIME]] 题目生成评估中，人工验证通过 Gradio Web 界面对生成的题目进行评分（1-5 分）和状态标注（approved/rejected/needs_revision），这是用户满意度的最终评判标准

## 主流基准测试

基准测试提供了标准化的评估环境和任务集，使得不同智能体之间的性能比较具有可比性和可重复性。以下三个基准分别侧重**通用能力**、**真实环境交互**和**代码修复**三大评估维度。

### AgentBench

> AgentBench 是由清华大学推出的综合性智能体评估基准（Benchmark），旨在全面评估大语言模型作为智能体的能力。该基准包含 8 个不同领域的任务，覆盖了智能体在真实世界中可能遇到的各种场景。

AgentBench 是智能体评估领域最具影响力的综合性基准之一。与其他专注于单一能力的基准不同，AgentBench 的设计理念是：**一个优秀的智能体应该具备跨领域、多任务的通用能力**。

AgentBench 的 8 个任务领域：

- **操作系统交互**：文件操作、进程管理等
- **数据库操作**：SQL 查询、数据处理等
- **知识图谱问答**：基于结构化知识的推理
- **数字游戏**：逻辑推理、策略规划
- **网页浏览**：信息检索、网页交互
- **代码执行**：程序理解、调试与执行
- **工具使用**：API 调用、函数调用
- **家居控制**：物联网设备管理

这种多维度的评估方式使得 AgentBench 能够揭示智能体在不同能力维度上的优势与不足。

**AgentBench 的重要性**：

- **全面性**：覆盖智能体的多种核心能力，避免"偏科"
- **标准化**：提供统一的评估框架，便于不同模型间的公平比较
- **指导性**：帮助研究者识别智能体的能力边界和改进方向
- **影响力**：被学术界广泛认可和使用，是智能体研究的重要参考

**实例：AgentBench 与单任务基准的对比**

| 特性 | AgentBench | 单任务基准（如 BFCL） |
|------|-----------|-------------------|
| 任务数量 | 8 个领域 | 单一领域 |
| 评估维度 | 多维度综合 | 单一维度深入 |
| 适用场景 | 通用能力评估 | 专项能力评估 |
| 评估成本 | 较高 | 较低 |

**AgentBench 评估流程**：

1. **环境准备**：搭建包含 8 个任务领域的评估环境
2. **数据集加载**：加载各领域的测试数据集
3. **智能体执行**：让智能体在模拟环境中完成各项任务
4. **结果评估**：根据任务完成情况计算得分
5. **综合分析**：生成各维度能力雷达图，分析优劣势

> Liu, X., Yu, H., Zhang, H., Xu, Y., Lei, X., Lai, H., ... & Zhang, D. (2023). AgentBench: Evaluating LLMs as Agents. *arXiv preprint arXiv:2308.03688*.

在 HelloAgents 评估体系设计中，AgentBench 作为通用能力评估的代表性基准，与 [[BFCL]]（工具调用能力评估）、[[GAIA]]（通用 AI 助手能力评估）共同构成三层评估体系。

- **与 BFCL 的对比**：BFCL 专注于工具调用准确性（使用 AST 匹配），AgentBench 关注综合通用能力（使用任务完成度、准确率等多指标）
- **与 GAIA 的对比**：GAIA 评估真实世界问题解决能力（使用准精确匹配），AgentBench 评估模拟环境中的多领域能力

> **实践建议**：在智能体开发的早期阶段，可使用 AgentBench 快速识别能力短板；在优化特定能力（如工具调用）时，应结合 BFCL 等专项基准进行精细调优；在评估成本有限时，优先选择 AgentBench 中与目标场景最相关的子集进行评估。

### WebArena

> WebArena 是由卡内基梅隆大学（CMU）推出的真实网页环境评估基准，专门用于评估智能体在真实网页环境中的自主任务完成能力和网页交互能力。与在模拟环境中进行评估不同，WebArena 让智能体在真实的网页环境中执行任务，从而更准确地反映其在现实世界中的表现。

WebArena 的核心设计理念是：**在真实环境中评估智能体**。它不是一个模拟的沙盒，而是让智能体真正地去浏览网页、填写表单、搜索信息、完成预订等。

WebArena 的主要特点：

- **真实环境**：使用真实的网页（而非模拟页面），包括 Wikipedia、GitHub、地图服务等
- **自主执行**：智能体需要自主规划、执行和反思，完成多步任务
- **端到端评估**：评估从任务理解到最终完成的整个流程
- **安全性**：通过特殊机制确保评估过程不会对真实服务造成影响

典型的 WebArena 任务包括：

- 在电商网站上搜索并比较商品
- 在地图上查找路线并获取交通信息
- 在 GitHub 上查找代码仓库并分析其内容
- 在预订网站上完成酒店或航班预订

**WebArena 的重要性**：

- **真实场景**：在真实网页环境中评估，结果更贴近实际应用
- **综合能力**：考验智能体的理解、规划、执行和反思能力
- **实用性**：评估结果直接对应实际应用场景，指导产品优化
- **挑战性**：真实环境的复杂性和不确定性对智能体提出了更高要求

**实例：WebArena 任务示例**

**任务**："帮我预订一家在纽约市中心、评分 4.5 以上、每晚价格不超过 200 美元的酒店，入住时间为下周五。"

智能体需要完成以下步骤：

1. 打开酒店预订网站
2. 搜索纽约市中心的酒店
3. 筛选评分 4.5 以上的酒店
4. 筛选价格不超过 200 美元/晚的酒店
5. 选择入住时间为下周五
6. 查看符合条件的酒店列表
7. 选择一家合适的酒店并尝试预订

评估维度：是否成功访问了正确的网站、是否正确使用了筛选条件、最终推荐的酒店是否符合所有要求、整个过程的交互效率和准确性。

> Zhou, S., Xu, F. F., Zhu, H., Zhou, X., Lo, R., Sridhar, A., ... & Neubig, G. (2023). WebArena: A Realistic Web Environment for Building Autonomous Agents. *arXiv preprint arXiv:2307.13854*.

WebArena 在 HelloAgents 评估体系中的定位：

- **真实环境评估的代表**：与 [[AgentBench]] 的模拟环境不同，WebArena 在真实网页环境中评估智能体，结果更贴近实际应用
- **与 GAIA 的关联**：GAIA 也评估真实世界任务，但 WebArena 更专注于网页交互能力，而 GAIA 覆盖更广的问题类型（文件处理、计算、推理等）
- **与 BFCL 的互补**：BFCL 评估工具调用准确性，WebArena 评估工具调用在真实环境中的实际效果

> **实践建议**：由于评估成本高，建议结合 [[AgentBench]] 等低成本基准进行初步筛选，再用 WebArena 进行最终验证；关注 WebArena 的最新变体和扩展，以获取更全面的评估能力。

### SWE-bench

> SWE-bench 是一个专门评估智能体（尤其是大语言模型）解决真实软件工程问题能力的基准测试。它收集了大量来自 GitHub 的真实 Issue 和对应的 Pull Request，要求智能体理解问题描述、定位代码缺陷、生成修复补丁，并验证修复是否成功。

SWE-bench 是软件工程领域最具影响力的智能体评估基准。与其他评估通用能力的基准不同，SWE-bench 专注于一个非常具体且高价值的场景：**自动修复代码 bug**。

SWE-bench 的核心设计：

- **真实数据**：全部来自真实的 GitHub 仓库，而非人工构造的测试用例
- **端到端评估**：从 Issue 描述到最终补丁验证的完整流程
- **可验证性**：每个任务都有明确的通过/失败判断标准（测试用例）
- **难度分级**：涵盖从简单 bug 修复到复杂架构问题的各种难度

智能体在 SWE-bench 上需要完成：

1. 理解 Issue 描述，明确问题所在
2. 在代码库中定位相关代码
3. 分析根因，设计修复方案
4. 生成代码补丁
5. 验证补丁是否通过所有测试用例

**SWE-bench 的重要性**：

- **实际价值**：代码自动修复是软件开发中最耗时、最昂贵的环节之一
- **技术挑战**：需要理解复杂代码、推理 bug 根因、生成正确补丁
- **商业应用**：成功的代码修复智能体可大幅降低软件开发成本
- **研究指导**：推动智能体在代码理解和生成领域的研究进展

**实例：SWE-bench 任务示例**

**Issue 描述**："当传入的参数 `data` 为 None 时，函数 `process_data` 会抛出 AttributeError。应该在处理前先检查 data 是否为 None。"

智能体需要完成的步骤：

1. **理解问题**：函数缺少对 None 的处理
2. **定位代码**：找到 `process_data` 函数的定义
3. **分析根因**：缺少空值检查
4. **生成补丁**：

```python
# 修复前
def process_data(data):
    return data.strip().lower()

# 修复后
def process_data(data):
    if data is None:
        return None
    return data.strip().lower()
```

5. **验证**：运行测试用例，确认修复成功

**SWE-bench 与其他代码评估基准的对比**

| 基准 | 任务类型 | 数据来源 | 评估重点 |
|------|---------|---------|---------|
| SWE-bench | 代码修复 | 真实 GitHub Issue | 理解、定位、修复 |
| HumanEval | 代码生成 | 人工构造 | 函数实现 |
| MBPP | 代码生成 | 众包 | 简单函数编写 |
| AgentBench-Code | 代码执行 | 模拟环境 | 执行与调试 |

SWE-bench 在 HelloAgents 评估体系中的定位：

- **软件工程能力评估的代表**：与 [[BFCL]] 的工具调用评估、[[GAIA]] 的通用能力评估不同，SWE-bench 专注于代码理解和修复这一高价值场景
- **与 AgentBench 的代码执行任务的对比**：AgentBench 包含代码执行任务，但 SWE-bench 更专注于真实代码修复，难度更高、价值更大
- **端到端评估**：从 Issue 描述到最终补丁验证的完整流程，与 [[WebArena]] 的端到端评估理念一致
- **可验证性**：每个任务都有明确的通过/失败判断标准（测试用例），这与 [[BFCL]] 的 AST 匹配评估类似

> **实践建议**：在评估智能体的代码能力时，SWE-bench 是比人工构造测试用例更可靠的选择；对于实际应用，可以从 SWE-bench 的任务中筛选与自己项目技术栈相关的子集进行评估；关注 SWE-bench 的最新变体（如 SWE-bench Lite），以降低评估成本；将 SWE-bench 的评估结果与 [[AgentBench]] 的代码执行任务结合分析，获得更全面的代码能力认知。

## RAG 专项评估：RAGAS 框架

> RAGAS（Retrieval Augmented Generation Assessment）是目前最流行的 RAG 效果量化评估框架。

| 指标 | 评估对象 | 含义 | 计算思路 |
|------|---------|------|---------|
| **Faithfulness** | 生成质量 | 回答忠实于检索内容的程度 | 答案中每个claim是否有context支持 |
| **Answer Relevancy** | 生成质量 | 是否真正在回答用户的问题 | 答案与原始问题的相关性 |
| **Context Precision** | 检索质量 | 检索结果中有多少是有用的 | Top-K中相关文档的排名位置 |
| **Context Recall** | 检索质量 | 是否检索到了所有必要信息 | ground truth中的信息被检索到的比例 |

详见 [[RAG]] — RAGAS评估框架位于RAG卡片的生产评估部分

## 三大基准对比总览

| 基准 | 机构 | 侧重维度 | 环境类型 | 核心评估能力 |
|------|------|---------|---------|-------------|
| [[AgentBench]] | 清华大学 | 通用能力 | 模拟环境 | 跨领域多任务综合表现 |
| [[WebArena]] | CMU | 真实交互 | 真实网页 | 网页环境端到端任务完成 |
| [[SWE-bench]] | Princeton | 代码修复 | 真实仓库 | 代码理解、定位与修复 |

三者构成了从**广度**（AgentBench 的多领域覆盖）到**深度**（WebArena 的真实环境交互、SWE-bench 的专业代码修复）的评估矩阵，建议根据智能体的应用场景选择合适的基准组合。

## 指标与基准的关联

评估指标与基准测试并非孤立存在，它们在实际评估中紧密关联：

- **[[AgentBench]]** 使用任务完成度和 [[准确率]] 作为核心评估指标
- **[[WebArena]]** 采用端到端的任务完成率，同时关注交互效率和准确性
- **[[SWE-bench]]** 使用测试通过率（本质上是 [[准确率]] 的一种形式）作为评估标准
- **[[F1 分数]]** 在 [[AgentBench]] 等多领域评估中，比单纯的 [[准确率]] 更能反映智能体在少数类别上的真实表现
- **[[用户满意度]]** 通过 [[LLM Judge]] 自动化代理和人工验证作为最终评判标准，连接技术指标与商业价值
