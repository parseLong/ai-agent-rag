# 七维确定性评分

> 把 harness 本身当成被测软件的评测体系——100% Python 确定性逻辑、零 LLM 调用、3 次跑分 hash 完全一致。==宁要可复现的"粗糙分"，不要会漂移的"精准分"。==

![[07-Attachments/harness-practice/七维评分图.jpg]]

---

## 核心定位

> 评测平台是==评估者==，不是执行者。它只检测被试是否走完了 harness 的每个节点（产物在不在、门禁过没过），而绝不替它去执行部署或测试。

---

## 七维评分体系

| 维度 | 权重 | 检查什么 | 怎么检查 |
|------|------|---------|---------|
| **流程完整性** | 22% | 该走的流程节点是否都走了？ | 检查产物文件是否存在 + 预设规则命中率，按 intent×risk 裁剪必需节点 |
| **产物质量** | 15% | 方案文档是否有实质内容？ | 结构检查（文件路径≥3、有代码块、有风险清单、有回滚方案）+ 反注水检测 |
| **代码正确性** | 22% | 代码能不能编译？单测能不能过？ | **真跑** `mvn compile` + `mvn test`，不信 AI 自报，编译失败=0分 |
| **效率** | 10% | 花了多少时间和 token？ | 耗时 + 成本 USD，支持与基线版本相对比 |
| **安全合规** | 8% | 有没有违反 harness 自身的规则？ | 扫描 transcript：Grep 搜 Java 文件、diff 含 ALTER TABLE 等 |
| **迭代能力** | 5% | 编译失败后能不能自己修好？ | 检测 BUILD FAILURE → BUILD SUCCESS 的恢复链条 |
| **接口验收** | 18% | 写的代码经得起集成测试吗？ | **真跑** ATDD 测试 + 检查接口测试证据（G7 门禁） |

### 为什么是确定性评分，不用 LLM 评委？

> 评测的唯一目的是==驱动迭代==——只有 3 次跑分完全一致，才能回答"这次改规范到底变好还是变坏"。一个偶尔波动 ±5 分的 LLM 评委，再"精准"也会让 A/B 对比彻底失去意义。

---

## 两个最关键的维度

### 流程完整性（22%）

不靠"模型说做了"，而靠"==产物文件在不在=="——文件系统不会说谎。

按 intent×risk 裁剪必需节点：
- `QUERY/NA`：不要求任何产物（满分）
- `BUG_FIX/LOW`：只查 5 个节点
- `FEATURE/HIGH`：查满 19 个

### 代码正确性（22%）

防注水的硬核维度：==真编译、真跑单测==。

还引入 **honesty gap**（诚实度差距）——对比 AI 自报结果和真实编译结果：

```
honesty_gap = AI自报G3通过 ? 实际编译挂了 ? 暴露差距
```

> [!warning] 评测环境不是越"干净"越好
> 给评测配了空隔离 Maven 仓库 → 依赖全解析失败、恒为 0 分；换回共享本地 6.9G `~/.m2` 缓存才跑通。==评测环境越"干净"，反而越不真实==。

## 三条轨道

| 轨道 | 入口 | 干什么 | 人工门 / 真部署 |
|------|------|--------|----------------|
| **评测** | `/eval` | 多版本×多case跑分对比、管基线 | ❌ headless 全自动 |
| **需求开发** | `/dev` | 用当前基线 harness 真实跑需求全链路 | ✅ 真 MCP + 确认门 + 真部署 |
| **问题排查** | `/query` | 输故障/traceId，只读排查根因 | 只读；真 bug 一键转 `/dev` |

---

## 评测平台解决什么

| 能力 | 证据 |
|------|------|
| **能区分好坏** | 删掉 Pre-Mortem/TDD 的劣化版：流程完整性 90→55、代码正确性 80→40，综合 C/65.2 vs 基线 B/80 |
| **不被自报欺骗** | honesty gap 检测 AI 虚报；无单测给中性分而非满分 |
| **失败响亮** | 不存在的 case → `invalid / FAIL / 0 分`，而非被兜底成"持平" |

---

## 自进化闭环

![[07-Attachments/harness-practice/自进化闭环.jpg]]

创建（AI生成/fork）→ 评测对比（7维×多case）→ 激活基线（留备份可回退）→ 收集弱项维度再优化。

甚至让 AI 拿"好配置"去改"待优化配置"生成候选版本——==用 AI 优化约束 AI 的规则，再用确定性分数验证优化是否有效==。

---

## 与现有评估框架的对照

| 维度 | 七维确定性评分 | [[智能体评估]] |
|------|--------------|--------------|
| **评估对象** | Harness 本身（流程纪律） | Agent 能力（任务完成度） |
| **评分方式** | 100% 确定性逻辑，零 LLM | 混合（测试通过率 + LLM 评委） |
| **可复现性** | 3 次跑分 hash 完全一致 | 受 LLM 评委随机性影响 |
| **参考基准** | SWE-bench + AgentBench + Anthropic Eval Guide + CMMI | AgentBench + WebArena + SWE-bench |
| **核心创新** | honesty gap + 流程完整性维度 | 超长会话串行评估方法 |

> [!important] 互补而非替代
> 七维评分评的是==Harness 质量==（流程纪律），[[智能体评估]] 评的是==Agent 能力==（任务完成度）。两者评估对象不同但互补：好 Harness → 好 Agent 表现。

---

## 欠账

- D2 产物质量只做结构检查，判不了方案设计优劣 → 拟引 LLM 评委 + 多次投票取中位数对冲非确定性
- eval 跑批单机单进程 → 需容器化 + 任务持久化
- 评测集仅 5 个 case → 长尾意图覆盖不足
