# 度量驱动演进闭环

> Harness 系统不是静态的城墙，而是==动态的免疫系统==——通过运行数据的采集、标注、回流，持续驱动自身的进化。这是 Harness Engineering 六大设计原则中最具前瞻性的一条："数据驱动进化"。

---

## 核心定义

度量驱动演进闭环是 Harness 系统的==自进化机制==——将 Agent 的每一次运行视为一次学习机会，建立从数据采集、标注、回流到策略更新的闭环，实现系统长期智能增长。

> [!important] 核心洞察
> ==Harness 也在经历"记忆到技能转化"==——作为一个系统，它也在从 L1（记住事实）→ L2（总结规则）→ L3（形成技能）的学习过程中，越运行越好用。详见 [[记忆到技能转化]]。

---

## 四类度量维度

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

度量不是"凑一张大表"，而是==诊断指针==——每种度量指向系统中需要改进的具体部位：

| 度量维度 | 衡量什么 | 关键指标 | 异常时指向 |
|---------|---------|---------|-----------|
| **任务效能** | Agent 做对了没有 | 任务成功率、指令遵循度、工具使用有效性 | → 回到[[驾驭工程#Harness 六大子系统|规划器]]和上下文策略 |
| **服务质量** | Agent 做快了没有 | 端到端延迟、首次响应延迟、错误率 | → 回到[[上下文注入]]和 Token 转化流水线 |
| **资源效率** | Agent 做省了没有 | 平均 Token 消耗、平均工具调用次数 | → 回到[[驾驭工程#Harness 六大子系统|资源配额]]和熔断策略 |
| **安全合规** | Agent 做安全了没有 | 策略拒绝率、安全事件数 | → 回到[[工程设计与安全#策略门控（Policy Gateway）|策略门控]]和沙盒 |

## 演进闭环的完整链路

```
Agent 运行 → 采集度量数据 → 标注失败模式 → 回流到 Harness → Agent 表现更好
    │              │              │              │              │
 PPAF循环     Tracing/         分类器训练     规则引擎更新     更优的
 产生数据     Metrics          模式识别       策略/权限调整    行为
    │              │              │              │              │
 L1记住事实 ─→ L2总结规则 ─→ L3形成技能 ─→ 自动化执行 ─→ 新度量数据
```

### 闭环的四个阶段

| 阶段 | 对应操作 | 工具/方法 | 与[[记忆到技能转化]]的对应 |
|------|---------|---------|--------------------------|
| **① 采集** | Tracing/Metrics 记录每次运行的完整事实 | LangSmith / Langfuse / OpenTelemetry / [[智能体评估|Helicone]] | ==L1 记住事实==——记录"发生了什么" |
| **② 标注** | 从事实中提炼失败模式和规则 | 错误分类器、模式识别、LLM 辅助标注 | ==L2 总结规则==——提炼"为什么会失败" |
| **③ 回流** | 规则内化为自动化的策略和机制 | 规则引擎更新、策略门控调整、上下文策略优化 | ==L3 形成技能==——自动化"如何避免失败" |
| **④ 验证** | 新度量数据验证改进是否有效 | 回归测试、A/B 对比、基准测试 | 闭环的==自我验证==——确认"改进真的有效" |

## 演进闭环与衰变定律的统一

度量驱动演进与 [[Harness Engineering#Harness 的衰变定律|衰变定律]] 不是矛盾而是==统一==——两者共同描述了 Harness 的==动态新陈代谢==：

| 力量 | 方向 | 触发条件 | 度量信号 |
|------|------|---------|---------|
| **衰变** | 规则==退出== | 模型能力增强，某类失败不再发生 | 对应失败率降为零 |
| **进化** | 规则==进入== | 新场景/新模型带来新失败模式 | 对应新出现的异常度量 |

> [!important] Harness 的新陈代谢
> Harness 的总量未必减少，但==组成会持续变化==——就像生物体的细胞：总数量大致稳定，但旧细胞凋亡、新细胞生成，组成持续更新。度量数据是驱动这一新陈代谢的==信号源==。

## 与[[智能体评估]]的交叉对照

度量驱动演进闭环的四类度量与 [[智能体评估]] 的现有评估体系有精确的对应关系：

| Harness 度量维度 | 对应评估体系 | 说明 |
|-----------------|------------|------|
| **任务效能** | [[AgentBench]] + [[SWE-bench]] | AgentBench 评估跨领域任务完成度，SWE-bench 评估代码修复成功率 |
| **服务质量** | 端到端延迟、TTFT | 传统性能评估指标 |
| **资源效率** | Token/成本跟踪 | [[智能体评估]] 中较少覆盖，是 Harness 特有维度 |
| **安全合规** | [[WebArena]] 安全场景 | WebArena 有真实环境中的安全交互评估 |

> [!tip] Harness 度量补充了评估体系的空白
> 传统[[智能体评估]]侧重于"模型能力"（准确率、F1），但 ==Harness 度量侧重于"系统表现"==——不是"模型在理想条件下能做到什么"，而是"系统在真实运行中实际表现如何"。两者互补：评估告诉你"模型的潜力"，度量告诉你"系统的现实"。

## 实践启示

> [!important] 构建度量闭环的关键步骤
> 1. **先做采集**：确保每次 Agent 运行都有完整的 Tracing 和 Metrics
> 2. **后做标注**：定期回顾度量数据，识别失败模式和新异常
> 3. **再做回流**：将标注结果转化为策略更新、权限调整、上下文策略优化
> 4. **最后验证**：用新度量数据验证改进是否有效，形成真正的闭环

> [!warning] 常见反模式
> - **只采集不标注**：有了 Tracing 数据但不分析——数据成了"数字垃圾"
> - **只标注不回流**：知道了问题但不修复——分析成了"纸上谈兵"
> - **只回流不验证**：做了改进但不验证效果——改进可能无效甚至有害
