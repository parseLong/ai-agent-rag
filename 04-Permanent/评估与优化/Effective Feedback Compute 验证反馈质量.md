# Effective Feedback Compute 验证反馈质量

> 决定 AI 干活靠不靠谱的不是"给它多少预算"，而是==检查做得多好==。原始 token 消耗仅解释 agent 成功率方差的 R²=0.33～0.42，而验证反馈质量达到 R²=0.94～0.99——三个数量级的差距。

---

## 核心发现（arxiv 2605.29682）

| 指标 | 解释成功率方差（R²） | 含义 |
|------|---------------------|------|
| **原始 Token 消耗** | 0.33～0.42 | 给 AI 多少预算几乎不影响成功率 |
| **工具调用次数** | 0.33～0.42 | AI 做了多少操作也几乎不影响成功率 |
| **Effective Feedback Compute** | **0.94～0.99** | ==检查做得多好几乎完全决定成功率== |

> [!important] 核心洞察
> ==钱不是关键，质检才是关键==。花 $100 不验证，不如花 $10 验证到位。这也印证了 [[Harness Engineering#护栏悖论]] 的核心论断：约束解空间反而让 Agent 更有生产力。

---

## 什么是 Effective Feedback Compute

不是"给 AI 多少反馈"（数量），而是==反馈的质量和精准度==：

| 低质量反馈 | 高质量反馈（Effective） |
|-----------|----------------------|
| "代码有 bug，请修复" | 编译报错信息 + 具体行号 + 失败测试名 |
| LLM 评委打 70 分（波动 ±5） | 确定性门禁：G3 编译 FAIL → 0 分，无波动 |
| 自报"已完成" | 真实验证：`mvn test` 实际运行结果 |
| 静默兜底成 70 分中性值 | 失败响亮：`invalid / FAIL / 0 分` |

## 对 Harness 设计的影响

这个发现直接解释了为什么 [[G1-G8 门禁墙参考卡]] 和 [[Honesty Gap]] 是 Harness 最值钱的部分：

```
给更多预算 (R²=0.33) → 几乎无效
做更好的验证 (R²=0.94~0.99) → 几乎完全决定成功率
```

| Harness 组件 | 对应的 Effective Feedback |
|-------------|-------------------------|
| G1-G8 门禁 | 确定性验证反馈——编译/测试的真实结果 |
| [[Honesty Gap]] | 自报 vs 真实的差距——暴露虚报 |
| [[七维确定性评分]] | 100% 确定性、零 LLM、3 次跑分 hash 一致 |
| hook 拦截 | 实时围栏——工具调用前拦截 |

## 实践启示

1. **验证优先于预算**：优化 AI Coding 效率的正确方向不是"买更好的模型"或"给更多 token"，而是"做更好的验证"
2. **确定性优于语义**：宁要可复现的"粗糙分"（R²=0.99），不要会漂移的"精准分"（LLM 评委 ±5 分波动）
3. **失败响亮优于静默兜底**：失败信息是最高质量的反馈——[[Honesty Gap]] 证明了这一点
4. **门禁外置优于 prompt 约束**：确定性代码检查（R²=0.94）远比让模型"记住"规则（R²=0.33）有效

> [!tip] 一句话总结
> ==AI 工程的核心不是让 AI 更聪明，而是让验证更到位==。Effective Feedback Compute 给了这个判断最硬的数据支撑。
