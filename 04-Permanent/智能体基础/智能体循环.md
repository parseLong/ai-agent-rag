# 智能体循环

> Agent的核心运行机制是一个While循环，每次迭代是LLM推理+工具调用+上下文更新，直到任务完成退出。是Agent系统的"心跳"。

## 核心要点

- **每次迭代的流程**：LLM Call推理→解析响应→执行工具→结果追加到上下文→继续循环或退出
- **终止条件**：LLM不再调用工具（直接返回文本答案）即为完成
- **安全设置**：设置最大迭代次数上限（通常10-20步），防止死循环

> [!important] 核心论点
> Agent框架设计的核心就是在Agent Loop这个While循环中设计如何管理上下文。上下文工程是智能的核心。

## Agent Loop 与六大模块的关系

创建一个最轻量级的 Agent，除了最关键的 Agent Loop 本身，还会涉及以下模块围绕循环协同运转：

```
┌─────────────────────────────────────────┐
│          Agent Loop (While循环)          │
│                                         │
│  ┌─→ LLM推理 ←─────────────────────┐   │
│  │   ↓                              │   │
│  │   解析响应                        │   │
│  │   ↓                              │   │
│  │   执行工具 ←── Tools(CLI/Script) │   │
│  │   ↓                              │   │
│  │   结果追加上下文                  │   │
│  │   ↓                              │   │
│  │   ← Prompt(渐进式加载) ──────┐   │   │
│  │   ← Planning(任务拆解) ─────┐ │   │   │
│  │   ← Memory(压缩/检索) ─────┐ │ │   │   │
│  │   ← Workflow(Skill编排) ───┐ │ │ │   │   │
│  │   ← Runtime(执行环境) ─────┐ │ │ │ │   │   │
│  └─────────────────────────────┘ │ │ │ │ │ │
│                                  │ │ │ │ │ │
│  终止？ ──→ Yes ──→ 返回结果      │ │ │ │ │ │
│         ──→ No  ──→ 继续循环  ←──┘ │ │ │ │ │
└──────────────────────────────────┘ │ │ │ │ │
                                     │ │ │ │ │
  外围模块:                           │ │ │ │ │
  • Prompt → 提供指令和上下文          │ │ │ │ │
  • Planning → 拆解任务和规划路径      │ │ │ │ │
  • Memory → 提供记忆检索和压缩       │ │ │ │ │
  • Tools → 提供可调用的工具能力      │ │ │ │ │
  • Workflow → 编排流程和Skill调用    │ │ │ │ │
  • Runtime → 提供执行环境和文件系统  │ │ │ │ │
```

> [!note] 各模块在循环中的角色
> - **Prompt**：每轮循环开始时注入上下文——从固化 System Prompt + 渐进式加载
> - **Planning**：每轮循环中 LLM 可能重新规划——Todo List 动态调整
> - **Memory**：循环过程中压缩对话、检索长期记忆——防止上下文溢出
> - **Tools**：每轮循环中 LLM 可能调用工具——CLI 命令 / Script 脚本
> - **Workflow**：循环中可能触发 Skill 执行——内聚逻辑 + 脚本编排
> - **Runtime**：提供所有操作的执行环境——文件读写、安全隔离

## 实战落地：Plan 即工具设计

> [!important] 核心方案：Plan 不是模式切换，而是 ReAct 循环中的普通工具
> 将 Planning 实现为循环内的 `todo` 工具，而非独立规划阶段。简单任务直接 ReAct 零开销，复杂任务自然触发 `todo` 创建计划——只有一套 ReAct Loop，没有模式切换的复杂度。

> [!example] todo 工具的设计要点
> - **每次返回完整列表+统计摘要**（而非 diff）：LLM 没有可靠"记忆上次结果"的能力，必须每次给出全量信息
> - **两种更新模式**：`todo(merge=False)` 替换模式——直接覆盖旧计划，适合重新规划；`todo(merge=True)` 增量更新模式——追加新任务，适合逐步扩展
> - **TodoStore 是纯内存 `list[dict]`**：不需要数据库，todo 是会话级的，会话结束即消亡
> - **每个 Agent 实例独立持有 TodoStore**：不用全局单例——主 agent 和子 agent 共享计划会导致冲突

> [!example] 上下文压缩后 todo 状态如何存活
> - 压缩后通过 `format_for_injection()` 自动将 todo 状态注入回 messages，确保长任务不因压缩"失忆"
> - **仅输出 pending 和 in_progress 的任务**：已完成和取消的任务会导致 LLM 误以为需要重做，必须过滤掉
> - **会话恢复时从历史重建 todo**：`hydrate_todo_store()` 从最近消息中找到包含 todos 的工具返回值，重建 todo 状态

> [!important] 核心 Loop 六重保障
> | 保障机制 | 说明 |
> |----------|------|
> | IterationBudget（迭代预算） | 限制最大循环次数，防止无限循环 |
> | 可中断 | 支持外部中断信号，避免死循环不可控 |
> | 上下文压缩 | 长对话自动压缩，防止上下文溢出 |
> | ErrorClassifier + jittered_backoff | 错误分类+抖动退避重试，区分可重试错误和致命错误 |
> | 空响应防护 | 连续 2 次空响应则终止，防止 LLM 无输出死循环 |
> | ToolDispatcher（工具修复） | 工具调用失败时自动修复参数，提高鲁棒性 |

> [!note] 对比传统方案的优势
> - 简单任务零开销：无需进入"规划模式"再退出
> - 只有一套 ReAct Loop：没有规划/执行双模式切换的复杂状态机
> - `todo(merge=False)` 随时替换计划：规划不是一次性承诺，而是可动态修正
> - `format_for_injection()` 确保压缩存活：传统方案压缩后计划信息丢失，本方案自动回注

## PPAF 闭环：Agent Loop 的认知模型

PPAF（Perception-Planning-Action-Feedback）闭环是 Agent Loop 的**认知层抽象**——Agent Loop 关注工程实现（LLM Call → 工具调用 → 结果反馈），PPAF 关注认知流程：

| PPAF 阶段 | 说明 | Agent Loop 对应 |
|-----------|------|----------------|
| **P — 感知（Perception）** | 感知当前世界状态：用户输入、工具结果、历史对话、任务进度 | 上下文注入（Prompt 组装） |
| **P — 规划（Planning）** | 基于感知信息，更新目标、拆解任务、选择下一步行动 | LLM 推理（生成 Thought 或 Function Call） |
| **A — 行动（Action）** | 执行内部操作（更新记忆）或外部操作（调用工具） | 工具执行 |
| **F — 反思（Feedback/Reflection）** | 行动结果注入上下文，供下一轮反思与重规划 | 结果追加到上下文 → 进入下一轮循环 |

> [!note] PPAF 与 Agent Loop 的互补关系
> - **Agent Loop**：工程视角——"每次迭代做什么"（LLM Call → Parse → Execute → Append）
> - **PPAF**：认知视角——"Agent 如何思考"（感知 → 规划 → 行动 → 反思）
> - 两者是同一机制的**不同维度描述**，理解两者能更完整地把握 Agent 的运行本质
