# 上下文窗口

> 模型能处理的最大Token数，决定了单次交互中能包含的信息量上限。

## 核心要点

- **定义**：上下文窗口 = 系统提示词 + 用户输入 + 模型推理 + 工具返回结果的总Token数上限
- **长上下文的挑战**：标准注意力计算复杂度O(L²)，[[KV 缓存]]与L成正比，上下文越长计算量和显存指数增长
- **1M上下文的意义**：Agent多轮任务轻松消耗数十万Token，整仓库代码理解需要百万级Token，长文档推理需要跨段落能力

## 1M 上下文的三大消耗场景

| 场景 | Token 消耗 | 传统方案的问题 |
|------|-----------|--------------|
| **Agent 多轮任务** | 30轮 Coding Agent 数十万起步 | 每轮塞入指令+源文件+shell输出+推理轨迹，128K窗口撑不住几轮 |
| **整仓库代码理解** | 中型项目 80–120万 | RAG 漏一个调用点就是一个 bug，超长上下文让模型自己看见所有调用关系 |
| **长文档推理** | 单文档 50–80万 | 切块摘要前后逻辑缺失，直接在原始材料上跨段落推理 |

> [!important] 1M 不是"能写更长的 prompt"
> 1M 上下文是让 Agent、整库代码、长文档等任务的执行**真正落地**——不是军备竞赛，而是实际需求驱动。
