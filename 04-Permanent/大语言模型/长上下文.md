# 长上下文（Long Context）

> 来源：罗福莉（小米）、姚顺雨（OpenAI）访谈

## 定义

长上下文能力是Agent时代的核心基础，模型能被输入更多Context、理解更多Context时潜力越高。

> "Agent范式很吃Post-train，Context越长也是产生智能的一种方式。模型能够被输入更多Context、理解更多Context的时候，潜力就越高。这跟原来Chat范式完全不一样，原来Chat的Context就是人输的，人输的很短，所以纯靠预训练。"
> — 罗福莉

## 关键洞察

### 1. 数据来源

> "真正能达到128K到一兆长度的数据，极大概率只有两类：一类是Code数据，一类是书籍。"
> — 罗福莉

| 数据类型 | 特点 | 适合建模 |
|----------|------|----------|
| Code数据 | 文件之间关联更强，信号更集中 | 首选 |
| 书籍 | 信号太发散 | 次选 |

### 2. Code的独特性

> "但书籍的信号太发散了，Code文件之间关联更强。在一个长上下文依赖更密集的数据集上去训练，模型自然对长上下文的建模会更好。"
> — 罗福莉

### 3. 成本考量

> "你训这个模型是几千卡。但实际上要投入大量卡做研究，研究的卡会是训练卡的几倍。"

- 100兆上下文太贵
- 需要在效果、成本、速度之间权衡
- TPS（每秒事务处理量）是关键指标

### 4. 当前领先者

> "现在Claude在一兆上下文能力的稳定度上，只有Claude 4.6 Opus和Sonnet是领先的。其他像Gemini，虽然号称上下文能力很好，其实都不行。"
> — 罗福莉

## 与Agent的关系

### Agent需要长上下文

> "Agent是一个非常长程和多轮的任务。"
> — 罗福莉

- Agent是非常长程和多轮的任务
- 必须依赖长上下文
- 更长的Context是产生智能的一种重要方式

### Chat vs Agent

| 维度 | Chat时代 | Agent时代 |
|------|----------|-----------|
| Context长度 | 短（用户输入） | 长（任务全貌） |
| 依赖 | 预训练 | 后训练+预训练 |
| 关键能力 | 检索能力 | 理解+推理+规划 |

## Memory层级视角

> "前年冬天，我读到冯诺依曼临终前写的一本书，The Computer and the Brain。最让我印象深刻的一句话是：Essentially, the Environment is always the most outer part of the Memory Hierarchy."
> — 姚顺雨

### Memory Hierarchy

| 层级 | 说明 | 例子 |
|------|------|------|
| CPU缓存 | 最快，最小 | Working Memory |
| 内存 | 中等 | 短期记忆 |
| 硬盘 | 慢，大 | 长期记忆 |
| **环境** | **最外层** | **外部世界** |

> "对于人，你有你的Memory Hierarchy，有Working Memory、Long-Term Memory在脑子里，但最外层是你的笔记本、Google Doc、Notion，这些是你最外层Memory Hierarchy的一部分。"
> — 姚顺雨

## Long Context vs Long-Term Memory

> "Long Context是实现Long-Term Memory的一种方式。如果你 能实现1亿或1千亿或无限长的Context，它是实现Long-Term Memory的一种方式。它是一种和人区别很大的方式，但这是有可能的。"
> — 姚顺雨
