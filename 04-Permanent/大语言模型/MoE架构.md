# MoE架构

> Mixture of Experts（专家混合模型）——用稀疏激活让万亿参数模型只激活一小部分，以可控计算成本获得超大模型容量。

## 核心思想

MoE 的关键洞见：**不是每个 token 都需要所有参数**。

传统 Dense 模型：每个 token 都要经过所有参数的计算——参数越多，算力成本越高。
MoE 模型：每个 token 只"激活"与它最相关的少数专家，其余专家保持沉默。

- **总参数量**决定模型的**知识容量**（"记住多少"）
- **激活参数量**决定模型的**计算成本**（"花多少算力"）

DeepSeek V4-Pro 的稀疏度：1.6T 总参数 / 49B 激活 ≈ **3% 激活率**——用 3% 的算力获得 1.6T 的知识容量。

## 路由/门控机制

MoE 的核心是**路由器（Router/Gate）**——一个小型网络，决定每个 token 应该发送给哪些专家。

```
token → Router → "你最适合专家 3 和专家 7" → 只激活专家 3 和 7 → 输出 = 专家3输出 + 专家7输出
```

路由器的输出通常是：
- **Top-K 选择**：每个 token 只选 K 个专家（如 K=2、K=4），其余忽略
- **加权组合**：被选中专家的输出按路由器给出的权重加权求和

> [!note] 路由器的设计挑战
> - **负载均衡**：如果所有 token 都涌向少数专家（"热门专家"），其他专家闲置→训练浪费。需要负载均衡损失（Load Balancing Loss）惩罚不均匀分配
> - **专家崩塌**：训练初期随机分配，某专家偶尔表现好→收到更多 token→表现更好→收到更多→正反馈循环→其他专家"饿死"

## DeepSeek V4 的 MoE 创新

### 1. 细粒度计算通信重叠

![[MoE_计算通信重叠时序.png]]

| 方案 | 做法 | 问题 |
|------|------|------|
| **原始 MoE** | 算完再通信，通信完继续算 | 通信时算力闲置，计算时通信闲置，资源利用率低 |
| **粗粒度重叠** | 矩阵运算的同时并行执行分发/回收通信 | 按阶段调度，计算与通信时间不对齐，仍有气泡 |
| **V4 细粒度重叠** | 随时中断、交接，碎片化调度 | 尽可能减少甚至完全消除气泡 |

> **比喻**：洗衣机和烘干机——洗衣和烘干耗时不一致，如果中间不能打断，总会一台空转等另一台。如果两台机器都可以随时中断、交接——洗衣机刚洗完就送进烘干机、自己立刻洗下一批——流水线最高效率运转。

### 2. 完全消除气泡的条件

$$\frac{V_{comp}}{V_{comm}} > \frac{C}{B}$$

- $V_{comp}/V_{comm}$：模型架构与量化方案确定后，计算量与通信数据量的比值（定值）
- $C/B$：硬件计算能力与通信带宽的比值（定值）
- MoE 是算力密集型的，当计算量占比足够大时，通信可完全"隐藏"进计算

> [!tip] 硬件选型启示
> 不必一味追求高带宽——当 $V_{comp}/V_{comm} > C/B$ 时，低带宽也够用。

### 3. FP4 量化最大收益点

[[FP4量化感知训练]] 精准打击：MoE 专家权重量化为 FP4——因为 MoE 的参数量远大于 ATTN 模块，显存占用的大头在这里，压缩收益最大。

## MoE vs Dense 模型对比

| 特性 | Dense 模型 | MoE 模型 |
|------|-----------|---------|
| 激活率 | 100%（所有参数都参与计算） | ~3%（只激活少量专家） |
| 知识容量 | 等于激活参数量 | 远大于激活参数量 |
| 计算成本 | 与总参数成正比 | 与激活参数成正比 |
| 通信开销 | 低（单机可完成） | 高（专家分布在多机器上） |
| 代表模型 | GPT-4、Claude | DeepSeek V4、Mixtral |
