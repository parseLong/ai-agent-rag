# 注意力机制

> 让模型在处理每个 token 时能够"关注"序列中所有其他 token，决定哪些信息更重要。

## 核心要点

- **自注意力**：每个 token 与所有其他 token 计算关联度，复杂度 O(L²)，是 [[Transformer 架构]] 的核心
- **CSA（压缩稀疏注意力）**：DeepSeek V4 引入，将 token 分组压缩后只对高关联度的压缩结果做精确计算，大幅降低 1M 上下文中的计算量
- **HCA（高度压缩注意力）**：比 CSA 更粗粒度的压缩，用于先定位"大块信息"，再在其中做 CSA 精细计算

## 1M 上下文为什么必须跳出"每对都算"

标准 Transformer 每层 attention 计算**序列中每一对 token 之间的关联**。4K 上下文时理论完美且跑得动，但 1M 上下文时：

### Prefill 阶段（输入处理）

整段上下文一口气进来，attention 计算复杂度是 **O(L²)**：
- L 从 4K 提升到 1M（提升 256×），Prefill FLOPs 提升 **6.5 万倍**
- 这一步是"一次性消化整本书"——书越长，消化成本指数级增长

### Decode 阶段（逐词生成）

每生成一个 token 都要把前面所有 L 组 KV 从 HBM 搬一遍：
- **显存占用和带宽压力与 L 成正比**（$\text{Mem}_{KV} \propto L$）
- L 从 4K 提升到 1M，KV Cache 膨胀 **256 倍**
- 这一步是"每说一句话都要回顾全书"——书越长，回顾成本线性增长

必须从"每一对都算"的范式跳出来——这就是 CSA + HCA 混合注意力的出发点。

## CSA：压缩稀疏注意力

> 形象比喻：引入一排"会议记录员"，不是参与者，而是站在外围做纪要的第三方。每位记录员把相邻两组发言综合成一份纪要。使用时先索引找到高关联度的纪要，再逐一精细查阅。

### 三步流程

**步骤一：压缩**（纪要编写）
- 将全部 token 分组，每组 $m$ 个 token
- 连续两组的压缩内容拼接在一起，保证切割后语义连贯性
- 组内每个 token 是**贡献加权的**，记录员给每个"发言维度"独立打分

![[标准残差公式.png]]

压缩数学表达式：

$$C_{group_i} = S_{group_i} \odot (W_v^{(C)} h_i + b_v^{(C)}) + S_{group_{i-1}} \odot (W_v^{(Z)} h_{i-1} + b_v^{(Z)})$$

- 左侧：当前组 token 的带权重加和压缩
- 右侧：前一组 token 的带权重加和压缩（**哈德玛积**，不是矩阵乘法——每个维度独立加权）
- 权重矩阵 $S$ 和值投影矩阵 $C$ 均由隐状态值投影得到，4 个 $W$ 和 2 个 $b$ 都是可学习参数

**步骤二：闪电索引**（纪要检索）
- 快速计算每个 token 与全部压缩纪要的关联度分数
- 每个注意力头加权后得到最终得分
- 选择关联度最高的 $k$ 份纪要

![[CSA_注意力计算公式.png]]

$$score = q \cdot (w \cdot K)$$

其中 $K$ 采用与压缩完全一致的计算思路（即压缩结果的"key 投影"），$q$ 和 $w$ 由隐状态投影得到。

> [!note] 闪电索引与 DSA
> 闪电索引思路与 DeepSeek V3.2 的 **DSA（Dynamic Sparse Attention）** 机制一致——快速计算关联度 + top-$k$ 选择 + 稀疏计算。V4 把 DSA 的思想融入了 CSA 的"纪要检索"步骤。

**步骤三：注意力计算**（精细查阅）
- 标准注意力计算，但**共享 KV**
- 只对闪电索引选出的 top-$k$ 组压缩信息做计算

## HCA：高度压缩注意力

> 形象比喻："速记员"版 CSA——每场参与人数更多，但只聚焦本场，不需要关注前后两场。纪要份数不多，直接逐一查阅。

### 两步流程

**步骤一：压缩**
- 每组 $m'$ 个 token（$m' \gg m$，比 CSA 粒度更粗）
- **不需要**连续两组压缩内容拼接
- 组内加权计算与 CSA 一致

$$C_{group_i} = S_{group_i} \odot (W_v^{(C)} h_i + b_v^{(C)})$$

**步骤二：注意力计算**
- 与 CSA 同样的公式，共享 KV
- 使用每个 token 的查询矩阵 $q$ 同 HCA 压缩的 $C$ 矩阵计算
- 由于每份纪要涵盖更多 token，总纪要份数不多，**直接逐一查阅**（无需闪电索引）

## 混合注意力的三层处理

实践中，先 HCA 定位上下文中关联度高的"大块信息"，再在这些信息中做 CSA 稀疏注意力计算：

> **内容海选（HCA）→ 内容精选（CSA 闪电索引）→ 稀疏采样 & 精确计算（CSA attention）**

这就是 [[DeepSeek V4]] 让 1M 上下文"算得过、存得下"的核心解法。

> [!important] 三层处理的精妙之处
> HCA 是粗粒度、低成本的全局扫描——"哪些大块可能相关？"CSA 是细粒度、中等成本的局部聚焦——"这些大块中哪些具体段落最重要？"标准注意力是精确计算——"对选出的少量段落做精确的注意力。"三层逐级缩小搜索范围，计算量和显存消耗逐级降低。

## CSA/HCA 如何同时降低 Prefill 和 Decode 成本

| 阶段 | 传统方案 | CSA/HCA 方案 | 效果 |
|------|---------|-------------|------|
| **Prefill** | O(L²) 全序列注意力 | HCA全局定位 + CSA局部精确 | 计算量大幅降低 |
| **Decode** | 每步搬运全部 L 组 KV | 只搬运 CSA/HCA 压缩后的少量 KV | [[KV 缓存]] 大幅减小 |

| 方案 | 团队 | 核心思路 | 效果 |

| **混合线性注意力** | Kimi、Qwen | 引入核函数打破 Softmax 约束 → 先算 $K \times V$ 再作 $Q$ 计算 → 复杂度从 O(L²) 降至 O(L) | 仅维护固定大小隐状态矩阵，不需随序列长度增长的 KV Cache |
