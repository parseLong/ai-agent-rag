# LLM 训练流程

## 训练流程全景

大语言模型的训练是一个多阶段递进的过程，每个阶段都有明确的目标和递进关系。整体流程如下：

```
预训练（Pre-training）        —— 基座：在海量无标注数据上学习通用语言表示
    ↓
SFT（监督微调）               —— 对齐：让模型学会遵循指令，从"文本补全"转变为"对话模型"
    ↓
微调（Fine-tuning）           —— 定制：在特定任务/领域数据上进一步适配
    ↓
强化学习（Reinforcement Learning）—— 决策引擎：通过试错与反馈优化策略
    ↓
RLHF / GRPO                  —— 对齐人类偏好：用人类反馈或组内比较来优化输出质量
    ↓
Post-train（后训练）           —— Agent时代的新重心：长程交互、环境适配、算力与预训练持平
```

> [!important] 核心洞见
> 在 Agent 时代，Post-train 的算力配比正与 Pre-train 赋予同等权重（1:1），训练周期也从数周拉长到半年至一年。RL Scaling —— 在 Agent 场景中大规模应用强化学习 —— 是当前的关键赛点。

## 预训练（Pre-training）

### 定义

预训练是指在大规模**无标注文本数据**上，通过自监督学习任务（如预测下一个词），让模型学习语言的通用表示和规律的过程。预训练是大语言模型训练的第一阶段，为后续的微调打下基础。

### 我的理解

预训练就像是让模型"读遍天下书"。通过在海量文本上学习，模型掌握了语言的语法、语义、逻辑和常识。这个过程不需要人工标注数据，只需要让模型阅读互联网上的文本、书籍、代码等。预训练后的模型就像一个"通识渊博的大学生"，虽然还没有专业技能，但已经具备了扎实的基础。

### 为什么重要

- **奠定了模型的知识基础**：预训练决定了模型的知识广度和语言理解能力
- **实现了知识的迁移**：预训练学到的通用知识可以迁移到各种下游任务
- **降低了对标注数据的依赖**：只需要少量标注数据就能完成特定任务
- **催生了大模型时代**：预训练 + 微调成为 NLP 的标准范式

### 预训练任务

**语言建模（Language Modeling）**：最经典的预训练任务，目标是预测序列中的下一个词。

$$P(w_1, w_2, ..., w_n) = \prod_{i=1}^{n} P(w_i | w_1, ..., w_{i-1})$$

**掩码语言建模（Masked Language Modeling）**：BERT 使用的预训练任务，随机掩盖输入中的一些词，让模型预测被掩盖的词。

### 缩放法则（Scaling Laws）

缩放法则揭示了模型性能与模型参数量、训练数据量以及计算资源之间存在着可预测的幂律关系。在对数-对数坐标系下，模型的性能（通常用损失 Loss 来衡量）与参数量、数据量和计算量这三个因素都呈现出平滑的幂律关系。

**Chinchilla 定律**（2022 年，DeepMind）：在给定的计算预算下，为了达到最优性能，模型参数量和训练数据量之间存在一个最优配比。最优的模型应该比之前普遍认为的要小，但需要用多得多的数据进行训练。

**能力的涌现**：当模型规模达到一定阈值后，会突然展现出在小规模模型中完全不存在或表现不佳的全新能力。例如，链式思考（Chain-of-Thought）、指令遵循（Instruction Following）、多步推理、代码生成等能力，都是在模型参数量达到数百亿甚至千亿级别后才显著出现的。

### 预训练数据

| 数据类型 | 示例 | 作用 |
|---------|------|------|
| 网页文本 | Common Crawl | 提供海量语言素材 |
| 书籍 | Project Gutenberg | 提升语言质量 |
| 学术论文 | arXiv | 增强专业知识 |
| 代码 | GitHub | 提升代码生成能力 |
| 对话数据 | Reddit、论坛 | 增强对话能力 |

### 预训练 vs 微调

| 阶段 | 数据 | 目标 | 计算成本 |
|------|------|------|---------|
| 预训练 | 海量无标注数据 | 学习通用语言表示 | 极高（数百万美元） |
| 微调 | 少量标注数据 | 适应特定任务 | 较低 |

### 实例

**GPT-3 的预训练**：
- **数据量**：约 45TB 的文本数据
- **数据来源**：Common Crawl、WebText、Books、Wikipedia 等
- **训练时间**：数周至数月
- **计算成本**：数百万美元
- **结果**：模型学会了语法、常识、推理、甚至编程

> 预训练是大语言模型最昂贵、最耗时的阶段，但也是最重要的阶段。一个好的预训练基座模型，即使只经过简单的微调，也能在特定任务上表现出色。在智能体开发中，选择预训练充分、数据质量高的基座模型至关重要。

## SFT（监督微调）

### 定义

SFT（Supervised Fine-Tuning，监督微调）是一种微调技术，它使用大量**人工标注的"指令-回答"数据对**，对预训练模型进行监督学习。经过 SFT 后，模型能够更好地理解并遵循用户的指令，从而从"文本补全模型"转变为"对话模型"。

### 我的理解

如果说预训练让模型学会了"说话"，那么 SFT 就是让模型学会了"听话"。未经 SFT 的模型可能只会续写文本，而经过 SFT 的模型能够理解你的指令并给出合适的回答。ChatGPT 之所以能进行对话，关键步骤之一就是 SFT。

### 为什么重要

- **对齐人机交互**：让模型从"文本补全"转变为"指令遵循"
- **提升任务执行能力**：模型学会按照用户意图完成特定任务
- **RLHF 的前置步骤**：SFT 通常是 RLHF 流程的第一步
- **构建对话模型的基础**：几乎所有对话模型都经过 SFT

### 核心要素

**指令数据格式**：

```json
{
  "instruction": "请将下面的英文翻译成中文",
  "input": "How are you?",
  "output": "你好吗？"
}
```

**数据质量要求**：
- **多样性**：覆盖各种任务类型和场景
- **准确性**：回答必须正确且高质量
- **一致性**：遵循统一的格式和风格
- **安全性**：避免有害或不当内容

**训练过程**：
1. 准备高质量的指令-回答数据集
2. 将数据格式化为模型可接受的输入
3. 使用交叉熵损失进行监督学习
4. 评估模型在测试集上的表现

### SFT 与预训练、RLHF 的关系

```
预训练（Pre-training）
    ↓
SFT（监督微调）
    ↓
RLHF（人类反馈强化学习）
    ↓
部署使用
```

| 阶段 | 数据类型 | 目标 |
|------|---------|------|
| 预训练 | 无标注文本 | 学习通用语言表示 |
| SFT | 指令-回答对 | 学会遵循指令 |
| RLHF | 人类偏好排序 | 对齐人类价值观 |

### 指令调优对提示的影响

**指令调优（Instruction Tuning）** 的出现，极大地简化了与模型交互的方式：

- **对"文本补全"模型的提示**：需要用少样本提示"教会"模型做什么
  ```
  这是一段将英文翻译成中文的程序。
  英文:Hello
  中文:你好
  英文:How are you?
  中文:
  ```

- **对"指令调优"模型的提示**：可以直接下达指令
  ```
  请将下面的英文翻译成中文:
  How are you?
  ```

### 实例

| 输入 | 预训练模型输出 | SFT 后模型输出 |
|------|-------------|--------------|
| "翻译：Hello" | "Hello 是一个英文单词，意思是..." | "Hello 的中文意思是：你好。" |
| "讲个笑话" | "笑话是一种文学体裁..." | "好的，给你讲个程序员笑话：0 和 1 走在路上，1 说..." |

> SFT 是大模型训练流程中承上启下的关键步骤。一个好的 SFT 数据集应该覆盖尽可能多的任务类型，保持回答的高质量和高标准，遵循统一的格式规范，避免偏见和有害内容。在智能体开发中，如果通用模型的表现不够理想，收集特定领域的数据进行 SFT 是提升效果的有效手段。

## 微调（Fine-tuning）

### 定义

微调（Fine-tuning）是在**预训练模型**的基础上，使用特定任务的**标注数据**进行进一步训练的过程。通过微调，预训练模型学到的通用知识可以被迁移到特定的下游任务上，从而在少量标注数据的情况下达到良好的性能。

### 我的理解

如果把预训练比作"大学毕业"，那么微调就是"岗前培训"。预训练让模型拥有了通用的语言能力，而微调则让模型学会在特定场景下如何应用这些能力。比如，一个预训练模型可以通过微调变成专门做情感分析的模型，或者变成专门写代码的模型。

### 为什么重要

- **高效利用预训练知识**：不需要从头训练，节省大量计算资源
- **少量数据即可生效**：通常只需要几千到几万条标注数据
- **显著提升特定任务性能**：在特定领域可以达到甚至超过人类水平
- **实现模型定制化**：让通用模型适应特定业务场景

### 微调类型

**1. 全量微调（Full Fine-tuning）**：更新模型的所有参数。效果最好，但计算成本最高。

**2. 部分微调（Partial Fine-tuning）**：只更新模型的部分参数（如最后几层）。平衡效果和效率。

**3. 参数高效微调（PEFT）**：
- **LoRA（Low-Rank Adaptation）**：只训练低秩矩阵，冻结原模型参数
- **Prompt Tuning**：优化输入提示的嵌入
- **Adapter**：在模型中插入小型适配层

**4. 指令微调（Instruction Tuning）**：使用"指令-回答"格式的数据训练模型，让模型学会遵循指令。这是从"文本补全模型"转变为"对话模型"的关键步骤。

### 模型幻觉与局限

经过微调后的模型仍可能产生**模型幻觉（Hallucination）**，即生成的内容与客观事实、用户输入或上下文信息相矛盾。

**幻觉类型**：
- **事实性幻觉**：模型生成与现实世界事实不符的信息
- **忠实性幻觉**：在文本摘要、翻译等任务中，生成的内容未能忠实地反映源文本的含义
- **内在幻觉**：模型生成的内容与输入信息直接矛盾

**缓解方法**：
1. **数据层面**：高质量数据清洗、引入事实性知识、RLHF
2. **模型层面**：探索新的模型架构，或让模型能够表达其对生成内容的不确定性
3. **推理与生成层面**：
   - **检索增强生成（RAG）**：在生成之前从外部知识库中检索相关信息
   - **多步推理与验证**：引导模型进行多步推理，并在每一步进行自我检查
   - **引入外部工具**：允许模型调用搜索引擎、计算器等工具

### 微调 vs 预训练

| 对比项 | 预训练 | 微调 |
|-------|-------|------|
| 数据类型 | 无标注数据 | 标注数据 |
| 数据量 | 海量（TB 级） | 较少（GB 级或更少） |
| 训练目标 | 学习通用语言表示 | 适应特定任务 |
| 计算成本 | 极高 | 较低 |
| 时间 | 数周至数月 | 数小时至数天 |

### 实例

**客服机器人的微调流程**：
1. **预训练模型**：选择一个通用的预训练模型（如 GPT-3.5）
2. **准备数据**：收集客服对话数据，标注意图和回答
3. **微调训练**：使用 LoRA 在客服数据上微调模型
4. **评估测试**：在测试集上评估模型性能
5. **部署上线**：将微调后的模型部署到客服系统

**效果对比**：预训练模型客服回答准确率 60% → 微调后 90%

> 微调是让通用模型"落地"的关键步骤。实际建议：先用提示工程尝试，如果效果不够再考虑微调；使用 LoRA 等参数高效微调方法降低计算成本；注意数据质量和多样性避免过拟合；微调后进行充分的测试和评估。

## 强化学习（Reinforcement Learning）

### 定义

强化学习（Reinforcement Learning，RL）是机器学习的三大范式之一，与监督学习、无监督学习并列。它最大的特点，是通过与环境互动，在不断试错中"学习"如何做出最优决策。

### 我的理解

强化学习是一种模拟人类决策行为的学习机制——不依赖明确的标签数据，而是通过试错和反馈让智能体不断改进自身策略。就像学骑自行车：没人给你标注"正确姿势"，你通过不断摔倒（惩罚）和成功骑行（奖励）来学会。

### 为什么重要

在人工智能领域，强化学习属于一种**高级的学习方式**，在**决策、控制、优化**等任务中占据了核心地位。[[智能体（Agent）]]的核心决策机制就依赖于强化学习。

### 发展历史

| 时间 | 里程碑 | 影响 |
|------|--------|------|
| 1989年 | Watkins提出Q-learning算法 | 为后续发展奠定基础 |
| 1990年代 | 蒙特卡洛方法、TD Learning理论完善 | 基础理论成型 |
| 2000年代 | 计算资源和仿真环境受限 | 实际应用受限 |
| 2013年 | DeepMind提出DQN | 将深度学习引入RL，Atari游戏超越人类 |
| 2016年 | AlphaGo击败李世石 | 融合RL+自我对弈+深度网络，成为研究热点 |

### 核心要素

| 要素 | 定义 | 通俗理解 |
|------|------|---------|
| **智能体（Agent）** | 执行动作的学习者 | "学生" |
| **环境（Environment）** | 智能体与之互动的对象 | "考场" |
| **状态（State）** | 当前环境的描述 | "当前局面" |
| **动作（Action）** | 智能体可以采取的行为 | "做出的选择" |
| **奖励（Reward）** | 执行动作后获得的反馈 | "得分" |
| **策略（Policy）** | 根据状态选择动作的规则 | "决策方式" |
| **价值函数（Value Function）** | 估计某状态的长期价值 | "前途评估" |

### 三大范式对比

| 范式 | 特点 | 比喻 |
|------|------|------|
| 监督学习 | 对照标准答案训练 | 学生对照答案写作业 |
| 无监督学习 | 发现数据规律 | 学生自己总结规律 |
| 强化学习 | 试错+反馈优化 | 学生在游戏中摸索规则 |

> [!note] 强化学习最适用于哪些任务？
> 那些无法提前列出正确答案，但可以通过长期观察"结果好不好"来评估行为的任务，比如下棋、开车、投资、打游戏等。

### 与深度学习的关系

> [!important] 深度强化学习（DRL）
> 强化学习和深度学习相结合，形成**深度强化学习（Deep Reinforcement Learning，DRL）**。通过将深度学习中的神经网络技术应用到强化学习中，能够处理更复杂的环境和任务，如图像驱动的机器人导航。

### 常见算法

| 算法 | 类型 | 特点 |
|------|------|------|
| **Q-learning** | 基于值 | 经典RL算法基础 |
| **DQN** | 基于值+深度学习 | 将Q-learning与深度网络结合 |
| **A2C / A3C** | 基于策略 | 异步策略梯度 |
| **PPO** | 基于策略 | 稳定策略梯度训练 |
| **DDPG** | 基于策略 | 连续动作空间 |

### 应用场景

| 领域 | 应用 | 例子 |
|------|------|------|
| 游戏 | AI博弈 | AlphaGo |
| 自动驾驶 | 优化驾驶策略 | 应对复杂路况 |
| 机器人控制 | 完成复杂任务 | 搬运物体、组装零件 |
| 推荐系统 | 优化推荐内容 | 根据点击/购买调整 |
| 金融决策 | 寻找最优策略 | 高频交易、投资组合管理 |

### 挑战与未来

**四大挑战**：

| 挑战 | 说明 |
|------|------|
| **训练效率低** | 需数百万次试验才能学出好策略，现实世界成本极高 |
| **不稳定性强** | 小扰动可能导致策略崩溃，训练易震荡 |
| **泛化能力差** | 一个环境训练的策略很难迁移到不同环境 |
| **安全性问题** | 自动驾驶、金融等敏感场景中策略不稳定后果严重 |

**四大发展方向**：

| 方向 | 说明 |
|------|------|
| **模仿学习** | 先观察人类行为，再微调RL |
| **元学习（Meta-RL）** | 学会"如何学习"，提升适应速度 |
| **层次强化学习** | 将任务分解为子任务，提高效率 |
| **多智能体强化学习** | 多个智能体间博弈协作 |

> [!note] 与[[智能体（Agent）]]的关联

> 强化学习的核心洞见：不是所有问题都有"标准答案"，有些问题只能通过"试错+反馈"来学习最优策略。这正是智能体和自动驾驶等复杂系统面临的现实。理解RL的本质——奖励驱动的策略优化——是理解[[智能体（Agent）]]行为逻辑的关键。

## RLHF（人类反馈强化学习）

### 定义

RLHF（Reinforcement Learning from Human Feedback，人类反馈强化学习）是一种训练大语言模型的方法，它通过**人类对模型输出的偏好反馈**来训练一个奖励模型（Reward Model），然后使用强化学习算法（如 PPO）优化语言模型，使其生成更符合人类价值观和期望的输出。

### 我的理解

RLHF 就像是给模型请了一位"人类导师"。模型生成多个回答，人类导师对这些回答进行排序（哪个更好），模型根据这些反馈学习什么样的回答是人类更喜欢的。经过反复迭代，模型逐渐学会了生成更有用、更无害、更诚实的回答。

### 为什么重要

- **对齐人类价值观**：让模型不仅"聪明"，还要"善良"
- **提升输出质量**：模型学会生成更有用、更准确的回答
- **减少有害内容**：降低模型生成偏见、歧视、有害内容的概率
- **ChatGPT 的核心技术**：ChatGPT 的成功很大程度上归功于 RLHF

### RLHF 的三个阶段

**阶段 1：收集人类偏好数据**
1. 模型对同一输入生成多个回答
2. 人类标注员对这些回答进行排序（从好到坏）
3. 收集大量这样的偏好数据

**阶段 2：训练奖励模型**
1. 使用人类偏好数据训练一个**奖励模型（Reward Model）**
2. 奖励模型学会预测人类会喜欢哪个回答
3. 奖励模型输出一个分数，分数越高表示回答越好

**阶段 3：使用 PPO 优化语言模型**
1. 语言模型生成回答
2. 奖励模型给回答打分
3. 使用 **PPO（Proximal Policy Optimization）** 算法更新语言模型
4. 目标是最大化奖励模型的评分
5. 同时加入 **KL 散度约束**，防止模型偏离太远

### PPO 算法原理

PPO 是一种策略梯度方法，其核心思想是通过**裁剪（Clipping）**机制来限制策略更新的幅度，防止模型变化过大而导致训练不稳定。

PPO 的目标函数包含两部分：
1. **裁剪后的策略梯度目标**：限制新旧策略之间的差异
2. **KL 散度惩罚项**：防止新策略偏离旧策略太远

### RLHF 流程图

```
预训练模型（GPT）
    ↓
SFT（监督微调）
    ↓
收集人类偏好数据
    ↓
训练奖励模型
    ↓
PPO 强化学习优化
    ↓
RLHF 模型（如 ChatGPT）
```

### RLHF 的挑战

| 挑战 | 说明 |
|------|------|
| 标注成本高 | 需要大量人工标注偏好数据 |
| 奖励黑客 | 模型可能找到"欺骗"奖励模型的方法 |
| 价值观对齐 | 不同文化、群体的价值观可能存在冲突 |
| 过度优化 | 模型可能过度迎合奖励模型，牺牲多样性 |

### 实例

**RLHF 在 ChatGPT 中的应用**：

1. **SFT 阶段**：使用人工编写的对话数据训练基础模型
2. **奖励模型训练**：
   - 输入："如何制作蛋糕？"
   - 模型生成 4 个回答
   - 人类排序：回答 B > 回答 A > 回答 D > 回答 C
   - 训练奖励模型学会这种排序偏好
3. **PPO 优化**：模型生成回答 → 奖励模型打分 → 使用 PPO 更新模型参数 → 迭代直至收敛

**效果对比**：

| 指标 | SFT 模型 | RLHF 模型 |
|------|---------|----------|
| 有用性 | 70% | 85% |
| 无害性 | 75% | 90% |
| 诚实性 | 80% | 88% |

> RLHF 是大模型训练中最复杂、最具争议的环节之一。它让模型从"会说"到"说得好"，但也带来了新的挑战：如何确保奖励模型本身没有偏见？如何避免模型过度优化（Reward Hacking）？如何在安全性和创造性之间取得平衡？这些问题也催生了 RLHF 的替代方案，如 DPO（Direct Preference Optimization）。

## GRPO（群组相对策略优化）

### 定义

GRPO（Group Relative Policy Optimization，群组相对策略优化）是由 DeepSeek 团队提出的一种高效强化学习算法。与 PPO 需要训练单独的价值模型（Critic）不同，GRPO 通过对**同一问题的多个回答进行组内相对比较**来计算优势值，从而省略了价值模型，大大降低了计算开销。

### 我的理解

传统的 PPO 算法需要两个模型：一个生成回答（Actor），一个评估回答好坏（Critic，价值模型）。GRPO 的洞见是：**我们不需要一个独立的评估模型，只需要让模型生成多个回答，然后比较这些回答的相对好坏即可**。

这就像考试评分：PPO 需要专门的阅卷老师（Critic），而 GRPO 采用"同组排名"的方式——同一道题，看看谁答得更好，不需要绝对分数，只需要相对排序。

### 为什么重要

- **大幅降低计算成本**：无需训练价值模型，节省 50% 以上的计算资源
- **简化训练流程**：减少了模型数量，降低了系统复杂度
- **提升训练稳定性**：避免了价值模型估计不准带来的噪声
- **推动了大模型普及**：让更多团队能够负担得起强化学习训练

### 核心思想

**1. 组内相对比较**：对于同一个问题，模型生成一组回答（如 16 个），然后通过奖励模型给每个回答打分。优势值不是绝对分数，而是**相对于组内其他回答的表现**。

$$A_i = \frac{R_i - \text{mean}(R)}{\text{std}(R)}$$

其中 $R_i$ 是第 i 个回答的奖励分数。

**2. 省略价值模型**：传统 PPO 的更新公式需要价值模型 $V(s_t)$ 计算。GRPO 直接用组内相对优势替代，不需要 $V(s_t)$。

$$\nabla_\theta J = \mathbb{E}[A_t \nabla_\theta \log \pi_\theta(a_t|s_t)]$$

**3. 策略更新约束**：与 PPO 类似，GRPO 也使用裁剪（clipping）机制来限制策略更新幅度，防止模型变化过大。

### GRPO 算法步骤

1. 对于输入问题，从当前策略中采样一组回答（如 16 个）
2. 使用奖励模型给每个回答打分
3. 计算组内平均分 $mean(R)$ 和标准差 $std(R)$
4. 计算每个回答的相对优势值：$A_i = \frac{R_i - mean(R)}{std(R)}$
5. 使用 PPO 的裁剪目标函数，但用 $A_i$ 替代价值模型的估计值
6. 更新策略模型参数

### GRPO vs PPO

| 特性 | PPO | GRPO |
|------|-----|------|
| 价值模型 | 需要独立的 Critic 模型 | 不需要 |
| 计算成本 | 高（需要训练两个模型） | 低（只需训练一个模型） |
| 优势值计算 | 基于价值模型的估计 | 基于组内相对比较 |
| 内存占用 | 大 | 小 |
| 训练稳定性 | 受价值模型质量影响 | 更稳定 |

### 实例

**GRPO 的训练过程**：
1. 输入问题："解释什么是机器学习"
2. 模型生成 16 个回答（组内采样）
3. 奖励模型给每个回答打分：回答 1: 0.8, 回答 2: 0.6, ..., 回答 16: 0.9
4. 计算组内平均值和标准差
5. 计算每个回答的相对优势值
6. 使用 GRPO 目标函数更新模型参数

> GRPO 是对 RLHF 训练流程的一次重要优化。它证明了在强化学习中，通过巧妙的设计可以大幅减少计算开销。不过，GRPO 也有一些局限性：组内样本数量影响估计的准确性；对于奖励模型质量的要求更高；在某些任务上可能不如 PPO 稳定。

## 强化学习训练环境设计

> 来源：姚顺宇（Anthropic）访谈

### 核心原则

在大规模[[强化学习]]领域，核心在于**找到合适的环境**。

> "得找到合适的环境，这个环境回馈信号足够清晰，本身也是一个很强的数据源。在这个上面能让训练非常稳定，这事就能做成。" — 姚顺宇

### 两个必要条件

| 条件 | 说明 |
|------|------|
| **回馈信号清晰** | 环境的反馈必须明确，能判断对错 |
| **数据干净** | 数据本身要足够干净、对模型来说是可学习的 |

**满足这两个条件，训练就能非常稳定地实现 scale up。**

### 为什么环境如此重要？

姚顺宇的早期洞察：

> "当我意识到环境有问题，我第一个比较重要的工作是 WebShop，首先要解决环境问题。如果你没有一个好的任务或环境，把这个游戏刷得再高，没有意义。" — 姚顺雨

> "当时，你如果有一个非常差的任务，你永远不可能学到非常好的东西。首先你要找一个足够有挑战的任务，这个任务能做出本质的新方法。" — 姚顺雨

### Coding 作为训练环境

| 优势 | 说明 |
|------|------|
| **回归信号清晰** | 代码有明确的正确/错误判断 |
| **数据充分** | 互联网上有大量代码数据 |
| **可验证性** | 可以运行代码验证结果 |
| **长程任务** | 软件开发是长程多轮任务 |

> "Coding是模型使用工具和环境交互的一个很好的抽象。这个抽象的好处在哪？回归信号清晰，数据充分。很难在别的场景下，找到能同时有这两个特质的使用工具场景。" — 姚顺宇

### 关键要素

**1. 训练数据产生**：环境需要能够产生足够的训练数据，数据量是 scaling 的基础。

**2. Reward 设计**：
> "Reward设计需要针对环境做精准的reward设定。这个事情比较有难度。" — 罗福莉

**3. 训练稳定性**：
> "你把这个环境反馈信号设计好了过后，怎么去让训练更稳定，这是一个很重要的命题。" — 姚顺宇

### 不同公司的差异化

> "很多算法设计并不独立于算法，它非常强的依赖于你的基础设施。" — 姚顺宇

采样器（产生 trace/token 的机器）与训练器（改变模型权重的机器）可能不一样，这导致了不同公司的算法设计差异。基础设施建设好的公司可以花更多精力在训练效果上，基础设施差的公司则主要精力花在控制差异、让训练稳定上。

> "很多这种小的 tips，很多 knowhow，其实没什么用。" — 姚顺宇

## RL Scaling（强化学习扩展）

> 来源：罗福莉（小米）访谈

### 定义

RL Scaling是指在 Agent 场景中大规模应用[[强化学习]]，通过环境交互和 reward 设计来提升模型能力。

> "现在大家在 Pre-train 上的代差是基本没有的，或者说非常接近。现在至少跟23年要去追平 Pre-train 的差距一样，大家很 all in，要去做好 Agent 的 Post-train。更具体说，是在 Agent 上怎么做好 RL 的 scaling。" — 罗福莉

### 核心要素

**1. 环境重要性**：需要构造真实的长程任务环境，环境是 RL 的基础设施。

**2. Reward 设计**：
> "怎么还原当时所有人用这个事情的环境。有这个环境，才能更好去进行更长程的交互。有了环境过后，才可以针对这个环境本身做更精准的reward设定。这个事情比较有难度。" — 罗福莉

**3. 关键赛点**：在 Agent 上做好 RL 的 scaling 是关键赛点，方向清晰但路径需要探索。

**4. 算力平衡**：
> "当RL算力与预训练算力达到同一个水位的时候，就会给大家分享。" — 罗福莉

当 RL 算力与预训练算力达到同一水位时，会有突破性进展。

### 挑战

| 挑战 | 说明 |
|------|------|
| 训练效率 | 一兆上下文训练非常慢，一两小时才能 reout 完 |
| 环境还原 | 还原真实使用环境很难 |
| Reward 精确性 | 精准的 reward 设定有难度 |

### Coding 是最佳训练场景

> "Codie有泛化力的一个最本质原因：Agent是一个非常长程和多轮的任务。" — 罗福莉

| 特性 | 说明 |
|------|------|
| **信号清晰** | 代码运行结果可以明确判断对错 |
| **数据充足** | 互联网上有海量代码数据 |
| **可验证性** | 可以运行代码验证结果 |
| **长程任务** | 软件开发是非常长程的任务 |

> "Code文件之间关联更强。在一个长上下文依赖更密集的数据集上去训练，模型自然对长上下文的建模会更好。" — 罗福莉

### Code 作为 AI 的 affordance

> "Code有点像人的手。它某种程度上，是AI最重要的affordance（环境给予行动者的可能性）。对于物理世界，人最重要的affordance是手——我们围绕它制造各种工具，比如锤子、笔、筷子。但对AI、对Digital Agent来说，最重要的affordance可能就是code。" — 姚顺雨

## Post-train（后训练）

> 来源：罗福莉（小米）访谈

### 定义

Post-train（后训练）是在模型预训练完成后进行的强化训练阶段，在 Agent 时代变得与预训练同等重要。

> "在卡的调配上：至少在Chat时代，for研究、for Pre-train和for Post-train的用卡比例非常夸张，比如3:5:1，现在一个非常合理的用卡比例可能是3:1:1。" — 罗福莉

### 算力配比变化

| 时代 | 研究 | 预训练 | 后训练 | 说明 |
|------|------|--------|--------|------|
| Chat时代 | 3 | 5 | 1 | 预训练主导 |
| 2026年 | 3 | 1 | 1 | 均衡配置 |
| 顶尖团队 | 1 | 1 | 1 | 完全均衡 |

> "预训练跟后训练一个比例，这是今年可能发生的很大变化。顶尖团队应该都是1:1了。" — 罗福莉

### 核心要点

**1. 训练周期拉长**：Post-train 周期正在拉长，可能需要**半年到一年**，相比 Chat 时代几天到几周大幅增加。

> "Agent范式很吃Post-train，你对框架理解，对于多Agent的协作，乱七八糟也好，可以理解为我们在Post-train上要投入的算力，可能跟Pre-train是相当的。所以Post-train时间周期在拉长。" — 罗福莉

**2. Agent 范式特点**：非常吃 Post-train，Context 越长越能激发智能潜力。

> "Context越长也是产生智能的一种方式。模型能够被输入更多Context、理解更多Context的时候，潜力就越高。" — 罗福莉

**3. 关键能力要求**：对框架的理解、多 Agent 协作能力、RL Infra 系统开发能力。

### 为什么需要这么长时间？

> "因为可能做了半年或一年Post-train过后，那些场景全变了。以前大家觉得128K就够，但过几个月可能觉得需要10兆上下文。类似这样的逻辑。" — 罗福莉

- Agent 框架快速迭代
- 场景需求不断变化
- 需要持续适配和优化

### Post-train 的重要性

> "接下来两三个月，大家怎么发生变化，是考验团队整体研究水平、技术敏捷程度，以及怎么拥抱新的范式来做研究的关键。" — 罗福莉

## 全流程关键对比

| 训练阶段 | 数据类型 | 核心目标 | 计算成本 | 关键算法/技术 |
|---------|---------|---------|---------|-------------|
| 预训练 | 海量无标注文本 | 学习通用语言表示 | 极高（数百万美元） | 语言建模、缩放法则 |
| SFT | 指令-回答对 | 学会遵循指令 | 中等 | 交叉熵损失 |
| 微调 | 特定任务标注数据 | 适配特定领域/任务 | 较低 | LoRA、Adapter、Prompt Tuning |
| 强化学习 | 环境交互+奖励反馈 | 通过试错优化策略 | 高 | PPO、DQN |

| Post-train | 环境交互+长程任务 | Agent 时代的全面强化 | 与预训练持平 | RL Scaling、环境设计 |
