# KV 缓存

> 缓存注意力计算的 Key 和 Value，避免 decode 阶段每生成一个 token 都要重算前面所有 KV——是 Transformer 推理加速的核心机制。

## 核心原理

在 Transformer 的推理过程中，两个阶段对 KV 的需求不同：

| 阶段 | KV 的角色 | 成本特点 |
|------|----------|---------|
| **Prefill**（输入处理） | 所有 token 的 KV **一次性计算** | 计算量 O(L²)，但只需算一次 |
| **Decode**（逐词生成） | 每个新 token 与**前面所有缓存的 KV** 做注意力 | 显存 $\text{Mem}_{KV} \propto L$，带宽压力大 |

KV Cache 的思路：**Prefill 时算好存下来，Decode 时只查不重算**。

| 指标 | 不用 KV Cache | 用 KV Cache |
|------|-------------|-----------|
| Decode 每步 | 重算前面所有 K 和 V | 只算新 token 的 Q，与缓存 KV 做注意力 |
| 显存占用 | 仅模型参数 | 模型参数 + $\text{Mem}_{KV} \propto L$ |
| 关键瓶颈 | — | **显存**是瓶颈（KV 存储量随上下文线性增长） |

> [!important] KV Cache 是显存瓶颈
> 模型参数是固定的，但 KV Cache 随上下文长度线性增长。1M 上下文时 KV Cache 膨胀 256 倍 → 显存成为推理的首要瓶颈。[[注意力机制]] 的 CSA/HCA 正是为了压缩 KV、降低这个瓶颈。

## 标准方案：PagedAttention

vLLM 等推理引擎使用 PagedAttention 管理 KV Cache：
- 将 KV Cache 切分成固定大小的"页"
- 按需分配，避免预分配浪费
- **假设所有层的 KV 形状一致**

## DeepSeek V4 的挑战与创新

V4 引入 [[注意力机制]] CSA/HCA 混合注意力后，每层的 KV Cache **长得都不一样**：
- CSA 的 KV 是压缩后的纪要（大小固定，结构紧凑）
- HCA 的 KV 是更粗粒度压缩（大小固定但结构不同）
- SWA（滑动窗口注意力）的 KV 是未压缩的原始数据（大小随窗口增长）
- 尾部 state 有特殊的存储需求

传统 PagedAttention 假设所有层 KV 形状一致，直接失效。

### V4 的 KV Cache 架构

拆开分类存储——不强制统一形状，而是让每种注意力机制有自己的 Cache 规则：

| KV 类型 | 来源 | 存储策略 |
|---------|------|---------|
| **SWA 缓存** | 滑动窗口注意力 | 窗口内的原始 KV，有**三档取舍方案** |
| **CSA 缓存** | 压缩稀疏注意力 | 压缩后的 KV，大小固定可预测 |
| **HCA 缓存** | 高度压缩注意力 | 更粗粒度压缩的 KV，大小固定 |
| **尾部 state** | 最后几层的特殊状态 | 特殊存储需求 |

### SWA 的三档取舍方案

滑动窗口注意力需要存储窗口内的原始 KV，但窗口大小直接影响显存消耗。V4 提出三档取舍：

| 档位 | 窗口大小 | 显存消耗 | 信息保留 | 适用场景 |
|------|---------|---------|---------|---------|
| **保守档** | 大窗口 | 高 | 最多 | 精度优先 |
| **均衡档** | 中窗口 | 中 | 较多 | 效率与精度兼顾 |
| **激进档** | 小窗口 | 低 | 最少 | 效率优先 |

> [!tip] 三档取舍的本质
> 这不是简单的"大小选择"，而是**根据任务需求动态切换**——简单对话用激进档，复杂推理用保守档，大多数场景用均衡档。

### KV Cache 持久化

所有用户的 system prompt 和知识库文档内容不经常变，每次请求都重算 KV 极浪费资源。

V4 的创新：面对异构 KV，**拆开分类持久化**：
- CSA/HCA 的压缩 KV → 直接持久化（体积小，磁盘存储友好）
- SWA 的原始 KV → 持久化但有三档取舍（不同档位对应不同的磁盘占用）
- 尾部 state → 持久化（固定体积）

> [!tip] 持久化的本质
> 从"每次都算"到"算一次用多次"。V4 的创新不是简单的"存到磁盘"，而是**让异构 KV 的每种类型都有合适的持久化策略**。
