# 团队知识库共建

> 团队知识库是独立 Git 仓库，不寄生于任何业务项目——==知识的生命周期独立于项目，知识是团队的资产而非项目的附属==。

---

## 独立 Git 仓库架构

```
team-knowledge.git                        ← 独立 Git 仓库
├── knowledge-catalog.md                  ← 全景目录（Agent 查询入口）
├── .knowledge-config.yaml                ← 团队配置（成员、冲突策略）
├── team-conventions/                     ← Layer 0-T: 团队约定
├── tech-wiki/                            ← Layer 1: 技术知识
│   ├── catalog.md                        ← 分类清单
│   ├── patterns/TK-PAT-001.md
│   └── anti-patterns/TK-AP-001.md
├── biz-wiki/                             ← Layer 2: 业务知识
│   └── {domain}/
│       ├── catalog.md
│       ├── entities/BK-AD-E001.md
│       └── pitfalls/BK-AD-P001.md
├── project-profiles/                     ← 项目画像
└── contributions/                        ← 贡献暂存区
    ├── pending/
    └── conflicts/
```

### 为什么要独立仓库？

| 原因 | 说明 |
|------|------|
| **跨项目共享** | 同一团队的多个项目连接同一知识仓库，项目 A 沉淀的知识，项目 B 自动受益 |
| **生命周期独立** | 业务项目可能归档或重构，但==知识不应该跟着项目消失== |
| **权限独立** | 知识库的贡献和消费权限可以独立于代码仓库管理 |

## 三种团队角色

| 角色 | 权限 | 适用人群 | 类比 |
|------|------|----------|------|
| **maintainer** | 裁决内容冲突、审批 proven 提升、管理成员 | 团队负责人、资深工程师 | 知识库的"法官" |
| **contributor** | 通过工作流自动贡献（创建/验证/标记矛盾） | 正式团队成员 | 知识库的"作者" |
| **reader** | 只消费知识（查询/注入），不贡献 | 新成员试用期 | 知识库的"读者" |

> [!tip] 设计理念
> contributor 的贡献==通过工作流自动发生==，不需要手动写知识条目。Agent 在 ARCHIVE 阶段自动提取知识并写入贡献暂存区。

## 贡献模式——区块链思想 × Git 实现

| 区块链思想 | AI Team 实现 | 机制 |
|-----------|-------------|------|
| **不可篡改的追加日志** | log.md 只追加不修改 | 每条变更记录贡献者、时间、会话哈希 |
| **贡献可溯源** | evidence.contributors[] | 类似 Git blame，粒度为知识条目级 |
| **共识机制** | maturity 多人验证提升 | draft→verified: 1 人验证; verified→proven: ≥2 人 + ≥2 项目 |

---

## 冲突解决策略

> [!important] 低摩擦设计
> 大多数情况可以==自动处理==，只有真正的内容矛盾才需要人工介入。

| 冲突类型 | 处理方式 |
|----------|----------|
| **纯新增**（不同条目） | 自动合并，两条都保留 |
| **证据追加**（同条目验证） | 自动合并，evidence 数组合并去重 |
| **成熟度提升** | 自动合并 |
| **内容矛盾** | 写入 `contributions/conflicts/`，通知 maintainer 裁决 |
| **成熟度冲突**（一升一降） | 保留较低成熟度 + 标记 contradiction |
