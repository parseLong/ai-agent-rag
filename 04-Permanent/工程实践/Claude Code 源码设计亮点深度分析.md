# Claude Code 源码设计亮点深度分析

> Claude Code 是一个约 512K 行 TypeScript 的生产级 AI Agent 系统。它不仅仅是"用 LLM 做事"，而是在**每一个架构决策**上都体现了深思熟虑。本文提炼了 12 个最具启发性的设计亮点，每个都从"设计动机 → 实现手法 → 可迁移的启发"三层递进分析。

---

## 设计亮点 1：死代码消除（Bun `bun:bundle`）

### 设计动机

传统的运行时配置检查 `if (config.enabled)` 有三个致命问题：
1. **代码始终存在**——即使功能未启用，代码仍占用包体积
2. **可被绕过**——修改配置即可启用未授权功能
3. **依赖仍被加载**——即使入口未调用，依赖模块仍会被解析

### 实现手法

```typescript
import { feature } from 'bun:bundle'

const voiceCommand = feature('VOICE_MODE')
  ? require('./commands/voice/index.js').default
  : null
```

`feature()` 宏在**构建时**求值——Bun 的打包器看到 `false` 分支，直接把整个 `require()` 连同其依赖树从产物中删除。未启用的功能**不是被隐藏，而是不存在**。

### 可迁移的启发

> [!important] 启发：安全始于构建时
> **不要相信运行时检查能保护你。** 如果一段代码不应该被执行，最安全的做法是让它根本不存在于产物中。这与"零信任"安全哲学一脉相承——代码的存在本身就是攻击面。在构建自己的 Agent 系统时，考虑将特性开关与构建管道深度集成，而非仅在运行时做开关判断。

## 设计亮点 2：Prompt Cache 稳定性的排序策略

### 设计动机

Anthropic 的 Prompt Caching 机制基于**前缀匹配**——只有连续不变的前缀才能命中缓存。工具列表的任何变化都会破坏缓存，导致重新计算全量 input token 的成本。

### 实现手法

```typescript
// assembleToolPool 的排序策略
return uniqBy(
  [...builtInTools].sort(byName).concat(allowedMcpTools.sort(byName)),
  'name'
)
```

关键设计决策：
- **内置工具在前**：内置工具极少变化，构成稳定的前缀
- **按名称排序**：确定性排序保证相同工具集合总是产生相同序列
- **MCP 工具在后**：MCP 工具经常增删，放在后段只破坏后半部分的缓存

### 可迁移的启发

> [!important] 启发：缓存友好的数据排列
> 在构建依赖缓存的外部系统时，**将最稳定的数据放在最前面，最易变的数据放在最后面**。这不是微优化——对于 Prompt Caching 来说，一次缓存失效意味着数万 token 的重新计算，可能产生数十美元的成本差异。在数据库索引设计中也适用：将最常查询但最少变化的字段作为索引前缀。

## 设计亮点 3：工具系统的自包含 + 统一接口

### 设计动机

Agent 系统的核心挑战是**工具扩展**——新工具不能破坏已有系统，第三方工具不能与内置工具冲突，权限系统必须对所有工具统一处理。

### 实现手法

每个工具是一个**完全自包含的模块**：

```typescript
type Tool = {
  name: string                    // 自描述
  inputJSONSchema: ToolInputJSONSchema  // 自描述
  isEnabled: () => boolean        // 自描述
  call: (input, context) => Promise<ToolResult>  // 自执行
  description?: string            // 自描述
  progress?: ProgressCallback     // 自控制
}
```

添加新工具是 **O(1) 复杂度**——写一个文件、注册一行，不需要修改任何已有代码。MCP 工具通过适配器模式转换为同样的 Tool 接口，LLM 无法区分来源。

### 可迁移的启发

> [!important] 启发：自描述组件 + 统一接口
> **让每个组件自己描述自己的能力、约束和行为，系统只定义接口。** 这是"策略模式"与"开放-封闭原则"的深度结合。在构建可扩展系统时，不要用中央配置描述所有组件——让每个组件自带元数据，系统只负责聚合和过滤。Claude Code 的工具注册不使用自动扫描，而是显式枚举，这看似"麻烦"但保证了构建时的确定性和审计友好性。

## 设计亮点 4：权限系统的责任链 + 分类器

### 设计动机

Agent 在真实环境中执行操作，真实环境有危险操作。不能"全开"或"全关"——需要**每个操作独立判断**，且判断必须快速（不阻塞用户体验）。

### 实现手法

六步决策链：

```
模式检查 → Deny 规则 → Allow 规则 → Hook 检查 → Classifier → 用户确认
```

Bash Classifier 是**非 LLM 的基于规则分类器**——几毫秒完成判断，不需要调用 API。两阶段设计：Stage 1 快速判断（`max_tokens=64`），Stage 2 深度思考（`max_tokens=4096`）只在模糊场景触发。

### 可迁移的启发

> [!important] 启发：分层快速决策 + LLM 只处理模糊
> **将确定性决策交给规则，将模糊决策交给模型。** 规则引擎处理 90% 的明确情况（ls 是安全的、rm -rf 是危险的），LLM 只在灰色地带介入。这避免了"每个操作都调 LLM 判断安全性"的性能和成本灾难。在构建自己的 Agent 安全层时，先建立规则引擎覆盖已知的安全/危险模式，只在规则无法判断时才动用更重的机制。

## 设计亮点 5：上下文压缩的分层递进策略

### 设计动机

LLM 的上下文窗口是有限资源。Agent 长时间运行后必须压缩历史，而压缩的质量直接决定 Agent 是否还能"记住"之前做了什么。**不能等到完全耗尽才做最重的压缩**。

### 实现手法

五层递进压缩：

| 层 | 策略 | 代价 | 触发条件 |
|----|------|------|---------|
| 1 | `applyToolResultBudget` | 无 | 每次消息 |
| 2 | `snipCompact` | 极低 | 功能开关 |
| 3 | `microCompact` | 低 | 旧 tool_result 替换为占位 |
| 4 | `contextCollapse` | 中 | PTL 错误时 |
| 5 | `autoCompact` | 高 | Token 低于缓冲量 |

### 可迁移的启发

> [!important] 启发：渐进式资源回收
> **不要等到资源耗尽才做最激进的回收——在轻度压力时做轻度回收，在重度压力时做重度回收。** 这与操作系统内存管理的思路一致：先回收缓存（低代价），再压缩内存（中代价），最后 OOM killer（高代价）。在构建长时间运行的 Agent 系统时，设计多层递进的上下文管理策略，而非单一的"到阈值就全量压缩"。

## 设计亮点 6：并行预取的启动优化

### 设计动机

CLI 工具的启动时间是用户体验的第一道门槛。慢启动 = 用户放弃。但 Claude Code 需要初始化 MDM 设置、读取 Keychain、预连接 API——这些都是 I/O 操作，串行执行会累积延迟。

### 实现手法

```typescript
// main.tsx — 作为 side-effects 在其他 imports 之前触发
startMdmRawRead()
startKeychainPrefetch()
```

这些不是完整的初始化，而是**发起 I/O 请求**。当后续代码真正需要数据时，结果已经就绪或接近就绪。同时，重模块（OpenTelemetry、gRPC、analytics）通过动态 `import()` 延迟加载。

### 可迁移的启发

> [!important] 启发：预取 + 惰性加载的组合
> **将启动路径分为三类：必须串行的（核心初始化）→ 可并行预取的（I/O）→ 可延迟加载的（重模块）。** 不要把所有初始化放在同一个同步路径上。在 Agent 系统中，API 预连接、配置读取、权限初始化都可以预取；而遥测、UI 渲染等可以延迟。这个策略组合让 Claude Code 在 ~1s 内完成启动。

## 设计亮点 7：桥接系统的透明代理 + 安全隔离

### 设计动机

AI Agent 不应该是独立工具，而应该嵌入开发者工作流。但 IDE 和 CLI 之间需要双向通信，且不能引入安全漏洞。

### 实现手法

Bridge 在 CLI 和 IDE 之间建立**透明通道**——双方感觉不到对方的存在。核心安全设计：

- **JWT 认证**：每个连接都需有效令牌
- **权限中继**：CLI 的权限请求通过 Bridge 转到 IDE，用户在 IDE 中审批
- **命令白名单**：远程模式下只有 `REMOTE_SAFE_COMMANDS` 可执行
- **requestId 关联**：多并发权限请求不会混淆

### 可迁移的启发

> [!important] 启发：透明代理 + 双向安全
> **桥接层应该让两端都觉得"我在直接工作"，而不是"我在通过代理工作"。** 权限中继是最值得学习的——不是让 IDE 自己做权限判断，而是把 CLI 的权限请求原样转发到 IDE，用户审批后结果回传 CLI。这保证了权限逻辑的**唯一来源**（CLI），同时让 UI 展示的**最佳位置**（IDE）。在自己的系统中，当需要桥接两个环境时，让核心逻辑驻留一侧，另一侧只负责展示和收集用户输入。

## 设计亮点 8：命令系统的 Prompt / Local / JSX 三分法

### 设计动机

用户意图有两种：需要 AI 推理的和确定性操作。不是所有意图都应该委托给 Agent——`/cost` 显示花费是确定性操作，不需要 LLM 参与。

### 实现手法

| 类型 | 执行方式 | 本质 |
|------|---------|------|
| `prompt` | 生成 Prompt → 发给 LLM | 提示词模板 |
| `local` | 直接执行 JS 函数 | 确定性操作 |
| `local-jsx` | 渲染 Ink UI | 交互式操作 |

> 这个分类看似简单，但解决了一个深层问题：**何时用 AI、何时不用**。prompt 命令利用 LLM 的推理能力，local 命令避免不必要的 API 调用——既省钱又更快。

### 可迁移的启发

> [!important] 启发：AI 与确定性操作的分离
> **在 Agent 系统中，明确区分"需要推理的任务"和"确定性任务"。** 后者不应经过 LLM——直接执行更快、更可靠、更便宜。在构建自己的 Agent 系统时，画出这条分界线：凡是能用确定性代码完成的，就不要委托给 LLM。LLM 是推理引擎，不是万能执行器。

## 设计亮点 9：子智能体的"Never Delegate Understanding"原则

### 设计动机

子智能体没有看过整个对话——它只看到任务描述。如果主智能体的提示词是"根据你的发现修复 bug"，子智能体只能产出浅薄、通用的工作，因为它缺乏上下文。

### 实现手法

Agent Tool 的 Prompt 中明确声明：

> 不要写"根据你的发现修复 bug"这类指令。提示词应该证明你已经理解了问题——包含文件路径、行号、具体要改什么。

这个原则确保主智能体始终保持对任务的**高层理解与控制**，子智能体只是执行已明确的任务片段。

### 可迁移的启发

> [!important] 启发：给子代理写提示词像给新来的聪明同事交代任务
> **它没看过这个对话历史。你的提示词必须是自包含的——包含足够的上下文让一个没有背景知识的人也能理解任务。** 在构建多智能体系统时，最常见的失败模式是主代理把"理解"也委托出去了——它自己没有深入理解问题，却期望子代理能"自己搞清楚"。结果是所有子代理都在表面层次工作，产出平庸的成果。

## 设计亮点 10：Fork 子代理的上下文占位替换

### 设计动机

子智能体需要继承父智能体的部分上下文，但完整的 tool_result 太大（可能数万 token），直接复制会膨胀子智能体的上下文。

### 实现手法

Fork 机制用**字节相同的占位文本**替换 tool_result：

- 保留原始 tool_result 的位置和结构（保证 prompt cache 键一致）
- 替换内容为轻量占位符
- 添加 per-child 指令文本块

两条核心戒律：
- **Don't peek**：不应读取 Fork 的 output_file，中间工具噪音会拉入主上下文
- **Don't race**：永远不伪造或预测 Fork 结果

### 可迁移的启发

> [!important] 启发：缓存键稳定性的精妙手法
> **用字节相同的占位文本替换内容而非删除，保持缓存键不变。** 这是 Prompt Cache 工程中最精妙的手法——删除一个消息会改变后续所有消息的序列号，导致缓存全量失效。替换保持序列和结构不变，只有内容变轻。在需要传递大量上下文但又要控制成本的场景中，这个"占位而非删除"的策略非常实用。

## 设计亮点 11：七层弹性状态机（Continue 站点）

### 设计动机

生产级的 Agent 循环不是简单的 while loop——它需要处理上下文溢出、输出超限、模型过载、Hook 拦截、长时间运行等各种异常情况。每种异常需要不同的恢复策略。

### 实现手法

7 个 Continue 站点形成一个**弹性状态机**：

| 站点 | 处理 | 恢复策略 |
|------|------|---------|
| 1 | 上下文溢出 + CONTEXT_COLLAPSE 开关 | 排空积累后重试 |
| 2 | 排空不足 | 压缩历史后重试 |
| 3 | 输出 token 超限 | 升级 max_tokens (8K→64K) |
| 4 | 升级后仍超限 | 注入恢复消息续写 |
| 5 | Stop Hook 拦截 | Hook 输出作为新用户消息 |
| 6 | Token Budget 接近目标 | 自动续费继续 |
| 7 | 正常工具完成 | tool_result 反馈循环 |

### 可迁移的启发

> [!important] 启发：弹性恢复而非硬性失败
> **每个失败都有对应的恢复策略，模型不会因为单次失败就终止会话。** 这与分布式系统的"可恢复性"设计哲学一致——不是避免所有失败（不可能），而是为每种失败准备恢复路径。在构建长时间运行的 Agent 系统时，列出所有可能的失败类型，为每种类型设计至少一个恢复策略，形成一个完整的"失败-恢复矩阵"。

## 设计亮点 12：System Prompt 的静态/动态分界线

### 设计动机

Anthropic 的 Prompt Caching 基于前缀匹配。如果整段 System Prompt 每次都变化（比如因为注入了新的工具结果），缓存永远无法命中。需要将**稳定部分**和**变化部分**物理分离。

### 实现手法

System Prompt 的组装不是拼接一个大字符串，而是**返回一个多段字符串数组**：

```
[7个静态段] → [缓存分界线 __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__] → [10+个动态段/条件段]
```

- 静态段（身份、规则、工具描述等）跨用户/组织共享缓存
- 动态段（会话指南、环境信息等）每次会话单独注入
- 缓存分界线标记让 API 知道前缀到此为止

### 可迁移的启发

> [!important] 启发：数据排列服从缓存语义
> **不是"内容逻辑上该怎么组织"，而是"缓存语义上该怎么排列"。** 这颠覆了传统的文档组织思维——内容应该按逻辑分组，但 Prompt 应该按**缓存稳定性**分组。在构建依赖外部缓存（无论是 Prompt Cache、HTTP Cache 还是 CDN Cache）的系统时，数据排列的第一原则是**缓存友好**，而非逻辑清晰。逻辑清晰可以在每个段内部实现，段与段之间的排列服从缓存语义。

## 设计亮点总结矩阵

| # | 设计亮点 | 核心原则 | 一句话启发 |
|---|---------|---------|-----------|
| 1 | 死代码消除 | 安全始于构建时 | 不该执行的代码不该存在于产物中 |
| 2 | Prompt Cache 排序 | 缓存友好排列 | 稳定在前、易变在后 |
| 3 | 工具自包含接口 | 开放-封闭原则 | 每个组件自描述能力，系统只定义接口 |
| 4 | 权限责任链 | 确定性优先 | 规则处理 90%，LLM 只处理模糊 |
| 5 | 分层压缩 | 渐进式回收 | 轻度压力轻度回收，重度压力重度回收 |
| 6 | 并行预取 | 启动路径分三类 | 必须串行 / 可并行预取 / 可延迟加载 |
| 7 | 桥接透明代理 | 核心逻辑唯一来源 | 逻辑驻留一侧，另一侧只展示和收集 |
| 8 | 命令三分法 | AI/确定性分离 | 确定性任务不经过 LLM |
| 9 | Never Delegate Understanding | 上下文自包含 | 提示词像给新同事交代任务 |
| 10 | Fork 占位替换 | 缓存键稳定性 | 替换而非删除，保持序列不变 |
| 11 | 弹性状态机 | 可恢复性优先 | 每种失败都有恢复策略 |
| 12 | 静态/动态分界线 | 排列服从缓存语义 | 缓存友好优先于逻辑清晰 |
