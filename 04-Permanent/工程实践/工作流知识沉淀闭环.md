# 工作流知识沉淀闭环

> 16 阶段状态机不是为了"好看"或"复杂"——它的==每一个阶段都与知识的流动紧密关联==。工作流是知识沉淀的载体，不是独立于知识的执行管道。

---

## 知识的完整生命周期：三通道沉淀

```
/flow-import（一次性冷启动）        /flow-run（每次需求）
      │                                │
      ▼                                ▼
 冷启动导入                       INIT: git pull 知识仓库
3 Agent 管道                          │  + 注入查询入口
 → 知识写入团队仓库                    │
                                       │  ← Agent 在各阶段按需查询
                                       │     （三级渐进式索引）
                                       ▼
                                 ARCHIVE: 知识提取 + 提升判定
                                       │
                                       ├→ Layer 3: docs/knowledge-base/
                                       ├→ Layer 1: tech-wiki/  ← git push
                                       └→ Layer 2: biz-wiki/  ← git push
                                                   │
                                                   ▼
                                            下一个人的 /flow-run 自动受益
```

### 三个关键时刻

| 时刻 | 阶段 | 知识行为 | 设计要点 |
|------|------|----------|----------|
| **① 知识注入** | INIT | `git pull` 团队知识仓库 + 注入查询入口 | 新启动的工作流==自动站在前人肩上== |
| **② 知识消费** | 各阶段执行 | Agent 在决策点按需查询知识库 | 查询预算控制，精准而非贪婪 |
| **③ 知识提取** | ARCHIVE | `@archiver` 自动从全流程产物提取知识 | 架构决策→decision，踩坑→pitfall，经验→guideline |

> [!important] 闭环的本质
> INIT 注入前人知识 → 各阶段消费知识做决策 → ARCHIVE 提取新知识回写仓库 → 下次 INIT 注入更多知识。这就是[[知识沉淀护城河]]中说的"复利效应"的工程实现。

---

## 各阶段查询什么知识

每个阶段的 Agent 有==独立的查询预算==，聚焦不同类型知识：

| 阶段 | 查询焦点 | 重点知识类型 | 为什么 |
|------|----------|-------------|--------|
| **ANALYSE_PRODUCT** | 业务知识 (Layer 2) + 历史需求 | model, process, pitfall | 理解业务实体和流程，避免重复踩坑 |
| **ANALYSE_TECH** | 技术知识 (Layer 1) + 归档索引 | decision, guideline(avoid), pitfall | 技术选型需参考已有决策和反模式 |
| **ARCHITECT** | 架构模式 + 实体关系 | decision, model | 架构设计依赖已有实体模型和决策 |
| **IMPLEMENT** | 编码实践 + 团队约定 | guideline, pitfall | 编码时需要推荐做法和已知风险提醒 |
| **BUILD_VERIFY** | 反模式库 | pitfall, guideline(avoid) | 验证阶段重点排查已知陷阱 |

> [!tip] 为什么限制查询预算？
> Agent 如果无限制读取知识库，会导致==上下文膨胀==——这恰恰是 Harness Engineering 要解决的核心问题。通过预算控制，让知识消费==精准而非贪婪==。这与 [[上下文预算与资源分配]] 中 Token Budget 作为"行为塑形器"的理念一致。

## 冷启动导入：`/flow-import`

历史项目（已有大量代码但没有知识库）的冷启动方案，通过 3 个 Agent 管道实现：

```
@doc-collector → 多源资料收集
  │              （Git/TAPD/iwiki/本地文档/口述）
  ↓
@codebase-profiler → 代码画像
  │                   （技术栈/模块/依赖/模式，60 次搜索预算）
  ↓
@knowledge-builder → 知识标准化
                      （4 维基线 + ≤13 条知识条目 + 归档总结）
```

### 冷启动策略

| 策略 | 说明 |
|------|------|
| **多源收集** | 支持 Git 仓库扫描、需求拉取、文档导入、口述录入等多种输入 |
| **渐进导入** | 所有导入知识初始 maturity 为 `draft`，通过后续工作流逐步验证 |
| **断点恢复** | 导入过程通过 `import-state.json` 持久化，支持中断后继续 |

> [!note] 与 [[知识成熟度与自动衰减]] 的关联
> 冷启动导入的所有条目初始 maturity 为 draft（置信度 0.5-0.6）。这体现了"不追求一次性导入完美"的理念——让工作流的实际使用来验证知识。
