# 成为 Harness Engineering 时代的工程师

> 工程师的职责正在从"写代码的人"进化为"设计让 AI 把代码写好的系统的人"。这并非取代，而是==价值向上迁移==——从体力劳动到系统杠杆。[[工程三次进化-关系衰变与新范式]]

四步进阶路线图，每一步都是前一步的扩展而非替代：

```
Prompt（基础语言能力） → Context（信息环境设计） → Harness（系统约束架构） → 动态 Harness（持续校准思维）
```

### 术语演进脉络

| 时间 | 里程碑 | 关键人物/机构 |
|------|--------|-------------|
| 2022-2024 | Prompt Engineering 主流 | 全行业 |
| 2025.12 | Andrej Karpathy 提出 Context Engineering（上下文管理理论化） | Andrej Karpathy |
| 2026.02 | Mitchell Hashimoto 首创 Harness Engineering（约束设计系统化） | Mitchell Hashimoto（HashiCorp/Terraform） |
| 2026.02 | OpenAI Ryan Lopopolo 正式定义 Harness Engineering（基于 Codex 生产实践） | OpenAI |
| 2026.04 | Martin Fowler 发布实践指南（提出 Feedforward/Feedback 框架） | Martin Fowler / Birgitta Böckeler |
| 2026.05 | DeepSeek 组建 Harness 研发团队，亲自下场挑战 Claude Code | DeepSeek |
| 2026.06 | 卡内基梅隆/耶鲁/亚马逊联合发表 Harness 工程学术综述 | CMU/Yale/Amazon 等 |

> [!quote] 一句话定位三层关系
> **Prompt 是输入技巧，Context 是信息架构，Harness 是系统工程。** — [[Prompt与Context工程（三次进化）]]
> - Prompt 解决"听不听得懂你在说什么"
> - Context 解决"有没有拿到完成任务所需的信息"
> - Harness 解决"即使会犯错，系统还能不能持续、稳定、可控地运行"

## 第一步：打牢 Prompt 基础，但不要执着于它

[[提示工程]] 的底层逻辑是==如何清晰、精准地表达意图==，是你与 AI 协作的基础语言能力。不必穷尽所有技巧，但要掌握核心：

### 三个必会核心

| 技能 | 说明 | 实践要点 |
|------|------|---------|
| **思维链引导** | 让模型显式推理而非直接给出答案 | 使用 "Let's think step by step" 或结构化的推理框架引导模型逐步拆解问题；复杂任务可结合 [[CoT]] 与 [[ToT]] 范式 |
| **角色设定** | 通过身份与专业领域描述提升输出质量 | 不是简单的"你是一个专家"，而是精确定义角色边界、知识域和行为规范；参见 [[系统提示词工程]] 中 Claude Code 如何用静态段定义 Agent 身份 |
| **结构化输出格式** | 指定 JSON、Markdown 等格式以便后续处理 | 从"自由文本"到"可被程序解析的确定性输出"；这直接衔接 [[工具调用（Function Calling）]] 的 Schema 设计 |

### 为什么不要执着？

> [!important] Prompt 的衰变特征
> 随着模型进化，今天的最优提示明天可能不再必要——这是 [[驾驭工程#Harness 的衰变定律|Harness 衰变定律]] 在 Prompt 层的直接体现：
> - Claude 3.0 时代需要精心设计的思维链提示，到了 3.5 时代模型已自带推理能力
> - 曾经需要 few-shot 示例的任务，现在 zero-shot 即可完成
> - 角色设定的精确度要求也在随模型理解能力提升而下降
>
> **核心原则：Prompt 是起点，不是终点。** 不要花 80% 的时间追求"最佳提示词"，而是尽快跃迁到 Context 和 Harness 层。

> [!tip] 何时该停止优化 Prompt？
> 当你发现：同一类任务换用不同 Prompt 后输出差异<10%，或模型在零提示下已能给出 70%+ 质量的答案——这时 Prompt 优化已进入边际收益递减区，应转向 Context Engineering。

### Prompt 层的 Feedforward 机制

Martin Fowler（2026.04）将 Prompt 的作用重新定位为 **Feedforward（前置约束）**——在 Agent 行动**前**施加影响，提高首次成功概率：

| 前置约束类型 | 说明 | 实例 |
|------------|------|------|
| **计算型前置约束** | 通过确定性工具在执行前检查 | LSP 语法检查、类型推断 |
| **推理型前置约束** | 通过上下文注入引导模型推理方向 | AGENTS.md 规范注入、角色设定、思维链引导 |
| **结构型前置约束** | 通过代码模板/脚手架约束输出结构 | 初始文件结构生成、接口 Schema 定义 |

> [!note] Feedforward vs Feedback
> Feedforward 在行动**前**引导方向，Feedback 在行动**后**纠正偏差。Prompt 属于 Feedforward，而 Lint/测试属于 Feedback——两者互补构成完整的约束闭环。详见[[#第三步：从系统视角思考 Agent 设计]]中的 Feedforward + Feedback 双重机制。

## 第二步：系统学习 Context Engineering

这是当下==最具差异化的技能之一==。Prompt 决定"说什么"，Context Engineering 决定"给模型什么信息、以什么格式、在什么时机"——从语言表达跃迁到==信息环境设计==。

### 四个核心模块

#### 模块 1：RAG 系统设计与调优

> [!quote] 核心论题
> RAG 不是"搜索+拼接"，而是==端到端的信息供给管道==——决定 Agent 在推理时能否获取正确、及时、可信的外部知识。

| RAG 关键环节           | 实践要点                                                                    | 常见误区                               |
| ------------------ | ----------------------------------------------------------------------- | ---------------------------------- |
| **分块策略**           | 语义分块优于固定长度分块；parent-child 层级分块保留上下文关系；重叠窗口防止语义断裂                        | 按固定 512 Token 切分，导致跨块语义丢失          |
| **Embedding 模型选择** | 通用模型（text-embedding-3-large）适合广域任务；领域微调模型在专业语料上显著优于通用模型                 | 所有场景都用同一个 Embedding 模型             |
| **检索优化**           | 混合检索（dense + BM25）覆盖语义与关键词；检索后用 cross-encoder 重排序提升精度；HyDE / 多查询扩展提高召回率 | 只用向量相似度（纯 dense retrieval），遗漏关键词匹配 |
| **检索精度评估**         | 用 Precision@k、MRR、Recall 等指标量化检索质量；建立检索质量 → 生成质量的因果链                    | 只看最终生成结果，不追溯检索环节的问题                |

> [!example] RAG 2.0 的方向
> Contextual AI（Douwe Kiela 团队）提出从"外挂式检索"走向端到端优化——Embedding 模型、检索器、生成器联合微调。关键转变：检索不再只看主题相似度，而是考虑==下游任务相关性==。

#### 模块 2：上下文窗口管理

> 上下文窗口是新的缓存——你需要精心决定什么放进去、什么被淘汰、什么持久保留。 — Simon Willison

| 管理维度     | 核心问题                  | 实践方案                                                    |
| -------- | --------------------- | ------------------------------------------------------- |
| **信息选择** | 在有限 Token 中做最优的信息取舍   | 优先注入：当前任务指令 > 最近交互 > 相关知识 > 对话历史；参见 [[上下文窗口管理]]         |
| **淘汰策略** | 窗口满了该丢什么？             | 从缓存管理理论借鉴：LRU（最近最少使用）、优先级驱逐、摘要压缩替代截断                    |
| **预取策略** | 主动注入 Agent 下一步可能需要的信息 | [[上下文注入管道]] 的 CLAUDE.md 多层加载机制：系统段（缓存）→ 项目段 → 用户段       |
| **预算分配** | 为不同信息类型设定 Token 预算    | [[上下文压缩]] 的渐进降级策略：预警(60%) → 均缩(70%) → 截断(80%) → 紧急(90%) |

> [!warning] 反模式：全量注入
> 将所有编码规范、架构文档、历史对话一次性塞入上下文 → 信息过载 → Agent 抓不住重点 → 输出质量反而下降。OpenAI 百万行代码实验的教训：将 `agent.md` 压缩至百行索引，按需动态加载子文档。参见 [[驾驭工程#OpenAI 百万行代码实验]]

#### 模块 3：记忆系统设计

| 记忆类型     | 定义           | 管理策略                                              |
| -------- | ------------ | ------------------------------------------------- |
| **短期记忆** | 当前对话轮次内的上下文  | 对话历史 + 工具结果；受窗口容量限制，需 [[上下文压缩]]                   |
| **长期记忆** | 跨会话的持久知识与偏好  | [[记忆系统]] 的分层管理：语义记忆（事实知识）→ 情景记忆（具体经历）→ 程序记忆（操作模式） |
| **工作记忆** | 当前任务的中间状态与进度 | Todo 状态注入（压缩后注入 pending/in_progress 任务防止遗忘）       |

> [!tip] 记忆系统的黄金法则
> ==LLM 是无状态的计算单元（CPU）==，所有跨轮次状态必须由 Harness 外部管理——[[驾驭工程#状态分离原则]]。试图让 LLM 自行维护复杂状态，会导致行为混乱且难以调试。

#### 模块 4：知识库治理

> [!quote] 知识库不是静态文档仓库，而是需要持续治理的活系统

| 治理维度       | 核心实践                           | 对应工程概念                 |
| ---------- | ------------------------------ | ---------------------- |
| **单一事实来源** | 确保所有决策记录、规范文档集中维护在版本受控的仓库中     | [[工程设计与安全\|代码-文档同步管道]] |
| **数据溯源**   | 每条知识可追溯至源头文档、版本和索引时间           | 知识块附带 provenance 元数据   |
| **新鲜度管理**  | 设定 TTL 策略；源文档更新时自动触发重新索引       | 防止过时信息污染 Agent 决策      |
| **质量审计**   | 自动检测矛盾、重复、低质量知识块               | 向量数据库的去重、漂移检测、定期重嵌入    |
| **访问控制**   | 基于角色的检索权限，确保 Agent 只获取授权范围内的知识 | RBAC/ABAC 权限模型应用于检索层   |

> [!example] OpenAI 的教训
> OpenAI 在百万行代码实验中强制将散落各处的决策记录迁移至代码仓库，确保 Agent 的唯一信息来源是可信的、版本受控的仓库——这就是"单一事实来源"原则的直接实践。

### AGENTS.md：知识库治理的落地方案

OpenAI/Martin Fowler 推荐在项目根目录放置 `AGENTS.md`，作为 Agent 的前置知识入口，采用三层约束模式：

| 约束层级 | 含义 | 典型条目 |
|---------|------|---------|
| **Always（始终执行）** | Agent 必须始终遵守的规则 | 使用 UTC 时区、记录所有操作、遵循命名约定 |
| **Ask First（询问后再执行）** | 需人类确认才能执行的操作 | 添加新通知渠道、修改重试间隔、引入新依赖 |
| **Never（禁止执行）** | 绝对不允许的操作 | 未验证 opt-in 发送通知、修改退订流程、删除生产数据 |

> [!tip] AGENTS.md 的设计哲学
> AGENTS.md 不是"给 Agent 写的说明书"，而是==结构化的前置约束==——它将散落在 Prompt 中的规则沉淀为版本受控的工程资产。参见 [[系统提示词工程]] 中 Claude Code 的多层加载机制：系统段（缓存）→ 项目段（AGENTS.md）→ 用户段。

## 第三步：从系统视角思考 Agent 设计

开始用 Harness 的视角审视 AI 项目——不再只关注"模型输出好不好"，而是关注==整个系统是否可靠运转==。

### ETCLOVG：Harness 的七层分类体系

卡内基梅隆大学、耶鲁大学、亚马逊等机构 2026 年 6 月联合发表的学术综述，将 Harness 分为七层——前四层是**结构核心**，后三层是**控制平面**：

#### 结构核心（前四层）

| 层级 | 名称 | 职责 |
|------|------|------|
| **E** | 执行环境与沙箱 | 决定 Agent 代码在哪里运行、受到什么约束 |
| **T** | 工具接口与协议 | 定义外部能力如何被描述、发现和调用 |
| **C** | 上下文管理 | 决定模型在短期、会话级和持久化层面能看到什么 |
| **L** | 生命周期与编排 | 负责组织状态读写控制流，覆盖单 Agent 循环到 issue → PR 工作流 |

#### 控制平面（后三层）

| 层级 | 名称 | 职责 |
|------|------|------|
| **O** | 可观测性 | 捕获轨迹、成本、失败和可靠性信号 |
| **V** | 验证 | 把任务和轨迹转化为评估、失败归因和回归反馈 |
| **G** | 治理 | 通过权限、身份、策略、安全加固、审计和人工监督约束系统行为 |

> [!important] 学术界的关键发现
> ==在不改模型权重的情况下，仅调整 harness 层本身，也可能显著改变 Agent 在 coding 和 terminal benchmark 上的表现。== 这意味着 Harness 不是"必要的恶"，而是一个==独立的优化维度==——即使模型不变，更好的 Harness 也能带来更好的结果。

> [!note] ETCLOVG 与已有框架的区别
> 与业界实践框架相比，ETCLOVG 将"可观测性"和"治理"作为==独立的架构层==看待，而非附属功能。这反映了学术界对生产级 Agent 系统的严谨认知：可观测性和治理不是"锦上添花"，而是架构基石。

### Feedforward + Feedback 双重机制

Martin Fowler（2026.04）提出 Harness 依赖两种互补控制机制——这为约束设计提供了系统化的思维框架：

| 机制 | 作用时机 | 计算型（确定性、快速、低成本） | 推理型（语义理解、高成本） |
|------|---------|--------------------------|------------------------|
| **Feedforward（前置约束）** | Agent 行动**前** | LSP 语法检查、类型推断 | AGENTS.md 规范注入、角色设定 |
| **Feedback（反馈循环）** | Agent 行动**后** | Linter、单测（快速确定性） | AI Code Review（语义理解成本高） |

> [!tip] "Lint 消息即 Prompt"
> Feedback 信号必须具备==可操作性==。优秀的 Lint 消息应包含修复指令，使 Agent 能自主修复，而非依赖人工解释：
> - ❌ `"violation detected"`
> - ✅ `"use logger.info({event: 'name', ...data}) instead of console.log"`
>
> 这一条原则将 Feedback 从"报错"升级为"指导"——每条 Lint 消息本质上都是一条微型的、高度精确的 Prompt。

### PEV（Plan-Execute-Verify）架构模式

Augment Code 提出摒弃传统"生成-检查"模式，采用 PEV 循环：

```
传统：Generate → Check（后置验证，发现问题时已浪费大量 Token）
PEV：  Plan → Execute → Verify（前置规划 + 受约束执行 + 多维验证）
```

| PEV 阶段 | 核心动作 | Harness 的角色 |
|----------|---------|---------------|
| **Plan（规划）** | 显式分解任务与验收标准，减少自由度 | 限制 Agent 的行动空间，提高首次成功率 |
| **Execute（执行）** | 受计划约束执行，Harness 在每次工具调用时触发 | Feedforward + Feedback 双重保障 |
| **Verify（验证）** | 执行前 + 运行中 + 事后 + 计划对齐，提供带上下文的错误消息 | 验证失败 → 结构化反馈 → Agent 原地修复 → 循环 |

> [!important] PEV 的核心洞察
> PEV 的价值不在于"多了一步验证"，而在于==将验证前置到规划和执行阶段==——不是等 Agent 做完了再检查，而是在它做的每一步都检查。这大幅降低了返工成本和 Token 浪费。

### Martin Fowler 的三类 Harness

| Harness 类型 | 说明 | 实施难度 | 典型实践 |
|-------------|------|---------|---------|
| **Maintainability Harness（可维护性约束）** | 最易实现 | ⭐ | Lint 规则、架构检查、复杂度限制 |
| **Architecture Fitness Harness（架构适配约束）** | 中等难度 | ⭐⭐ | 模块边界检查、性能基准、可观测性标准 |
| **Behaviour Harness（行为约束）** | 最具挑战 | ⭐⭐⭐ | 功能规范、AI 生成测试验证、回归测试 |

> [!tip] 渐进式实施路径
> 从易到难推进：先建立 Maintainability Harness（最快见效），再补齐 Architecture Fitness Harness，最后攻克 Behaviour Harness。每类 Harness 建立后，应立即观察其对 Agent 行为的改善效果，再决定是否投入下一类。

### 四个核心问题

#### 问题 1：约束设计——防止 Agent 跑偏

> [!quote] 核心思维
> 不是"Agent 会跑偏吗？"而是=="Agent 在哪些具体地方会跑偏？我如何为每个风险点设计针对性约束？"==

| 典型跑偏场景 | 约束设计 | 对应 Harness 组件 |
|-------------|---------|-----------------|
| "顺手"修改不该动的东西 | 最小权限 + 白名单路径限制 | [[护栏与钩子系统]] 的 PreToolUse 钩子 |
| 声称"测试通过"而未实际运行 | 强制验证闭环，工具结果必须可验证 | [[重试与恢复机制]] + 可观测性 |
| 上下文耗尽后"假装完成" | 进展评估 + 里程碑机制 | [[驾驭工程#子系统 3：迭代控制]] |
| 重复执行相同操作 | 模式检测（同工具+同参数连续N次触发干预） | 迭代控制的 Layer 2 |
| 工具参数格式错误 | 参数校验 + 修复策略（四策略递进） | [[驾驭工程#子系统 4：工具治理]] |

#### 问题 2：验证机制——从"声称完成"到"验证完成"

> [!important] 不为 Agent 的善意假设买单，只信任可验证的结果

```
传统思路：Agent 说"完成了" → 人类检查 → 发现问题 → 手动修复
Harness 思路：Agent 说"完成了" → 自动验证（Lint + 测试 + 截图对比）→ 验证失败 → 结果自动反馈 → Agent 原地修复 → 循环直到验证通过
```

**关键转变**：从"人类做验证"到"系统做验证"。Anthropic F-Harness 的 Evaluator 角色就是这一思想的体现——独立第三方审核，消除 AI 的自评偏见。参见 [[驾驭工程#Anthropic F-Harness]]

#### 问题 3：单 Agent vs 多 Agent 协作

| 任务特征 | 推荐模式 | 理由 |
|---------|---------|------|
| 简单、线性、低风险 | 单 Agent | 协作成本（通信、状态同步）大于收益 |
| 高复杂度、需多视角 | 多 Agent（Planner → Generator → Evaluator） | [[驾驭工程#Anthropic F-Harness]] 的代价对比：20x 时间、22x 成本，但质级飞跃 |
| 流水线式、步骤可独立 | 多 Agent 串行协作 | 每步专注、上下文隔离、失败可定位 |
| 并行探索、需多方案比较 | 多 Agent 并行 | 多视角同时探索，避免单点偏见 |

> [!warning] 多 Agent 不是万能解
> Anthropic 的实验表明：单 Agent 模式耗时约 20 分钟、成本约 $9，而三 Agent F-Harness 模式耗时约 6 小时、成本约 $200。只有在任务的复杂度==超过单 Agent 的可靠性边界==时，多 Agent 协作才是唯一可行的工程解法。

#### 问题 4：可观测性——及时发现异常

> [!quote] 可观测性三支柱

| 支柱 | 关键记录项 | 目的 |
|------|-----------|------|
| **Logging** | 每轮决策、工具调用入参出参、错误上下文、安全事件 | 事后复盘与问题定位 |
| **Tracing** | 端到端请求链路、父子 Agent 关系、工具耗时、压缩/降级事件 | 理解数据流与瓶颈 |
| **Metrics** | Token/轮/成本消耗、成功率/重试率、延迟分布、循环检测触发率 | 系统健康度量化监控 |

追踪粒度：`Session → Turn → LLM Call → Tool Call → Sub-Agent`

代表工具：LangSmith / Langfuse / OpenTelemetry / Helicone

## 第四步：培养"动态 Harness 思维"

这是==最难培养、也是最有价值的能力==。它要求你不仅设计 Harness，还要持续校准 Harness 的"厚度"——随模型能力变化而动态调整。

### 两个核心问题

#### 问题 1：约束的本质——模型能力不足 vs 业务逻辑需要

> [!important] 约束分类框架
> 每一条约束/规则，都应该被归类到以下两类之一：

| 约束类型 | 定义 | 生命周期 | 典型示例 |
|---------|------|---------|---------|
| **能力补偿型约束** | 因为模型目前做不到而必须人工补的 | ==会被模型进化淘汰== → 衰变 | 严格的思维链提示（模型已自带推理）、硬编码的参数校验（模型逐渐学会正确格式）、强制分步执行（模型全局统筹能力提升后不再必要） |
| **业务逻辑型约束** | 即使模型能力完美也必须存在的 | ==长期有效== → 不衰变 | 权限边界（任何系统都需要）、合规要求（法规硬性规定）、数据隐私规则（业务底线）、审计追踪（组织治理需要） |

> [!tip] 动态分类的实践
> 建一个约束清单表，定期审视每条约束：标注其类型（能力补偿 or 业务逻辑）、预估衰变时间、设定简化触发条件。当新一代模型发布时，逐条验证"这条约束是否仍然必要"。

#### 问题 2：模型进化后的简化策略

> [!quote] 衰变定律的反向思考
> "如果下一版模型变强了 20%，我的 Harness 里哪些部分可以被简化？"

| 简化策略 | 说明 | 风险控制 |
|---------|------|---------|
| **约束降级** | 从硬约束 → 软约束 → 提示建议 → 移除 | 每次降级后运行回归测试，确保系统可靠性不降 |
| **验证自动化** | 从人工验证 → 半自动验证 → 全自动验证 | 保留最后一道人工审核作为安全网 |
| **编排简化** | 从刚性 Workflow → 弱引导编排 → 自主 Agent | 渐进式，每步验证输出质量是否维持 |
| **上下文精简** | 从详细注入 → 紧要注入 → 按需检索 | 检索质量指标（MRR、Recall）作为精简依据 |

> [!warning] 简化的边界
> 模型能力提升不意味着==所有约束都可以简化==。两类约束的衰减速度截然不同：
> - **能力补偿型约束**：随模型进化快速衰变（可能每代模型淘汰 20-30%）
> - **业务逻辑型约束**：几乎不衰变（除非业务本身发生变化）
>
> 过度简化业务逻辑型约束 = 在生产环境裸奔。过度保留能力补偿型约束 = 浪费工程资源。

### 动态 Harness 思维的成熟度模型

| 阶段 | 特征 | 典型行为 |
|------|------|---------|
| **Level 1：静态思维** | Harness 设计后就不再变更 | "规则定了就定了"，不管模型升级 |
| **Level 2：被动调整** | 模型升级后发现旧规则多余才删 | "这个提示好像不需要了" |
| **Level 3：主动预判** | 新模型发布前就规划哪些约束可简化 | "Claude 4.0 发布后，我准备把这3条硬约束降级为提示" |
| **Level 4：系统化治理** | 约束清单 + 分类标注 + 衰变追踪 + 简化触发条件 → 形成闭环 | 持续校准 Harness 厚度，保持"刚好够用" |

### OpenAI 实战案例：从空仓库到百万行代码

OpenAI 2026 年 2 月发布的生产实践，是 Harness Engineering 最具说服力的标杆案例：

| 维度 | 数据 |
|------|------|
| **团队规模** | 从 3 人增长到 7 人 |
| **开发模式** | 完全禁止人工手写代码，全部由 Codex Agent 生成 |
| **开发周期** | 约 5 个月 |
| **代码产出** | 生成近 100 万行代码 |
| **PR 数量** | 约 1500 个 PR |
| **人均效率** | 日均处理 3.5 个 PR |
| **整体效率提升** | 约 10 倍 |

> [!important] 案例的核心启示
> 这个案例证明：==当 Harness 足够健壮时，"完全由 AI 写代码"不是幻想，而是可行的工程实践==。但前提是——团队的主要工作从"写代码"转向了"设计 Harness"：
> - 为 Agent 配置测试、Lint、CI
> - 设计多轮执行和反馈闭环
> - 将规则程序化（不只写在文档里，还能被 Lint/CI 强制执行）
> - 设计回滚、审批、人工兜底机制

## 杠杆力升级的本质

四步进阶的底层逻辑是==杠杆力的层层升级==：

```
Prompt → 个人杠杆（我 + AI = 更快的我）         1x → 3-5x
Context → 团队杠杆（团队 + AI + 上下文 = 更高效的团队） 5x → 10x+
Harness → 系统杠杆（系统 + AI + 人 + 流程 = 自进化的工程体系） 10x → 100x+
动态 Harness → 持续杠杆（随模型进化自动校准，永不过时）   → 长期可持续
```

衡量标准也从"个人产出"切换到"系统杠杆率"——[[工程三次进化-关系衰变与新范式#衡量标准的切换]]：

| 旧标准 | 新标准 |
|-------|-------|
| 每天能写多少行代码 | Harness 能支撑多高的代码产出率 |
| 能实现多复杂的业务逻辑 | 能设计多健壮的 Agent 系统 |
| 能处理多难的 Bug | 能构建多完善的自动闭环机制 |
| 对某语言/框架的熟悉程度 | 对 AI 系统边界和失效模式的理解深度 |

## 核心哲学

> [!quote] Human Steer, Agents Execute
> **人类掌舵，Agent 执行。**
>
> 工程师的价值不在手工劳作，而在==设计让 AI 把代码写好的系统==。— OpenAI 实验总结 & [[工程三次进化-关系衰变与新范式]]

从"写代码的人"，进化为"设计让 AI 把代码写好的系统的人"。这，才是这个时代工程师最该掌握的技能。

### 工程师价值迁移对照表

| 过去（工程师主要价值） | 现在/未来（工程师主要价值） |
|---|---|
| 写代码 | 定义目标 |
| 改 Bug | 设计边界 |
| 调接口 | 组织知识 |
| 补文档 | 制定规范 |
| 查日志 | 安排工具、设置检查点、处理例外情况，并最终对结果负责 |

### 三步启动法：从零开始建设 Harness

基于 Martin Fowler 和 OpenAI 的实践，推荐从最小可行 Harness（MVH）起步：

```
第 1 步：审计 → 检查最近 5 个 AI 生成的 PR，找出重复出现的失败模式
第 2 步：选择 → 挑选 3 个能阻止这些失败的约束
第 3 步：实施 → 将约束编码为 Linter 规则，CI 失败时阻止合并
```

> [!tip] 渐进式生长路径
> 不是一次性搭建庞大平台，而是**从最小闭环起步**：
> 1. **先生成**初始脚手架与 `AGENTS.md`
> 2. **再补齐**测试、CI 与规则约束
> 3. **持续清理**冗余代码和过期上下文
> 4. 在此过程中逐步把 Harness **"生长"**出来

## 常见陷阱与规避

| 常见陷阱 | 解决方案 |
|---------|---------|
| 过度约束导致低吞吐量 | 分层约束：关键路径严格，非关键路径宽松 |
| 缺乏人类监督 | 实现分级审批机制 |
| 上下文爆炸 | 使用 RAG + 记忆压缩 |
| 错误累积 | 设置检查点 + 回滚机制 |
| Lint 消息不明确 | 将修复指令嵌入错误消息 |
| 规则文件被 Agent 忽略 | 结合确定性约束（Linter）而非仅依赖指令注入 |
| 信息过载 | 给模型太多无关信息反而让它分心，压缩至核心 |
| 上下文污染 | 历史上下文不断膨胀，污染决策 |
| 缺少闭环 | Agent 完成任务后无测试、审查和回归机制 |
| 系统漂移 | 任务变长后系统开始偏离目标 |

> [!warning] 最重要的认知升级
> 当 Agent 表现不好时，你的第一反应可能是"改 Prompt"。但更多时候，问题出在==错误的层级==：
> - 如果模型不知道该做什么 → Feedforward（AGENTS.md、规范注入）
> - 如果模型做错了但没有被纠正 → Feedback（Lint、测试、验证闭环）
> - 如果模型反复犯同类错误 → Harness（将规则编码为不可绕过的硬约束）
>
> **你当然可以继续改 Prompt，但很多时候那只是"在错误层级上努力"。**
