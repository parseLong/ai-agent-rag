# 驾驭工程

> 驾驭工程（Harness Engineering）是为 AI 智能体设计和实施**控制层/驾驭层**的工程实践——将原始 LLM 转化为可控、安全、高效的智能体。它是 AI 工程化的第三次进化，也是将大语言模型从"对话工具"转化为"可控生产系统"的关键一跃。

> [!quote] 核心论题
> ==通过工程化手段构建确定性，以承载模型不确定性==——这是 Agent 技术从"魔法调优"走向"系统工程"的核心思想，也将是未来很长一段时间内构建高质量 Agent 的基石。虽然各个模块的具体实现方式仍在快速演进，但这条核心思想始终未变：即在保证安全、可控的前提下，最大化释放模型的推理与执行潜力。
>
> 来源：[[Agent技术范式演变（2023-2026）]]

---

## 驾驭层的隐喻与定位

### 给 AI 套上"马具"

Harness，字面意思是"马具"。套在马身上的那套装备——缰绳、鞍具、辔头。没有马具的马骑起来信马由缰，套上马具的马才能指哪打哪。

驾驭工程就是**研究如何为大模型设计一套合适的"马具"**。

另一种理解：原始 LLM 就像一辆没有方向盘的汽车——动力强大但无法控制方向。驾驭工程就是在这辆汽车上安装：方向盘（系统提示词）、刹车（权限系统）、导航仪（上下文注入）、仪表盘（流式渲染控制）、安全带（护栏与钩子）、自动恢复系统（重试与压缩）。

### 核心公式

**AI Agent System = LLM + Harness**

一个完整的 AI Agent 系统，除了大模型本身之外的所有东西，都属于 Harness：

- **工具接口**：模型调用外部工具的能力（API、数据库、文件系统等）
- **验证机制**：确保输出正确性、符合规范的自动化校验流程
- **上下文管理**：动态组装、压缩、注入上下文信息的策略
- **权限控制**：限制模型可执行操作的范围和边界
- **多 Agent 协作机制**：角色分工、任务分配、结果审核的编排层

核心原则：**LLM 是引擎，驾驭层是方向盘和安全带**。

### Philipp Schmid 的操作系统类比

Philipp Schmid 给了一个更精确的类比：

| 计算机组件 | Agent 对应物 | 角色 |
| --- | --- | --- |
| **CPU** | 模型 | 原始算力 |
| **内存** | 上下文窗口 | 有限的、易失的工作记忆 |
| **操作系统** | Agent Harness | 管理上下文、处理启动流程、提供标准驱动 |
| **应用程序** | Agent | 跑在操作系统上的用户逻辑 |

模型就是 CPU，算力再强，没有操作系统也跑不起来。**而 Harness，就是给 AI Agent 套上的那层操作系统。**

> [!important] 核心洞察
> 过去两年一直在升级 CPU（更大模型、更长上下文、更强推理能力），但操作系统呢？**还停留在 DOS 时代。**大多数人用 AI Agent 的方式还是「打开终端，输入命令，看结果」——没有文件系统、没有进程管理、没有标准驱动、没有权限控制。

### 为什么需要 Harness？

[[提示工程]]解决了"说清楚"的问题，[[上下文窗口管理 | 上下文工程]]解决了"给够信息"的问题，但即使做到了这两点，Agent 依然会在复杂任务中失控：

- 它可能"顺手"修改不该动的东西
- 它可能声称"测试通过"而根本没有运行测试
- 它可能生成风格不一致、功能重复的代码
- 它在长任务中途耗尽上下文，宣称完成实际未完成

> [!note] Workflow Agent（阶段二）是"早期的 Harness"
> 来源：[[Agent核心技术概念与范式发生了哪些演变以及背后的思考]]
>
> 2024年时期的工作流 Agent（Agentic Workflow），虽然当时没有 Harness 这个概念，但所做的事情本质上就是==一套比较重的 Harness==——用工程化的硬约束来弥补模型的不确定性。Workflow 的"刚性编排"特征——无论外部环境如何变化，仍==死板严格遵循第一步、第二步==——正是早期 Harness "重约束、轻自主"的表现。
>
> 这与 Harness 的衰变定律形成呼应：随着模型能力提升（从阶段二到阶段三），Harness 从"重"变"轻"——自主 Agent 阶段的 Harness 是==轻量级==的，配合自我校验机制而非刚性 Workflow。详见 [[Agent技术范式演变（2023-2026）]]。

这些问题的根源不在"说什么"或"给什么信息"，而在于：**系统层面缺乏约束、验证和反馈机制**。这恰恰是 Harness Engineering 填补的盲区。

### Mitchell Hashimoto 的采纳五阶段

Harness Engineering 这个词的起源，来自 Mitchell Hashimoto（HashiCorp 联合创始人、Terraform 缔造者）2026 年 2 月 5 日写的一篇博客，记录自己用 AI 编程的进化史：

| 阶段 | 名称 | 核心做法 |
|------|------|---------|
| 第一阶段 | Drop the Chatbot | 别用聊天界面了 |
| 第二阶段 | Reproduce Your Own Work | 强迫用 Agent 重新做一遍手动完成的工作 |
| 第三阶段 | Warm Start | 利用下班时间跑 Agent，第二天早上收获成果 |
| 第四阶段 | Background Agents | 简单任务交给后台 Agent，自己专注深度工作 |
| **第五阶段** | **Engineer the Harness** | **每发现 Agent 犯一个错，就工程化一个解决方案让它再不犯** |
| 第六阶段 | 全天候 Agent | Agent 全天候自主运行 |

> [!quote] 第五阶段的定义
> ==每当你发现 Agent 犯了一个错误，你就花时间去工程化一个解决方案，让它再也不会犯同样的错。==

> [!note] Ghostty 项目实践
> Mitchell 在 Ghostty 项目中实践了这个理念。AGENTS.md 文件里的每一行规则，背后都对应着 Agent 曾经犯过的一个错。他说：==那个文件里的每一行都基于一次 Agent 的不良行为，而且几乎完全解决了这些问题。==

> [!important] 击中的痛点
> 这句话击中了很多人隐隐感觉到但没说清楚的痛点：**模型能力够了，可它就是不听话。** Harness Engineering 的核心操作就是——犯错后反哺，而不是反复提示。

## 三层技术定义（arxiv 论文 2603.05344）

arxiv 论文 *Building Effective AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned* 给出了更精确的三层架构定义：

| 层级 | 名称 | 核心职责 | 传统对应 |
|------|------|---------|---------|
| **Scaffolding**（脚手架） | 预执行阶段组装 | 系统 prompt 编译、工具 schema 构建、sub-agent 注册表填充 | BIOS 和引导程序 |
| **Harness**（运行时编排） | 核心推理循环包装层 | 协调工具执行、上下文管理、安全执行、会话持久化 | 操作系统内核 |
| **Context Engineering**（上下文工程） | Token 预算管理 | 决定什么信息该进来、什么该压缩、什么该丢弃 | 操作系统内存管理 |

核心公式：**coding agent = AI model(s) + harness**

> [!important] 三层与现有概念的映射
> 三层定义中 **Scaffolding** ≈ 本文中 [[Harness Engineering#驾驭层架构全景|七大组件]]的初始化阶段；**Harness** ≈ 运行时的六大子系统；**Context Engineering** ≈ 上下文维度（①）的深化。三层不是替代关系，而是==层层包裹、嵌套运行==——Scaffolding 启动 Harness，Harness 内部调用 Context Engineering。

## Harness 的四个核心维度（与 PPAF 紧密耦合）

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]（原文图片不可读，基于上下文重构）

Harness Engineering 将 Agent 的工程化体系解构为四个核心维度，每个维度都与 PPAF 闭环的一个或多个环节紧密耦合：

| 维度 | 覆盖 PPAF 环节 | 核心关注 | 对应实现 |
|------|---------------|---------|---------|
| **① 上下文维度** | 感知 + 反思 | 从无限状态到有限 Token 的转化 | [[上下文注入]]、[[上下文压缩]]、[[上下文窗口管理]] |
| **② 行动维度** | 行动（+ 规划→行动的转化） | 从文本预测到物理执行的闭环 | [[工具调用（Function Calling）]]、[[Plan-and-Execute]] |
| **③ 约束维度** | 全闭环（贯穿 PPAF） | 安全、权限、预算等硬约束 | [[工程设计与安全]]、策略门控、沙盒、熔断 |
| **④ 度量维度** | 反思（闭环的自我审视） | 可观测性→度量→驱动 Harness 演进 | [[度量驱动演进闭环]]、[[智能体评估]] |

> [!important] 四维度与 R.E.S.T. 目标的映射
> 四维度是==实现视角==（"怎么做"），R.E.S.T. 是==目标视角==（"做到什么"）：
> - **Reliability** ← ③约束维度 + ④度量维度
> - **Efficiency** ← ①上下文维度 + ③约束维度
> - **Security** ← ③约束维度
> - **Traceability** ← ④度量维度

## 关键工程位点（Harness 架构的关键组件）

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]（原文图片不可读，基于上下文重构）

为实现 REPL 闭环并落地设计原则，Harness 需要在架构中部署一系列关键组件（"工程位点"）：

| 工程位点 | 在 REPL 中的位置 | 功能 | PPAF 环节 |
|---------|----------------|------|-----------|
| **上下文管理器** | Read 端 | 规约规则 + 注入边界 + Token 转化流水线 | 感知 |
| **调用拦截器** | Eval→Print 之间 | FC 捕获→路由→监控执行（超时、配额、错误） | 规划→行动 |
| **反馈汇编器** | Print 端 | 观测注入→结构化反馈→重新注入上下文 | 反思 |
| **策略门控** | 规划器→执行层之间 | 权限 + 数据过滤 + 注入防御 + 审计 | 规划→行动（硬约束） |
| **迭代控制器** | Loop 外围 | 预算限制 + 模式检测 + 进展评估 | 全闭环 |
| **状态管理器** | Loop 外部 | 持久化状态 + 记忆分层 + 会话恢复 | 全闭环（跨轮次） |
| **沙盒执行器** | 数据平面 | 工具隔离执行 + 资源配额 | 行动 |

> [!note] 工程位点与 Claude Code 实现的映射
> Claude Code 的架构几乎一对一映射了这些位点——[[Claude Code 源码架构]] 对此有完整剖析。

## 驾驭层的核心矛盾

| 矛盾 | 驾驭层策略 | Claude Code 实现 |
|------|----------|-----------------|
| **自主 vs 安全** | 最小权限 + 信任递增 | default → auto → bypassPermissions |
| **灵活 vs 可控** | 工具描述定义行为边界 | Zod schema + 权限标记 + 描述文本 |
| **效率 vs 透明** | 流式渲染 + 选择性折叠 | 工具自渲染 + 搜索命令折叠 |
| **记忆 vs 预算** | 智能压缩而非截断 | snip compaction + reactive compact |
| **单一 vs 多元** | 上下文优先级合并 | CLAUDE.md 多层加载 + 驳回覆盖 |

## 构建可靠 Agent 的 R.E.S.T. 模型

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

要让 Agent 从"有趣的玩具"变为"可靠的工具"，必须满足四个核心目标——**R.E.S.T.** 模型：

| 目标 | 定义 | 关键要求 |
|------|------|---------|
| **R — 可靠性（Reliability）** | 系统在各种预期和非预期输入、环境变化和内部故障时，能持续稳定地完成既定任务 | 失败可恢复（检查点恢复）、操作幂等性（写操作可安全重试）、行为一致性（相同输入→可预测输出） |
| **E — 效率（Efficiency）** | 在满足功能和可靠性的前提下，系统使用计算、存储、网络等资源的有效性 | 资源可控（Token/API调用/计算时间有精确预算）、低延迟响应（交互式场景快速反馈）、高吞吐量（批处理单位时间处理更多任务） |
| **S — 安全性（Security）** | 保护系统及数据免受未经授权的访问、使用、泄露或破坏 | 最小权限（仅授予当前子任务必需权限）、沙盒执行（不授信代码严格隔离）、输入/输出过滤（防指令注入、敏感信息泄露） |
| **T — 可观测性（Traceability）** | 系统提供足够数据使开发和运维人员能理解其内部状态、决策过程和行为轨迹 | 全链路追踪（请求→结果每环节清晰可追溯）、决策可解释（每个关键决策有明确归因记录）、状态可审计（任意历史时间点的完整状态可查询） |

> [!important] R.E.S.T. 与驾驭层架构的关系
> R.E.S.T. 模型是驾驭层架构的**目标层**——六大子系统（错误分类与恢复→可靠性、上下文工程→效率、迭代控制→可靠性+效率、工具治理→安全性、安全防护→安全性、可观测性→可观测性）共同服务于这四个目标。

## Agent 运行的 PPAF 闭环

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

一个完备的 Agent 系统，其核心运行机制可抽象为一个持续循环的四阶段过程——**PPAF 闭环**：

| 阶段 | 英文 | 说明 | 与驾驭层的对应 |
|------|------|------|--------------|
| **感知** | Perception | 感知当前世界状态：用户输入、工具结果、历史对话、任务进度 | 上下文管理器（Context Manager）将外部信息和内部记忆"翻译"成结构化 Prompt |
| **规划** | Planning | 基于感知信息，由规划器更新目标、拆解任务、选择下一步行动 | LLM 推理 + 调用拦截器（Call Interceptor）捕获规划意图 |
| **行动** | Action | 执行内部操作（更新记忆）或外部操作（调用工具、发出回复） | 工具执行器执行具体操作，受权限和配额约束 |
| **反思** | Feedback / Reflection | 行动结果注入上下文，供 LLM 下一轮反思与重规划 | 反馈汇编器（Feedback Assembler）将执行结果结构化注入 |

> [!note] PPAF 与 Agent Loop 的关系
> PPAF 是对 [[智能体循环]] 的更高层抽象——Agent Loop 的 While 循环是 PPAF 的实现机制，PPAF 是 Agent Loop 的认知模型。两者视角互补：Agent Loop 关注工程实现（LLM Call → 工具调用 → 结果反馈），PPAF 关注认知流程（感知 → 规划 → 行动 → 反思）。

## 战略分析矩阵：认知循环 × 上下文效率

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

理解不同类型 Agent 的能力边界和工程挑战，可以从两个维度构建战略分析矩阵：

- **横轴：AI 认知循环（Cognitive Loop）**
  - **被动响应（React）**：行为由外部单次触发驱动，缺乏自主规划和反思能力
  - **主动规划与反思（Proactive Plan & Reflect）**：能基于长期目标自主多步规划、执行、反思和动态调整

- **纵轴：上下文处理效率（Context Efficiency）**
  - **低效（人工/单点投喂）**：上下文依赖人工手动提供或有限接口获取
  - **高效（沙盒化/全自动注入）**：通过系统级接口（文件系统、API网关、状态引擎）高效自动捕获和注入

| 象限 | 认知循环 | 上下文效率 | 典型应用 | Harness 需求 |
|------|---------|-----------|---------|-------------|
| **第一象限** | 主动规划与反思 | 高效全自动注入 | 复杂自主 Agent（如 DeepResearch、Claude Code） | 高级 Harness：状态分离、多Agent协作、策略门控 |
| **第二象限** | 主动规划与反思 | 低效人工投喂 | 研究型 Agent、需要人类引导的复杂任务 | 重规划机制、记忆分层、Token转化流水线 |
| **第三象限** | 被动响应 | 高效全自动注入 | 简单自动化工具（如天气查询、翻译） | 基础 Harness：错误分类、重试恢复 |
| **第四象限** | 被动响应 | 低效人工投喂 | 简单聊天机器人 | 最小 Harness：基本重试和超时 |

> [!important] Harness 的成熟度决定跃迁
> Harness 的成熟度直接决定了 Agent 能否从低效、被动的第三/四象限跃迁至高效、主动的第一/二象限。这正是 Harness Engineering 的核心价值所在。

## 顶层抽象：Harness 作为 REPL 容器

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

在架构层面，Harness 的本质可以被抽象为一个**带有边界控制、工具路由与确定性反馈的 REPL（Read-Eval-Print Loop）容器**。它包裹在 LLM 这个非确定性的"大脑"之外，负责管理 PPAF 的完整生命周期，将 LLM 的推理能力接入到确定性的工程世界：

| REPL 环节 | 对应 PPAF 阶段 | Harness 组件 | 说明 |
|-----------|---------------|-------------|------|
| **Read（读取）** | 感知 | 上下文管理器（Context Manager） | 将外部世界和内部记忆"翻译"成 LLM 可理解的、高度结构化的 Prompt |
| **Eval（评估/执行）** | 规划→行动 | 调用拦截器（Call Interceptor） | 捕获 LLM 的规划意图（Function Calling），路由到正确的工具执行器，严密监控超时、配额和错误 |
| **Print（打印/反馈）** | 反思 | 反馈汇编器（Feedback Assembler） | 将执行结果（成功或异常）封装成结构化"观测结果"，重新注入上下文 |
| **Loop（循环）** | PPAF 全闭环 | — | 读取→评估→打印不断循环，直到目标状态或终止条件，构成 PPAF 的核心驱动力 |

> [!tip] REPL 与 Agent Loop 的关系
> REPL 容器是对 [[智能体循环]] 的**架构视角**抽象——Agent Loop 关注"每次迭代做什么"，REPL 关注"数据如何流动"。两者是同一机制的不同维度描述。

## 状态分离原则

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

| 原则 | 说明 |
|------|------|
| **LLM 是无状态的计算单元（CPU）** | 必须将 LLM 严格视为无状态计算单元，所有需要跨轮次保持一致性的状态（用户会话、任务进度）存储在 Harness 控制的外部上下文状态管理器或其他持久化引擎中 |

> [!warning] 反模式
> 试图通过 Prompt Engineering 让 LLM 在长对话中自行维护复杂状态，会导致系统行为混乱、不可预测且难以调试。这正是 [[上下文压缩]] 和 [[记忆系统]] 存在的根本原因——状态必须由 Harness 管理，而非 LLM 自行维护。

## 六大设计原则（咸鱼版）

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

与本文已有的 7 条核心设计原则（来自 OpenHarness/Claude Code 等项目分析）互补，咸鱼从理论框架层面提炼了六大设计原则：

| 原则 | 含义 | 与现有7原则的对应 |
|------|------|-----------------|
| **为失败而设计（Design for Failure）** | 将异常和失败视为系统运行的常态而非个例，所有组件都应具备容错、重试和优雅降级能力 | ↔ 分类先于处理 + 渐进式降级 + 预算有限 |
| **契约优先（Contract-First）** | 所有交互由明确的、机器可读的契约（Schema、API、Event）定义 | ↔ 声明式策略 |
| **默认安全（Secure by Default）** | 安全是设计的出发点，遵循最小权限、零信任和纵深防御 | ↔ 最小权限 |
| **决策与执行分离（Decision vs. Execution）** | 将"决定做什么"（规划）与"如何做"（执行）在逻辑和物理上解耦 | ↔ [[Plan-and-Execute]] 范式的工程化体现 |
| **万物皆可度量（Everything is Measurable）** | 每个行为、决策、资源消耗都应可度量 | ↔ 可观测 |
| **数据驱动进化（Data-driven Evolution）** | 将每次运行视为学习机会，建立数据采集→标注→回流→更新的闭环 | 新增视角：Harness 不是静态的，而是通过数据反馈持续进化 |

## 系统架构：控制平面与数据平面

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

一个成熟的 Harness 通常拆分为**控制平面（Control Plane）**与**数据平面（Data Plane）**：

| 平面 | 职责 | 核心关注 |
|------|------|---------|
| **控制平面** | "决定做什么" | 任务调度、资源配额、行为规划、策略与权限 |
| **数据平面** | "如何去做" | 实际的 Agent 运行实例、状态存储、记忆存储和沙盒执行环境 |

> [!note] 控制平面与数据平面在 Claude Code 中的映射
> Claude Code 的架构天然体现了这一分离：QueryEngine（控制平面）负责查询编排、权限策略、错误分类；StreamingToolExecutor（数据平面）负责实际工具执行、结果返回。[[Claude Code 源码架构]] 对此有完整剖析。

## 策略门控（Policy Gateway）

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

除了沙盒本身，Harness 还需要一个位于"规划器 → 执行层"之间的**策略门控（Policy Gateway）**，在每一次行动前做最后的安全与合规检查：

| 检查类型 | 说明 | 对应驾驭层组件 |
|---------|------|--------------|
| **权限检查** | 基于 RBAC/ABAC 判定 Agent 是否有权访问目标资源或执行敏感操作 | [[护栏与钩子系统]]的 PreToolUse 钩子 |
| **敏感数据过滤** | 对参数和返回结果做 PII/密钥检测与脱敏 | 输出流过滤（凭证泄漏检测） |
| **指令注入防御** | 识别潜在恶意 Prompt/命令拼接，禁止危险模式进入执行层 | 模式匹配+语义分类器 |
| **审计日志** | 记录每一次"谁在何时尝试做什么、结果如何" | [[护栏与钩子系统]]的 PostToolUse 钩子 |

> [!tip] 策略门控 vs 护栏钩子
> 策略门控是[[护栏与钩子系统]]的**理论抽象**——钩子是策略门控的实现机制。策略门控强调"规划与执行之间必须有安全检查层"这一设计原则，钩子则提供了具体的19种事件拦截机制。

## Harness 7条核心设计原则

基于对 OpenHarness、DeerFlow、Parlant、SWE-agent、Claude Code 等项目的分析：

| 原则 | 含义 | 反面教材 |
|------|------|---------|
| **分类先于处理** | 先诊断错误类型，才选正确策略 | 所有错误一律重试3次 |
| **渐进式降级** | 从温和到激进，逐级响应 | token超限直接清空历史 |
| **静默优先** | 能内部恢复的异常不暴露给用户 | 每次retry都弹toast |
| **预算有限** | 任何自动恢复都必须有上限 | 无限重试直到成功 |
| **可观测** | 静默≠无记录，所有操作必须可审计 | 静默吞掉异常无日志 |
| **最小权限** | 工具权限按需授予，危险操作需审批 | 所有工具默认全权执行 |
| **声明式策略** | 安全规则外置为配置，非硬编码 | 每个安全检查写死在代码里 |

## Harness 六大子系统

完整 Harness 通常包含以下子系统（来自 OpenHarness、DeerFlow、Parlant、SWE-agent 等项目的共性分析）：

### 子系统 1：错误分类与恢复

核心原则：**不是所有错误都值得重试，但所有错误都需要被分类处理**。

| 错误类型 | 典型示例 | 处理策略 | 进阶策略 |
|---------|---------|---------|---------|
| **瞬态 (Transient)** | 网络超时、连接断开、500服务器错误、响应截断 | 指数退避重试+随机抖动，max 3次 | 提供商切换（OpenAI超时→Anthropic）；请求拆分 |
| **限流 (Throttle)** | 429 Too Many Requests、模型并发限制、Token配额耗尽、账户欠费 | 尊重Retry-After头或递增等待 | — |
| **永久 (Fatal)** | API Key无效、模型不存在、内容违规拒绝、Schema不匹配 | 终止+报告给用户 | — |

> [!tip] 为什么需要"抖动"（Jitter）
> 如果多个客户端在失败后等待相同时间重试，会在同一瞬间再次涌入服务器——这就是**惊群效应（Thundering Herd）**。加随机抖动使重试时间离散分布，避免雪崩。AWS、Google Cloud、Anthropic SDK 均采用指数退避+抖动策略（抖动幅度 0~50% 随机偏移，退避上限 60s）。

> [!example] 真实案例
> Anthropic 官方 SDK 内置 2 次自动重试+指数退避；OpenAI SDK 默认重试 `429`/`500`/`503` 错误——这些都是 Harness 的最小实现。

> [!example] 错误分类器三层匹配实现
> **FailoverReason枚举**涵盖15+种错误类型：`auth`、`auth_permanent`、`billing`、`rate_limit`、`overloaded`、`server_error`、`timeout`、`context_overflow`、`model_not_found`、`format_error`等。
>
> 三层匹配机制：
> - **Layer1 HTTP状态码**：429→`rate_limit`，402→`billing`，401→`auth`
> - **Layer2 错误消息模式匹配**：含中文模式（如"超过最大长度"适配阿里云DashScope）
> - **Layer3 异常类型名兜底**：按异常类名映射到枚举
>
> 上下文溢出**不重试而是压缩后重试**——调用`compress_with_fallback`先压缩再重新提交。

> [!example] 线上错误场景速查
> | 场景 | 错误表现 | 分类结果 |
> |------|---------|---------|
> | OpenAI限流 | `429 Too Many Requests` | `rate_limit` |
> | 阿里云DashScope | "rate increased too quickly" | `rate_limit` |
> | Anthropic过载 | `529 Overloaded` | `overloaded` |
> | vLLM上下文溢出 | "exceeds the max_model_len" | `context_overflow` |
> | 网络断连 | `ConnectionResetError` | `timeout` |

### 子系统 2：上下文工程

上下文管理不只是防溢出，更是**让 Agent 在有限窗口内保持最优决策能力**。Parlant 将其定义为："getting the right context, no more and no less, into the prompt at the right time"。

| 阶段 | 触发条件 | 操作 | 信息损失 | 代表实现 |
|------|---------|------|---------|---------|
| 预警 | 60% 窗口 | 标记冗余区域，准备压缩 | 无 | Claude Code auto-compact |
| 均缩 | 70% 窗口 | LLM生成结构化摘要替换早期消息 | 低 | LangChain ConversationSummaryMemory |
| 截断 | 80% 窗口 | 工具返回只保留摘要，删除原始内容 | 中 | OpenHarness context trim |
| 紧急 | 90% 窗口 | 仅保留 system prompt+最近3轮 | 高 | SWE-agent window reset |

> [!important] 范式转变：从"事后压缩"到"事前精选"
> Parlant 提出**上下文窄化（Context Narrowing）**——根据当前对话轮次的主题，只注入相关的规则、知识和工具描述，从源头控制 token 消耗。这是比被动压缩更根本的范式转变。

> [!tip] 上下文压缩三步策略
> **Step1 划分保护区**：HEAD区（system prompt + 首轮对话）和TAIL区（最近3轮）不压缩。
> **Step2 修剪MIDDLE区**：对工具输出按类型智能摘要——`terminal`提取命令和退出码，`read_file`提取路径和偏移量，`write_file`提取路径和行数，`search_files`提取模式和匹配数。
> **Step3 LLM生成摘要**：将修剪后的MIDDLE区交给LLM，生成13字段结构化摘要替换原始消息。

> [!important] 三级降级与防抖
> **降级①**：尾部保护从6条减到3条 → **降级②**：删除最早10条工具结果 → **降级③**：只保留system prompt + 最近3轮（相当于重启对话）。
>
> **防抖机制**：记录最近5次压缩平均节省率，低于10%跳过压缩，避免压无可压时浪费LLM调用。
> **触发时机为75%而非90%**——要给completion tokens和压缩本身的LLM调用留25%余量。

> [!tip] Todo状态注入
> 压缩后把未完成的todo注入回messages，防止LLM忘记做到哪了。只注入`pending`/`in_progress`状态，不注入`completed`/`cancelled`。

### 子系统 3：迭代控制

Agent 的无限循环是生产中最常见的"静默杀手"——不会崩溃，但持续消耗 Token 和时间，直到预算耗尽。

| 防护层 | 检测机制 | 触发动作 |
|-------|---------|---------|
| **Layer 1: 硬性预算** | 最大迭代次数(200轮)、最大Token消耗($5/次)、最大执行时间(30min) | 强制终止+输出当前摘要 |
| **Layer 2: 模式检测** | 重复检测(同一工具+同一参数连续N次)、震荡检测(A→B→A→B反复切换)、空转检测(连续N轮无新信息/无副作用) | 注入反思提示/强制re-plan/终止 |
| **Layer 3: 进展评估** | 里程碑机制(每N轮评估"离目标更近了吗")、衰减因子(每轮价值递减)、置信度(LLM自评完成确定性) | 交还控制权给用户+附带进展报告 |

> [!example] 生产实践
> SWE-agent 在代码修复任务中设置 cost limit（$3/实例），超出后强制停止并提交当前最优 patch。DeerFlow 引入"迭代衰减"——前10轮权重1.0，之后每轮权重递减 0.95^n，累积低于阈值时自动终止。

### 子系统 4：工具治理

工具操作**有副作用**（删文件、发邮件、执行代码）且**不可逆**，因此需要完整的治理生命周期。

| 步骤 | 关键动作 | 具体检查项 |
|------|---------|-----------|
| ① 参数校验 | &清洗 | JSON Schema、必填参数、类型转换、注入检测 |
| ② 权限检查 | &审批 | 白名单/黑名单、路径范围限制、危险操作需审批、并发/频率限制 |
| ③ 参数修复 | (容错) | 大小写归一化、camelCase→snake_case、模糊匹配(Levenshtein)、缺失参数填默认值 |
| ④ 沙箱执行 | (隔离) | Docker容器(SWE-agent/OpenHands)、WASM沙箱、文件系统虚拟化、seccomp/AppArmor |
| ⑤ 结果验证 | &后处理 | 返回大小限制、敏感信息脱敏、格式标准化 |

> [!tip] 声明式安全规则
> avakill 通过 YAML 定义工具策略，在执行前拦截判定（如阻止 `rm -rf` 等破坏性操作、限制外部URL访问需用户确认）。saifctl 先定义 Agent 能力边界规格（spec），运行时确保不越界——类似形式化验证思路。

> [!example] 工具名修复四策略递进
> - **策略1**：直接小写转换
> - **策略2**：标准化（替换`-`和空格为`_`）
> - **策略3**：候选集交叉组合（camelToSnake + stripSuffix + normalize）
> - **策略4**：difflib模糊匹配，相似度≥70%才采纳

> [!example] 参数JSON修复三策略
> - **策略1**：清理surrogate字符 → **策略2**：补全右括号 → **策略3**：删除尾随逗号 → 全部失败返回`raw_input`

> [!tip] 嵌套参数自动包装
> MCP工具的inputSchema常有嵌套结构（如`{request:{query:...}}`），但LLM常展平为`{query:...}`。系统检测inputSchema的嵌套层级，自动将展平参数包装回嵌套结构。

> [!example] 修复案例速查
> | LLM输出 | 修复结果 | 所用策略 |
> |---------|---------|---------|
> | `QueryWeather` | `query_weather` | 策略1+2 |
> | `qeury_weather` | `query_weather` | 策略4(difflib≥70%) |
> | `search-files` | `search_files` | 策略2 |

### 子系统 5：安全防护

Agent 面临独特且严峻的安全威胁——**间接提示注入**（工具返回的内容本身就可能是攻击载荷）。

| 3个攻击面 | 常见威胁 | 代表防护方案 |
|---------|---------|-------------|
| **输入侧** | 用户直接注入、对抗性prompt绕过安全限制 | Constitutional AI |
| **工具返回侧**（最危险） | 网页/文件/API响应中嵌入"Ignore previous instructions..."等恶意指令操纵Agent行为 | Anthropic Constitutional AI、模式匹配+语义分类器 |
| **输出/持久化侧** | 泄漏system prompt/内部标签/凭证、记忆投毒（写入恶意信息影响后续行为） | Claude Code memory validation、Parlant response filtering、OpenHarness output scrubber |

| 威胁 | 检测方法 | 防护位置 |
|------|---------|---------|
| 间接提示注入 | 模式匹配+语义分类器 | 工具返回后、送入LLM前 |
| 凭证泄漏 | 正则(API Key/Token模式) | 输出流过滤 |
| 路径穿越 | 路径规范化+白名单 | 文件操作工具前 |
| 命令注入 | Shell元字符检测 | Bash工具参数清洗 |

> [!warning] 深层防御原则
> 安全不能只靠一层。业界最佳实践是 **Defense in Depth**——输入、处理、输出三层各设独立检测，任何一层失守，后面的层仍能拦截。

### 子系统 6：可观测性

同一输入可能产生不同的执行路径，没有可观测性只能"重现一下看看"。

| 三支柱 | 关键记录项 |
|-------|-----------|
| **Logging** | 每轮决策记录、工具调用入参出参、错误上下文、安全事件审计 |
| **Tracing** | 端到端请求链路、父子Agent关系、工具调用耗时、压缩/降级事件 |
| **Metrics** | Token/轮/成本、成功率/重试率、延迟分布、循环检测触发率 |

追踪维度：`Session → Turn → LLM Call → Tool Call → Sub-Agent`（用户意图→决策轮次→模型响应→外部操作→委托任务）

代表工具：LangSmith / Langfuse / OpenTelemetry / Helicone

## 三次进化的关系

三者不是替代关系，而是**层层包裹、相互依存的嵌套关系**：

- **[[提示工程]]** 回答：我该跟模型说什么？
- **[[上下文窗口管理 | 上下文工程]]** 回答：模型在回答时该知道什么？
- **Harness Engineering** 回答：整个 AI 系统该如何可靠地运转？

没有好的 Prompt，Context 注入的信息无法被正确理解；没有好的 Context，Harness 中的 Agent 在信息真空中瞎跑；没有好的 Harness，再好的 Prompt 和 Context 在复杂任务中也会累积错误、最终崩溃。

## OpenAI 百万行代码实验

> OpenAI 内部的一支 3 到 7 人小团队，在短短五个月内，让 AI 生成了将近 100 万行生产级别的代码。据称全程，没有一个工程师亲手写过一行业务逻辑代码。

### 实验数据

| 维度 | 数据 |
|------|------|
| **时间** | 5 个月 |
| **团队规模** | 3-7 人 |
| **AI 生成代码量** | 近 100 万行（生产级） |
| **效率对比** | 约为纯人工的 10 倍 |

### 转折点：真正的瓶颈

实验初期，Agent 频繁跑偏、反复犯同类错误，进展远不如预期。团队意识到，**真正的瓶颈不在于模型能力，而在于 Harness 设计**。这证明了：即使拥有最先进的大模型，没有合适的 Harness，Agent 也无法在生产环境中可靠运行。

### 三大 Harness 策略

#### 策略一：上下文治理（Context Governance）

**问题**：初期将所有编码规范、架构设计、业务逻辑堆进一个巨大的 `agent.md` 文件，信息太多导致 Agent 抓不住重点。

**改进方案**：
- 将 `agent.md` 压缩至百行以内，仅保留索引和分类
- 当 Agent 需要特定规范时，系统动态加载对应子文档
- 强制将散落各处的决策记录全部迁移至代码仓库，确保 Agent 的唯一信息来源是可信的、版本受控的仓库

这与 [[上下文注入管道]] 的核心理念一致：不是把所有信息一次性塞进去，而是按需、精准地注入。

#### 策略二：验证闭环（Verification Loop）

**问题**：Agent 自我声称"测试通过"而实际上根本没运行测试。

**改进方案**——为系统配备完善的工具栈，将"声称完成"变成"验证完成"：
- **Chrome DevTools 集成**：AI 可自行截图、模拟操作，视觉验证 UI 是否符合预期
- **可观测性工具**：AI 读日志、查性能指标，主动排查问题
  - 每个worktree都有一套本地日志、指标和链路追踪（基于 Vector、Victoria Logs/Metrics/Traces）
  - Agent 可以自己查报错、看性能数据、定位问题在哪一层
- **强制 Lint + 自动化测试**：代码不符合规范，报错信息自动反馈给 AI，要求原地修复，形成闭环

这套机制对应 [[重试与恢复机制]] 和 [[护栏与钩子系统]] 的核心思想：**不为 Agent 的善意假设买单，只信任可验证的结果**。

#### OpenAI 团队的四个核心理念

1. **仓库就是大脑**：仓库是 Agent 唯一的知识来源。代码、markdown、schema、可执行计划全都版本化存储。Agent 看不到仓库之外的东西，所以仓库必须包含一切。隐性知识必须显性化推入仓库——Agent 不会来问你，也不会去茶水间聊天。

2. **给 Agent 写的代码**：代码不仅要对人类可读，更要对 Agent 可读（**application legibility**）。Agent 更依赖结构化线索：严格的分层架构、一致的命名模式、显式类型定义。对人类"一看就懂"的隐含约定，对 Agent 可能是致命盲区。

3. **自主性分级**：从简单任务开始逐步提升自主权限。后期单个 Codex 任务可**连续运行 6 小时以上**自主完成复杂功能开发。==自主性提升必须建立在约束系统成熟基础上==。

4. **合并哲学**：工程师做「审查」而非「修改」。如果一个 PR 需大量修改才能合并，反思 Harness 哪里出了问题，把修复方案反哺回 Harness。**团队增长反而提升吞吐量**——每个工程师都是独立的「Harness 调参师」，协调成本很低。

> [!quote] Ryan Lopopolo（OpenAI 核心工程师）
> **Agent 不难，Harness 才难。**

#### 筋略三：技术债清理（Tech Debt Cleanup）

**问题**：大规模 AI 代码生成不可避免地引入重复命名、风格不一致、废弃文档等问题。

**解决方案**：设置后台运行的 Codex 任务，像操作系统的垃圾回收（GC）机制一样，定期扫描代码库，自动修复偏离规范的代码和过时文档。

## Anthropic F-Harness

### AI 的"自恋问题"

> AI 倾向于给自己的 Bug 打高分。

Anthropic 在尝试克隆 Claude.ai 复杂界面的实验中发现，单 Agent 模式下存在三大致命缺陷：

1. **上下文耗尽**：任务量过大，Agent 在中途耗尽上下文，"记不住"之前做了什么
2. **虚假声明完成**：功能只完成了一半，Agent 就宣称"已全部完成"
3. **自评过度乐观**：让 Agent 自评输出质量，结果是惊人的过高评价

### F-Harness 三角色分工

| 角色 | 职责 | 解决的核心问题 |
|------|------|---------------|
| **Planner（规划者）** | 将模糊需求拆解为精细的、可逐项追踪的功能列表 | 任务量过大导致中途迷失 |
| **Generator（生成者）** | 按功能列表逐项执行，完成一项才标记一项 | 稳扎稳打，避免跳步或遗漏 |
| **Evaluator（评估者）** | 独立第三方审核 Agent，与 Generator 完全独立 | AI 的自评偏见 |

### 代价对比

> 20 倍的时间代价，22 倍的成本代价，换来的是质的飞跃。

| 维度 | 单 Agent 模式 | F-Harness 三 Agent 模式 |
|------|--------------|------------------------|
| **耗时** | 约 20 分钟 | 约 6 小时 |
| **成本** | 约 $9 | 约 $200 |
| **输出质量** | 逻辑残缺，勉强可用 | 生产环境级别，逻辑完整 |

**核心结论**：当任务的复杂度超过单 Agent 的可靠性边界，多 Agent 协作的 Harness 是唯一可行的工程解法。

F-Harness 的分工思想在 [[Claude Code 整体架构]] 中有直接体现：
- **Planner** 对应 [[任务分解]] 机制
- **Generator** 对应 [[工具系统]] 的逐项执行模式
- **Evaluator** 对应 [[验证与测试]] 和 [[Reflection 范式]]

这与 [[Plan-and-Solve]] 范式一脉相承，但走得更远：将 Plan、Execute、Evaluate 三个环节彻底解耦并通过 [[角色分配]] 分配给独立 Agent，是 [[多智能体协作]] 的典型应用。

## Harness 的衰变定律

> 模型能力越强，所需的 Harness 越简单。

Anthropic 的研究者发现了一条反直觉的规律：

- 在 Claude 3.0 时代，为保证 Agent 不在复杂任务中途崩溃，需要极严格的 Harness 约束
- 当模型升级到 Claude 3.5，其全局统筹能力、长上下文处理能力和自我校验能力大幅提升，原本不可或缺的许多 Harness 规则自然变得不再必要

**实践启示**：
1. **Harness 是当下的现实答案**——在模型能力尚未完美的今天，Harness 是让 AI 系统在生产环境可靠运行的必要条件
2. **Harness 可能是一项过渡性技术**——随着模型能力提升，今天需要精心设计的许多 Harness 规则，未来会被模型能力自然吸收
3. **核心策略：动态调节 Harness 的"厚度"**——不过度设计那些模型未来能自我解决的问题，将精力集中在模型短期内无法自行解决的业务逻辑边界和外部环境接口上

## 行业实践案例

### Stripe Minions：每周 1300 个 PR

Stripe 内部的「Minions」系统是目前规模最大的 Agent 编程实践之一：**每周合并 1300+ 个 PR，全部由无人值守 Agent 完成**。

核心架构设计——**Blueprint 编排系统**：
- **确定性节点**（运行 linter、推送更改）按固定路径执行，不调用 LLM
- **Agentic 节点**（实现功能、修复 CI 失败）让 LLM 行使判断
- 就像一条生产线：传送带是确定性的，工人的手艺活是 Agentic 的

关键限制：
- **CI 最多两轮**：第一轮失败→Agent 自动修复→再跑一次→还失败→交人类。不允许 Agent 在 CI 上无限重试
- **Toolshed** 挂载约 500 个 MCP 工具，但每个 Agent 只给精心筛选过的子集。==更多工具 ≠ 更好表现==

> [!quote] Stripe 工程团队总结
> 成功取决于可靠的开发者环境、测试基础设施和反馈循环，==跟模型选择关系不大==。如果对人类友好，对 LLM 也一样友好。

### Cursor Self-Driving：每小时 1000 个 commit

Cursor 的「Self-Driving Codebases」研究：约每小时 1000 个 commit，一周超过 **1000 万次工具调用**，启动后无需人工干预。

经历的四版迭代（说明 Harness 设计的难度）：
1. **单 Agent** → 复杂任务扛不住
2. **多 Agent 共享状态文件** → 锁竞争严重，Agent 之间互相打架
3. **结构化角色分工**（Planner → Executor → Workers → Judge）→ 太僵硬
4. **持续执行器** → 角色过载导致诡异行为

最终版本：**递归 Planner-Worker 模型**——根级 Planner 拥有全局视野，生成子 Planner 处理细分任务，Worker 在仓库副本上独立操作。

> [!warning] 关键发现
> ==差的初始指令会在数百个 Agent 间被放大==。一条模糊的指令，一个 Agent 犯的错，乘以几百个并发 Agent……结果可想而知。

### Peter Steinberger：一个人的军队

前 iOS 圈知名开发者，2026 年 1 月单月产出 **6600+ 个 commit**，同时运行 5-10 个 Agent。OpenClaw 项目 4 个月内拿到 18 万 stars（GitHub 增长最快仓库）。

激进做法：**不逐行审查 Agent 生成的代码**——Code Review 变成「Prompt Review」。更关心生成代码的提示词写得好不好。

> [!note] 反常识发现
> ==喜欢算法谜题的工程师反而很难适应 Agent 工作流，产品导向的工程师适应更快。==

2026 年 2 月加入 OpenAI——这件事本身也是对 Harness Engineering 价值的认可。

## 拐杖论（Bitter Lesson 阵营）

Latent Space 将反对 Harness Engineering 的观点归为「Bitter Lesson 阵营」，出自 Rich Sutton 那篇著名 AI 随笔：==别在工程花活上下太多功夫，算力的增长终究会碾平一切==。

### Noam Brown（OpenAI）的观点

> Harness 就像一根拐杖，我们终将能够超越它。

历史支撑：推理模型出现前，开发者搭建了大量 Agentic 系统模拟推理能力（路由器、编排器、multi-agent 协作）。推理模型一出，这些基础设施一夜之间就不需要了。

> [!quote] Noam Brown 预判
> 我们公开说过，我们想要走向一个==单一统一模型==的世界。你不应该需要在模型上面再加一个路由器。

给开发者的建议：==别花六个月搭建一个可能六个月后就被淘汰的东西==。

### METR 数据的质疑

METR 找了 scikit-learn、Sphinx、pytest 的 4 位活跃维护者审查 296 个 AI 生成 PR：

- **维护者合并率 ≈ 自动评分通过率的一半**
- Claude Sonnet 4.5：自动评分对应 ~50 分钟任务范围，维护者实际合并对应仅 ~8 分钟 → **7 倍能力高估**
- Claude Code 和 Codex 并不比基础脚手架表现更好，Harness 选择基本落在误差范围内

## 路线之争：Big Model vs Big Harness

Latent Space 2026 年 3 月专题分析《Is Harness Engineering Real?》，把行业分成两大阵营：

### Big Model 阵营

**核心观点**：模型能力的增长才是主旋律，Harness 只是权宜之计。

Claude Code 团队 Boris Cherny & Cat Wu：

> 所有的秘密武器都在模型本身。我们追求的是==最薄的那层包装==。

强调持续简化，别往上堆复杂性。Claude Code 的 Harness 层尽可能地薄，把更多事情交给模型自己判断。

### Big Harness 阵营

**核心观点**：模型是引擎，Harness 是方向盘和刹车。引擎再强，没有方向盘你也到不了目的地。

Jerry Liu（LlamaIndex 创始人）：

> Model Harness 就是一切。从 AI 那里获取价值的最大障碍，是你自己为模型做上下文工程和工作流工程的能力。

数据支撑：开发者在 60% 的工作中使用 AI，但**完全委托给 AI 的任务只有 0-20%**。Cursor $50B 估值（Harness 公司的代表）也在印证这个方向。

## 护栏悖论

**车速越快，护栏越重要。**

| 交通方式 | 速度 | 护栏需求 |
|---------|------|---------|
| 自行车道 | 30 km/h | 无护栏 |
| 高速公路 | 120 km/h | 标配护栏 |
| 磁悬浮列车 | 300 km/h | 全封闭轨道 |

模型就是引擎。引擎越强，速度越快，就越需要精心设计的约束系统确保它跑在正确方向上。

> [!important] 护栏悖论的核心洞察
> ==约束解空间反而让 Agent 更有生产力==。当 Agent 可以生成任何东西时，它会浪费 token 探索死胡同；当 Harness 定义了清晰的边界，Agent 反而更快收敛到正确答案。这正是 Cursor 团队的发现：告诉 Agent「不要留 TODO」比告诉它「完成所有实现」效果更好。

Noam Brown 说得对：很多脚手架确实会随模型进化被淘汰。但==架构约束、反馈循环、熵管理不会消失，只会换形态==。从马车到汽车：马鞭消失了，方向盘和刹车不会消失。

**Harness 阵营真正应该担心的**：自己搭建的 Harness 六个月后是否就过时了。这反而成了最好的 Harness Engineering 建议：==Harness 要轻，要模块化，要随时准备被拆掉重来==。

Philipp Schmid 三词建议：**Start Simple. Build to Delete.**

## 七个配置杠杆

从行业实践来看，Harness 的配置杠杆有七类（与 7 条核心设计原则互补——原则是「应该追求什么」，杠杆是「具体怎么做」）：

| 杠杆 | 说明 | 关键要点 |
|------|------|---------|
| **AGENTS.md / CLAUDE.md** | 仓库根目录 markdown 文件，写架构约定、命名规范、测试期望 | ==控制在 60 行以内==，写「目录」别写「百科全书」。每次 Agent 犯错就加一条规则 |
| **确定性约束** | linter、类型检查、结构化测试、pre-commit hooks | ==硬约束比软性 prompt 指令更可靠==。OpenAI 和 Stripe 的经验都证实这一点。linter 报错信息嵌入修复指引 |
| **工具精简** | 别给 Agent 塞太多工具 | Vercel 从 15 砍到 2，准确率反而升。Stripe 500 工具只给 Agent 精选子集 |
| **Sub-Agent 隔离** | 复杂任务拆成子任务，各有独立上下文窗口 | 防止中间噪声在主线程累积，不同子任务可用不同模型 |
| **反馈循环** | Agent 自己跑测试、看截图、查日志 | 让 Agent ==自验证产出==，不靠人工盯。OpenAI 给 Agent 配了完整可观测性栈 |
| **CI 限速** | CI 最多两轮 | 跑两次还不过→交人类。既控制成本，也防止 Agent 在错误方向越跑越远 |
| **垃圾回收** | 定期启动 Agent 扫描技术债、过时文档、架构漂移 | 容易忽略但极重要，尤其代码量大了之后。修复 PR 多能在 1 分钟内审查合并 |

## 驾驭层的三个底层思维

> [!important] 思维1：能力层尽可能抽象，编排层尽可能简洁
> 核心Loop不到50行，所有复杂逻辑下沉到子系统。编排层只做"调谁、传什么、拿到什么"，脏活累活全在子系统里。

> [!important] 思维2：静默优先，但静默≠无记录
> 修复、压缩、重试都是静默的——用户看不到，但都有日志、指标、追踪。静默是用户体验，记录是工程质量。

> [!important] 思维3：概率系统嫁接在确定性工程上
> 观测比开发更重要，好架构思路比堆人力调参有效10倍。LLM是概率引擎，Harness用确定性工程兜底概率波动——这是整个驾驭层的哲学根基。

---

## 为什么驾驭层是产品

2025-2026 年，AI Agent 的差异化不再是 LLM 本身，而是**驾驭层的质量**。同样的 Claude 模型，没有驾驭层就是一个聊天机器人；有好的驾驭层就是一个能自主编码、安全执行、持续工作的智能体。驾驭层就是产品。

Claude Code 的整个架构就是一个驾驭工程的**完整实现**。512K 行代码中，大部分是驾驭层而非模型调用——**驾驭层的复杂度远超模型调用本身**。构建有用的 Agent，90% 的工作在驾驭层，10% 在模型调用。

---

## 未来趋势

- **Harness 成为新的「服务模板」**：未来组织可能从预制 Harness 模板中选择并定制（类似今天的 CI/CD 模板或 cookiecutter 项目脚手架，只是层级更高）
- **技术栈收敛**：当写代码本身不再是瓶颈时，团队会更偏向选择那些「有好 Harness 可用」的技术栈。AI 友好性可能超过开发者个人偏好，成为选型首要考量
- **Harness 反哺模型训练**：Agent 失败轨迹可成为模型训练的高质量数据——哪些约束被反复触犯？哪些错误模式在不同项目中重复出现？这些数据对模型改进的价值可能远超传统训练数据
- **「旧代码」问题**：OpenAI 的实验是从空仓库开始的，Harness 从第一天就存在。但对于已有几十万行代码的老项目，给老代码加 Harness 可能像给从不跑测试的项目补测试一样痛苦
- **学科化**：AIE Europe 已设立全球首个 Harness Engineering 专题赛道，arxiv 上有了专门论文。概念正从 buzzword 向正式工程学科演进

## 本质是管理

Harness Engineering 说的这些——上下文管理、架构约束、反馈循环、定期清理——**不就是管理吗？**

| 管理 | Harness 对应 |
|------|-------------|
| 给新人写 onboarding 文档 | AGENTS.md |
| 定代码规范和架构原则 | linter + 结构化测试 |
| 做 code review 确保质量 | CI/CD 检查 |
| 定期做技术债清理 | 垃圾回收 |
| 给工具做精简和选型 | 工具链管理 |
| 遇到反复出现的问题写进 wiki | 反馈循环 |

> [!important] 核心洞察
> ==AI Agent 越强，就越像一个能力很强但需要管理的员工。==你不会把一个刚入职的天才工程师扔进没有文档、没有规范、没有 CI 的项目里指望他写完美代码。同样，也不该把强大的 AI 模型扔进没有 Harness 的环境里然后抱怨它不好用。
>
> 未来最稀缺的，可能不是训练模型的人，而是==管理模型的人==。
