# Harness Engineering 完全指南

> 驾驭工程（Harness Engineering）是为 AI 智能体设计和实施**控制层/驾驭层**的工程实践——将原始 LLM 转化为可控、安全、高效的智能体。它是 AI 工程化的第三次进化，也是将大语言模型从"对话工具"转化为"可控生产系统"的关键一跃。

> [!quote] 核心论题
> ==通过工程化手段构建确定性，以承载模型不确定性==——这是 Agent 技术从"魔法调优"走向"系统工程"的核心思想，也将是未来很长一段时间内构建高质量 Agent 的基石。虽然各个模块的具体实现方式仍在快速演进，但这条核心思想始终未变：即在保证安全、可控的前提下，最大化释放模型的推理与执行潜力。
>
> 来源：[[Agent技术范式演变（2023-2026）]]

> [!quote] 一句话定位三层关系
> **Prompt 是输入技巧，Context 是信息架构，Harness 是系统工程。** — [[Prompt与Context工程（三次进化）]]
> - Prompt 解决"听不听得懂你在说什么"
> - Context 解决"有没有拿到完成任务所需的信息"
> - Harness 解决"即使会犯错，系统还能不能持续、稳定、可控地运行"

## 一、定义与核心概念

### 1.1 给 AI 上"马具"

Harness，字面意思是"马具"。套在马身上的那套装备——缰绳、鞍具、辔头。没有马具的马骑起来信马由缰，套上马具的马才能指哪打哪。

驾驭工程就是**研究如何为大模型设计一套合适的"马具"**。

另一种理解：原始 LLM 就像一辆没有方向盘的汽车——动力强大但无法控制方向。驾驭工程就是在这辆汽车上安装：方向盘（系统提示词）、刹车（权限系统）、导航仪（上下文注入）、仪表盘（流式渲染控制）、安全带（护栏与钩子）、自动恢复系统（重试与压缩）。

### 1.2 核心公式

**AI Agent System = LLM + Harness**

一个完整的 AI Agent 系统，除了大模型本身之外的所有东西，都属于 Harness：

- **工具接口**：模型调用外部工具的能力（API、数据库、文件系统等）
- **验证机制**：确保输出正确性、符合规范的自动化校验流程
- **上下文管理**：动态组装、压缩、注入上下文信息的策略
- **权限控制**：限制模型可执行操作的范围和边界
- **多 Agent 协作机制**：角色分工、任务分配、结果审核的编排层

核心原则：**LLM 是引擎，驾驭层是方向盘和安全带**。

### 1.3 Philipp Schmid 的操作系统类比

| 计算机组件 | Agent 对应物 | 角色 |
| --- | --- | --- |
| **CPU** | 模型 | 原始算力 |
| **内存** | 上下文窗口 | 有限的、易失的工作记忆 |
| **操作系统** | Agent Harness | 管理上下文、处理启动流程、提供标准驱动 |
| **应用程序** | Agent | 跑在操作系统上的用户逻辑 |

模型就是 CPU，算力再强，没有操作系统也跑不起来。**而 Harness，就是给 AI Agent 套上的那层操作系统。**

> [!important] 核心洞察
> 过去两年一直在升级 CPU（更大模型、更长上下文、更强推理能力），但操作系统呢？**还停留在 DOS 时代。**大多数人用 AI Agent 的方式还是「打开终端，输入命令，看结果」——没有文件系统、没有进程管理、没有标准驱动、没有权限控制。

### 1.4 为什么需要 Harness？

[[提示工程]]解决了"说清楚"的问题，[[上下文窗口管理|上下文工程]]解决了"给够信息"的问题，但即使做到了这两点，Agent 依然会在复杂任务中失控：

- 它可能"顺手"修改不该动的东西
- 它可能声称"测试通过"而根本没有运行测试
- 它可能生成风格不一致、功能重复的代码
- 它在长任务中途耗尽上下文，宣称完成实际未完成

这些问题的根源不在"说什么"或"给什么信息"，而在于：**系统层面缺乏约束、验证和反馈机制**。这恰恰是 Harness Engineering 填补的盲区。

> [!note] Workflow Agent（阶段二）是"早期的 Harness"
> 来源：[[Agent核心技术概念与范式发生了哪些演变以及背后的思考]]
>
> 2024年时期的工作流 Agent（Agentic Workflow），虽然当时没有 Harness 这个概念，但所做的事情本质上就是==一套比较重的 Harness==——用工程化的硬约束来弥补模型的不确定性。Workflow 的"刚性编排"特征——无论外部环境如何变化，仍==死板严格遵循第一步、第二步==——正是早期 Harness "重约束、轻自主"的表现。
>
> 这与 Harness 的衰变定律形成呼应：随着模型能力提升（从阶段二到阶段三），Harness 从"重"变"轻"——自主 Agent 阵段的 Harness 是==轻量级==的，配合自我校验机制而非刚性 Workflow。详见 [[Agent技术范式演变（2023-2026）]]。

### 1.5 Mitchell Hashimoto 的采纳五阶段

Harness Engineering 这个词的起源，来自 Mitchell Hashimoto（HashiCorp 联合创始人、Terraform 缔造者）2026 年 2 月 5 日写的一篇博客，记录自己用 AI 编程的进化史：

| 阶段 | 名称 | 核心做法 |
|------|------|---------|
| 第一阶段 | Drop the Chatbot | 别用聊天界面了 |
| 第二阶段 | Reproduce Your Own Work | 强迫用 Agent 重新做一遍手动完成的工作 |
| 第三阶段 | Warm Start | 利用下班时间跑 Agent，第二天早上收获成果 |
| 第四阶段 | Background Agents | 简单任务交给后台 Agent，自己专注深度工作 |
| **第五阶段** | **Engineer the Harness** | **每发现 Agent 犯一个错，就工程化一个解决方案让它再不犯** |
| 第六阶段 | 全天候 Agent | Agent 全天候自主运行 |

> [!quote] 第五阶段的定义
> ==每当你发现 Agent 犯了一个错误，你就花时间去工程化一个解决方案，让它再也不会犯同样的错。==

> [!note] Ghostty 项目实践
> Mitchell 在 Ghostty 项目中实践了这个理念。AGENTS.md 文件里的每一行规则，背后都对应着 Agent 曾经犯过的一个错。他说：==那个文件里的每一行都基于一次 Agent 的不良行为，而且几乎完全解决了这些问题。==

> [!important] 击中的痛点
> 这句话击中了很多人隐隐感觉到但没说清楚的痛点：**模型能力够了，可它就是不听话。** Harness Engineering 的核心操作就是——犯错后反哺，而不是反复提示。

### 2.2 三层技术定义（arxiv 论文 2603.05344）

arxiv 论文 *Building Effective AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned* 给出了更精确的三层架构定义：

| 层级 | 名称 | 核心职责 | 传统对应 |
|------|------|---------|---------|
| **Scaffolding**（脚手架） | 预执行阶段组装 | 系统 prompt 编译、工具 schema 构建、sub-agent 注册表填充 | BIOS 和引导程序 |
| **Harness**（运行时编排） | 核心推理循环包装层 | 协调工具执行、上下文管理、安全执行、会话持久化 | 操作系统内核 |
| **Context Engineering**（上下文工程） | Token 预算管理 | 决定什么信息该进来、什么该压缩、什么该丢弃 | 操作系统内存管理 |

核心公式：**coding agent = AI model(s) + harness**

> [!important] 三层与现有概念的映射
> 三层定义中 **Scaffolding** ≈ 七大组件的初始化阶段；**Harness** ≈ 运行时的六大子系统；**Context Engineering** ≈ 上下文维度的深化。三层不是替代关系，而是==层层包裹、嵌套运行==——Scaffolding 启动 Harness，Harness 内部调用 Context Engineering。

### 2.3 学术七层分类体系（CMU/Yale/Amazon 2026.06）

卡内基梅隆大学、耶鲁大学、亚马逊等机构 2026 年 6 月联合发表的学术综述，将 Harness 分为七层——前四层是**结构核心**，后三层是**控制平面**：

#### 结构核心（前四层）

| 层级 | 名称 | 职责 |
| --- | --- | --- |
| **E** | 执行环境与沙箱 | 决定 Agent 代码在哪里运行、受到什么约束 |
| **T** | 工具接口与协议 | 定义外部能力如何被描述、发现和调用 |
| **C** | 上下文管理 | 决定模型在短期、会话级和持久化层面能看到什么 |
| **L** | 生命周期与编排 | 负责组织状态读写控制流，覆盖单 Agent 循环到 issue → PR 工作流 |

#### 控制平面（后三层）

| 层级 | 名称 | 职责 |
|------|------|------|
| **O** | 可观测性 | 捕获轨迹、成本、失败和可靠性信号 |
| **V** | 验证 | 把任务和轨迹转化为评估、失败归因和回归反馈 |
| **G** | 治理 | 通过权限、身份、策略、安全加固、审计和人工监督约束系统行为 |

> [!important] 学术界的关键发现
> ==在不改模型权重的情况下，仅调整 harness 层本身，也可能显著改变 Agent 在 coding 和 terminal benchmark 上的表现。== 这意味着 Harness 不是"必要的恶"，而是一个==独立的优化维度==——即使模型不变，更好的 Harness 也能带来更好的结果。

### 2.4 四个核心维度（与 PPAF 紧密耦合）

| 维度 | 覆盖 PPAF 环节 | 核心关注 | 对应实现 |
|------|---------------|---------|---------|
| **① 上下文维度** | 感知 + 反思 | 从无限状态到有限 Token 的转化 | [[上下文注入]]、[[上下文压缩]]、[[上下文窗口管理]] |
| **② 行动维度** | 行动（+ 规划→行动的转化） | 从文本预测到物理执行的闭环 | [[工具调用（Function Calling）]]、[[Plan-and-Execute]] |
| **③ 约束维度** | 全闭环（贯穿 PPAF） | 安全、权限、预算等硬约束 | [[工程设计与安全]]、策略门控、沙盒、熔断 |
| **④ 度量维度** | 反思（闭环的自我审视） | 可观测性→度量→驱动 Harness 演进 | [[度量驱动演进闭环]]、[[智能体评估]] |

> [!important] 四维度与 R.E.S.T. 目标的映射
> 四维度是==实现视角==（"怎么做"），R.E.S.T. 是==目标视角==（"做到什么"）：
> - **Reliability** ← ③约束维度 + ④度量维度
> - **Efficiency** ← ①上下文维度 + ③约束维度
> - **Security** ← ③约束维度
> - **Traceability** ← ④度量维度

### 2.5 关键工程位点

为实现 REPL 闭环并落地设计原则，Harness 需要在架构中部署一系列关键组件（"工程位点"）：

| 工程位点 | 在 REPL 中的位置 | 功能 | PPAF 环节 |
|---------|----------------|------|-----------|
| **上下文管理器** | Read 端 | 规约规则 + 注入边界 + Token 转化流水线 | 感知 |
| **调用拦截器** | Eval→Print 之间 | FC 捕获→路由→监控执行（超时、配额、错误） | 规划→行动 |
| **反馈汇编器** | Print 端 | 观测注入→结构化反馈→重新注入上下文 | 反思 |
| **策略门控** | 规划器→执行层之间 | 权限 + 数据过滤 + 注入防御 + 审计 | 规划→行动（硬约束） |
| **迭代控制器** | Loop 外围 | 预算限制 + 模式检测 + 进展评估 | 全闭环 |
| **状态管理器** | Loop 外部 | 持久化状态 + 记忆分层 + 会话恢复 | 全闭环（跨轮次） |
| **沙盒执行器** | 数据平面 | 工具隔离执行 + 资源配额 | 行动 |

> [!note] 工程位点与 Claude Code 实现的映射
> Claude Code 的架构几乎一对一映射了这些位点——[[Claude Code 源码架构]] 对此有完整剖析。

### 2.6 8 个核心元件（简洁版）

所有不属于 model weights、也不只是 prompt string 本身的工程元件都算 Harness 范围：

| # | 元件 | 做什么 | 关键点 |
|---|---| ---| ---|
| 1 | **[[智能体循环]]** | "LLM → tool → result → LLM" 循环 | `max_iter` safety net 必须设 |
| 2 | **Tool registry** | 动态 tool dispatch、permission gate | Schema 质量决定 LLM 选择准确度 |
| 3 | **[[上下文窗口管理]]** | message history 管理、auto-compact | `/compact` 释放 context |
| 4 | **Safety layer** | permission prompts、sandboxed exec | 破坏性操作必须拦截 |
| 5 | **Retry / recovery** | tool fail 怎么处理 | error 回传**结构化 dict**、不要 `raise` |
| 6 | **Telemetry / Observability** | metrics、logging、trace | 没 observability 的 agent debug = 黑盒 |
| 7 | **[[智能体评估]] harness** | regression test、quality gate | 自有 eval set > 外部 benchmark |
| 8 | **Cost / Latency optimization** | prompt caching、model routing、batching | 2026必修，会用省50-90% |

### 2.7 Framework vs Harness

| 维度 | Framework（Stage 4） | Harness（本概念） |
|---|---| ---|
| 规范 | **API** — 你调用的接口长什么样 | **runtime** — 怎么跑、怎么 recovery、怎么观测 |
| 例子 | [[LangGraph]] 的 StateGraph API | Claude Code 的 permission + retry + telemetry |

**Framework 是脚手架，Harness 是运行时**。两者不冲突、但解决的是不同层级的问题。

### 2.8 控制平面与数据平面

一个成熟的 Harness 通常拆分为**控制平面（Control Plane）**与**数据平面（Data Plane）**：

| 平面 | 职责 | 核心关注 |
|------|------|---------|
| **控制平面** | "决定做什么" | 任务调度、资源配额、行为规划、策略与权限 |
| **数据平面** | "如何去做" | 实际的 Agent 运行实例、状态存储、记忆存储和沙盒执行环境 |

> [!note] 控制平面与数据平面在 Claude Code 中的映射
> Claude Code 的架构天然体现了这一分离：QueryEngine（控制平面）负责查询编排、权限策略、错误分类；StreamingToolExecutor（数据平面）负责实际工具执行、结果返回。[[Claude Code 源码架构]] 对此有完整剖析。

## 三、运行模型

### 3.1 PPAF 闭环

一个完备的 Agent 系统，其核心运行机制可抽象为一个持续循环的四阶段过程——**PPAF 闭环**：

| 阶段 | 英文 | 说明 | 与驾驭层的对应 |
|------|------|------|--------------|
| **感知** | Perception | 感知当前世界状态：用户输入、工具结果、历史对话、任务进度 | 上下文管理器将外部信息和内部记忆"翻译"成结构化 Prompt |
| **规划** | Planning | 基于感知信息，由规划器更新目标、拆解任务、选择下一步行动 | LLM 推理 + 调用拦截器捕获规划意图 |
| **行动** | Action | 执行内部操作（更新记忆）或外部操作（调用工具、发出回复） | 工具执行器执行具体操作，受权限和配额约束 |
| **反思** | Feedback / Reflection | 行动结果注入上下文，供 LLM 下一轮反思与重规划 | 反馈汇编器将执行结果结构化注入 |

> [!note] PPAF 与 Agent Loop 的关系
> PPAF 是对 [[智能体循环]] 的更高层抽象——Agent Loop 的 While 循环是 PPAF 的实现机制，PPAF 是 Agent Loop 的认知模型。

### 3.2 Harness 作为 REPL 容器

在架构层面，Harness 的本质可以被抽象为一个**带有边界控制、工具路由与确定性反馈的 REPL（Read-Eval-Print Loop）容器**：

| REPL 环节 | 对应 PPAF 阶段 | Harness 组件 | 说明 |
|-----------|---------------|-------------|------|
| **Read（读取）** | 感知 | 上下文管理器 | 将外部世界和内部记忆"翻译"成 LLM 可理解的、高度结构化的 Prompt |
| **Eval（评估/执行）** | 规划→行动 | 调用拦截器 | 捕获 LLM 的规划意图（Function Calling），路由到正确的工具执行器，严密监控超时、配额和错误 |
| **Print（打印/反馈）** | 反思 | 反馈汇编器 | 将执行结果（成功或异常）封装成结构化"观测结果"，重新注入上下文 |
| **Loop（循环）** | PPAF 全闭环 | — | 读取→评估→打印不断循环，直到目标状态或终止条件 |

### 3.3 PEV（Plan-Execute-Verify）架构模式

Augment Code 提出摒弃传统"生成-检查"模式，采用 PEV 循环：

```
传统：Generate → Check（后置验证，发现问题时已浪费大量 Token）
PEV：  Plan → Execute → Verify（前置规划 + 受约束执行 + 多维验证）
```

| PEV 阶段 | 核心动作 | Harness 的角色 |
| --------------- | --------------------------------- | ------------------------------ |
| **Plan（规划）** | 显式分解任务与验收标准，减少自由度 | 限制 Agent 的行动空间，提高首次成功率 |
| **Execute（执行）** | 受计划约束执行，Harness 在每次工具调用时触发 | |
| **Verify（验证）** | 执行前 + 运行中 + 事后 + 计划对齐，提供带上下文的错误消息 | 验证失败 → 结构化反馈 → Agent 原地修复 → 循环 |

> [!important] PEV 的核心洞察
> PEV 的价值不在于"多了一步验证"，而在于==将验证前置到规划和执行阶段==——不是等 Agent 做完了再检查，而是在它做的每一步都检查。这大幅降低了返工成本和 Token 浪费。

### 3.4 OTA vs PPAF：微观与宏观的互补

| 模型 | 分析目的 | 核心差异 |
|------|---------|---------|
| **PPAF** | 理论解构——将 Harness 解构为四个维度 | ==强调闭环的完整性==——必须有反思环节，否则 Agent 只会"执行但不会改进" |
| **OTA**（Observe-Think-Act） | 工程实现——描述 Agent 的单轮运行逻辑 | ==强调循环的实时性==——观察→思考→行动是一个紧凑的实时循环 |

> [!important] 深层含义
> PPAF 是==宏观视角==——跨越多轮的完整认知闭环。OTA 是==微观视角==——单轮内的实时决策循环。
> ==反思在 OTA 中不是独立阶段，而是隐含在 Observe 中==——上一轮的行动结果作为下一轮的观察输入。
> 生产实现中，Agent 核心循环采用 OTA（简洁、实时），但系统整体设计必须遵循 PPAF（完整、进化）。

### 3.5 战略分析矩阵：认知循环 × 上下文效率

理解不同类型 Agent 的能力边界和工程挑战，可以从两个维度构建战略分析矩阵：

| 象限 | 认知循环 | 上下文效率 | 典型应用 | Harness 需求 |
|------|---------|-----------|---------|-------------|
| **第一象限** | 主动规划与反思 | 高效全自动注入 | 复杂自主 Agent（如 DeepResearch、Claude Code） | 高级 Harness：状态分离、多Agent协作、策略门控 |
| **第二象限** | 主动规划与反思 | 低效人工投喂 | 研究型 Agent、需要人类引导的复杂任务 | 重规划机制、记忆分层、Token转化流水线 |
| **第三象限** | 被动响应 | 高效全自动注入 | 简单自动化工具（如天气查询、翻译） | 基础 Harness：错误分类、重试恢复 |
| **第四象限** | 被动响应 | 低效人工投喂 | 简单聊天机器人 | 最小 Harness：基本重试和超时 |

> [!important] Harness 的成熟度决定跃迁
> Harness 的成熟度直接决定了 Agent 能否从低效、被动的第三/四象限跃迁至高效、主动的第一/二象限。

## 四、设计原则

### 4.1 7 条核心设计原则（来自 OpenHarness/Claude Code 等项目分析）

| 原则 | 含义 | 反面教材 |
|------|------|---------|
| **分类先于处理** | 先诊断错误类型，才选正确策略 | 所有错误一律重试3次 |
| **渐进式降级** | 从温和到激进，逐级响应 | token超限直接清空历史 |
| **静默优先** | 能内部恢复的异常不暴露给用户 | 每次retry都弹toast |
| **预算有限** | 任何自动恢复都必须有上限 | 无限重试直到成功 |
| **可观测** | 静默≠无记录，所有操作必须可审计 | 静默吞掉异常无日志 |
| **最小权限** | 工具权限按需授予，危险操作需审批 | 所有工具默认全权执行 |
| **声明式策略** | 安全规则外置为配置，非硬编码 | 每个安全检查写死在代码里 |

### 4.2 咸鱼六大设计原则（理论框架层面）

与 7 条核心设计原则互补——咸鱼从理论框架层面提炼了六大设计原则：

| 原则 | 含义 | 与7原则的对应 |
|------|------|-----------------|
| **为失败而设计（Design for Failure）** | 将异常和失败视为系统运行的常态而非个例 | ↔ 分类先于处理 + 渐进式降级 + 预算有限 |
| **契约优先（Contract-First）** | 所有交互由明确的、机器可读的契约定义 | ↔ 声明式策略 |
| **默认安全（Secure by Default）** | 安全是设计的出发点，遵循最小权限、零信任和纵深防御 | ↔ 最小权限 |
| **决策与执行分离（Decision vs. Execution）** | 将"决定做什么"与"如何做"在逻辑和物理上解耦 | ↔ [[Plan-and-Execute]] 范式的工程化体现 |
| **万物皆可度量（Everything is Measurable）** | 每个行为、决策、资源消耗都应可度量 | ↔ 可观测 |
| **数据驱动进化（Data-driven Evolution）** | 将每次运行视为学习机会，建立数据采集→标注→回流→更新的闭环 | 新增视角：Harness 不是静态的，而是通过数据反馈持续进化 |

> [!important] 两套原则互补而非矛盾
> 咸鱼六原则提供==理论方向==（"应该追求什么"），七原则提供==实践手段==（"具体怎么做"）。

### 4.3 构建可靠 Agent 的 R.E.S.T. 模型

要让 Agent 从"有趣的玩具"变为"可靠的工具"，必须满足四个核心目标：

| 目标 | 定义 | 关键要求 |
|------|------|---------|
| **R — 可靠性（Reliability）** | 系统在各种预期和非预期输入、环境变化和内部故障时，能持续稳定地完成既定任务 | 失败可恢复、操作幂等性、行为一致性 |
| **E — 效率（Efficiency）** | 在满足功能和可靠性的前提下，系统使用资源的有效性 | 资源可控、低延迟响应、高吞吐量 |
| **S — 安全性（Security）** | 保护系统及数据免受未经授权的访问、使用、泄露或破坏 | 最小权限、沙盒执行、输入/输出过滤 |
| **T — 可观测性（Traceability）** | 系统提供足够数据使开发和运维人员能理解其内部状态、决策过程和行为轨迹 | 全链路追踪、决策可解释、状态可审计 |

> [!important] R.E.S.T. 的真正创新
> R.E.S.T. 的大部分维度可以从传统软件工程找到对应，但==Agent 系统有一个独特挑战==：**模型本身的不确定性**。传统系统可以假设代码逻辑是确定性，但 Agent 系统必须假设"大脑"的输出是概率性的。这使得 R.E.S.T. 的每个维度都比传统系统更难实现。

### 4.4 状态分离原则

| 原则 | 说明 |
|------|------|
| **LLM 是无状态的计算单元（CPU）** | 必须将 LLM 严格视为无状态计算单元，所有需要跨轮次保持一致性的状态存储在 Harness 控制的外部上下文状态管理器或其他持久化引擎中 |

> [!warning] 反模式
> 试图通过 Prompt Engineering 让 LLM 在长对话中自行维护复杂状态，会导致系统行为混乱、不可预测且难以调试。

> [!important] 状态分离是所有其他 Harness 的基础
> 如果你不接受"LLM 是无状态 CPU"这个前提，你就不会投入精力去设计状态管理器、进度追踪器、策略门控。==接受状态分离，是从 Prompt Engineering 心智模型切换到 Harness Engineering 心智模型的关键转折点==。

### 4.5 驾驭层的核心矛盾

| 矛盾 | 驾驭层策略 | Claude Code 实现 |
|------|----------|-----------------|
| **自主 vs 安全** | 最小权限 + 信任递增 | default → auto → bypassPermissions |
| **灵活 vs 可控** | 工具描述定义行为边界 | Zod schema + 权限标记 + 描述文本 |
| **效率 vs 透明** | 流式渲染 + 选择性折叠 | 工具自渲染 + 搜索命令折叠 |
| **记忆 vs 预算** | 智能压缩而非截断 | snip compaction + reactive compact |
| **单一 vs 多元** | 上下文优先级合并 | CLAUDE.md 多层加载 + 驳回覆盖 |

### 4.6 策略门控（Policy Gateway）

除了沙盒本身，Harness 还需要一个位于"规划器 → 执行层"之间的**策略门控**，在每一次行动前做最后的安全与合规检查：

| 检查类型 | 说明 | 对应驾驭层组件 |
|---------|------|--------------|
| **权限检查** | 基于 RBAC/ABAC 判定 Agent 是否有权访问目标资源或执行敏感操作 | [[护栏与钩子系统]]的 PreToolUse 钩子 |
| **敏感数据过滤** | 对参数和返回结果做 PII/密钥检测与脱敏 | 输出流过滤（凭证泄漏检测） |
| **指令注入防御** | 识别潜在恶意 Prompt/命令拼接 | 模式匹配+语义分类器 |
| **审计日志** | 记录每一次"谁在何时尝试做什么、结果如何" | [[护栏与钩子系统]]的 PostToolUse 钩子 |

### 4.7 驾驭层的三个底层思维

> [!important] 思维1：能力层尽可能抽象，编排层尽可能简洁
> 核心Loop不到50行，所有复杂逻辑下沉到子系统。编排层只做"调谁、传什么、拿到什么"，脏活累活全在子系统里。

> [!important] 思维2：静默优先，但静默≠无记录
> 修复、压缩、重试都是静默的——用户看不到，但都有日志、指标、追踪。静默是用户体验，记录是工程质量。

> [!important] 思维3：概率系统嫁接在确定性工程上
> 观测比开发更重要，好架构思路比堆人力调参有效10倍。LLM是概率引擎，Harness用确定性工程兜底概率波动——这是整个驾驭层的哲学根基。

## 五、六大子系统详解

### 子系统 1：错误分类与恢复

核心原则：**不是所有错误都值得重试，但所有错误都需要被分类处理**。

| 错误类型 | 典型示例 | 处理策略 | 进阶策略 |
|---------|---------|---------|---------|
| **瞬态 (Transient)** | 网络超时、连接断开、500服务器错误、响应截断 | 指数退避重试+随机抖动，max 3次 | 提供商切换（OpenAI超时→Anthropic）；请求拆分 |
| **限流 (Throttle)** | 429 Too Many Requests、模型并发限制、Token配额耗尽、账户欠费 | 尊重Retry-After头或递增等待 | — |
| **永久 (Fatal)** | API Key无效、模型不存在、内容违规拒绝、Schema不匹配 | 终止+报告给用户 | — |

> [!tip] 为什么需要"抖动"（Jitter）
> 如果多个客户端在失败后等待相同时间重试，会在同一瞬间再次涌入服务器——这就是**惊群效应（Thundering Herd）**。加随机抖动使重试时间离散分布，避免雪崩。

> [!example] 错误分类器三层匹配实现
> **FailoverReason枚举**涵盖15+种错误类型。
> 三层匹配机制：
> - **Layer1 HTTP状态码**：429→`rate_limit`，402→`billing`，401→`auth`
> - **Layer2 错误消息模式匹配**：含中文模式（如"超过最大长度"适配阿里云DashScope）
> - **Layer3 异常类型名兜底**：按异常类名映射到枚举
>
> 上下文溢出**不重试而是压缩后重试**——调用`compress_with_fallback`先压缩再重新提交。

### 子系统 2：上下文工程

上下文管理不只是防溢出，更是**让 Agent 在有限窗口内保持最优决策能力**。

| 阶段 | 触发条件 | 操作 | 信息损失 | 代表实现 |
|------|---------|------|---------|---------|
| 预警 | 60% 窗口 | 标记冗余区域，准备压缩 | 无 | Claude Code auto-compact |
| 均缩 | 70% 窗口 | LLM生成结构化摘要替换早期消息 | 低 | LangChain ConversationSummaryMemory |
| 截断 | 80% 窗口 | 工具返回只保留摘要，删除原始内容 | 中 | OpenHarness context trim |
| 紧急 | 90% 窗口 | 仅保留 system prompt+最近3轮 | 高 | SWE-agent window reset |

> [!important] 范式转变：从"事后压缩"到"事前精选"
> Parlant 提出**上下文窄化（Context Narrowing）**——根据当前对话轮次的主题，只注入相关的规则、知识和工具描述，从源头控制 token 消耗。

> [!tip] 上下文压缩三步策略
> **Step1 划分保护区**：HEAD区（system prompt + 首轮对话）和TAIL区（最近3轮）不压缩。
> **Step2 修剪MIDDLE区**：对工具输出按类型智能摘要。
> **Step3 LLM生成摘要**：将修剪后的MIDDLE区交给LLM，生成13字段结构化摘要替换原始消息。

> [!important] 三级降级与防抖
> **降级①**：尾部保护从6条减到3条 → **降级②**：删除最早10条工具结果 → **降级③**：只保留system prompt + 最近3轮。
> **防抖机制**：记录最近5次压缩平均节省率，低于10%跳过压缩。
> **触发时机为75%而非90%**——要给completion tokens和压缩本身的LLM调用留25%余量。

> [!tip] Todo状态注入
> 压缩后把未完成的todo注入回messages，防止LLM忘记做到哪了。只注入`pending`/`in_progress`状态。

### 子系统 3：迭代控制

Agent 的无限循环是生产中最常见的"静默杀手"——不会崩溃，但持续消耗 Token 和时间，直到预算耗尽。

| 防护层 | 检测机制 | 触发动作 |
|-------|---------|---------|
| **Layer 1: 硬性预算** | 最大迭代次数(200轮)、最大Token消耗($5/次)、最大执行时间(30min) | 强制终止+输出当前摘要 |
| **Layer 2: 模式检测** | 重复检测(同一工具+同一参数连续N次)、震荡检测(A→B→A→B反复切换)、空转检测(连续N轮无新信息/无副作用) | 注入反思提示/强制re-plan/终止 |
| **Layer 3: 进展评估** | 里程碑机制(每N轮评估"离目标更近了吗")、衰减因子(每轮价值递减)、置信度(LLM自评完成确定性) | 交还控制权给用户+附带进展报告 |

> [!example] 生产实践
> SWE-agent 在代码修复任务中设置 cost limit（$3/实例）。DeerFlow 引入"迭代衰减"——前10轮权重1.0，之后每轮权重递减 0.95^n。

### 子系统 4：工具治理

工具操作**有副作用**且**不可逆**，因此需要完整的治理生命周期。

| 步骤 | 关键动作 | 具体检查项 |
|------|---------|-----------|
| ① 参数校验 | &清洗 | JSON Schema、必填参数、类型转换、注入检测 |
| ② 权限检查 | &审批 | 白名单/黑名单、路径范围限制、危险操作需审批、并发/频率限制 |
| ③ 参数修复 | (容错) | 大小写归一化、camelCase→snake_case、模糊匹配(Levenshtein)、缺失参数填默认值 |
| ④ 沙箱执行 | (隔离) | Docker容器(SWE-agent/OpenHands)、WASM沙箱、文件系统虚拟化 |
| ⑤ 结果验证 | &后处理 | 返回大小限制、敏感信息脱敏、格式标准化 |

> [!tip] 声明式安全规则
> avakill 通过 YAML 定义工具策略，在执行前拦截判定。saifctl 先定义 Agent 能力边界规格（spec），运行时确保不越界——类似形式化验证思路。

> [!example] 工具名修复四策略递进
> - **策略1**：直接小写转换
> - **策略2**：标准化（替换`-`和空格为`_`）
> - **策略3**：候选集交叉组合
> - **策略4**：difflib模糊匹配，相似度≥70%才采纳

### 子系统 5：安全防护

Agent 面临独特且严峻的安全威胁——**间接提示注入**（工具返回的内容本身就可能是攻击载荷）。

| 3个攻击面 | 常见威胁 | 代表防护方案 |
|---------|---------|-------------|
| **输入侧** | 用户直接注入、对抗性prompt绕过安全限制 | Constitutional AI |
| **工具返回侧**（最危险） | 网页/文件/API响应中嵌入"Ignore previous instructions..."等恶意指令 | Anthropic Constitutional AI、模式匹配+语义分类器 |
| **输出/持久化侧** | 泄漏system prompt/凭证、记忆投毒 | Claude Code memory validation、Parlant response filtering |

| 威胁 | 检测方法 | 防护位置 |
|------|---------|---------|
| 间接提示注入 | 模式匹配+语义分类器 | 工具返回后、送入LLM前 |
| 凭证泄漏 | 正则(API Key/Token模式) | 输出流过滤 |
| 路径穿越 | 路径规范化+白名单 | 文件操作工具前 |
| 命令注入 | Shell元字符检测 | Bash工具参数清洗 |

> [!warning] 深层防御原则
> 安全不能只靠一层。业界最佳实践是 **Defense in Depth**——输入、处理、输出三层各设独立检测，任何一层失守，后面的层仍能拦截。

### 子系统 6：可观测性

| 三支柱 | 关键记录项 |
|-------|-----------|
| **Logging** | 每轮决策记录、工具调用入参出参、错误上下文、安全事件审计 |
| **Tracing** | 端到端请求链路、父子Agent关系、工具调用耗时、压缩/降级事件 |
| **Metrics** | Token/轮/成本、成功率/重试率、延迟分布、循环检测触发率 |

追踪维度：`Session → Turn → LLM Call → Tool Call → Sub-Agent`

代表工具：LangSmith / Langfuse / OpenTelemetry / Helicone

## 六、实践架构

### 6.1 五层架构实践（杜学友）

> 核心设计思想只有一句：==把上下文当预算来管理==。分层标准不是"按功能分类"，而是"按何时被读取"——常驻的极小，深的按需加载。

![[07-Attachments/harness-practice/harness-五层架构图.jpg]]

#### 第一层：常驻入口层（CLAUDE.md + CLAUDE.local.md）

放角色、代码偏好、流程触发规则、G1–G8 门禁速查。**`CLAUDE.local.md` 自包含、不依赖全局 `@import`**——新项目5分钟接入。

| 目标 | 效果 |
|------|------|
| 项目流程规范彼此隔离、互不串味 | 主会话常驻上下文压到 ≤8K |

#### 第二层：原子规则层（rules/，7 个）

每个规则单一职责、可被按需引用。本质是把踩过的坑固化成强制约束。

| 规则 | 约束 | 背后的事故 |
|------|------|-----------|
| `build.md` | 禁用 `mvn -am`，只编译单模块 | `-am` 触发全量解析卡死 |
| `branch-hygiene.md` | 提预发前必须先合 master | 多分支集成构建报"找不到符号" |
| `code-search.md` | 禁 Grep 搜 Java 结构、禁 LSP，强制走 kbase | 符号搜索误命中、上下文浪费 |

> 每条规则都是一次事故的墓志铭。坑只踩一次，之后由规则兜底——这是 harness 最朴素也最值钱的复利。

#### 第三层：角色 Agent 层（agents/）

把"全能主会话"拆成职责清晰的流水线。**最终精简为 12 个 agent**（从早期 24 个合并而来）。

| 类别 | 角色 | 为什么不能合并 |
|------|------|--------------|
| 调度+合成 | dispatcher（路由）、orchestrator（合成） | dispatcher 只读 state.json 做路由决策，orchestrator 只读 phases/*.md 做合成确认——职责正交 |
| 三角色评审 | requirement-analyst、tech-architect、quality-guardian | 业务/技术/质量必须独立思考、互不看到对方初稿，才有对抗性 |
| 流程执行 | plan-generator → developer → verifier → deployer → tester | 每个节点的 system prompt 和可用工具完全不同 |

**薄主会话三条铁律**：
1. 主会话只听 dispatcher，禁止自己 Read phases/*.md / evidence.json
2. 职责隔离：dispatcher 只管路由、orchestrator 只管合成、developer 只管编码
3. 上下文 ≤8K

![[07-Attachments/harness-practice/19节点链与门禁.jpg]]

#### 第四层：按需上下文层（context/，10 个）

完整流程详情、Pre-Mortem 模板、对抗辩论模板、证据链规范、TDD/ATDD 指南——只在进入对应阶段时才被 Read。

> 设计原则：上下文不是越大越好的"免费缓冲区"，而是需要精心管理的稀缺资源。每份 context 只含该阶段所需最小集，用完即释放。

理论支撑：
- LLM 注意力呈 U 型分布，中部准确率显著下降（[[上下文腐蚀与注意力衰减]]）
- 声称支持 32K+ 的模型仅半数能在该长度保持可靠性能（NVIDIA RULER）

#### 第五层：执行支撑层（skills/ 22 个 + commands/ 12 个 + evals/）

- **skills/（22 个）**：CLI 和研发工具链封装成 AI 可调用能力
- **commands/（12 个）**：`/init-harness` 一键接入、`/harness-audit` 体检、`/learn` 沉淀规则
- **经验三级进化**：踩坑 → lesson → pattern → instinct → 详见 [[经验三级进化]]

#### 稳定性支点：G1–G8 门禁墙

每个门禁是确定性 Python 函数，检查产物存不存在、编译过不过、单测通没通。任一 gate FAIL → 流程退回 DEVELOPING——**不是"建议"，是"阻断"**。

| 门禁 | 检查什么 | 检查方式 |
|------|---------|---------|
| **G1** 产物存在性 | 关键产物文件是否生成 | 文件路径检查 |
| **G2** 规则命中率 | 必要规则是否被触发 | 预设规则命中率统计 |
| **G3** 编译通过 | 代码能否编译 | **真跑** `mvn compile` |
| **G4** 单测通过 | 单元测试是否通过 | **真跑** `mvn test` |
| **G5** 代码覆盖率 | 测试覆盖是否达标 | 覆盖率阈值检查 |
| **G6** ATDD 通过 | 验收测试是否通过 | **真跑** ATDD 测试 |
| **G7** 接口测试通过 | 接口测试证据是否完整 | 部署预发后接口测试 |
| **G8** 生产上线确认 | 生产环境是否健康 | 目前不强制，人工兜底 |

> 详见 [[G1-G8 门禁墙参考卡]]

> [!important] 核心原则
> ==流程强制执行必须从 LLM 推理中外置到确定性基础设施==。门禁必须是确定性代码，独立于上下文窗口，fail-closed（默认拒绝，只放行显式允许）。

#### 19 节点链 × intent×risk 动态裁剪

完整链路：需求评审→需求确认→方案设计→方案确认→Pre-Mortem→实施计划→验收标准确认→拉变更→建分支→建 worktree→开发→编译→单测→ATDD→证据链→部署预发→接口测试→上线确认→验收报告

| intent × risk | 实际走的节点 |
|--------------|-------------|
| `QUERY / NA` | 0 个——纯查询不该启动流程 |
| `BUG_FIX · LOW` | FAST_PATH：开发→编译→单测→证据→报告 |
| `FEATURE · MEDIUM` | 再加方案设计、对抗辩论、TDD |
| `FEATURE · HIGH` | 19 节点拉满 + ADR |

### 6.2 四阶段演进实践（杜学友）

> 不是设计出来的，是被现实一路逼出来的。==真正推动架构演进的从来不是"想要更好"，而是"现在的做法已经崩了"。==每一阶段的切换都不是优化，是止损。

![[07-Attachments/harness-practice/四阶段演进图.jpg]]

核心转变只有一句话：==从"用更多的字约束 AI"，到"用更好的结构约束 AI"。==

#### 第一阶段：拿来主义

用社区项目的 OpenSpec 规范直接上手——别人总结好的 CLAUDE.md 模板、多 agent 示例、最佳实践合集。

| 效果 | 局限 |
|------|------|
| 从"裸用 AI"跨进"有章法地用 AI" | 通用规范覆盖不了特定开发流程 |

> 触发词：==每次我要写的额外 prompt 比规范本身还长时，就意味着该自己造了。==

#### 第二阶段：重 prompt 约束

把所有流程规矩写进 CLAUDE.md。**三天后崩了**。三个症状：

| 症状 | 机制 |
|------|------|
| **不听话** | 规则太多 → 模型"选择性遵守" |
| **上下文爆炸** | 所有规则常驻窗口 → 留给代码的空间被挤压 |
| **自我矛盾** | 规则间偶尔冲突 → 模型编造折中方案 |

![[07-Attachments/harness-practice/上下文爆炸示意图.jpg]]

> ==核心教训：prompt 约束是说服，不是强制==。模型"理解"了规则 ≠ "遵守"了规则。

#### 第三阶段：减负 + 分层加载

问题的根因：==把"有状态的流程"硬塞进了"无状态的对话窗口"==——本质上用错了工具。

做了两件事：
1. 给 harness 减负：常驻 prompt 砍到 ≤8K，深度内容移到 `context/` 层
2. 整理三层加载链路：常驻入口层 → 原子规则层 → 按需上下文层

![[07-Attachments/harness-practice/减负分层效果.jpg]]

新问题：长程会话中规则虽还在但已被稀释到注意力衰减区。

#### 第四阶段：Agent 调度编排

最后一跳：==不再约束模型"你该怎么做"，而是让不同的 agent 各司其职、互相制衡==。

![[07-Attachments/harness-practice/Agent调度编排.jpg]]

核心设计：dispatcher 作为大脑，只负责"算下一步该谁上场"；其他 agent 各管一段。

==状态外置、决策收敛给 dispatcher，即使单次会话崩了、上下文被压缩了，状态不丢、流程能续==。

#### 踩坑清单

| 坑 | 根因 | 教训 |
|---|------|------|
| headless 四连坑 | 无头环境里每个"理所当然"的东西都是地雷 | 必须显式接管浏览器、工作目录、进程生命周期 |
| 评分假"持平" | 空产物被兜底成 70 分中性值 | ==失败必须响亮==，绝不静默兜底 |
| 子 agent stall | 让它自由探索 270+ 文件直接卡死 | 给精确文件清单，别让它自己找 |
| 云端"找不到符号" | 分支没先合 master | 提预发前强制合主干 |

#### 四阶段与行业演进的对照

| 维度 | 本框架四阶段 | [[Prompt与Context工程（三次进化）]] | 行业趋势 |
|------|------------|--------------------------------|----------|
| 第一阶段 | 拿来主义 | Prompt Engineering | Prompt 基础 |
| 第二阶段 | 重 prompt | Context Engineering | Context 工程 |
| 第三阶段 | 减负分层 | Harness Engineering | Agent 系统设计 |
| 第四阶段 | Agent 编排 | — | 动态 Harness 思维 |

> [!important] 个人实践 vs 行业趋势
> 四阶段是==个人实践的微观路径==，三次进化是==行业趋势的宏观路径==。两者同步但视角不同。

### 6.3 长程 Coding 自动化实践（胡韶山）

#### 两条直觉经验

> [!important] 直觉 1：三环节必不可少
> 一条任务精简到最后必不可少三个环节——==规划、执行、验证==，除此之外的环节可能都是多余的复杂度。

> [!important] 直觉 2：充分必要的信息传递
> ==不足的信息会出现猜测幻觉，多余的信息会误导任务==。

> [!warning] 不要复杂化流程
> 随着模型能力提升，很多复杂化会变得过时。凭直觉经验觉得==必不可少==的东西，才去构建。

#### 上下文甜区与上下文焦虑

> [!quote] 上下文甜区
> 让每个 Agent 运行在==上下文窗口的 40%== 左右。消除"上下文焦虑"——当 Agent 认为自己接近上下文极限时，会==过早地开始收尾==，牺牲质量以求"完成"。

每个子功能任务都会启动全新的 Developer 和 Evaluator，一个复杂任务拆解成 5-10 个 Task Sprint。

#### Rubric 评测框架（L1-L4）

源自 Scale AI 论文 *Rubrics as Rewards* (2025)。

**Rubric 四原则**：

| 原则 | 含义 |
|------|------|
| **基于专家知识** | Rubric 必须反映领域专家的判断标准 |
| **全面覆盖** | 涵盖多个维度，包括==负向标准（Pitfall）== |
| **分级权重** | Essential(1) → Important(0.7) → Optional(0.3) → ==Pitfall(0.9)== |
| **自包含可评判** | 每条 Rubric 独立可操作，==不依赖外部上下文== |

**四层评测体系**：

$$
总分 = L1(功能正确性) \times 0.40 + L2(健壮性) \times 0.25 + L3(UI呈现) \times 0.20 + L4(交互体验) \times 0.15
$$

| 层级 | 权重 | 关注维度 |
|------|------|---------|
| **L1: 功能正确性** | 0.40 | 数据逻辑、状态管理、条件逻辑、API 集成 |
| **L2: 偏健壮性** | 0.25 | 边界输入、错误处理、并发与时序 |
| **L3: UI 偏呈现** | 0.20 | 页面渲染、布局、组件完整性 |
| **L4: 交互体验** | 0.15 | 操作反馈、加载指示、动画过渡 |

> [!important] L1 的关键区分
> 不能只看界面变了，还要==确认数据真的变了==——同时检查 DOM 状态和底层数据状态，两者交叉验证。

#### 四轮迭代历程

| 迭代 | 改进 | 效果/新问题 |
|------|------|-----------|
| 1 | 流程跑不下来 | 没有显式指定迭代循环 |
| 2 | 增加显式迭代循环 | 交互效果原始，50% 功能仅做了 UI 占位 |
| 3 | 引入 Rubric 四层评测 | 评测从主观变为==结构化可量化== |
| 4 | quality-vision 品质锚定 + Rubric 精细化 | 从"做什么"到"做成什么样" |

**迭代 3 实测数据**：5 小时自动化运行，4 轮迭代（3.35 → 4.11 → 4.52 → 4.66）

**迭代 4 关键改进**：新增 ==`quality-vision.md`== 产物——品质定位（MVP/精打磨/生产级）、参考锚点（正面+反面截图）、设计意图（风格关键词、配色方向）。

#### Skills 开发 Harness 实践

**三角色信息隔离**：

| 角色 | 职责 | 可见信息 |
|------|------|----------|
| **主 Agent** | 调度编排 | 全局信息 |
| **执行 Agent** | 装配 Skill 并以真实 Agent 身份处理请求 | SKILL.md + 用户请求（==不可见评测标准==） |
| **评估 Agent** | 基于 Rubric 独立判分 | 评测标准 + 执行结果（==不可见执行者内部状态==） |

> [!important] 避免自出题、自执行、自判卷
> 三者之间==严格信息隔离==——执行者不可见评测标准，从根本上避免分数虚高。

#### 实测数据

| 维度 | 数据 |
|------|------|
| **全流程时长** | 7 小时全自动执行 |
| **Token 消耗** | 2.2 亿 Token（2.1 亿命中 cache） |
| **迭代轮次** | 4-7 轮评估迭代 |
| **人类参与** | 除最初目标澄清，后续过程==人类没有一点参与== |

> [!important] 模型能力到达水位后，Harness 占非常大的权重
> 侧面说明：模型能力到达一定水位后，Agent 的效果发挥，==harness 会占非常大的权重==。

### 6.4 杜学友实践的核心判断与金句

- 主会话的上下文是最贵的资源，不是免费的草稿纸
- 主会话应退化为纯执行器，只听 dispatcher 指令
- 流程强制执行必须从 LLM 推理中外置到确定性基础设施
- 评测平台是评估者，不是执行者
- ==宁要可复现的"粗糙分"，不要会漂移的"精准分"==
- ==失败必须响亮，绝不静默兜底==

> 每条规则都是一次事故的墓志铭。
> 真正需要警惕的不是"agent 多"，而是"agent 间耦合多"。
> prompt 约束是说服，不是强制——你无法用更多的字来对抗概率性的遗忘。

### 6.5 关键研究引用

| 引用 | 核心发现 |
|------|---------|
| VILA-Lab Claude Code 逆向 | 记忆完全基于文件系统，上下文管理靠 5 层渐进式压缩管线 |
| Devin CPO (Latent Space) | 记忆检索到正确记忆——"尚未解决" |
| Stanford "Lost in the Middle" (TACL 2024) | LLM 注意力呈 U 型分布，中部准确率显著下降 |
| NVIDIA RULER | 声称支持 32K+ 的模型仅半数能在该长度保持可靠性能 |
| arxiv 2605.29682 | 验证反馈质量 R²=0.94～0.99，远超 token 消耗 R²=0.33～0.42 |
| VikingMem (VLDB 2026) | 更少 Token 留存 + 更智能组织 > 全量保留 |

## 七、行业实践案例

### 7.1 OpenAI 百万行代码实验

OpenAI 内部的一支 3-7 人小团队，在短短五个月内，让 AI 生成了将近 100 万行生产级别的代码。据称全程，没有一个工程师亲手写过一行业务逻辑代码。

#### 实验数据

| 维度 | 数据 |
|------|------|
| **时间** | 5 个月 |
| **团队规模** | 3-7 人 |
| **AI 生成代码量** | 近 100 万行（生产级） |
| **效率对比** | 约为纯人工的 10 倍 |

#### 转折点：真正的瓶颈

实验初期，Agent 频繁跑偏、反复犯同类错误。团队意识到，**真正的瓶颈不在于模型能力，而在于 Harness 设计**。

#### 三大 Harness 策略

**策略一：上下文治理（Context Governance）**
- 将 `agent.md` 压缩至百行以内，仅保留索引和分类
- 当 Agent 需要特定规范时，系统动态加载对应子文档
- 强制将散落各处的决策记录全部迁移至代码仓库

**策略二：验证闭环（Verification Loop）**
- Chrome DevTools 集成：AI 可自行截图、模拟操作
- 可观测性工具：每个 worktree 都有一套本地日志、指标和链路追踪
- 强制 Lint + 自动化测试：报错信息自动反馈给 AI，要求原地修复

**策略三：技术债清理（Tech Debt Cleanup）**
- 设置后台运行 Codex 任务，定期扫描代码库
- 自动修复偏离规范的代码和过时文档

#### OpenAI 团队的四个核心理念

1. **仓库就是大脑**：仓库是 Agent 唯一的知识来源，隐性知识必须显性化推入仓库
2. **给 Agent 写的代码**：代码不仅要对人类可读，更要对 Agent 可读（**application legibility**）
3. **自主性分级**：从简单任务开始逐步提升自主权限，自主性提升必须建立在约束系统成熟基础上
4. **合并哲学**：工程师做「审查」而非「修改」。如果一个 PR 需大量修改才能合并，反思 Harness 哪里出了问题

> [!quote] Ryan Lopopolo（OpenAI 核心工程师）
> **Agent 不难，Harness 才难。**

### 7.2 Anthropic F-Harness

#### AI 的"自恋问题"

Anthropic 在尝试克隆 Claude.ai 复杂界面的实验中发现，单 Agent 模式下存在三大致命缺陷：

1. **上下文耗尽**：任务量过大，Agent 在中途耗尽上下文
2. **虚假声明完成**：功能只完成了一半就宣称"已全部完成"
3. **自评过度乐观**：让 Agent 自评输出质量，结果是惊人的过高评价

#### F-Harness 三角色分工

| 角色 | 职责 | 解决的核心问题 |
|------|------|---------------|
| **Planner（规划者）** | 将模糊需求拆解为精细的功能列表 | 任务量过大导致中途迷失 |
| **Generator（生成者）** | 按功能列表逐项执行 | 稳扎稳打，避免跳步或遗漏 |
| **Evaluator（评估者）** | 独立第三方审核 Agent | AI 的自评偏见 |

#### 代价对比

| 维度 | 单 Agent 模式 | F-Harness 三 Agent 模式 |
|------|--------------|------------------------|
| **耗时** | 约 20 分钟 | 约 6 小时 |
| **成本** | 约 $9 | 约 $200 |
| **输出质量** | 逻辑残缺，勉强可用 | 生产环境级别，逻辑完整 |

**核心结论**：当任务的复杂度超过单 Agent 的可靠性边界，多 Agent 协作的 Harness 是唯一可行的工程解法。

### 7.3 Stripe Minions：每周 1300 个 PR

Stripe 内部的「Minions」系统：**每周合并 1300+ 个 PR，全部无人值守 Agent 完成**。

核心架构——**Blueprint 编排系统**：
- **确定性节点**（运行 linter、推送更改）按固定路径执行
- **Agentic 节点**（实现功能、修复 CI 失败）让 LLM 行使判断

关键限制：
- **CI 最多两轮**：跑两次还不过→交人类
- **Toolshed** 挂载约 500 个 MCP 工具，但每个 Agent 只给精选子集。==更多工具 ≠ 更好表现==

> [!quote] Stripe 工程团队总结
> 成功取决于可靠的开发者环境、测试基础设施和反馈循环，==跟模型选择关系不大==。

### 7.4 Cursor Self-Driving：每小时 1000 个 commit

约每小时 1000 个 commit，一周超过 **1000 万次工具调用**。

经历的四版迭代：
1. **单 Agent** → 复杂任务扛不住
2. **多 Agent 共享状态文件** → 锁竞争严重
3. **结构化角色分工** → 太僵硬
4. **持续执行器** → 角色过载导致诡异行为

最终版本：**递归 Planner-Worker 模型**。

> [!warning] 关键发现
> ==差的初始指令会在数百个 Agent 间被放大==。

### 7.5 Peter Steinberger：一个人的军队

2026 年 1 月单月产出 **6600+ 个 commit**，同时运行 5-10 个 Agent。OpenClaw 项目 4 个月内 18 万 stars。

激进做法：**不逐行审查代码**——Code Review 变成「Prompt Review」。

> ==喜欢算法谜题的工程师反而很难适应 Agent 工作流，产品导向的工程师适应更快。==

## 八、路线之争与衰变定律

### 8.1 数据对比：壳比模型值钱

同一模型，只改 Harness，性能差异惊人：

| 来源 | 变量 | 效果 |
|------|------|------|
| **Nate B Jones** | 同一模型，只改 Harness | 编程基准成功率 42% → 78%（≈换一代模型） |
| **LangChain** | gpt-5.2-codex，只改 Harness | Terminal Bench 2.0: 52.8% → 66.5%，排名从 #30+ → 前 5 |
| **Pi Research** | 15 个不同 LLM，只改 Harness | 一个下午提升所有 LLM 编程能力 |
| **Vercel** | 工具数 15 → 2 | 准确率 80% → 100% |
| **Terminal Bench 2.0** | Claude Opus 4.6，换 Harness | 排名 #33 → #5（差 28 位） |

> [!important] 模型对特定 Harness 可能过拟合
> Claude Opus 4.6 在 Claude Code Harness 上排 #33，换不同 Harness 冲到 #5。==在 A 环境下如鱼得水的模型，换到 B 环境可能就水土不服==。

### 8.2 拐杖论（Bitter Lesson 阵营）

Latent Space 将反对观点归为「Bitter Lesson 阵营」，出自 Rich Sutton 的 AI 随笔：==别在工程花活上下太多功夫，算力的增长终究会碾平一切==。

**Noam Brown（OpenAI）**：
> Harness 就像一根拐杖，我们终将能够超越它。

历史支撑：推理模型出现前，开发者搭建了大量 Agentic 系统模拟推理能力。推理模型一出，这些基础设施一夜之间就不需要了。

预判：OpenAI 将走向==单一统一模型==的未来——不应该需要在模型上面再加一个路由器。

给开发者建议：==别花六个月搭建一个可能六个月后就被淘汰的东西==。

### 8.3 METR 数据的质疑

4 位活跃维护者审查 296 个 AI 生成 PR：
- **维护者合并率 ≈ 自动评分通过率的一半**
- Claude Sonnet 4.5：自动评分对应 ~50 分钟任务范围，维护者实际合并对应仅 ~8 分钟 → **7 倍能力高估**
- Claude Code 和 Codex 并不比基础脚手架表现更好

### 8.4 路线之争：Big Model vs Big Harness

#### Big Model 阵营

**核心观点**：模型能力的增长才是主旋律，Harness 只是权宜之计。

Claude Code 团队 Boris Cherny & Cat Wu：
> 所有的秘密武器都在模型本身。我们追求的是==最薄的那层包装==。

#### Big Harness 阵营

**核心观点**：模型是引擎，Harness 是方向盘和刹车。

Jerry Liu（LlamaIndex 创始人）：
> Model Harness 就是一切。

数据支撑：开发者 60% 工作使用 AI，但完全委托的只有 0-20%。Cursor $50B 估值印证 Harness 方向价值。

### 8.5 护栏悖论

**车速越快，护栏越重要。**

| 交通方式 | 速度 | 护栏需求 |
|---------|------|---------|
| 自行车道 | 30 km/h | 无护栏 |
| 高速公路 | 120 km/h | 标配护栏 |
| 磁悬浮列车 | 300 km/h | 全封闭轨道 |

> [!important] 护栏悖论的核心洞察
> ==约束解空间反而让 Agent 更有生产力==。Cursor 团队发现：告诉 Agent「不要留 TODO」比告诉它「完成所有实现」效果更好。

Noam Brown 说得对：很多脚手架会随模型进化被淘汰。但==架构约束、反馈循环、熵管理不会消失，只会换形态==。

Philipp Schmid 三词建议：**Start Simple. Build to Delete.**

### 8.6 Harness 衰变定律

> 模型能力越强，所需的 Harness 越简单。

Anthropic 的研究者发现了一条反直觉的规律：

- Claude 3.0 时代需要极严格的 Harness 约束
- 升级到 Claude 3.5 后，许多 Harness 规则自然变得不再必要

**两层深意**：
1. **Harness 是当下的现实答案**——在模型能力尚未完美的今天不可或缺
2. **Harness 可能是一项过渡性技术**——未来会被模型能力自然吸收

> [!important] 衰变定律的深层含义
> 衰变不是单向的"越来越少"，而是==有进有出的动态过程==——旧规则退出，新规则进入。总量未必减少，但==组成会持续变化==。

### 8.7 动态 Harness 思维

这是==最难培养、也是最有价值的能力==——随模型能力变化而动态调整 Harness 的"厚度"。

#### 约束的本质区分

| 约束类型 | 定义 | 生命周期 | 典型示例 |
|-----------|------|---------|---------|
| **能力补偿型约束** | 因为模型目前做不到而必须人工补的 | ==会被模型进化淘汰== → 衰变 | 严格的思维链提示、硬编码参数校验、强制分步执行 |
| **业务逻辑型约束** | 即使模型能力完美也必须存在的 | ==长期有效== → 不衰变 | 权限边界、合规要求、数据隐私规则、审计追踪 |

#### 模型进化后的简化策略

| 简化策略 | 说明 | 风险控制 |
|---------|------|---------|
| **约束降级** | 从硬约束 → 软约束 → 提示建议 → 移除 | 每次降级后运行回归测试 |
| **验证自动化** | 从人工验证 → 半自动验证 → 全自动验证 | 保留最后一道人工审核 |
| **编排简化** | 从刚性 Workflow → 弱引导编排 → 自主 Agent | 渐进式，每步验证输出质量 |
| **上下文精简** | 从详细注入 → 紧要注入 → 按需检索 | 检索质量指标作为精简依据 |

## 九、深度解析（哲学层）

> 对咸鱼《万字干货：理解 Harness Engineering》的十层深度解析——穿透表层框架，抵达思想内核。

### 9.1 核心哲学："从有序驾驭无序"

- **LLM 的本质是无序的**：基于 Transformer 的自回归架构，原始输出本质上是随机且无序的——它生成的是"最可能的续写"，而非"最正确的答案"
- **Harness 的本质是有序的**：通过工程化手段为无序的算力注入确定性
- **两者的结合才是生产力**：无序的创造力 + 有序的可靠性 = 可规模化的生产力

> [!important] 深层洞察
> 这不是简单的"加约束"——而是在==保留模型的创造力（无序的优势）==的同时，==为其铺设一条可靠的生产轨道（有序的保障）==。Harness 的工程艺术，就在于找到创造力和可靠性的==平衡点==。

### 9.2 "超级实习生"隐喻的深层含义

| 隐喻层 | 含义 | 对工程实践的要求 |
|--------|------|----------------|
| **超级** | 能力极强 | 不要低估 Agent 的能力上限 |
| **实习生** | 经验不足，判断力不稳定 | 不要高估 Agent 的可靠性 |
| **需要管理** | 不能放任，也不能窒息 | 管理的尺度要动态调整——这就是衰变定律的来源 |

### 9.3 三个关键悖论

#### 悖论 1：束缚即释放

> "这套'缰绳'的终极目的，从来不是束缚，而是==为了更安全、更彻底地释放==。"

没有刹车的汽车跑不快——不是因为刹车限制了速度，而是因为没有刹车时驾驶员不敢全力加速。同理，没有 Harness 的 Agent 无法被信任执行复杂任务。

#### 悖论 2：临时即永恒

Harness 既是"当下的现实答案"又是"过渡性技术"。这不是矛盾，而是==时间的辩证法==——在任何一个历史节点都需要当前最合适的 Harness，但也必须清醒地知道今天的 Harness 在未来会被淘汰。

> [!important] 历史类比
> 这与软件工程中的"技术债"管理如出一辙：你永远有技术债（临时方案），但你通过持续重构来管理它。Harness 就是对 AI 系统的"技术债管理"。

#### 悖论 3：软约束必须被硬约束兜底

| 约束类型 | 机制 | 失效模式 |
|---------|------|---------|
| **软约束** | Prompt 规则、角色设定 | 模型可能忽略或误解 |
| **硬约束** | 权限系统、沙盒隔离、熔断 | 可能过度限制灵活性 |

==软约束告诉模型"应该怎么做"，硬约束确保模型"不能怎么做"==。两者缺一不可。

### 9.4 概念谱系——每个框架的思想源头

| Harness 概念 | 思想源头 | 深层含义 |
|-------------|---------|---------|
| **REPL 容器** | Lisp/函数式编程传统 | 代码即数据——LLM 的输出既是"数据"也可能是"代码指令"（Function Call），Harness 核心职责之一就是判断：这次输出是纯数据还是代码指令 |
| **控制平面/数据平面** | 云原生/网络工程传统（SDN、Kubernetes） | Agent 系统面临的挑战与分布式系统惊人相似：不确定性、需要动态调度、需要资源隔离 |
| **状态分离原则** | 函数式编程/数据库传统 | LLM 不是纯函数（温度引入随机性），但核心不是要求 LLM 是纯函数，而是==要求 LLM 不承担状态管理职责== |
| **R.E.S.T. 模型** | 传统软件工程（CAP/SOLID） | Agent 系统有一个独特挑战：**模型本身的不确定性**——需要在不确定性上构建可靠性 |

### 9.5 "无限→有限"桥梁——Harness 的根本挑战

| 子系统 | 如何服务于"无限→有限"转化 |
|-------|------------------------|
| **上下文管理** | 直接将无限状态压缩为有限 Prompt |
| **Function Calling** | 将 LLM 的无限输出空间收窄为有限的结构化调用 |
| **记忆分层** | 将无限的历史信息分层为有限的短期/长期知识 |
| **迭代控制** | 限制 Agent 的无限循环倾向 |

> [!important] 根本矛盾
> Transformer 架构操作的是==有限的、线性的 Token 序列==，而 Agent 需要理解的现实世界状态是==无限的、非线性的信息网络==。真正的解法是==一套智能的信息选择和压缩机制==——这就是 Token 转化流水线存在的根本理由。

> [!tip] FC 是"有限→无限"方向的最脆弱桥梁
> LLM 在有限的 Token 中生成文本，这段文本需要被"反序列化"为在无限复杂的现实世界中执行的操作。==两个方向的脆弱点不同，但都需要 Harness 的保障==。

## 十、实践路线图与配置杠杆

### 10.1 四步进阶路线图

```
Prompt（基础语言能力） → Context（信息环境设计） → Harness（系统约束架构） → 动态 Harness（持续校准思维）
```

#### 第一步：打牢 Prompt 基础，但不要执着于它

三个必会核心：

| 技能 | 说明 | 实践要点 |
|------|------|---------|
| **思维链引导** | 让模型显式推理而非直接给出答案 | 使用 "Let's think step by step" 或结构化推理框架 |
| **角色设定** | 通过身份与专业领域描述提升输出质量 | 精确定义角色边界、知识域和行为规范 |
| **结构化输出格式** | 指定 JSON、Markdown 等格式 | 从"自由文本"到"可被程序解析的确定性输出" |

> [!important] Prompt 的衰变特征
> 随着模型进化，今天的最优提示明天可能不再必要——这是 Harness 衰变定律在 Prompt 层的直接体现。
> **核心原则：Prompt 是起点，不是终点。** 不要花 80% 的时间追求"最佳提示词"，尽快跃迁到 Context 和 Harness 层。

#### 第二步：系统学习 Context Engineering

四个核心模块：

1. **RAG 系统设计与调优**：语义分块优于固定长度分块；混合检索（dense + BM25）覆盖语义与关键词
2. **上下文窗口管理**：优先注入当前任务指令 > 最近交互 > 相关知识 > 对话历史
3. **记忆系统设计**：短期记忆（对话历史）→长期记忆（跨会话知识）→工作记忆（任务进度）
4. **知识库治理**：单一事实来源、数据溯源、新鲜度管理、质量审计、访问控制

#### 第三步：从系统视角思考 Agent 设计

开始用 Harness 的视角审视 AI 项目——关注==整个系统是否可靠运转==。

四个核心问题：
1. **约束设计**：Agent 在哪些具体地方会跑偏？如何为每个风险点设计针对性约束？
2. **验证机制**：从"声称完成"到"验证完成"——不为 Agent 的善意假设买单
3. **单 Agent vs 多 Agent**：只有超过单 Agent 的可靠性边界时，多 Agent 才是唯一可行的工程解法
4. **可观测性**：及时发现异常

#### 第四步：培养"动态 Harness 思维"

核心问题：
1. "这个约束是因为模型能力不足而存在的，还是因为业务逻辑本身需要它？"
2. "如果下一版模型变强了 20%，我的 Harness 里哪些部分可以被简化？"

### 10.2 七个配置杠杆

| 杠杆 | 说明 | 关键要点 |
|------|------|---------|
| **AGENTS.md / CLAUDE.md** | 仓库根目录 markdown 文件 | ==控制在 60 行以内==，写「目录」别写「百科全书」 |
| **确定性约束** | linter、类型检查、结构化测试 | ==硬约束比软性 prompt 指令更可靠== |
| **工具精简** | 别给 Agent 塞太多工具 | Vercel 15→2，准确率反而升 |
| **Sub-Agent 隔离** | 复杂任务拆成子任务 | 防止噪声累积，可用不同模型 |
| **反馈循环** | Agent 自己跑测试、看截图 | 让 Agent ==自验证产出== |
| **CI 限速** | CI 最多两轮 | 跑两次还不过→交人类 |
| **垃圾回收** | 定期扫描技术债 | 容易忽略但极重要 |

### 10.3 构建者 Checklist

```
□ 1. 定位：我的 Agent 在战略矩阵的哪个象限？需要哪种级别的 Harness？
□ 2. 状态分离：我是否已经将 LLM 视为无状态 CPU？状态管理在哪里？
□ 3. R.E.S.T. 目标：我对可靠性/效率/安全性/可观测性的优先级排序是什么？
□ 4. 约束平衡：软约束（Prompt）和硬约束（策略门控）的覆盖范围？
□ 5. 失败设计：我对每种失败类型是否有分类、降级路径和恢复策略？
□ 6. Token 预算：我是否有 Token 转化流水线？
□ 7. 度量闭环：我是否能从运行数据中驱动 Harness 的持续进化？
□ 8. 动态调节：我是否能识别哪些 Harness 是"临时性"的，哪些是"永久性"的？
```

### 10.4 Cost / Latency Optimization 速查

| 技巧 | 怎么省 | 2026状态 |
|------|--------|---------|
| Prompt caching | 重复 prefix 一次计费、后续 ~90%折扣 | Anthropic/OpenAI/Gemini 全支持 |
| Model routing / cascade | 简单→小model、难→frontier | RouteLLM / OpenRouter |
| Thinking budget | 控制 reasoning model token 上限 | Claude/Gemini API 参数 |
| Semantic caching | 相似 query 共享回答 | GPTCache / Helicone 内建 |
| Batching | 多 query 并行处理 | vLLM / production inference |

## 十一、常见陷阱与规避

| 常见陷阱 | 解决方案 |
|---------|---------|
| 过度约束导致低吞吐量 | 分层约束：关键路径严格，非关键路径宽松 |
| 缺乏人类监督 | 实现分级审批机制 |
| 上下文爆炸 | 使用 RAG + 记忆压缩 |
| 错误累积 | 设置检查点 + 回滚机制 |
| Lint 消息不明确 | 将修复指令嵌入错误消息 |
| 规则文件被 Agent 忽略 | 结合确定性约束（Linter）而非仅依赖指令注入 |
| 信息过载 | 给模型太多无关信息反而让它分心，压缩至核心 |
| 上下文污染 | 历史上下文不断膨胀，污染决策 |
| 缺少闭环 | Agent 完成任务后无测试、审查和回归机制 |
| 系统漂移 | 任务变长后系统开始偏离目标 |

> [!warning] 最重要的认知升级
> 当 Agent 表现不好时，你的第一反应可能是"改 Prompt"。但更多时候，问题出在==错误的层级==：
> - 如果模型不知道该做什么 → Feedforward（AGENTS.md、规范注入）
> - 如果模型做错了但没有被纠正 → Feedback（Lint、测试、验证闭环）
> - 如果模型反复犯同类错误 → Harness（将规则编码为不可绕过的硬约束）

## 十二、三次进化的关系

三者不是替代关系，而是**层层包裹、相互依存的嵌套关系**：

- **[[提示工程]]** 回答：我该跟模型说什么？
- **[[上下文窗口管理|上下文工程]]** 回答：模型在回答时该知道什么？
- **Harness Engineering** 回答：整个 AI 系统该如何可靠地运转？

没有好的 Prompt，Context 注入的信息无法被正确理解；没有好的 Context，Harness 中的 Agent 在信息真空中瞎跑；没有好的 Harness，再好的 Prompt 和 Context 在复杂任务中也会累积错误、最终崩溃。

## 十四、为什么驾驭层是产品

2025-2026 年，AI Agent 的差异化不再是 LLM 本身，而是**驾驭层的质量**。同样的 Claude 模型，没有驾驭层就是一个聊天机器人；有好的驾驭层就是一个能自主编码、安全执行、持续工作的智能体。**驾驭层就是产品。**

Claude Code 的 512K 行代码中，大部分是驾驭层而非模型调用——**驾驭层的复杂度远超模型调用本身**。构建有用的 Agent，90% 的工作在驾驭层，10% 在模型调用。

---

## 十五、新范式下的工程师：Human Steer, Agents Execute

> [!quote] 核心哲学
> **"Human steer, agents execute."** 人类掌舵，Agent 执行。

工程师的核心职责重构为三件事：

1. **定方向（Steering）**：清楚地知道要建什么、为什么建、最终形态是什么
2. **搭架子（Harnessing）**：为 Agent 构建可靠的运行支架——制定规则、提供可信上下文、设计自动化验证和反馈回路
3. **做判别（Decision Making）**：在关键的架构决策点进行人工干预和确认

### 衡量标准的切换

| 过去的衡量标准 | 新的衡量标准 |
|---------------|-------------|
| 每天能写多少行代码 | Harness 能支撑多高的代码产出率（系统杠杆率） |
| 能实现多复杂的业务逻辑 | 能设计多健壮的 Agent 系统 |
| 能处理多难的 Bug | 能构建多完善的自动闭环机制 |
| 个人产出（Individual Output） | 系统杠杆（System Leverage） |

## 十七、本质是管理

Harness Engineering 说的这些——上下文管理、架构约束、反馈循环、定期清理——**不就是管理吗？**

| 管理 | Harness 对应 |
|------|-------------|
| 给新人写 onboarding 文档 | AGENTS.md |
| 定代码规范和架构原则 | linter + 结构化测试 |
| 做 code review 确保质量 | CI/CD 检查 |
| 定期做技术债清理 | 垃圾回收 |
| 给工具做精简和选型 | 工具链管理 |
| 遇到反复出现的问题写进 wiki | 反馈循环 |

> [!important] 核心洞察
> ==AI Agent 越强，就越像一个能力很强但需要管理的员工。==你不会把一个刚入职的天才工程师扔进没有文档、没有规范、没有 CI 的项目里指望他写完美代码。
>
> 未来最稀缺的，可能不是训练模型的人，而是==管理模型的人==。

## 三层工程分工总结

| 层级 | 工程的对象 | 核心问题 | 对应 stage |
|------|----------|---------|---------|
| **1. [[提示工程]]** | 送进 LLM 的**字符串** | 这一次要怎么问？ | Stage 2 |
| **2. [[Context Engineering]]** | 窗口里装的**信息** | 这次该给模型哪些信息？ | Stage 6 |
| **3. 驾驭工程**（本概念） | 模型**外围的执行与控制层** | 整个流程怎么跑起来？ | Stage 7 |

> [!important] 三层正交
> 一次 call 的 RAG app 也在做 Context Engineering（重点是组 context）；50 次 call 但没做 retrieval 的 chatbot，仍然只是在做提示工程。**三层是不同工程位置，不是 call 几次的区别**。

## AGENTS.md：知识库治理的落地方案

OpenAI/Martin Fowler 推荐在项目根目录放置 `AGENTS.md`，采用三层约束模式：

| 约束层级 | 含义 | 典型条目 |
|---------|------|---------|
| **Always（始终执行）** | Agent 必须始终遵守的规则 | 使用 UTC 时区、记录所有操作 |
| **Ask First（询问后再执行）** | 需人类确认才能执行的操作 | 添加新依赖、修改重试间隔 |
| **Never（禁止执行）** | 绝对不允许的操作 | 删除生产数据、未验证发送通知 |

> [!tip] AGENTS.md 的设计哲学
> AGENTS.md 不是"给 Agent 写的说明书"，而是==结构化的前置约束==——将散落在 Prompt 中的规则沉淀为版本受控的工程资产。
