# 批无关性与计算确定性

> 训练过程中，无论 batch 如何切分、无论执行顺序如何，输出结果在 bit 级别完全一致——这是 DeepSeek V4 训练可复现性的基石。

## 批无关性（Batch-Invariant）

无论 batch 如何切分，某个 token 落在哪个 batch，它的输出结果在 **bit 级别完全一致**。

> **比喻**：买奶茶——无论店员做奶茶的顺序如何、朋友有几个人，我的那杯必须是精准 50% 糖，不能是 51% 或 49%。店员必须用同一套严格的固定工序做每一杯，不能因为订单多就开"批量快捷模式"走捷径。

### 具体措施

**注意力计算**：
- 单条序列全程在单个 SM 内完成注意力计算（保证满载计算波的高吞吐）
- 多 SM 协同处理单序列的 kernel 计算顺序**完全对齐**
- 同一条 query 无论走哪个 kernel → bit 级完全一致

**矩阵乘法**：
- 通用 GEMM 库 cuBLAS 是动态调整乘法计算顺序和切分模式来追求性能最优的
- 同一个 GEMM 在不同 batch 下可能走不同路径 → 天然批相关
- 解决方案：使用 DeepSeek 开源的 **DeepGEMM** 矩阵乘法库 → 确保矩阵乘法也是批无关的

## 计算确定性（Determinism）

浮点加法不满足结合律 $(a+b)+c \neq a+(b+c)$，同一个数学等价的计算因执行顺序不同产出不同结果。

> **比喻**：同一杯"50% 糖"奶茶——按"糖浆→牛奶→冰块"顺序调 vs 按"冰块→糖浆→牛奶"顺序调，用料完全一样，但尝起来糖度微妙不同。

### V4 的解法

反向传播中三个典型的"多对一"累加场景，**强制累加顺序固定**：

| 累加场景 | 说明 |
|---------|------|
| 注意力反向传播 | 大量 token 的梯度汇聚 |
| [[MoE架构]] 反向传播 | 多个专家的梯度汇聚 |
| [[mHC（多流约束残差连接）]] split-k 归约 | 多个流的归约累加 |

> [!important] 关键区分
> **"执行顺序"依然允许乱序**（GPU 可以乱序执行以提高并行度），但**"加法的括号结构"必须钉死**（即数学上的累加路径必须固定）。这样既保性能又保确定性。

## 为什么重要

- **训练可复现**：相同数据 + 相同参数 → 完全相同的训练结果，便于 debug 和实验对比
- **部署一致性**：推理结果不随 batch 大小变化，生产环境稳定可靠
- **跨硬件可验证**：不同硬件上的结果可以 bit 级比对
