# 工程设计与安全

> AI Agent 的生产级运行需要两大支柱：**工程设计哲学**（如何做出更好的工程决策）和**安全防护模式**（如何确保 Agent 可控、安全、持续工作）。从构建者信条到代码-文档同步、工具描述、输出路由、重试恢复、护栏钩子与流式渲染——七个模块构成了驾驭层的完整实践体系。

---

## Agent 安全威胁模型

Agent面临独特且严峻的安全威胁——==间接提示注入==（Indirect Prompt Injection）。与传统Web安全不同，攻击面在于Agent处理的内容本身就可能是攻击载荷。

### 三个攻击面

| 攻击面 | 威胁类型 | 防护位置 | 代表方案 |
|--------|---------|---------|---------|
| **① 输入侧** | 用户直接注入（直接 Prompt Injection）、对抗性 prompt 绕过安全限制 | 用户消息处理前 | Anthropic Constitutional AI |
| **② 工具返回侧（最危险）** | 网页内容嵌入"Ignore previous instructions..."、文件内容嵌入隐藏指令、API 响应中的恶意 payload、邮件/消息内容操纵 Agent 行为 | 工具返回后、送入LLM前 | 模式匹配+语义分类器 |
| **③ 输出/持久化侧** | Agent 输出泄漏系统 prompt / 内部标签、凭证信息（API Key/Token）出现在回复中、记忆投毒——写入恶意信息影响后续行为 | 输出流过滤 | OpenHarness output scrubber |

### 策略门控（Policy Gateway）

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

除了沙盒本身，Harness 还需要一个位于**"规划器 → 执行层"之间**的策略门控，在每一次行动前做最后的安全与合规检查：

| 检查类型 | 说明 |
|---------|------|
| **权限检查** | 基于 RBAC/ABAC 判定 Agent 是否有权访问目标资源或执行敏感操作 |
| **敏感数据过滤** | 对参数和返回结果做 PII/密钥检测与脱敏 |
| **指令注入防御** | 识别潜在恶意 Prompt/命令拼接，禁止危险模式进入执行层 |
| **审计日志** | 记录每一次"谁在何时尝试做什么、结果如何"，便于事后追溯和合规审计 |

> [!note] 策略门控与护栏钩子的关系
> 策略门控是理论抽象——强调"规划与执行之间必须有安全检查层"。[[护栏与钩子系统]]的 PreToolUse/PostToolUse 钩子是策略门控的具体实现机制。两者互补：策略门控定义了**该检查什么**，钩子提供了**如何检查**。

### 沙盒执行框架

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

为了让 Agent 可以"动手做事"而不破坏系统，需要给 Agent 一个安全环境让它独立运行。沙盒按隔离强度分为四级：

| 级别 | 隔离方式 | 特点 | 适用场景 |
|------|---------|------|---------|
| **Level 1 进程级隔离** | chroot / Linux namespaces / seccomp-bpf | 启动快但仍共享内核 | 可信内部工具 |
| **Level 2 容器级隔离** | Docker / containerd | 生态成熟，大多数工具执行的默认选择 | 通用场景（**推荐默认**） |
| **Level 3 轻量级虚拟机** | Firecracker 等 | 独立虚拟内核 | 多租户或执行不可信代码 |
| **Level 4 完整虚拟机** | KVM / QEMU | 安全性最高但成本最大 | 极少数特殊任务 |

> [!tip] 推荐策略
> 默认采用容器级隔离（Level 2），配合严格的系统安全内核与只读根文件系统；对不可信代码或高敏感数据，引入轻量级虚拟机（Level 3）作为加强版沙盒。

### 资源管理与弹性策略

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

在资源与成本控制方面，关键机制包括：

| 机制 | 说明 |
|------|------|
| **预算与配额** | 为 Token、外部 API 调用次数、CPU 时间分别设置配额，支持"平台/租户/Agent/单任务"多层级配置 |
| **超时控制** | 所有网络请求和工具执行必须设置合理超时，避免因下游卡死拖垮整个 Agent |
| **重试策略** | 对可恢复的临时错误使用带退避的重试，对明显的永久性错误快速失败并上报 |
| **熔断机制** | 某个依赖连续失败时暂时熔断，防止级联故障 |
| **优雅降级** | 关键能力不可用时自动降级为"弱但安全"模式（如从"可执行代码"退回到"只读+建议"） |

### 防护策略矩阵

| 威胁 | 检测方法 | 防护位置 | 代表方案 |
|------|---------|---------|---------|
| 间接提示注入 | 模式匹配+语义分类器 | 工具返回后、送入LLM前 | Constitutional AI |
| 凭证泄漏 | 正则(API Key/Token模式) | 输出流过滤 | OpenHarness output scrubber |
| 路径穿越 | 路径规范化+白名单 | 文件操作工具前 | SWE-agent chroot |
| 命令注入 | Shell元字符检测 | Bash工具参数清洗 | DeerFlow sandbox |
| 记忆投毒 | 内容可信度评估 | Memory write前 | Claude Code memory validation |
| System Prompt泄漏 | 输出标签检测+围栏去除 | 流式输出管线 | Parlant response filtering |

### 声明式安全策略实现

防护策略矩阵中的方案各有实现方式。两个代表性实践值得关注：

**avakill 的 YAML 策略引擎**——通过声明式 YAML 定义工具策略，在执行前拦截判定。策略规则匹配工具名和参数模式，动作包括 deny（阻断）、ask_user（交由用户决定）等。这本质上是[[护栏与钩子系统]]中 PreToolUse 钩子的声明式表达——用配置而非代码定义安全规则：

```yaml
policies:
  - name: "prevent-destructive-file-ops"
    match:
      tool: "bash"
      args_contains: ["rm -rf", "dd if=", "mkfs"]
    action: deny
    message: "Destructive file operation blocked"
  - name: "limit-network-access"
    match:
      tool: "fetch_url"
      args_match:
        url: "^(?!https://(api\\.github\\.com|.*\\.internal)).*"
    action: ask_user
    message: "Agent wants to access external URL: {url}"
```

**saifctl 的规格驱动方式**——先定义 Agent 的能力边界规格（spec），运行时 harness 确保 Agent 行为不越界。类似形式化验证的思路，从"事后检测"转向"事前约束"。

> [!important] 深层防御原则（Defense in Depth）
> 安全不能只靠一层。最佳实践是==输入、处理、输出三层各设独立检测==，任何一层失守，后面的层仍能拦截。这与护栏与钩子的多层关卡思想一致。

## 构建者信条

构建者信条（Builder Ethos）是从 gstack 的 ETHOS.md 中提取的 AI 辅助开发哲学，包含三个核心原则和一个量化框架。它不是技术规范，而是**思考方式**——在 AI 将边际成本推到零的时代，如何做出更好的工程决策。

### 煮沸湖泊（Boil the Lake）

**核心洞察：** 在 AI 辅助下，完整实现的边际成本趋近于零。

面对"完整方案 A（~150 行）vs 90% 方案 B（~80 行）"时，**永远选 A**。

关键区分：
- **湖泊（可煮沸）**：模块级完整实现——AI 可以一次性完成，边际成本几秒
- **海洋（不可煮沸）**：全系统重写——需要多季度，不是 AI 单次能完成的

> "最后 10% 的完整性过去是团队会跳过的，现在只需几秒钟。"

### 构建前搜索（Search Before Building）

面对并发、陌生模式、基础设施等场景，先搜三层知识：

1. **Layer 1 — 经过验证的标准**："{runtime} {thing} built-in"
2. **Layer 2 — 当前流行的实践**："{thing} best practice {current year}"
3. **Layer 3 — 第一性原理**：最 valuable 的是发现**"传统做法为何错误"**的 Eureka Moment

> Prize Layer 3 above all. 当第一性原理推理揭示传统智慧是错的，命名这个"顿悟时刻"并记录它。

### 用户主权（User Sovereignty）

**AI 推荐，用户决策。**引用 Andrej Karpathy 的"钢铁侠战衣"哲学——AI 增强用户而非替代用户。

- 每个决策以**用户影响**收尾："这对你的用户意味着什么"
- AskUserQuestion 格式：上下文 + 问题 + 推荐 + 选项
- 永远不自动合并需要用户判断的变更

### 量化框架

| 任务类型 | 人类团队 | CC+gstack | 压缩比 |
|----------|---------|-----------|--------|
| 样板代码 | 2 天 | 15 min | ~100x |
| 测试编写 | 1 天 | 15 min | ~50x |
| 功能实现 | 1 周 | 30 min | ~30x |
| Bug 修复 + 回归测试 | 4 小时 | 15 min | ~20x |
| 架构设计 | 2 天 | 4 小时 | ~5x |
| 研究探索 | 1 天 | 3 小时 | ~3x |

**关键洞察：** 完整性是廉价的。不要推荐捷径，当完整实现是一个"湖泊"而非"海洋"时。

> "煮沸湖泊"隐喻的精妙之处在于区分了两类"大任务"：湖泊是一个模块的完整实现，AI 能一次性煮沸；海洋是系统级重写，无论多少 AI 也需要多季度。这避免了两种常见错误：(1) 对所有"大任务"都退缩（把湖泊当海洋），(2) 对所有"大任务"都冲上去（把海洋当湖泊）。

> "构建前搜索"的三层知识模型最有价值的是 Layer 3 的 Eureka Moment——发现传统做法错误的时刻。这不是简单的"搜索更快"，而是**改变了搜索的目的**：从"找到怎么做"变成"验证这么做是否正确"。

- **前置知识**：[[AI辅助开发]]、[[效率压缩比]]
- **相似概念**：[[完整性原则]]、[[第一性原理思考]]
- **相反概念**：[[过早优化]]、[[90%方案]]

## 代码-文档同步管道

代码-文档同步管道（Code-Doc Sync Pipeline）是一种从**源代码元数据**自动生成文档的架构模式。核心原则：如果命令存在于代码中，它就出现在文档中；如果代码中没有该命令，它就不可能出现在文档中。**结构性正确性**优于手动维护的准确性。

### 餐厅菜单的隐喻

- **手写菜单**：厨师新增了菜品但忘了更新菜单 → 顾客点了不存在的东西 → 漂移（drift）
- **自动生成菜单**：菜单从厨房库存系统实时生成 → 厨房有什么，菜单就写什么 → 永不漂移

但菜单上的**描述文字**仍然需要厨师亲自写。自动生成只负责**结构性内容**，人类负责**解释性内容**。

### 三阶段管道

```
Template (.tmpl)          ← 人类编写的流程 + 占位符
       ↓
Generator (gen-docs.ts)   ← 读取源代码元数据
       ↓
Document (.md)            ← 提交的、自动生成 + 手写混合
```

### 占位符类型

| 占位符 | 来源 | 性质 | 自动/手写 |
|--------|------|------|----------|
| `{{COMMAND_REFERENCE}}` | `commands.ts` | 结构性（命令列表） | **自动** |
| `{{SNAPSHOT_FLAGS}}` | `snapshot.ts` | 结构性（标志说明） | **自动** |
| `{{PREAMBLE}}` | 序言组合根 | 基础设施 | **自动** |
| 工作流步骤、示例、提示 | `.tmpl` 文件 | 解释性（怎么用） | **手写** |

### 关键约束

1. **文档是生成的，编辑模板**：永远修改 `.tmpl` 而非 `.md`
2. **CI 验证新鲜度**：`gen-docs --dry-run` + `git diff --exit-code`
3. **Git blame 可用**：可追溯命令何时添加
4. **合并冲突原则**：NEVER 直接接受生成的文档任意一方。先解决模板冲突，再重新生成

### 何时使用

| 场景 | 推荐此模式？ | 原因 |
|------|------------|------|
| API 文档（命令/标志/参数） | 强烈推荐 | 结构性内容与代码紧密绑定 |
| 教程/指南 | 不推荐 | 解释性内容为主，代码绑定少 |
| 配置参考 | 推荐 | 配置键与代码中的定义直接对应 |
| 设计哲学文档 | 不推荐 | 纯手写内容，无代码绑定 |

> 此模式最重要的洞察是**区分结构性内容和解释性内容**。结构性内容与代码 1:1 对应，必须自动生成；解释性内容需要人类判断，必须手写。试图用自动生成覆盖解释性内容会产出空洞的文档；试图手写结构性内容会永远漂移。

- **相似概念**：[[API文档自动生成]]、[[Swagger/OpenAPI]]、[[JSDoc]]
- **相反概念**：[[手维护文档]]、[[文档与代码分离]]

## 工具描述工程

工具描述工程是驾驭层的**操作手册**——通过工具的 name、description、inputSchema、权限标记来定义 Agent 可执行的动作及其边界。在 Claude Code 中，工具不是简单的函数签名，而是**三位一体**：schema（输入验证）、权限标记（安全边界）、描述文本（行为塑形）。

> 工具描述不是给程序员看的 API 文档，而是给模型看的"行为引导手册"。每一句话都在塑形模型的行为——"先用 ls 检查目录"不是对人类的建议，而是对模型的行为约束。

### 工具定义的三位一体

```typescript
interface Tool {
  // 行为塑形层
  name: string           // 工具名（模型看到的标识）
  description(): string  // 工具描述（模型理解用途的文本）
  searchHint: string     // 搜索提示（帮助模型找到工具）

  // 安全边界层
  isReadOnly(): boolean      // 只读标记 → 不需要写权限确认
  isDestructive(): boolean   // 破坏性标记 → 需要额外确认
  isConcurrencySafe(): bool  // 并发安全标记 → 可并行执行
  checkPermissions()         // 动态权限检查 → 运行时判断

  // 输入验证层
  inputSchema: ZodSchema    // 输入类型和约束
  strict: boolean           // 严格模式（API 侧验证）

  // 输出控制层
  outputSchema: ZodSchema   // 输出类型约束
  renderToolUseMessage()    // 工具调用时的 UI 渲染
  renderToolResultMessage() // 工具结果的 UI 渲染
}
```

### 描述文本如何塑形行为

**BashTool**：描述中包含 "If your command will create new directories or files, first use this tool to run ls to verify the parent directory exists" → 模型在创建文件前会先检查目录，减少错误。

**FileReadTool**：描述中说明可以读取 images、PDF、Jupyter notebooks → 模型更全面地利用工具能力。

**AgentTool**：描述中包含 "Terrible command-style prompts produce shallow, generic work" 和 "NEVER delegate understanding" → 模型知道要给子代理足够的上下文，不会把理解任务本身交给子代理。

### 权限标记与并发策略

```
isReadOnly = true  →  读操作 → 权限系统默认允许
isReadOnly = false →  写操作 → 需要用户确认
isDestructive = true →  破坏性操作 → 需要额外确认 + 警告

isConcurrencySafe = true  → 并行执行（如 GlobTool + GrepTool）
isConcurrencySafe = false → 串行执行（如 FileWriteTool + BashTool）
```

## 输出解析与路由

输出解析与路由是驾驭层的**信号处理器**——从模型的流式输出中实时检测工具调用请求，解析参数，路由到对应的工具执行器，并决定并行/串行执行策略。核心组件是 `StreamingToolExecutor`。

> 输出解析与路由就是"翻译+调度"。模型输出的是文本流，驾驭层需要：(1) 实时检测"模型想调用工具了"；(2) 解析出工具名和参数；(3) 决定哪些工具可以并行、哪些必须串行；(4) 执行后把结果喂回模型。

### 流式输出解析

```
API 流式事件序列：
  message_start → content_block_start
  │
  ├─ type: "text"    → 文本内容 → 直接渲染到终端
  ├─ type: "thinking" → 思考内容 → 内部处理
  ├─ type: "tool_use" → 工具调用 → 解析+路由
  │   content_block_start → 工具名 + 输入 schema 开始
  │   content_block_delta → 参数片段逐字解析
  │   content_block_stop → 完整工具调用参数可用
  │
  stop_reason 决定下一步：
    "end_turn" → 模型完成
    "tool_use" → 模型请求工具调用 → 继续循环
```

### StreamingToolExecutor 的调度策略

```
只读工具（isConcurrencySafe=true）→ 并行执行
  示例：同时 GlobTool + GrepTool → 加速搜索

写操作工具（isConcurrencySafe=false）→ 串行执行
  示例：FileWriteTool → BashTool → 防止冲突

混合策略：
  Tool_A (read-only) ──→ 并行执行 ──→ 结果 A
  Tool_B (read-only) ──→ 并行执行 ──→ 结果 B
  Tool_C (write)     ──→ 等待 A+B 完成 → 串行执行 → 结果 C
```

### 工具执行的生命周期

```
工具状态流转：
  queued      → 工具调用已检测，等待执行
  executing   → 工具正在执行
  completed   → 工具执行完成，结果可用
  yielded     → 结果已返回给模型

  异常路径：
  cancelled   → 被中止（sibling abort 或用户 abort）
  failed      → 执行失败
```

### Sibling Abort 机制

当一个工具执行失败时，StreamingToolExecutor 取消所有同级工具执行。原因：如果 BashTool 执行失败，后续的 FileEditTool 可能依赖 BashTool 的结果，继续执行会产生不一致的结果。

### 消息类型体系

```
AssistantMessage → 模型输出（文本/思考/工具调用）
UserMessage → 工具结果 + 用户输入
ProgressMessage → 工具进度（实时，不持久化）
SystemMessage → 系统通知（API 错误、压缩边界）
AttachmentMessage → 文件变更通知
ToolUseSummaryMessage → 工具结果摘要
```

> 流式解析是生产级 Agent 的必需能力。驾驭层必须在流式输出中实时检测工具调用——不是等模型"说完"才解析，而是在模型"说到"工具调用时就检测。这要求解析器对每个 content_block_delta 做增量解析。

## 重试与恢复机制

重试与恢复机制是驾驭层的**自动修复系统**——当 API 调用失败、上下文溢出、工具执行出错时，自动触发恢复策略，确保 Agent 持续工作而不中断。

> 重试与恢复就是 Agent 的"自我修复能力"。生产级 Agent 与 demo Agent 的关键区别：demo 遇到错误就停了，生产 Agent 遇到错误就恢复。

### API 重试策略

```
配置：
  ├─ 最大重试次数：10 次
  ├─ 退避策略：指数退避 + jitter（防雷群效应）
  ├─ 529 错误：最多 3 次（前台查询）
  ├─ 持久重试：无值守会话可无限重试 + keep-alive

  示例退避序列：
    第1次重试：1s + jitter
    第2次重试：2s + jitter
    第3次重试：4s + jitter
    ...
    第10次重试：1024s + jitter（约17分钟）
```

### Fast Mode 降级

遇到 429/529 错误时：retryAfter 较短 → 等待后重试（保持 Fast Mode）；retryAfter 较长 → 降级到 Standard Mode 继续工作。关键设计：不放弃——只是降级速度，用户仍能看到结果。

### 上下文溢出恢复策略（5 层逐级升级）

```
级别 1：HISTORY_SNIP（摘要压缩）
  → 删除早期消息，保留关键决策和结果

级别 2：Microcompact（微压缩）
  → 将旧消息摘要为简短描述

级别 3：Context Collapse（上下文折叠）
  → 将粒度化的上下文折叠为摘要

级别 4：Reactive Compact（响应式压缩）
  → 在 prompt-too-long 错误后触发，紧急压缩

级别 5：Max Output Tokens Recovery（输出 token 恢复）
  → 从 8K → 16K → 32K → 64K 逐步增加输出限制
  → 解决"模型想说很多但输出窗口不够"的问题

阻断极限：当上下文超过硬限制 → 阻止 API 调用 → 必须先压缩才能继续
```

### 错误分类体系

```
PROMPT_TOO_LONG → 上下文溢出 → 触发压缩策略
429 (Rate Limit) → 重试 + 可能降级 Fast Mode
529 (Overloaded) → 重试（最多 3 次）
500 (Server Error) → 重试
其他 → 不重试，直接报错
```

### 工具执行失败的恢复

- **单工具失败**：返回错误消息给模型，触发 PostToolUseFailure 钩子，模型看到错误可以调整策略（换工具、改参数）
- **Sibling Abort**：一个工具失败 → 取消所有同级工具，模型看到所有结果/错误 → 重新规划
- **流式 Abort**：用户主动中止 → AbortController → 工具检查 signal.aborted → 立即停止 → 生成 tombstone

> 5 层逐级升级策略是关键创新。不是"压缩一次就够"，而是"如果第一层不够，自动升级到第二层"。另一个洞见：**错误不是终止，而是信号**——工具失败不是让 Agent 停下来，而是让模型看到错误并重新规划。这体现了 ReAct 范式的深层精神：观察 → 思考 → 行动。

- **关联机制**：[[输出解析与路由]]、[[护栏与钩子系统]]

## 护栏与钩子系统

护栏与钩子系统是驾驭层的**安全带**——在 Agent 执行链的关键节点插入拦截机制，实现事前检查（权限）、事后审计（钩子）、人工确认（交互式权限）和自动分类（ML 分类器）。Claude Code 支持 19 种事件钩子，覆盖全生命周期。

> 护栏就是"关卡"——每个关键行动前都要过关卡。钩子就是"记录器"——每个关键事件都被记录和审计。两者组合形成完整的监督体系：事前拦截防止危险行动，事后审计提供行为透明度。

### 19 种事件钩子

| 类别 | 钩子事件 | 驭驶功能 |
|------|---------|---------|
| **工具执行** | `PreToolUse`（工具调用前） | 拦截、修改参数、拒绝 |
| | `PostToolUse`（工具调用后） | 审计结果、通知、日志 |
| | `PostToolUseFailure`（工具调用失败后） | 错误处理、重试策略 |
| **上下文管理** | `PreCompact` / `PostCompact` | 控制和审计压缩策略 |
| **会话生命周期** | `SessionStart` / `SessionEnd` / `Setup` | 初始化、清理、配置验证 |
| **停止与错误** | `Stop` / `StopFailure` | 结果验证、错误分析 |
| **多代理** | `SubagentStart` / `SubagentStop` / `TeammateIdle` | 上下文注入、结果收集、资源调度 |
| **任务管理** | `TaskCreated` / `TaskCompleted` | 依赖追踪、进度追踪 |
| **系统事件** | `ConfigChange` / `CwdChanged` / `FileChanged` | 动态调整、上下文更新、缓存失效 |
| **指令加载** | `InstructionsLoaded` / `UserPromptSubmit` | 合并验证、输入预处理 |
| **权限与交互** | `PermissionRequest` / `Elicitation` / `ElicitationResult` | 自动审批、MCP 认证 |

### 钩子的输出验证

```
事件触发 → 执行用户配置的 Shell 命令 → 输出验证
  ├─ { "decision": "approve" } → 允许继续
  ├─ { "decision": "deny", "reason": "..." } → 拒绝执行
  ├─ { "decision": "block", "reason": "..." } → 阻断并通知
  ├─ { "decision": "modify", "input": {...} } → 修改工具参数
  └─ 无输出/非法输出 → 忽略钩子，继续默认流程
```

**关键设计**：钩子可以**修改工具参数**——用户可以通过钩子在工具执行前自动调整输入。

### 权限检查的三层机制

```
1. 规则引擎层（hasPermissionsToUseTool）
   ├─ allow → 直接允许
   ├─ deny → 直接拒绝
   ├─ ask → 需要进一步判断

2. 分类器层（TRANSCRIPT_CLASSIFIER）
   ├─ 仅在 "ask" 决策后触发
   ├─ 2 秒超时：高置信度 → 自动决定，超时/低置信度 → 降级为用户确认

3. 交互确认层（handleInteractivePermission）
   ├─ 展示工具名称、参数、风险评估
   ├─ 用户选择：允许/拒绝/本次允许/永久允许
   ├─ 永久允许 → 写入规则引擎 → 后续自动允许
```

### Abort 机制

AbortController 穿透整条执行链：QueryEngine → ToolUseContext → Tool.call()。工具执行中检查 abortController.signal.aborted：已 abort → 立即停止，返回 tombstone；未 abort → 继续执行。

> 护栏与钩子的核心洞见：**安全不是一层墙，而是多层关卡**。PreToolUse 拦截事前、PostToolUse 审计事后、分类器自动判断、用户交互最终确认——四层关卡形成纵深防御。特别是"分类器竞赛"设计：2 秒超时机制既利用了分类器的自动化能力，又防止了分类器延迟影响用户体验。

- **关联机制**：[[查询引擎]]、[[子代理系统]]

## 流式渲染控制

流式渲染控制是驾驭层的**仪表盘**——决定用户在 Agent 工作过程中看到什么、何时看到、如何呈现。Claude Code 的渲染是**工具自渲染 + 进度折叠 + Spinner 状态 + 选择性展示**的精密系统。

> 流式渲染就是"导演剪辑"。驾驭层决定：哪些实时显示（进度），哪些折叠（搜索结果），哪些隐藏（内部思考），哪些强调（最终答案）。

### 工具自渲染机制

```
每个工具定义自己的渲染方式：
  renderToolUseMessage(input)     → 工具调用时的展示
  renderToolResultMessage(result) → 工具完成后的展示
  renderToolUseProgressMessage()  → 执行中的实时进度
  renderToolUseQueuedMessage()    → 等待执行时的状态
  isResultTruncated()             → 结果是否截断（可点击展开）
  isSearchOrReadCommand()         → 是否是搜索/读命令（可折叠）
```

### 折叠与选择性展示

```
搜索类命令：默认折叠 → 用户看到"搜索完成，找到 23 条结果" → 可展开查看完整结果
写入类命令：默认展示 → 用户看到"文件已修改"的摘要 → 包含 diff
Bash 执行：长输出自动截断 → 只显示开头和结尾 → 可展开
静默命令（isSilentBashCommand=true）：不显示输出 → 信息已注入上下文
```

### Spinner 状态控制

```
'requesting' → API 请求中 → 等待动画 + "Thinking..."
'responding' → 模型正在生成 → 流式文本输出
Compact/Reactive Compact → "Compressing context..." + Spinner
```

### "看到过程"的设计哲学

Claude Code 的渲染哲学：让用户看到过程，但不过度展示。

- 实时知道 Agent 在做什么（Spinner + 进度）
- 关键操作透明展示（写入、删除、权限请求）
- 大量信息选择性折叠（搜索、长输出）
- 最终答案优先展示

与 ChatGPT 的"隐藏过程"哲学形成对比：ChatGPT 只展示最终答案（黑箱），Claude Code 展示关键步骤（灰箱）。编码任务需要透明性——用户需要知道 Agent 看了什么文件、执行了什么命令，才能判断 Agent 是否在正确的轨道上。

> 流式渲染控制的核心洞见：**透明度是可控的**。不是"要么全展示，要么全隐藏"——Claude Code 用工具自渲染、折叠、截断、静默等策略实现了**分级透明**。编码场景需要透明度但不需要所有信息，分级透明是"灰箱"设计——介于黑箱和白箱之间，是最实用的折衷。

## 七模块之间的关系

七个模块构成了驾驭层的完整闭环：

```
构建者信条 → 提供决策哲学（"怎么做"的价值观）
    ↓
代码-文档同步 → 确保知识与代码的一致性（信息源可靠）
    ↓
工具描述工程 → 定义 Agent 可执行的动作边界（行为塑形）
    ↓
输出解析与路由 → 检测工具调用、调度执行（信号处理）
    ↓
护栏与钩子系统 → 关键节点拦截与审计（安全保障）
    ↓
重试与恢复机制 → 自动修复错误与溢出（韧性保障）
    ↓
流式渲染控制 → 控制用户看到什么（透明度控制）
```

每个模块既独立运作，又与其他模块紧密耦合。构建者信条为所有决策提供价值观锚点；代码-文档同步确保 Agent 的信息源可靠；工具描述定义行为边界后，输出解析才能正确路由；护栏在执行前后拦截，重试在失败后修复；流式渲染则将所有模块的内部状态翻译成用户可理解的信息。
