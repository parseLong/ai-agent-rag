# 团队知识库共建共享

> 团队知识库是一个==独立 Git 仓库==，不寄生于任何业务项目。它是知识的"单一事实来源"（System of Record），项目 A 沆淀的知识，项目 B 自动受益。

---

## 独立 Git 仓库架构

```
team-knowledge.git                        ← 独立 Git 仓库
├── knowledge-catalog.md                  ← 全景目录（Agent 查询入口）
├── .knowledge-config.yaml                ← 团队配置（成员、冲突策略）
├── team-conventions/                     ← Layer 0-T: 团队约定
│   ├── coding-standards.md
│   └── commit-conventions.md
├── tech-wiki/                            ← Layer 1: 技术知识
│   ├── catalog.md                        ← 分类清单
│   ├── patterns/TK-PAT-001.md
│   └── anti-patterns/TK-AP-001.md
├── biz-wiki/                             ← Layer 2: 业务知识
│   └── {domain}/
│       ├── catalog.md
│       ├── entities/BK-AD-E001.md
│       └── pitfalls/BK-AD-P001.md
├── project-profiles/                     ← 项目画像
└── contributions/                        ← 贡献暂存区
    ├── pending/
    └── conflicts/
```

### 为什么独立仓库？

| 原因 | 说明 |
|------|------|
| **跨项目共享** | 多个项目连接同一知识仓库，A 沆淀的知识 B 自动受益 |
| **生命周期独立** | 业务项目可能归档，但==知识不应该跟着项目消失== |
| **权限独立** | 知识库的贡献和消费权限可独立于代码仓库管理 |

> [!important] 与 [[5条驭驭工程原则]] 的呼应
> 独立 Git 仓库 = **System of Record** 原则的工程实现——所有真实知识住外部 docs、不住 prompt。`knowledge-catalog.md` = Progressive Disclosure 的入口点。

## 三种团队角色

| 角色 | 权限 | 适用人群 | 对应知识行为 |
|------|------|----------|-------------|
| **maintainer** | 裁决内容冲突、审批 proven 提升、管理成员 | 团队负责人、资深工程师 | 只在矛盾出现时介入 |
| **contributor** | 通过工作流自动贡献（创建/验证/标记矛盾） | 正式团队成员 | ==工作流自动提取知识并推送== |
| **reader** | 只消费知识（查询/注入），不贡献 | 新成员试用期 | 查询但不写入 |

> [!note] 角色与 [[知识成熟度与自动衰减]] 的关系
> maintainer 负责审批 verified→proven 的提升（需要 ≥2 人 + ≥2 项目验证）；contributor 通过工作流自动验证知识（引用即验证）。

## 贡献模式：区块链思想 + Git 实现

借鉴区块链三个核心思想，用 Git 作为实现载体：

| 区块链思想 | AI Team 实现 | 机制 |
|-----------|-------------|------|
| **不可篡改的追加日志** | `log.md` 只追加不修改 | 每条变更记录贡献者、时间、会话哈希 |
| **贡献可溯源** | `evidence.contributors[]` | 类似 Git blame，粒度为==知识条目级== |
| **共识机制** | maturity 多人验证提升 | draft→verified: 1 人; verified→proven: ≥2 人 + ≥2 项目 |

### log.md 示例

```markdown
## [2026-04-09] ingest | [Steven] | 门店履约视图归档 | +1 decision, +2 guideline | #a3f8c2
- 新增 DEC-005: 地图组件选型（腾讯地图 GL JS SDK）
- 新增 GL-012: fitBounds 在 flexbox 布局中的替代方案 (polarity=recommend)

## [2026-04-12] verify | [Alice] | 跨项目验证 | maturity↑ 2 | #c5f0e2
- TK-SB-003 "分页查询延迟关联优化" (verified→proven, 2 projects)
```

## 冲突解决策略

| 冲突类型 | 处理方式 | 是否需要人工 |
|----------|----------|-------------|
| **纯新增**（不同条目） | 自动合并，两条都保留 | ❌ |
| **证据追加**（同条目验证） | 自动合并，evidence 数组合并去重 | ❌ |
| **成熟度提升** | 自动合并 | ❌ |
| **内容矛盾** | 写入 `contributions/conflicts/`，通知 maintainer | ✅ |
| **成熟度冲突**（一升一降） | 保留较低成熟度 + 标记 contradiction | ❌（但标记需后续裁决） |

> [!tip] 设计理念
> ==大多数情况可以自动处理==，只有真正的内容矛盾才需要人工介入。让知识共建过程尽可能低摩擦。
