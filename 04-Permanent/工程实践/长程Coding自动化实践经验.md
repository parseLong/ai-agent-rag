## 两条直觉经验

在 Coding Harness 的实践中，作者提炼出两条至关重要的直觉经验：

> [!important] 直觉 1：三环节必不可少
> 一条任务精简到最后必不可少三个环节——==规划、执行、验证==，除此之外的环节可能都是多余的复杂度。
>
> Anthropic 的实践也按此三环节设置 3 个独立职责的 Agent，"能不协同就不协同"是很好的经验判断依据。

> [!important] 直觉 2：充分必要的信息传递
> ==不足的信息会出现猜测幻觉，多余的信息会误导任务==。
>
> "目标要求信息"每个 Agent 都要传递。但 e2e 评估 Agent 不需要知道开发实现思路——多了反倒干扰评估，类似 QA 同事过多了解开发思路会被带偏，更关注 PRD 才合理。

> [!warning] 不要复杂化流程
> 随着模型能力提升，很多复杂化会变得过时，这条经验已被反复验证。凭直觉经验觉得==必不可少==的东西，才去构建能力——理性分析总是在模型的迭代中被一次次打脸。

## 上下文控制的具体实践

> 已有笔记覆盖了"不要全量注入""单一事实来源"等原则，这里补充==具体实现方案==。

### 上下文甜区与上下文焦虑

> [!quote] 上下文甜区
> 让每个 Agent 运行在==上下文窗口的 40%== 左右。消除"上下文焦虑"——当 Agent 认为自己接近上下文极限时，会==过早地开始收尾==，牺牲质量以求"完成"。

每个 Agent 唤起时拥有全新上下文，每个子功能任务都会启动全新的 Developer 和 Evaluator。一个复杂任务拆解成 5-10 个 Task Sprint，每个任务创建单独的子 Agent 开发测试。

### 知识摘要：工作区的知识地图

> [!note] 不是大而全文档，而是工作区知识摘要
> - **知识地图**：一张地图，指向更深层"事实来源"。高度精炼、长期稳定且具备强制性，成为编码 rule 的一部分，让 AI 每次编码时都注意到，成为==肌肉记忆==
> - **知识目录和索引**：渐进式披露，规范框架知识而非大而全的详细说明书（指导太多等于没有指导）
> - **代码仓库作为单一事实来源**：好的代码即文档，细节知识通过清晰的架构设计、标准化接口定义和直观逻辑流沉淀在代码库

实际实现：`.workspace` 目录下定义全局前后端规范（架构规范、接口规范、编码规范、组件规范等）和项目功能概览 + 关键文件索引。需求相关文档在 `requirements/` 下（需求文档、设计文档、开发计划、测试用例）。

## Rubric 评测框架

> 这是本文==最大的新增概念==——一种结构化、可量化的 Agent 产出评测方案，源自 Scale AI 论文 *Rubrics as Rewards* (2025)。

### Rubric 四原则

| 原则 | 含义 |
|------|------|
| **基于专家知识** | Rubric 必须反映领域专家的判断标准，捕捉正确性所需的关键事实、推理步骤和结论。作为产研，AI Coding 领域我们本身就是专家 |
| **全面覆盖** | 涵盖多个维度：事实准确性、逻辑连贯性、完整性、风格、安全性等，以及==负向标准（Pitfall）==——识别常见或高风险的错误 |
| **分级权重** | Essential(关键, 权重1) → Important(重要, 权重0.7) → Optional(可选, 权重0.3) → ==Pitfall(陷阱, 权重0.9)== |
| **自包含可评判** | 每条 Rubric 独立可操作——无论是人类还是自动化 Judge，都能==不依赖外部上下文==独立判断该条是否满足 |

> [!tip] 核心论题
> 如果 Agent 的产出不能被有效地评估，那么它就没有改进的方向，也就不会去改进。如德鲁克所说："==无法衡量的无法改进==。"真正拉开差距的是增加一个挑剔专业 Evaluator，能够基于真实效果给出客观明晰反馈。

### e2e 测试的四层评测体系

评测划分为 4 层，前两层偏功能，后两层偏表现，但有==交叉验证==关系：

#### L1: 功能正确性（核心层，权重 0.40）

| 子维度 | 关注点 | 验证方式 |
|--------|--------|----------|
| **数据逻辑** | CRUD 是否正确（增删改查后数据状态是否符合预期） | 操作 + 断言自动验证 |
| **状态管理** | 组件间数据流是否正确（props 传递、全局状态同步、表单绑定） | 检查 DOM 状态 + 底层数据交叉验证 |
| **条件逻辑** | 分支判断是否符合需求（不同条件下的行为差异） | 构造不同条件输入验证 |
| **API 集成** | 请求参数、响应渲染、loading/error 状态处理 | 检查请求参数 + 响应数据 + 状态 |

> [!example] L1 Rubric 示例（"带筛选的用户管理列表"）
> | 类别 | 条目 |
> |------|------|
> | Essential | 用户列表是否正确渲染了全部数据（条数一致、字段无缺漏） |
> | Essential | 搜索筛选后列表是否只显示匹配项 |
> | Essential | 新增用户后列表是否自动刷新且包含新记录 |
> | Essential | 删除操作是否真正从数据源移除记录 |
> | Pitfall | 连续快速操作是否导致数据不一致或重复请求 |
> | Pitfall | 空数据/无搜索结果时是否有合理兜底展示 |

> [!important] L1 的关键区分
> 不能只看界面变了，还要==确认数据真的变了==——同时检查 DOM 状态和底层数据状态（通过 API 或直接读取 store），两者交叉验证。

#### L2: 偏健壮性（防御层，权重 0.25）

| 关注点 | 典型测试 |
|--------|----------|
| **边界输入** | 空字符串、超长文本、特殊字符（XSS payload）、负数、零值、极大值 |
| **错误处理** | 网络断开时是否有提示而非静默失败、必填字段空是否阻止提交、API 500 时是否有降级 |
| **并发与时序** | 快速连续点击是否重复提交、异步操作 race condition、组件卸载后 setState |

> [!example] L2 Rubric 示例
> | 类别 | 条目 |
> |------|------|
> | Pitfall | 输入 `<script>alert(1)</script>` 是否会触发 XSS |
> | Pitfall | 表单提交按钮在请求进行中是否禁用（防重复提交） |
> | Important | 必填字段的校验是否在提交前执行并有明确提示 |
> | Important | 网络异常时是否展示用户可理解的错误信息 |

自动化方式：构造异常输入数据 + 模拟网络故障（拦截请求返回 500），然后检查页面行为。

#### L3: UI 偏呈现（表现层，权重 0.20）

页面渲染是否正确、布局是否合理、组件是否完整展示。截图交给 LLM，配合 Rubric 评判视觉效果。

> [!example] L3 Rubric 示例
> | 类别 | 条目 |
> |------|------|
> | Essential | 页面完整渲染，无白屏/布局塌陷/元素缺失 |
> | Pitfall | 无元素重叠遮挡、文字溢出截断 |
> | Important | 布局层次清晰，信息密度合理 |
> | Important | 响应式适配（移动端/桌面端均可正常使用） |
> | Optional | 视觉风格一致，符合设计规范 |

#### L4: 交互体验（体感层，权重 0.15）

> [!example] L4 Rubric 示例
> | 类别 | 条目 |
> |------|------|
> | Important | 关键操作（保存、删除）是否有明确反馈（toast/状态变更） |
> | Important | 加载状态是否有 loading 指示（而非页面冻结） |
> | Optional | 动画过渡是否平滑、操作是否可撤销 |
> | Optional | 键盘导航和无障碍访问是否支持 |

### 四层权重公式

$$
总分 = L1(功能正确性) \times 0.40 + L2(健壮性) \times 0.25 + L3(UI呈现) \times 0.20 + L4(交互体验) \times 0.15
$$

> [!tip] 权重可调
> 权重根据产品在不同视角的关注度分配。如果对交互创新型要求高，希望 AI 更多发挥，可以增加 UI 交互权重。

## 四轮迭代历程

### 迭代 1：流程跑不下来

| 问题 | 原因 |
|------|------|
| 一轮就结束 | 没有显式指定一定要执行反馈迭代循环 |
| 遇到环境问题放弃评估 | 环境操作问题导致评估中断 |

### 迭代 2：多轮迭代但产出质量差

| 改进 | 效果 |
|------|------|
| 主 Agent 增加显式指定开启迭代循环 | 复杂任务维持 2 小时长时运行 |
| Plan 测试用例阶段增加 AI 主动问询用户补充信息 | 环境准备更充分 |

| 新问题 | 原因 |
|------|------|
| 交互效果不令人满意，UI 交互原始 | 没有结构化评测标准，AI 缺少可量化基准，==自己主观发挥== |
| 50% 功能仅做了 UI 占位，逻辑缺失 | AI 基于"需求已实现"自我评估，缺乏客观验证 |

### 迭代 3：引入 Rubric → 效果显著提升

| 改进 | 效果 |
|------|------|
| 采用 Rubric 四层评测方案 | 评测从主观变为==结构化可量化== |
| 四层权重公式 | 各维度改进方向明确 |

实测数据（评测问卷系统）：5 小时自动化运行，4 轮迭代：

| 轮次 | 评分 | 说明 |
|------|------|------|
| 第 1 轮 | 3.35 | 全方位都比较差，还白屏 |
| 第 2 轮 | 4.11 | 功能和 UI 维度提升明显 |
| 第 3 轮 | 4.52 | 每个维度继续小幅提升 |
| 第 4 轮 | 4.66 | 边际效果很低，可退出循环 |

> [!important] 关键发现
> 迭代 2 时 AI 声称功能已实现，但 ==50% 是页面上仅做了功能占位==，逻辑根本没实现。引入 Rubric 后，功能逻辑基本都已实现，界面和交互也有较大提升。

### 迭代 4：品质锚定 + 精细 Rubric → 从"做什么"到"做成什么样"

| 问题分析 | |
|---------|---|
| **目标端**：知道"做什么"但不知道"做成什么样子" | PRD 只定义了做什么，没有"做成什么级别"；没有参照物机制 |
| **评估端**：有 Rubric 框架但不够挑剔和细致 | 偏功能性验收缺乏品质性验收；"布局合理"这种条目太模糊 |

#### 目标端改进：quality-vision.md 品质锚定

在需求澄清阶段增加"品质锚定"环节，新增 ==`quality-vision.md`== 产物：

| 品质锚定内容 | 说明 |
|-------------|------|
| **品质定位** | MVP / 精打磨 / 生产级 |
| **参考锚点** | 正面参考截图/URL/竞品 + 注解"好在哪"；反面参考 + 注解"为什么不好" |
| **设计意图** | 风格关键词、配色方向、信息密度、交互范式 |

#### 评估端改进：Rubric 精细化

- Rubric 从需求拆出具体条目，和 ==`quality-vision.md`== 锚定
- 增强 Evaluator 挑刺能力，增加 UI 设计专家视角、UX 交互专家视角的==逐项检查清单==

改进后效果：首轮需求澄清从原来 10 题增加到 ==31 题==，涵盖非功能性并发、安全合规、多端适配、品质期待等。最终 7 迭代，连续两轮评分超过 0.95 目标后退出。

> [!warning] 边际迭代效益递减
> 从第 4 轮后边际迭代效益很低，但 Agent 仍耗费很长时间改进——后续优化方向：如何在边际成本很低的情况下==尽早结束循环==，不让 AI 为了改进而改进。

## 应用环境对 Agent 的可读性

> 已有笔记覆盖了可观测性的一般原则，这里补充==具体工具和实践==。

| 维度 | 具体实践 |
|------|----------|
| **UI 交互** | 通过 ==Playwright / Puppeteer / CDP== 接入 Agent 运行时，让 AI 截图、查看 DOM 快照、分析交互问题 |
| **测试数据** | Agent 自行生成测试数据，但测试账号、测试库链接等需预设并说明每个账号角色 |
| **CI/CD** | 为 Agent 集成测试环境的 DevOps 工具（CLI），让它自行完成 CI/CD，自动化飞轮高效运转 |
| **运行过程** | 日志、指标、链路追踪通过本地可观测性栈，让 AI 检查执行过程 |

> [!important] Agent 不一定每次都会调用环境工具
> 即使配置了环境工具，Agent 也不一定每次都会调用执行。例如 e2e 测试环节让 Agent 截图分析交互并和前一轮对比，Agent 不能一致遵循。
>
> 解决方案：==显式提供工具脚本==，让 Evaluator 的全量/E2E 阶段强制执行。

## Skills 开发 Harness 实践

Skills 开发流程与 AI Coding 本质类似，但针对 Skill 本体和 Tool 的开发评测改进循环。

### 驱动循环的核心要素

| 要素 | 说明 |
|------|------|
| **多 Agent 协作** | 单一 Agent 有明显的过早结束倾向，很难开启多轮循环 |
| **对效果基准和迭代轮次的强制要求** | 必须"达到多少分"或"至少执行多少轮"，强制避免 AI 过早放弃 |
| **效果基准的判断科学化** | 使用 Rubric 评测方法，强制避免 AI 评估过于正向迎合 |

### 三角色信息隔离

| 角色 | 职责 | 可见信息 |
|------|------|----------|
| **主 Agent** | 调度编排 | 全局信息 |
| **执行 Agent** | 装配 Skill 并以真实 Agent 身份处理请求 | SKILL.md + 用户请求（==不可见评测标准==） |
| **评估 Agent** | 基于 Rubric 独立判分 | 评测标准 + 执行结果（==不可见执行者内部状态==） |

> [!important] 避免自出题、自执行、自判卷
> 三者之间==严格信息隔离==——执行者不可见评测标准，从根本上避免分数虚高。这正是 [[Harness Engineering#Anthropic F-Harness|F-Harness]] 分工思想的 Skill 开发落地。

### 评测流程

```
Skill PRD → 初版 Skill → 上传评测平台 → 设置得分阈值 + 迭代轮次
    → Agent 认领任务 → 自动生成 12-20 个结构化测试用例
    → 每个用例附带 7-15 条实例级 Rubric 检查项（含正样本 + near-miss 负样本）
    → 逐个派遣执行者真实运行 → 评估者独立判定每条 Rubric
    → 平台自动聚合评分 → 未达标 → Agent 根据 结构化改进建议 优化 SKILL.md
    → 新版本进入下一轮循环 → 达标或达到最大迭代轮次 → 自动退出
```

> [!note] 实测数据
> 一次自动化多轮反馈改进循环约 ==20-30 分钟==，同等效果相当于原来测试和开发各投入半人日的工作。

> [!warning] Agent 工具必须具备子 Agent 能力
> 切记 Agent 工具要具备创建"子 Agent"的能力。否则主调度和执行都是同一个 Agent，带来上下文污染和正向评价倾向，过早结束。

## 结构化执行回路

> 已有笔记覆盖了 PEV 和 PPAF 的理论框架，这里补充==状态机流程控制==的具体实现。

### 状态机流程控制

```
用户发出需求 → start-session hook 强制开启 Harness 研发流程
  → 主 Agent 创建研发流程状态机
  → Planner（clarify-requirements / write-prd / write-tech-design / write-dev-plan / write-acceptance-criteria / write-test-cases）
  → 大循环开始（macro_cycle.current_index++）
  → Developer（implement-task / fix-from-report / systematic-debugging / verification-before-completion / git commit 闭环）
  → Evaluator（api-testing → case_id 逐条 PASS/FAIL/SKIP）
      FAIL → 回 Developer 修复（小循环，≤50 次）
      PASS → 主 Agent 校验 SHA → 下一任务
  → 全量 E2E Testing（e2e-testing skill + ui_evidence 截图 + 多维度评估 + AI 读图）
      replan_required=true → Planner sync-overview → 更新概览 → 回大循环
      replan_required=false → phase: done（需求完成）
```

### 人类审查：控制和放权

| 环节 | 人类审查程度 |
|------|-------------|
| 需求研究和产品规划（偏目标性质） | ==严格审查== |
| 自测、CR、端到端测试（偏执行过程） | 交由 AI 检查 |
| 交付结果 | 人类==最终审查== |

> [!quote] 像管理者管理员工一样管理 Agent
> 管理者关注任务两端，偶尔看过程。新手多看，老手少看。这要求人类转变思维，成为 Agents 的管理者，==放弃一些过程掌控感==。

### 强制约束机制

| 约束 | 说明 |
|------|------|
| **强制迭代轮次** | 避免 Agent ==过早结束==的倾向 |
| **强制 Evaluator 审核** | 避免 Developer ==自我评估的自我表扬==倾向 |
| **强制 git commit 闭环** | 精确 add + SHA 记录，确保每步可追溯 |

## 实测数据与洞察

| 维度 | 数据 |
|------|------|
| **全流程时长** | 7 小时全自动执行 |
| **Token 消耗** | 2.2 亿 Token（2.1 亿命中 cache） |
| **迭代轮次** | 4-7 轮评估迭代 |
| **人类参与** | 除最初目标澄清，后续过程==人类没有一点参与== |
| **模型配置** | 主 Agent 用旗舰模型，子 Agent 用 cursor 微调 composer 2 模型 |

> [!important] 模型能力到达水位后，Harness 占非常大的权重
> 侧面说明：模型能力到达一定水位后，Agent 的效果发挥，==harness 会占非常大的权重==。

## 适用场景与局限

| 适用 | 不适用（暂无成熟案例） |
|------|----------------------|
| 新产品 0→1 阶段 | 十年以上历史的遗留代码 |
| 新模块 0→1 开发 | 复杂微服务系统 |
| 前后端一体化项目 | 前后端深度分离且架构复杂的项目 |

> [!tip] 前后端一体化建议
> 如果前后端分离仅出于岗位划分的历史原因而非架构需要，建议优先==整合为统一代码库==。如果 DevOps 暂不支持，可在 IDE 开发阶段先做逻辑层面统一管理。

## 三个不变的重要原则

无论模型如何演进，以下三点始终至关重要：

1. **环境支持**：为 Agent 提供完善的工具、充分且必要的数据，让它具备==可读、可分析、可操作==的能力
2. **专业化分工**：多个职责清晰的 Agent 协同工作，优于单个 Agent 在超大上下文中完成全流程
3. **上下文控制**：无论 Agent 的记忆管理发展到什么程度，初始上下文依然==极其关键==

> [!warning] Agent Loop 应越简单越好
> 模型和 Agent 的能力迭代速度极快，今天构建的部分能力未来可能很快变得冗余甚至产生负作用。==尽可能减少人为植入的特定规则==。
