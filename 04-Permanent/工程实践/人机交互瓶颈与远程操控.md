# 人机交互瓶颈与远程操控

> ==好的 Harness 工程不仅要设计"Agent 怎么跑"，还要设计"人怎么随时参与"。== 工作流的流转依赖于人的在场——16 阶段状态机设计得再精密，如果 Agent 在执行过程中需要人工确认而你恰好在开会、通勤，工作流就卡住了。

---

## 问题：Harness 工作流的"在场依赖"

传统 Agent 工作流有一个隐含假设：==操作者坐在电脑前，IDE 打开着，随时可以响应 Agent 的请求==。

现实：一天 8 小时工作，真正能"坐在工位操控 Agent"的时间可能不到 4 小时。碎片时间（会议间隙 5 分钟、通勤 30 分钟）恰恰是 Agent 需要确认的黄金窗口。

**如果工作流在你离开时就暂停，在你回来时才继续，那工作流的效率至少折半。**

---

## 三层影响

| 影响 | 说明 |
|------|------|
| **交付周期拉长** | 原本 Agent 可连续推进的需求，因卡在人工确认节点，从 1 天变 3 天——工作流没出错，只是在"等人" |
| **知识沉淀时效性下降** | ARCHIVE 阶段的知识提取依赖工作流完整走完。流程卡得越久，新知识沉淀速度越慢 |
| **碎片时间浪费** | 会议间隙有 5 分钟、通勤有 30 分钟，这些碎片时间本可推进工作流，但因不在工位、没有 IDE 环境而白白流失 |

---

## 解法：远程操控 + 跨设备接管

引入 Hapi 内网版：==在办公网下用手机远程接管运行在开发机上的 AI 编程会话==。

| 能力 | 说明 | 对 Harness 工作流的意义 |
|------|------|------------------------|
| **跨设备会话接管** | 手机/平板/电脑均可接管同一 Agent 会话 | 工作流不因设备切换而中断 |
| **24 小时待机** | 开发机上的 Agent 持续运行 | 工作流可以 7×24 小时流转 |
| **PWA 原生体验** | 安装到桌面后像原生 App | 降低远程操控的使用门槛 |
| **多助手切换** | 支持 Codebuddy/Codex/Gemini 等 | 适配不同 Agent 引擎的工作流 |
| **自主模式** | YOLO 模式让 Agent 自主执行 | 减少人工确认频率 |

---

## 工程架构设计启示

> [!important] 四条设计原则
> 好的 Harness 工程不仅要设计"Agent 怎么跑"，还要设计"人怎么随时参与"。

1. **状态持久化**：工作流状态必须是持久化的（==文件系统即状态机==），不依赖内存。无论从哪个设备接入，都能看到一致的状态。
2. **断点恢复**：每个阶段的入口和出口都有明确的持久化产物，支持从任意断点恢复。
3. **异步审批**：人工确认节点应设计为==异步模式==——Agent 提交产物、暂停等待，人类可在任意时间、任意设备审批后，Agent 继续执行。
4. **通知触达**：关键节点（如架构评审、产物验收）应通过企业微信等渠道==主动推送==，而非被动等待人来检查。

> [!note] 与 AI Team 哲学的契合
> 这些设计与 AI Team 的"文件系统即状态机"哲学天然契合——所有状态都在文件中，不依赖内存或特定进程，远程设备通过 Web 界面看到的就是真实的工作流状态。

---

## 与知识沉淀闭环的结合

远程操控能力不仅解决"人机交互"效率问题，更重要的是==保障知识沉淀闭环的完整性==：

```
INIT（知识注入）→ 各阶段执行（知识消费）→ ARCHIVE（知识提取）
```

由于 Harness 工作流采用"文件系统即状态机"设计，==暂停本身不会丢失任何进度==——所有阶段产物和状态持久化在文件中。但暂停过久的真正问题是==效率和时效性==。

有了远程操控后，碎片时间都能被利用——工作流更紧凑地走完，从 INIT 到 ARCHIVE 大幅缩短交付周期，==加速知识沉淀闭环的流转==。
