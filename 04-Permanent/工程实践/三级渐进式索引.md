# 三级渐进式索引

> 从"推送"到"主动查询"的范式转变——==Agent 不被动接收固定数量的知识推荐，而是通过三级渐进式索引主动按需查阅==。用 ~50 行成本了解知识库全貌，对比"一次性推送 50 条完整知识"（5000-10000 行），上下文效率提升了一个数量级。

---

## 传统推送模式的两个问题

| 问题 | 说明 |
|------|------|
| **信息过载** | 推送太多知识，Agent 反而被淹没，找不到关键信息 |
| **不精准** | 预先推送的知识不一定是 Agent ==当前决策点==需要的 |

> [!important] 与 [[上下文工程]] 的呼应
> 这与上下文工程的核心洞察一致：上下文窗口不是"能装多少"的问题，而是"装什么、何时装、何时清"的系统设计问题。三级渐进式索引正是"何时装"的工程化实现。

---

## 三层索引结构

借鉴 Karpathy 的 [LLM Wiki Pattern](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)，设计三层索引：

| 层级 | 文件 | 大小 | 作用 | 成本 |
|------|------|------|------|------|
| **Layer A: 全景目录** | `knowledge-catalog.md` | ~50 行 | "知识库有什么？"——分类统计 + 按阶段推荐查阅路径 | 零成本 |
| **Layer B: 分类清单** | 各目录下 `catalog.md` | ~100-300 行 | "这个分类有哪些条目？"——每条一行摘要（ID + 标题 + 成熟度 + 标签） | 低成本 |
| **Layer C: 完整条目** | `TK-*.md` / `BK-*.md` | ~50-200 行 | "这条知识说了什么？"——完整内容 + 背景 + 适用场景 | 按需 |

---

## 渐进查询流程

```
Step 1: 读全景目录（~50 行，零成本）
  → 了解知识库有什么分类、每类多少条
  → 定位当前阶段推荐查阅的 catalog.md 路径

Step 2: 读分类清单（~100-300 行，低成本）
  → 每条知识一行摘要
  → 按 tags / applicable_phases 过滤相关条目

Step 3: 读完整条目（按需，每条 50-200 行）
  → 获取完整知识内容

Step 4: 读原始产物（深入，可选）
  → 沿 source_references 追溯原始推导过程
```

> [!tip] 效率对比
> - 传统推送：一次性推送 50 条完整知识 ≈ **5000-10000 行**上下文
> - 渐进索引：~50 行了解全貌 + ~300 行定位条目 + 按需读取 ≈ **~350 行**基础成本
> - ==上下文效率提升一个数量级==

## 知识引用追踪闭环

Agent 查询知识后，在输出产物中记录引用：

```json
{
  "knowledgeReferences": [
    { "id": "TK-SB-003", "title": "分页查询延迟关联优化", "usedIn": "复用评级 Step 2" },
    { "id": "BK-AD-G004", "title": "广告预算扣减并发控制规则", "usedIn": "业务规则参考" }
  ]
}
```

ARCHIVE 段读取所有产物中的 `knowledgeReferences`，批量更新 `evidence.last_referenced`——形成==自动化引用追踪闭环==。被引用的知识 maturity 自动提升，长期未引用的自动衰减。

> [!note] 与 [[知识成熟度与自动衰减]] 的关联
> 引用追踪闭环是成熟度提升和自动衰减的==数据来源==——没有引用数据，衰减机制就无法运作。

## 与已有概念的映射

### 与 Progressive Disclosure 的精确对应

| 三级渐进式索引 | [[5条驭驭工程原则]] 中的 PD | 共同精神 |
|---------------|----------------------------|---------|
| Layer A: 全景目录 | AGENTS.md 100 行只放最高层 | ==懒加载胜过热加载== |
| Layer B: 分类清单 | `docs/` 子目录按需拉 | 给 Agent 导航地图而非倾倒全部内容 |
| Layer C: 完整条目 | Skill `references/` 需要才进 | 精准而非贪婪地消费知识 |

### 与 GSSC 流水线的对应

| 渐进式索引 | [[GSSC流水线]] | 共同逻辑 |
|-----------|---------------|---------|
| Layer A（全景目录） | Gather 阶段 | 先知道有什么，再决定要什么 |
| Layer B（分类清单） | Select 阶段 | 在预算内选择最有价值的信息 |
| Layer C（完整条目） | Structure + Compress | 按需获取结构化的完整信息 |

### 与 System of Record 的对应

| 渐进式索引 | [[5条驭驭工程原则]] 中的 SoR | 共同精神 |
|-----------|----------------------------|---------|
| 知识存储在独立 Git 仓库 | 知识住外部 docs、不住 prompt | ==唯一权威来源== |
| `knowledge-catalog.md` 是入口 | 100 行 entry map 是导航 | prompt 绝不重复 docs 内容 |
