# Agent 遗忘三重根因

> Agent "遗忘"不是 bug，是当前架构的==必然代价==。遗忘有三重根因——harness 的三层设计（规则外置、状态持久化、门禁阻断）恰好对应这三个根因，逐一堵漏。

---

## 三重根因

| 根因 | 机制 | 典型表现 | Harness 对策 |
|------|------|---------|-------------|
| **① 压缩丢失** | Auto-Compact 省略"看似不重要"的流程步骤 | 写完代码后忘记该走什么流程——"先跑单测再提交"被几十屏代码输出挤到模型"看不见"的位置 | 规则外置：关键规则不靠上下文常驻，靠 `rules/` 按需加载 |
| **② 检索失败** | 记忆文件在但没被加载进上下文 | `mvn -am` 卡死踩过三次，每次换会话都忘 | 状态持久化：`state.json` + 文件交接，Read 即续 |
| **③ 指令遵循失败** | 信息都在但模型仍然跳步 | 规则太多，模型"选择性遵守"：这次记得先写单测，下次又跳过 | 门禁阻断：确定性代码检查，不管听不听话，越界就被拦 |

> [!important] 根因是结构性的，不是偶然的
> 这三个根因有明确的技术支撑：
> - **压缩丢失**：VILA-Lab 逆向证实 Claude Code 上下文管理靠 5 层渐进式压缩管线——流程状态细节在 Auto-Compact 阶段被丢失
> - **检索失败**：Devin CPO 坦承"数千条记忆中如何在正确时机检索到正确的记忆——尚未解决"
> - **指令遵循失败**：模型"理解"了规则 ≠ "遵守"了规则——[[上下文焦虑症与自我评估偏差]]揭示了 AI 的自评过度乐观

---

## 对应 Harness 三层设计

```
遗忘根因          →  Harness 对策
① 压缩丢失        →  规则外置（rules/ 按需加载，不靠上下文常驻）
② 检索失败        →  状态持久化（state.json + 文件交接）
③ 指令遵循失败     →  门禁阻断（确定性代码检查，fail-closed）
```

> [!tip] 核心洞察
> ==对付遗忘的正确思路不是"让模型记住更多"，而是"让关键信息不再依赖模型的记忆"==。规则外置到文件系统、状态外置到 JSON、门禁外置到确定性代码——三重外置，把模型从"必须记住"降级为"只需执行"。
