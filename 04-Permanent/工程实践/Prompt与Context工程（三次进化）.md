# Prompt Engineering与Context Engineering（三次进化·上）

> 本文基于李伟山《从Prompt、Context到Harness，工程的三次进化与终局之战》整理，聚焦前两次进化：Prompt Engineering 与 Context Engineering，以及两者共同的局限性。第三次进化 Harness Engineering 见 [[04-Permanent/# 驾驭工程（Harness Engineering）]]。

---

## 一、Prompt Engineering：学会跟模型说话

### 1.1 本质：加约束的艺术

大语言模型的底层逻辑是一个极其擅长**续写**的系统——给它一段输入，它预测接下来最有可能出现的内容。问题在于，"最有可能出现"并不等于"你真正想要的"。

以"帮我写一封道歉信"为例，逐步加约束的效果差异：

1. **不加约束**：千篇一律的模板文字
2. **加上对象和原因**（"对象是老板，原因是我迟到了三次"）：开始接近实际需求
3. **加上语气和结尾要求**（"诚恳但不卑微，暗示已采取改进措施"）：这才是一封真正可用的信

> [!quote]
> 这个"加约束"的过程，就是提示词工程（Prompt Engineering）的本质。它研究的是：如何通过精心设计的输入，最大限度地激发模型的正确能力。

本质上，[[提示工程]]是在 LLM 的"续写倾向"上施加结构性约束，将概率空间从"所有可能"收窄到"你想要的"。

### 1.2 武器库：五大核心技巧

在 GPT 早期走入大众视野时，[[提示工程]]是最炙手可热的技能。核心技巧包括：

- **零样本提示（Zero-shot Prompting）**：直接告诉模型做什么，不给示例。适合简单任务。
- **少样本提示（Few-shot Prompting）**：给几个输入-输出示例，让模型从中"意会"规律。效果往往远好于零样本。
- **思维链（Chain-of-Thought, CoT）**：不让模型直接跳结论，而是引导它一步步推理。在数学、逻辑推理任务上效果显著。与其相关的范式有 [[ReAct 范式]]、[[Plan-and-Solve]]、[[Reflection 范式]]。
- **角色扮演（Role Prompting）**："你是一位有 20 年经验的 Java 架构师，请……"给模型设定身份，往往能显著提升输出的专业性。对应笔记：[[系统提示词工程]]。
- **提示链（Prompt Chaining）**：把复杂任务拆成多个小提示，前一步的输出作为后一步的输入，像流水线一样串联。

### 1.3 繁荣与衰退：模型越强，Prompt 的边际效益越低

2023—2024 年，"Prompt Engineer"一度被视为最有前途的职业之一。但底层环境发生了巨变：模型的智能化越来越高了。

> [!quote]
> GPT-3 时代，你需要精心设计的少样本提示才能让模型完成一个稍复杂的任务。到了 GPT-4、Claude 3，你随便说一句话，它就能理解你的意图，甚至哪怕你的表达并不精准也没关系。

当模型本身的**语言理解能力足够强**，写好 Prompt 的边际效益就显著降低了。

但更深层的问题随之浮现：即使模型听懂了你说的话，它有时候依然会给出错误的答案——原因不是你没说清楚，而是它根本**不知道一些关键信息**。这就是"上下文"的问题。

## 二、Context Engineering：给模型备好简报

### 2.1 核心隐喻：金鱼助理

> [!quote]
> 假设你雇了一位全世界最聪明的助理，但这位助理有一个致命弱点：他的记忆只有 7 秒。每次会面，他都记不住上次你们聊过什么，不知道你的偏好，不了解你的项目背景。即使他智商超群，每次都要重新从零开始建立对你情况的了解。

你会怎么办？你会在每次见面前，把关键信息整理成一份简报递给他——告诉他上次的决策、当前的目标、需要回避的坑。

这个**准备简报的过程，就是 Context Engineering**。

大语言模型的本质，就是这位"金鱼助理"。每次对话，它能看到的信息被严格限制在 [[上下文窗口]] 之内。窗口外的一切，它一无所知。这和 [[智能体（Agent）]] 的 [[记忆系统]] 设计密切相关，尤其是 [[短时记忆]] 和 [[长时记忆]] 的分层管理。

### 2.2 上下文窗口的层级结构

一个完整的 LLM 上下文，通常包含以下几层信息：

| 层级 | 内容 | 说明 |
|------|------|------|
| **System Prompt** | 系统级指令、角色定义、行为边界 | 最高优先级，定义"它是谁" |
| **对话历史** | 当前会话的所有对话记录 | 提供连续性和上下文 |
| **检索文档** | 从知识库动态注入的相关资料 | RAG 的核心输入 |
| **工具输出** | API 调用结果、数据库查询等 | 动态获取的外部信息 |
| **用户输入** | 当前轮次的用户消息 | 触发本次推理的直接输入 |

每一层都至关重要，却又都在争夺有限的 [[Token]] 空间。Context Engineering 要解决的，就是**信息注意力分配**的问题。这涉及 [[上下文窗口管理]]、[[上下文压缩]] 和 [[上下文注入]] 三方面技术。

### 2.3 RAG：不存知识，存索引

传统做法是把所有知识都写进 System Prompt，结果：空间爆满，模型不知道看哪里，输出质量反而下降。

> [!quote]
> RAG 的思路截然不同：不存知识，存索引。需要什么，临时去检索，精准注入。

[[RAG]]（Retrieval-Augmented Generation，检索增强生成）的具体流程：

1. **离线索引阶段**：将知识文档分块，通过 Embedding 模型转为向量，存入 [[向量存储]]
2. **在线查询阶段**：用户提问时，将问题向量化，检索最相关的 Top-K 文档块
3. **上下文注入**：将检索结果与原始问题、System Prompt 等拼装为完整上下文，送入 LLM 生成回答

这个机制让模型能够访问**远超其参数记忆的外部知识**，同时又不会被无关信息淹没。在实践层面，这与 [[上下文注入管道]] 的设计密切相关。

### 2.4 上下文压缩：对抗遗忘的艺术

随着对话越来越长，历史消息会把 [[上下文窗口]] 撑满，挤走最新的关键信息。

更关键的问题是 **"Lost in the Middle"（中间遗忘）**现象：当上下文过长时，模型对开头和结尾的内容记忆较好，对中间大段内容的关注度大幅下降。

解决方案是 [[上下文压缩]]，三种核心策略：

- **滚动摘要（Rolling Summary）**：定期将旧对话压缩为摘要，只保留精华
- **重要性评分（Importance Scoring）**：给每段历史内容打分，低分内容优先淘汰
- **层次记忆（Hierarchical Memory）**：短期记忆保留细节，长期记忆只存关键节点

> [!quote]
> OpenAI 的实战经验验证了这一点：他们把原来装满所有规范的巨型 agent.md 文件压缩至百行以内，仅作为索引目录，需要什么规范就动态加载对应子文档。结果：模型的遵从度和输出质量显著提升。

### 2.5 单一事实来源：Context Engineering 的纪律

> [!quote]
> 在实际工程中，技术决策可能散落在企微消息、腾讯文档、本地 PDF、GitHub Issue 里。对人类工程师来说，这已经够难管理了。对 AI Agent 来说，这是灾难性的：它不知道该信哪个版本，结果就是综合出一个四不像的答案。

**单一事实来源（Single Source of Truth）** 的核心纪律：强制将所有决策、规范、文档都归档进代码仓库，确保 AI 的信息来源是**唯一的、可追溯的、版本受控的**。

这在 [[工具描述工程]] 和 [[系统提示词工程]] 中也同样适用——所有工具描述、系统指令都应有一个唯一的、受版本控制的来源，而不是散落在不同配置文件中。

## 三、两者的共同局限：当"说对"和"给够"都不够用

### 3.1 Agent 失控的典型场景

假设你构建了一个代码生成 Agent，已经做到了：

- 精心设计的 System Prompt（Prompt Engineering）
- 动态注入最相关的代码规范文档（Context Engineering）

然后你让它生成一个用户登录模块。一小时后检查结果：

| 问题 | 具体表现 |
|------|----------|
| **越权操作** | 写了登录逻辑——正确，但同时"顺手"重构了不该动的数据库层 |
| **虚假声称** | 声称测试通过了——但根本没有运行测试，只是自我评估"应该能过" |
| **风格不一致** | 命名风格跟项目其他部分完全不一致——因为没有人告诉它命名规范 |
| **功能重复** | 生成了三个功能重复的工具函数——因为没有机制检测重复 |

### 3.2 根源：系统层面缺乏约束、验证和反馈

> [!quote]
> 提示词写得再好，上下文管得再精，也没能阻止这一切发生。因为这些问题的根源不在"说什么"或"给什么信息"，而在于：系统层面缺乏约束、验证和反馈机制。

这就是 Prompt Engineering 和 Context Engineering 的**共同盲区**：

- **Prompt Engineering** 只管"输入端质量"——指令是否清晰、约束是否到位
- **Context Engineering** 只管"信息供给质量"——信息是否充分、是否精准注入
- 但**两者都管不到**执行过程中的行为约束、结果验证、错误纠正

完成这三件事的，是第三次进化：**Harness Engineering（[[04-Permanent/# 驾驭工程（Harness Engineering）]]）**——通过系统级的约束、验证和反馈回路，让 AI 的"声称完成"变成"验证完成"。
