# MCP 协议实现

## 📖 定义

MCP 协议实现是 Claude Code 对 **Model Context Protocol** 的内部支持层，位于 `src/services/mcp/`。它负责 MCP 服务器连接管理、工具/资源/命令发现与执行、多种传输协议（stdio、SSE、HTTP、WebSocket、SDK）、认证机制（OAuth、XAA、API Key）等。这使 Claude Code 能与任何 MCP 服务器交互，扩展其能力边界。

## 🧠 我的理解

MCP 协议实现就是 Agent 的"外部连接器"。[[MCP 服务器集成]]（已有笔记）讲的是**配置层**——如何配置 MCP 服务器。这篇笔记讲的是**实现层**——Claude Code 内部如何与 MCP 服务器通信。就像 USB 接口：配置层是"插哪个设备"，实现层是"USB 协议怎么传输数据"。

## 🎯 为什么重要

MCP 是 AI Agent 与外部世界交互的标准协议。理解 Claude Code 的 MCP 实现，就理解了**如何让 Agent 安全可靠地调用外部工具**——不只是配置，而是传输、认证、发现、执行的完整链路。

## 🔑 传输协议

| 协议 | 适用场景 | 特点 |
|------|---------|------|
| `stdio` | 本地 CLI 工具 | 最简单，通过标准输入/输出通信 |
| `SSE` (Server-Sent Events) | HTTP 长连接服务 | 单向推送 + 请求 |
| `HTTP` | RESTful 服务 | 请求-响应模式 |
| `WebSocket` | 双向实时通信 | 低延迟双向通信 |
| `SDK` | 内嵌式服务 | 直接调用 SDK，无需独立进程 |

## 🔑 认证机制

```
认证层级：
  1. API Key — 最简单，直接在配置中提供密钥
  2. OAuth 2.0 — 标准授权流程，适合需要用户授权的服务
  3. XAA (Cross-App Access) — 跨应用访问认证
  4. Elicitation — URL-based 认证流程，MCP 服务器可以要求用户访问特定 URL 完成认证
```

## 🔑 工具发现与执行流程

```
1. 启动时：读取配置 → 连接 MCP 服务器 → 发现可用工具/资源/命令
2. 运行时：
   模型请求 MCP 工具
     │
     ▼
   MCPTool 查找工具定义
     │
     ▼
   选择传输协议 → 构建请求
     │
     ▼
   发送到 MCP 服务器 → 等待响应
     │
     ├─ 成功 → 返回工具结果 → 模型继续
     ├─ 需认证 → 触发认证流程 → 用户交互 → 重新请求
     ├─ 失败 → 返回错误消息 → 模型调整策略
```

## 📝 个人笔记

MCP 的核心设计思想：**标准化 AI Agent 与外部工具的接口**。就像 USB 让任何设备都能连接电脑，MCP 让任何工具都能连接 Agent。Claude Code 的实现特别值得关注的是**Elicitation 认证机制**——它让 MCP 服务器可以要求用户在浏览器中完成认证，然后 Agent 才能继续调用。这解决了"Agent 需要调用需要认证的服务"这个难题。
