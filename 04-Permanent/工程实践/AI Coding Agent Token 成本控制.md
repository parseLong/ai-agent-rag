# AI Coding Agent Token 成本控制

> 适用：CodeBuddy、Cursor、Codex、Gemini CLI 等 AI Coding Agent。

## 🧠 核心心智模型

> **AI Coding Agent 的成本，本质上不是"你问了什么"，而是"系统为了回答你，重复搬运了多少上下文"。**

你问的那句话，在每次请求里可能连 1% 都不到。真正的成本大头藏在系统自动带上的那一大坨东西里——System Prompt、Skill 定义、Tool/MCP 定义、历史会话、代码文件。

### 一轮请求的成本分布

```
System Prompt         5K
项目说明文档          10K
Skill 定义            20K
Tool / MCP 定义       30K
历史会话             100K
代码文件              50K
用户问题               0.1K
```

### 成本公式

```
总成本 ≈ 固定前缀 + 会话历史 + 运行时检索 + 工具往返 + 模型输出
```

三层上下文结构：

| 层级 | 内容 | 特征 |
|------|------|------|
| **固定前缀** | System Prompt、Skill 定义、Tool/MCP 定义、稳定背景文档 | 每轮都带，适合缓存命中 |
| **半固定上下文** | 项目说明、Repo Map、Memory、长期约束 | 变化少，可缓存 |
| **动态上下文** | 聊天历史、代码片段、检索结果、工具返回、新问题 | 变化大，难以缓存 |

> [!important] 关键推论
> 你的提问只是"点火器"，它负责触发任务，但通常不是成本主体。因此优化重点不是把提示词写短，而是**减少重复上下文**。

## 🔑 五种成本类型

| 成本类型 | 它是什么 | 为什么会放大 |
|----------|----------|--------------|
| **输入 Token** | 系统提示词、历史消息、代码片段、检索结果、工具定义 | 每轮都可能重复携带，是最大头 |
| **输出 Token** | 模型最终回答的文字 | 回答越长、越啰嗦、越容易失控 |
| **推理 Token** | 模型内部思考和规划的预算 | 简单任务走高推理档位，明显溢价 |
| **工具往返成本** | 工具描述、调用参数、返回值进入上下文 | 一次工具调用，常常比原问题还长 |
| **重试成本** | 第一次做错后重新来一轮 | 整包上下文被反复付款 |

> [!warning] 最容易被低估的是后两项
> **工具往返**：你以为只是"让它查个文件"，但模型经历的是理解工具定义→生成参数→接收返回值→再结合返回值继续思考。详见 [[工具调用回路]]。
> **重试成本**：第一次不合格→重新描述→重新附带历史→重新拉代码→再跑一轮。每次修正 = 整包上下文重新付款。详见 [[减少重试是成本优化]]。

## 🔑 AI Agent 最怕的五件事

| 模式 | 为什么可怕 | 对应优化层 |
|------|-----------|-----------|
| 会话越滚越长 | 每轮都要重新带过去全部历史 | 第一层：使用习惯 |
| 工具越堆越多 | 每多一个 MCP → 多一段常驻定义 + 更高错误调用概率 | 第一层 + [[工具调用回路]] |
| 背景越塞越满 | 噪音越多 → 模型分神 → 越贵但不越准 | 第三层：上下文压缩 |
| 每次都从零找信息 | grep → 读文件 → 发现漏了 → 再 grep 的循环 | 第四层：代码图谱 |
| 输出不合格反复重试 | 每修一次 = 整包上下文重新付款 | [[减少重试是成本优化]] |

## 🔑 Prompt Cache：所有优化的基础

缓存不是缓存"答案"，而是缓存**稳定前缀的处理结果**——如果两次请求前半段几乎一样，服务端不必每次从头处理。

![[笔记同步助手/images/98739db0eae5682966914c15c54f6544_MD5.png]]

三个关键推论：

1. **省的不是首次成本，而是重复成本**——价值在第二次及后续的复用
2. **缓存不是"写短"，而是"写稳"**——天天改前缀，缓存很难命中
3. **缓存优化 = 上下文治理**——稳定内容前置、变化内容后置，提升可复用比例

> **Token 优化重点，不是把提示词写短，而是把前缀写稳。**

## 🔑 "它明明记得"只是一种幻觉

大模型本身通常**无状态**。真正"记得"的，不是模型，而是包在模型外面的 Agent/CLI/平台层——它在每一轮请求前，把历史、规则、工具、代码、文档重新拼起来，再发给模型。

![[笔记同步助手/images/789da6fdc6673cef1e10faa9ac102f4e_MD5.png]]

> **不是它记得，而是系统在一遍遍提醒它。**

这决定了成本结构：会话越长→后续每一轮越贵；工具越多→常驻定义越重；工具调用形成回路→上下文越塞越满。

## 📐 五层成本优化框架

![[笔记同步助手/images/02ce689dfd212ee767fbdea57c9ce6fc_MD5.png]]

| 层级 | 解决什么问题 | 关键手段 |
|------|-------------|----------|
| **第一层：使用习惯** | 无意义的历史和废 Token | 一件事一个Session、`/compact`、意图一次说完 |
| **第二层：模型路由** | 贵模型干便宜活 | [[模型路由]]：任务→模型匹配 |
| **第三层：上下文压缩** | 同样前缀重复发送 | [[RTK]]、[[Caveman]]、[[headroom]]、[[context-mode]] |
| **第四层：代码图谱** | 每次都从零找代码 | [[Graphify]]、[[CodeGraph]] |
| **第五层：Agent架构** | 所有任务挤进同一个上下文 | Orchestrator-Worker、subagent隔离 |

五层背后的总目标：
- 让模型少看无关内容
- 让便宜模型多干标准活
- 让贵模型只做高价值推理

贯穿线索：**观测与预算治理**——没有数据，优化就是体感。详见 [[上下文预算与资源分配]]。

### 第一层：使用习惯（零工程投入，最大收益）

| # | 习惯 | 省什么 |
|---|------|--------|
| 1 | 一个 Session 一件事 | 减少无关历史继承 |
| 2 | 长会话及时 `/compact` | 压缩历史，防止越滚越贵 |
| 3 | 聊天记录不是数据库——长期信息外置到 Memory/Summary/Repo Map | 让会话只承载当前工作状态 |
| 4 | 少说废话，指定输出格式 | 省输出Token + 减少重试 |
| 5 | Skill ≠ 免费能力——高频常驻，低频按需 | 减少常驻上下文 |
| 6 | MCP 越多不一定越强——只常驻高频工具 | 减少工具定义重量 |
| 7 | CLI 优先于 MCP | 成熟命令链路更轻量 |
| 8 | 引用文件用 `@路径` 而非只写文件名 | 免去搜索链路 |
| 9 | 意图一次说完，避免聊天式拆碎 | 减少多轮上下文往返 |

![[笔记同步助手/images/b183732a0162cc0b5e575990185ffd3a_MD5.jpg]]

> [!tip] CLI 优先原则
> 能 CLI 解决→尽量 CLI，能脚本解决→尽量脚本，最后才考虑 MCP。已有成熟命令链路时，不要为了看起来高级而过度工具化。

### 第二层：模型路由 → 详见 [[模型路由]]

### 第三层：上下文压缩 → 详见 [[RTK]]、[[Caveman]]、[[headroom]]、[[context-mode]]

四个压缩工具互补，不互斥：

| 工具 | 压缩什么 | 典型节省 |
|------|----------|----------|
| [[RTK]] | 终端命令输出 → 上下文 | 89% |
| [[Caveman]] | AI 回复输出 | 65-75% |
| [[headroom]] | 所有进上下文的内容 | 47-92% |
| [[context-mode]] | MCP 工具结果 + 会话连续性 | 98%（工具输出） |

最低成本起点：先装 RTK + Caveman，五分钟见效。详见 [[上下文压缩]]。

### 第四层：代码图谱 → 详见 [[Graphify]]、[[CodeGraph]]

核心洞察：**真正贵的是"找代码"，不是"读代码"**。没有图谱时，AI 反复 grep → 读文件 → 发现漏了 → 再 grep，消耗大量工具调用和上下文。图谱让 AI 在动手读文件之前，就知道该读哪里。

### 第五层：多 Agent 架构

单 Agent 为什么越用越贵？规划、写代码、跑测试、Review 全在一个会话→上下文越积越长→所有任务用同一个模型配置→职责越混→上下文越胖→成本越高。

**Orchestrator-Worker 模式**的核心：让每个 Agent 只看当前步骤的相关内容，而不是把整个任务的所有背景都塞进每一次调用。

```
Orchestrator（协调器）
  ├── 负责规划、拆解、调度、汇总，不亲自读大量文件
  └── Worker A：定位 Bug（只看图谱和入口文件）
      Worker B：修复代码（只看有 Bug 的文件）
      Worker C：补单测（只给修复后文件 + 现有测试）
      Worker D：写 changelog（只给 git diff + 模板）
```

**数据流转四原则**（详见 [[Agent间数据流转约定]]）：
1. **输出格式结构化**——JSON 传递，比自然语言更紧凑可靠
2. **进度文件追踪状态**——`.agent/progress.json`，Orchestrator 不需要回放历史
3. **Worker context 精心裁剪**——明确告知"只读哪些东西"
4. **临时文件及时清理**——任务完成后归档或删除 `.agent/`

**并行执行收益**：

| 并行 Worker 数 | 实际加速倍数 |
|----------------|-------------|
| 2 个 | 1.5–1.8 倍 |
| 3 个 | 2.2–2.6 倍 |
| 4 个 | 2.8–3.4 倍 |

**成本对比**（API层重构端到端示例）：

| 模式 | Token 消耗 | 说明 |
|------|-----------|------|
| 单 Agent 全程跑 | 800K–1.2M | 带全程历史，每轮重新处理同一批背景 |
| Orchestrator-Worker | 100K–150K | 每步只看相关内容，**节省 70-85%** |

## ⚠️ 六大误区

| 误区 | 真相 |
|------|------|
| 上下文越多越好 | 噪音越多→模型分神→越贵但不越准。该读哪个文件比读多少更重要 |
| MCP 越多越强 | 选择空间变大→决策变慢→错误调用概率更高→低频工具贡献更多成本 |
| 所有 Agent 都上最强模型 | 这不是优化质量，是放弃路由。每个环节默认最贵 = 直接烧钱 |
| 聊天记录当长期记忆 | 长会话方便但不是记忆系统→成本越来越高→关键信息越来越难提取 |
| 只看单价不看总成本 | 便宜模型如果导致更多重试和搜索，总成本未必更低。看**完成一次任务的总成本** |
| Prompt 越短越好 | 丢掉必要 few-shot 或格式说明→第一次输出不合格→反复重试。压缩要精准不是一味缩短 |
