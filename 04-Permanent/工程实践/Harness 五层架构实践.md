# Harness 五层架构实践

> 核心设计思想只有一句：==把上下文当预算来管理==。分层标准不是"按功能分类"，而是"按何时被读取"——常驻的极小，深的按需加载。

![[07-Attachments/harness-practice/harness-五层架构图.jpg]]

---

## 第一层：常驻入口层（CLAUDE.md + CLAUDE.local.md）

放角色、代码偏好、流程触发规则、G1–G8 门禁速查。

关键设计：**`CLAUDE.local.md` 自包含、不依赖全局 `@import`**——新项目只需拷一份模版就能独立运作，**5 分钟接入**即可让全新项目跑通 harness。

| 目标 | 效果 |
|------|------|
| 项目流程规范彼此隔离、互不串味 | 主会话常驻上下文压到 ≤8K |

> [!tip] 与 [[Harness Engineering#驾驭层架构全景]] 的对照
> 常驻入口层对应"系统提示词工程"组件的精简版——只留索引级信息，深度内容下沉到后续层。

---

## 第二层：原子规则层（rules/，7 个）

每个规则单一职责、可被按需引用。本质是把踩过的坑固化成强制约束。

| 规则 | 约束 | 背后的事故 |
|------|------|-----------|
| `build.md` | 禁用 `mvn -am`，只编译单模块 | `-am` 触发全量解析卡死 |
| `branch-hygiene.md` | 提预发前必须先合 master | 多分支集成构建报"找不到符号" |
| `code-search.md` | 禁 Grep 搜 Java 结构、禁 LSP，强制走 kbase | 符号搜索误命中、上下文浪费 |

> 每条规则都是一次事故的墓志铭。坑只踩一次，之后由规则兜底——这是 harness 最朴素也最值钱的复利。

> [!note] 与 [[护栏与钩子系统]] 的关系
> 原子规则是"声明式策略"的具体实例——规则文件定义了"不该做什么"，hook 系统在运行时强制执行这些约束。两者配合形成"规则外置 + hook 拦截"的完整防线。

---

## 第三层：角色 Agent 层（agents/）

全套框架的发动机，把"全能主会话"拆成职责清晰的流水线。**最终精简为 12 个 agent**（从早期 24 个合并而来）。

### 四类角色

| 类别 | 角色 | 为什么不能合并 |
|------|------|--------------|
| 调度+合成 | dispatcher（路由）、orchestrator（合成） | dispatcher 只读 state.json 做路由决策，orchestrator 只读 phases/*.md 做合成确认——职责正交，合并会导致"裁判兼选手" |
| 三角色评审 | requirement-analyst、tech-architect、quality-guardian | 业务/技术/质量必须独立思考、互不看到对方初稿，才有对抗性；合成后反而更省 token |
| 流程执行 | plan-generator → developer → verifier → deployer → tester | 每个节点的 system prompt 和可用工具完全不同——合并意味着同时暴露所有工具，等于没约束 |

### 薄主会话三条铁律

1. **主会话只听 dispatcher**：dispatcher 读 state.json 返回"下一步调谁"，主会话照做，禁止自己 Read phases/*.md / evidence.json
2. **职责隔离**：dispatcher 只管路由、orchestrator 只管合成、developer 只管编码、verifier 只管检查，每个 agent 的可用工具严格受限
3. **上下文 ≤8K**：主会话只加载 CLAUDE.md + 触发规则 + 最近一条 dispatcher 指令

![[07-Attachments/harness-practice/19节点链与门禁.jpg]]

### 代价诚实清单

| 得到 | 付出 |
|------|------|
| 职责隔离，杜绝上下文污染与越权 | 链路变长，需求要在多个 agent 间转交 |
| 状态外置，可中断可续跑 | 调试更难，bug 要跨 agent 追 |
| 单 agent 上下文可控、可并行 | agent 间通信有 token 与协调开销 |

> [!important] 与 Devin "脑机分离"的对照
> Devin 从第一天做了"脑机分离"：推理（大脑）在沙箱外执行，执行环境（机器）无权访问大脑状态。我的 harness 走了更轻量的路——不隔离进程，而是通过 agent 职责隔离 + 文件交接达到类似效果。详见 [[Dispatcher 状态机与文件交接]]。

## 第四层：按需上下文层（context/，10 个）

完整流程详情、Pre-Mortem 模板、对抗辩论模板、证据链规范、TDD/ATDD 指南、记忆进化机制——只在进入对应阶段时才被 Read。

> 设计原则：上下文不是越大越好的"免费缓冲区"，而是需要精心管理的稀缺资源。每份 context 只含该阶段所需最小集，用完即释放。

**上下文的两个服务对象**：context 层既为"模型工作记忆"提供即时信息，又为"人类审计需求"提供可追溯记录——这两层目的不同但共享同一文件系统。

理论支撑：
- LLM 注意力呈 U 型分布，中部准确率显著下降（[[上下文腐蚀与注意力衰减]]）
- 声称支持 32K+ 的模型仅半数能在该长度保持可靠性能（NVIDIA RULER）

> [!note] 与 [[上下文注入]] 的关系
> 按需上下文层是"上下文注入管道"的实践落地——不是把所有信息一次性塞进去，而是按需、精准、按阶段注入。这正是 [[Context Engineering]] 的核心理念。

---

## 第五层：执行支撑层（skills/ 22 个 + commands/ 12 个 + evals/）

- **skills/（22 个）**：CLI 和研发工具链封装成 AI 可调用能力。核心 `ubase` 全家桶——一句自然语言就能自动拼查询、做换算、聚类异常摘要。
- **commands/（12 个）**：`/init-harness` 一键接入、`/harness-audit` 体检、`/learn` 沉淀规则。
- **经验三级进化**：踩坑 → lesson（单次）→ pattern（归纳）→ instinct（自动注入）→ 详见 [[经验三级进化]]

## 稳定性支点：eval 检测 + hook 拦截

规则定义"该怎么做"，验证机制确保"有没有做到"。

### G1–G8 门禁墙

每个门禁是确定性 Python 函数，检查产物存不存在、编译过不过、单测通没通。任一 gate FAIL → 流程退回 DEVELOPING——**不是"建议"，是"阻断"**。

| 门禁 | 检查什么 | 检查方式 |
|------|---------|---------|
| **G1** 产物存在性 | 关键产物文件是否生成 | 文件路径检查 |
| **G2** 规则命中率 | 必要规则是否被触发 | 预设规则命中率统计 |
| **G3** 编译通过 | 代码能否编译 | **真跑** `mvn compile` |
| **G4** 单测通过 | 单元测试是否通过 | **真跑** `mvn test` |
| **G5** 代码覆盖率 | 测试覆盖是否达标 | 覆盖率阈值检查 |
| **G6** ATDD 通过 | 验收测试是否通过 | **真跑** ATDD 测试 |
| **G7** 接口测试通过 | 接口测试证据是否完整 | 部署预发后接口测试 |
| **G8** 生产上线确认 | 生产环境是否健康 | 目前不强制，人工兜底 |

> 详见 [[G1-G8 门禁墙参考卡]]

### hook 拦截

Claude Code hook 在工具调用执行前拦截：
1. 状态文件写操作只允许编排层 agent 触发（其他绕过直接 reject）
2. 危险操作（`git push --force`、`rm -rf`）弹确认

> [!important] 核心原则
> ==流程强制执行必须从 LLM 推理中外置到确定性基础设施==。门禁必须是确定性代码，独立于上下文窗口，fail-closed（默认拒绝，只放行显式允许）。

业界共识对照：
- sd0x-dev-flow："hook-enforced dual review, state-machine gates that survive context compaction, and fail-closed safety"
- Apache Burr：每个 Agent 决策表达为状态机节点 + 可插拔持久化 + 实时追踪 UI

## 19 节点链 × intent×risk 动态裁剪

完整链路：需求评审→需求确认→方案设计→方案确认→Pre-Mortem→实施计划→验收标准确认→拉变更→建分支→建 worktree→开发→编译→单测→ATDD→证据链→部署预发→接口测试→上线确认→验收报告

| intent × risk | 实际走的节点 |
|--------------|-------------|
| `QUERY / NA` | 0 个——纯查询不该启动流程 |
| `BUG_FIX · LOW` | FAST_PATH：开发→编译→单测→证据→报告 |
| `FEATURE · MEDIUM` | 再加方案设计、对抗辩论、TDD |
| `FEATURE · HIGH` | 19 节点拉满 + ADR |

外加硬规则"改完必部署"：检测到真实代码改动 → 自动追加部署预发 + 接口测试。
