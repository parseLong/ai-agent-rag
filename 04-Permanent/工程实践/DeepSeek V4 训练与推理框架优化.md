# DeepSeek V4 训练与推理框架优化

> V4 的架构创新（[[mHC（多流约束残差连接）]]、[[注意力机制]] CSA/HCA、[[MoE架构]]）需要训练和推理框架的配套改造——新机制与旧框架打架的地方，都需要逐一解决。

## 训练框架的四大冲突与解法

### 1. Muon + 分布式训练冲突

**冲突**：[[Muon优化器]] 需要一次性拿到**完整参数梯度**才能更新参数，但传统分布式训练会把参数切成小块分到各机器上——两者天然冲突。

**解法**：重新设计参数分组和分配方式，让 Muon 和分布式训练共存，同时把机器间传输梯度的通信量**砍了一半**。

### 2. mHC 拖慢流水线

**冲突**：[[mHC（多流约束残差连接）]] 虽然效果更好，但会占更多显存、机器之间要传更多数据，拖慢流水线。

**解法**：三招组合：
- 定制计算内核
- 选择性"算过的不存、回头再算一遍"（用算力换显存）
- 调整流水线节奏

**结果**：mHC 带来的额外耗时仅 **6.7%**——几乎无感。

### 3. 长文本压缩段跨机器

**冲突**：训练超长文本时，一条文本被切开分给不同机器处理。[[注意力机制]] CSA/HCA 的 KV 压缩需要连续的一小段 token 一起压缩，但这段有时正好跨在两台机器的边界上。

**解法**：两步走通信——相邻机器先交换边界数据，再把压缩结果收拢重排。

### 4. 显存与算力的精细权衡

**冲突**：训练大模型时经常要在"多存一点中间结果省算力"和"少存一点省显存、用到了再算一次"之间做取舍。传统做法粒度太粗——要么整层都存、要么整层都重算。

**解法**：精细到**单个 tensor** 的自动化机制——开发者只需要：
1. 写一次前向逻辑
2. 标注几个关键 tensor

框架自动帮你算出"重算哪些最省"——不是全存也不是全重算，而是**按 tensor 粒度择优**。

## 推理框架优化

### KV Cache 架构

[[注意力机制]] CSA/HCA + SWA + 尾部 state 导致每层 KV 形状不同，传统 PagedAttention 失效 → 需要专门设计异构 KV Cache 架构。

详见 [[KV 缓存]]

### KV Cache 持久化

所有用户的 system prompt / 知识库文档内容不经常变，每次请求都重算 KV 极浪费资源。通用的做法是把算过的 KV 持久化到磁盘，下次命中直接读，跳过重算。

V4 的创新在于面对自己的异构 KV（CSA/HCA 压缩 + SWA 未压缩 + 尾部 state），**拆开分类存储**，特别是给 SWA 提出了**三档取舍方案**，让这套机制在 V4 的混合架构下依然可用。
