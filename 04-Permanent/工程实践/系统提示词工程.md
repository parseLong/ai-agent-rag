# 系统提示词工程

## 📖 定义

系统提示词工程是驾驭层的**方向盘**——通过精心构造的系统提示词定义 Agent 的身份、行为规则、输出风格和边界约束。Claude Code 的系统提示词不是一段静态文本，而是由**静态段（可缓存）+ 动态段（每会话生成）+ 缓存边界**组成的精密架构，位于 `src/constants/prompts.ts` 和 `src/utils/systemPrompt.ts`。

## 🧠 我的理解

系统提示词就是给 Agent 写的"岗位说明书"。但它比普通说明书更精密：(1) 有缓存边界——静态内容缓存以节省 token，动态内容每次生成；(2) 有优先级覆盖——不同来源的指令有不同优先级；(3) 有合并策略——多个来源的指令如何合并而不冲突。

## 🔑 Prompt架构的演进脉络

> 来源：[[Agent技术范式演变（2023-2026）]]

系统提示词的架构经历了从"深耦合"到"动静分离+渐进式加载"的根本性重构：

### 早期：单体大 System Prompt（深耦合）

早期构建 Agent 时，做法是**一个任务创建一个 Agent**。例如写一篇文章，会拆出"写作 Agent""编辑 Agent""绘图 Agent"等，每个 Agent 对应一段精心调试、独立存在的 System Prompt，包含：
- 人设定义
- 任务目标
- 任务要求
- 约束条件
- 注意事项
- 各种示例

这种模式下，Prompt Engineering 几乎等同于为每个任务写一段"小作文"——维护成本极高，随场景增多管理极其混乱。

### 当前：System Prompt 固化 + 渐进式加载（动静分离）

核心思路：==尽量固化 System Prompt，将动态内容剥离到外部文件==。

**固化层**：System Prompt 只保留最底层、最通用的系统级指令和基本行为规范——不太容易变化的部分，保持极度"稳定"。

**动态层**：原本堆积在 System Prompt 中的大量动态内容，被拆解并存储到外部文件系统，通过渐进式披露（Progressive Disclosure）按需加载：

| 动态内容类型 | 存储文件 | 说明 |
|------------|---------|------|
| Skill层面的沉淀 | `SKILL.md` | 执行某项具体任务的方法论、步骤要求、领域约束，构成 Agent 的"技能库"/"要求库" |
| 人设定义、用户偏好 | `USER.md` / `SOUL.md` | 用户个性偏好、人设规范 |
| 项目级规范、开发规范 | `CLAUDE.md` / `AGENTS.md` | 项目约束、代码风格、团队规范 |

Agent 在执行特定任务时，==动态渐进式加载==对应的 Markdown 文件，获取具体的操作指南。

> [!tip] 核心变化
> 不是"内容变了"，而是==上下文的组织形式变了==——从单体大 Prompt 到模块化挂载。System Prompt 变得更加纯粹和稳定，而易变的业务逻辑和领域知识通过结构化 Markdown 文件灵活挂载，实现了真正的"动静分离"。

### Claude Code 的实现

Claude Code 将这一理念落地为具体的分层架构（即本卡片上方所述的静态段+动态段+缓存边界），与演进方向完全一致：
- **静态段** ≈ 固化层：Agent 身份、行为规则、工具指南、风格规则（跨会话缓存）
- **动态段** ≈ 动态层：环境信息、用户记忆、MCP 配置、Skill 指令（每次会话生成）

详见 [[Agent技术范式演变（2023-2026）]]

## 🔑 系统提示词的分层构造

### 静态段（缓存，`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 前）

| 段 | 功能 | 行为塑形效果 |
|----|------|------------|
| `getSimpleIntroSection()` | Agent 身份定义 | "你是交互式 Agent" → 定义自我认知 |
| `getSimpleSystemSection()` | 系统行为规则 | 工具执行、权限模式、系统提醒处理规则 |
| `getSimpleDoingTasksSection()` | 任务执行哲学 | 代码风格、错误处理、安全原则 |
| `getActionsSection()` | 风险行动指南 | 删除、force-push 等危险操作需确认 |
| `getUsingYourToolsSection()` | 工具使用指南 | "优先用 Read 而非 cat" → 指导工具选择 |
| `getSimpleToneAndStyleSection()` | 输出风格规则 | 简洁、有用、不啰嗦 |
| `getOutputEfficiencySection()` | 效率指令 | 短响应、不重复 |

### 动态段（不缓存，每次会话生成）

| 内容 | 来源 | 作用 |
|------|------|------|
| Agent 工具指令 | 当前可用 Agent 类型 | 告诉模型可用子代理类型 |
| Skill 指令 | 当前安装的 Skills | 告诉模型可用技能 |
| Memory 指令 | `loadMemoryPrompt()` | 加载持久记忆 |
| 环境信息 | git status, cwd, model | 注入当前环境状态 |
| 语言偏好 | 用户配置 | 输出语言选择 |
| MCP 指令 | MCP 服务器配置 | 外部工具使用指南 |
| Token 预算 | 当前预算状态 | 上下文窗口管理 |
| Brief/Proactive 模式 | 特征开关 | 行为模式切换 |

## 🔑 缓存边界机制

```
系统提示词结构：
  ┌─────────────────────────────────┐
  │ 静态段 (cacheScope: 'global')    │ ← 缓存，跨会话复用
  │ - Agent 身份                     │    节省 ~2000 tokens/次
  │ - 行为规则                       │    API 缓存命中率高
  │ - 工具指南                       │
  │ - 风格规则                       │
  ├─ SYSTEM_PROMPT_DYNAMIC_BOUNDARY ─┤ ← 缓存分界线
  │ 动态段 (cacheScope: null)        │ ← 不缓存，每次生成
  │ - 当前环境信息                    │    包含变化的内容
  │ - 用户记忆                       │    git status, date, memory
  │ - MCP 配置                       │
  └─────────────────────────────────┘
```

**关键设计**：静态段利用 Anthropic API 的 prompt caching 机制，跨会话缓存。动态段每次重新生成。这节省了大量 API 成本——静态段约 2000 tokens，每次缓存命中都省一次输入 token 计算。

## 🔑 优先级覆盖系统

`buildEffectiveSystemPrompt()` 按优先级选择系统提示词：

```
优先级（从高到低）：
  1. Override system prompt（强制覆盖）    ← 用于测试或特殊场景
  2. Coordinator mode prompt（协调器模式）  ← 多代理协调场景
  3. Agent system prompt（Agent 提示词）   ← 子代理场景
  4. Custom system prompt（自定义提示词）   ← 用户自定义
  5. Default system prompt（默认提示词）   ← 标准场景
```

## 🔑 行为塑形的关键技术

### "工具选择指南"示例

```
系统提示词中写入：
  "Prefer dedicated tools over Bash when one fits"
  "Use Read (NOT cat/head/tail)"
  "Use Edit (NOT sed/awk)"
  "Use Glob (NOT find or ls)"

效果：模型倾向于使用结构化工具而非原始 Shell 命令
原因：结构化工具有更好的权限控制、输出格式、错误处理
```

### "风险行动指南"示例

```
系统提示词中写入：
  "Carefully consider reversibility and blast radius"
  "For destructive operations, check with the user before proceeding"

效果：模型在危险操作前主动征求用户同意
原因：LLM 本身不会"害怕"删文件——需要提示词注入这个"恐惧"
```

## 📝 个人笔记

系统提示词工程的核心洞见：**提示词不是"指令"，而是"行为规格书"**。Claude Code 的提示词长达数千 tokens，覆盖身份、行为、风格、安全、工具选择等维度——这不是简单的"你是一个编码助手"，而是**一份完整的 Agent 行为规范**。特别是缓存边界的设计：静态段缓存，动态段每次生成。这把 prompt caching 从"碰巧缓存"变成了**有意设计的缓存架构**。
