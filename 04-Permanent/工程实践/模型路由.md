# 模型路由

> 模型路由不是"一律用便宜的"，而是先把任务类型分对：复杂→强模型、简单→便宜模型、重复→稳定模型。

## 🧠 核心原则

**匹配优先，不是便宜优先**。架构设计需要长链路推理，该用贵的就用。写单测、补注释、生成提交信息的，便宜模型完全够用。

> 不是所有任务都要深度思考，也不是所有任务都要长答案。让一个模型用高推理档位做分类、摘要、提交说明，本质上是在拿推土机扫地。

## 🔑 任务→模型匹配表

| 任务 | 推荐模型档位 | 理由 |
|------|-------------|------|
| 写 UT | 便宜模型 | 模式稳定、模板化 |
| 写 commit | 便宜模型 | 输出短、风险低 |
| Code Review | 中高档模型 | 需要语义理解 |
| 架构设计 | 强模型 | 长链路推理 |
| 复杂 Bug 分析 | 强模型 | 多步假设和验证 |
| 批量分类/摘要 | 低价或 Batch | 规模大、成本敏感 |

## 🔑 升级链路：先过便宜模型，再升级

```
便宜模型先跑 → 判断复杂度 → 需要才升级
```

例子：
- 先让便宜模型分类（重构？Bug？文档？）
- 先做初筛（PR 有没有高风险？）
- 先做摘要（20页 → 1页）
- 再交给强模型做推理

这种**级联工作流**既省钱，又让贵模型花在刀刃上。

## 🔑 调预算旋钮

现在很多模型提供成本旋钮：

| 旋钮 | 影响 |
|------|------|
| `reasoning effort` | 想多深 |
| `thinking budget` / `thinkingBudget` | 思考 Token 预算 |
| `verbosity` | 答多长 |
| `max output tokens` | 输出长度上限 |

它们影响同一组东西：**想多深、答多长、花多少钱**。

Google Gemini 的 `thinkingBudget` 允许按任务控制思考 Token，简单任务甚至可以压到 0。OpenAI 建议很多工作负载从 `low` 的 reasoning effort 起步。

## 🔑 Skill / Agent / Command 都应绑定模型

模型路由不该只存在于"主对话框"的心智里，要**落实到执行单元**。

### Skill 绑定模型

以 CodeBuddy 为例，SKILL.md 文件头部可以直接声明模型和上下文策略：

```yaml
---
context: fork
model: deepseek-v4-pro
---
```

`model` 字段指定该 Skill 使用的模型，`context: fork` 表示把这类固定流程放到独立执行上下文里——适合任务边界清晰、可单独完成的 Skill。低复杂度 Skill 自动走便宜模型，不必把主会话的全部历史都带进去。

### Agent 绑定模型

```
Planner  → 中高档
Coder    → 中档 / 便宜
Reviewer → 强
```

### Command 绑定模型

很多命令本身就是固定模板，适合预设便宜模型。`/commit`、`/gen-ut` 这类命令，团队里每个人用都默认走便宜模型，不需要依赖个人记得切换。

> [!important] 从"靠人记得切"到"系统默认更省"
> 这类设计一旦固化，成本治理就从个人行为变成系统默认。
