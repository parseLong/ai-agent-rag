# 5条驭驭工程原则

> 这些原则不是任何单一厂商独有的——Anthropic、OpenAI、Cognition 都在各自文章中谈过，只是用词不同，但指向**同一组设计约束**。

## 🔑 4 大类别

| 类别 | 核心问题 | 该类别下的原则 |
|---|---|---|
| **① [[上下文工程]]** | 上下文不爆炸，agent 永远拿到对的信息 | System of Record / Memory Persistence / Progressive Disclosure |
| **② Interface / 沟通** | agent 看得懂 codebase，也能说清楚自己在做什么 | Legibility / ACI / Transparency |
| **③ Quality / 验证** | 写得对，不能 hallucinate | Taste Invariants / Evaluator-Optimizer loop / "Evals are everything" |
| **④ Process 纪律** | scale + iterate 不爆 | Simplicity / Throughput Changes Merge / Don't Build Multi-Agents |

## 🔑 5 条 OpenAI 原则

### 1. Legibility — 让 agent 能读懂 codebase / docs

> "Because the repository is entirely agent-generated, it's optimized first for **Codex's legibility**."

**核心精神**：把 agent 当新进工程师、为它优化 navigability——**为 agent 优化、不是为人**。

- **一致 schema 命名**：`get_user_by_id` 永远用这个格式
- **文件大小限制**：< 500 行、agent 一次能完整读进[[上下文窗口]]
- **清楚 tool description**：每个 tool 一行讲"干什么"
- **Poka-yoke 工具设计**：强制 absolute path / ISO 日期格式——让 agent 想犯错都犯不了

### 2. System of Record — 知识的唯一权威来源

> "The repository's knowledge base lives in a structured `docs/` directory **treated as the system of record**."

**核心精神**：所有真实知识住外部 docs、**不住 prompt**。

- **100 行 entry map**：AGENTS.md / CLAUDE.md 只放"地图"
- **`docs/` 结构化**：实际内容住子目录、agent 按需拉
- **prompt 绝不重复 docs 内容**：避免版本对不上
- **唯一权威、单向同步**：SoR 改了所有 agent 下次都读新版本

### 3. Progressive Disclosure — 从 small entry point 渐进深挖

> "Agents start with a small, stable entry point and **are taught where to look next**."

**核心精神**：**懒加载胜过热加载**——能晚一刻塞 context、就晚一刻。

- **AGENTS.md 100 行**：只放最高层
- **brief 而非 dump**：给 agent 任务时用 brief、不是倾倒整个 codebase
- **Skill `references/` 机制**：默认不进 prompt、需要才进
- **`@-import` 语法**：CLAUDE.md 按需拉

> SoR 提供目的地，PD 是导航机制——三者配对。

### 4. Taste Invariants — 用 lint 强制不变量

> "By enforcing invariants, not micromanaging implementations, we let agents ship fast."

**核心精神**：**定义边界、不细管实作**——让 agent 在你划好的格子里自由冲。

- **黄金准则 list**："偏好 composition 而非 inheritance"、"文件 < 500 行"
- **自动化 Linter**：违规 code → CI 自动拒绝 merge
- **拒绝"AI slop"**：设定品味基准要求 agent 不断重构、简化

### 5. Throughput Changes Merge Philosophy — agent 高吞吐 → QA 变瓶颈

> "...3.5 PRs per engineer per day... **the bottleneck became human QA capacity**."

**核心精神**：**Quality gate 从"人读过"变成"机器跑过 + 人 spot-check"**。

- **自动 lint + test + acceptance gate**：CI 跑过才能 merge
- **agent 内建 critique step**：写完 code 再自审
- **LLM-as-judge 自动评分**：低于门槛退回
- **Human spot-check only**：人类只看最终样子

## 🔑 Anthropic 额外 3 条原则

| 原则 | 白话 | 来源 |
|---|---|---|
| **Simplicity** | 先用 basic LLM call、不要跳 multi-step agent | [Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents) |
| **Transparency** | agent 自己秀 plan、不要默默做完 | 同上 |
| **Memory Persistence** | context 满前先存外部、下次从读档开始 | [Multi-Agent Research](https://www.anthropic.com/engineering/built-multi-agent-research-system) |

> **Simplicity 是隐性 root** — 一上来就堆 multi-agent，其他所有原则的复杂度都会暴涨。Cognition "Don't Build Multi-Agents" = Anthropic "Simplicity"。

## 🔑 原则之间的关键关系

| 关系 | 描述 | 为什么重要 |
|---|---|---|
| **SoR + Memory + PD 三者配对** | SoR 提供目的地、Memory 跨 session 保存、PD 是导航 | 三者单独用不够、必须一起设计 |
| **Legibility ↔ Transparency 双向** | agent 能读才能 self-report；能 self-report 才能验证 legibility 有效 | 互为前提、缺一不可 |
| **Quality 是 Process 自动化前置** | 没写死 invariants + eval loop、人类无法把 review 交给 automation | 是 Process 纪律的必要条件 |

## 🔑 痛点 → 原则 → 实作

| 痛点 | 原则 | 实作 |
|---|---|---|
| Context 200k 满 | Progressive Disclosure + Memory Persistence | Skills `references/` / `CLAUDE.md` `@-import` |
| Agent 看不懂自家 codebase | Legibility + ACI | `AGENTS.md` 100 行 / poka-yoke 工具设计 |
| 多 agent desync | System of Record | `docs/` + `.coord/` shared-memory |
| 随机 drift / Review 漏抓 | Taste Invariants + Transparency | acceptance gate YAML / evaluator-optimizer loop |
| Agent PR 速度 > Human QA | Throughput Changes Merge | mandatory preset / LLM-as-judge |
