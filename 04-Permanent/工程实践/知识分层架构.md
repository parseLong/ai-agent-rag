# 知识分层架构

> 在 AI Team 系统中设计的三维正交知识体系——回答三个问题：==知识在哪（存储层）？知识是什么（类型）？知识多可信（成熟度）？==

---

## 三维正交设计

| 维度 | 回答的问题 | 划分 |
|------|------------|------|
| **存储层** | 知识存在哪里？ | Layer 0-P → 0-T → 1 → 2 → 3 |
| **知识类型** | 知识描述的是什么？ | model / decision / guideline / pitfall / process |
| **成熟度** | 知识经过多少验证？ | draft → verified → proven |

> [!important] 设计哲学
> 三个维度==正交==（互不影响），任何一条知识都可以独立在三个维度上定位。这遵循 MECE（互斥且完全穷尽）原则，确保知识条目只属于一个类型、一个层级、一个成熟度。

---

## 五层存储架构

```
┌──────────────────────────────────────────────────────────────┐
│                      五层知识存储                              │
├──────────┬──────────────────────────────┬────────────────────┤
│ Layer 0-P │ 个人偏好 (～/.ai-team/)       │ 纯本地，不共享     │
│ Layer 0-T │ 团队约定 (team-conventions/) │ 团队级，Git 共享   │
│ Layer 1   │ 技术知识 (tech-wiki/)        │ 团队级，跨项目     │
│ Layer 2   │ 业务知识 (biz-wiki/{domain}/)│ 团队级，按领域     │
│ Layer 3   │ 项目知识 (docs/knowledge/)   │ 项目级，随项目走   │
└──────────┴──────────────────────────────┴────────────────────┘
```

### 各层详解

| 层级 | 范围 | 共享边界 | 示例 | 生命周期 |
|------|------|----------|------|----------|
| **Layer 0-P** | 个人偏好 | 纯本地，不共享 | "我喜欢 4 空格缩进" | 随个人走 |
| **Layer 0-T** | 团队约定 | 团队级，Git 共享 | 代码规范、Commit 规范 | 相对稳定，团队"宪法" |
| **Layer 1** | 技术知识 | 跨项目通用 | "Spring Boot 多租户拦截器设计模式" | 跨项目持久 |
| **Layer 2** | 业务知识 | 按领域 | "广告审核流程：提交→机审→人审→上线" | 随业务领域持久 |
| **Layer 3** | 项目知识 | 仅当前项目 | "本项目数据库用的是 TencentDB for MySQL 8.0" | 随项目归档 |

### 知识"向上提升"机制

> [!tip] 关键设计
> Layer 3 的项目知识，如果被判定为==跨项目通用==，会自动提升到 Layer 1 或 Layer 2。

```
Layer 3 (项目内)
  │  所有类型，maturity 为 draft
  │
  ├──→ Q1: 是否项目特有？ → 是：留在 Layer 3
  ├──→ Q2: 是否通用技术？ → 是：提升到 Layer 1 (tech-wiki)
  └──→ Q3: 是否通用业务？ → 是：提升到 Layer 2 (biz-wiki)
```

## 五种知识类型

遵循 MECE 原则，每条知识==只属于一个类型==：

| 类型 | 定义 | 示例 | 最常查询的阶段 |
|------|------|------|----------------|
| **model** | 实体定义、数据结构、关系图 | "广告计划包含预算/出价/投放时段三个核心字段" | ANALYSE_PRODUCT, ARCHITECT |
| **decision** | 技术选型、架构决策及理由 | "选择事件驱动而非 RPC 同步，因为广告状态变更需要解耦" | ANALYSE_TECH, ARCHITECT |
| **guideline** | 推荐做法(recommend) 或禁止做法(avoid) | "公共模块变更后的兼容性检查清单" | IMPLEMENT, BUILD_VERIFY |
| **pitfall** | 已知风险、故障模式、排查步骤 | "广告预算扣减在高并发下会超扣" | ANALYSE_TECH, BUILD_VERIFY |
| **process** | 业务流程、状态机、操作步骤 | "广告审核：提交→机审→人审→上线" | ANALYSE_PRODUCT |
