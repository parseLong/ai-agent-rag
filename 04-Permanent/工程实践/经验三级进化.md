# 经验三级进化

> harness "越用越聪明"的核心机制——踩坑经验从一次性记录逐步晋升为自动注入的规则。每一级晋升需人工确认，防止错误经验扩散。

---

## 三级模型

| 级别 | 名称 | 定义 | 生命周期 | 实例 |
|------|------|------|---------|------|
| **L1 lesson** | 单次记录 | 第一次踩坑，写成教训记录 | 单次、项目内 | "mvn -am 在这个项目卡死了" |
| **L2 pattern** | 归纳 | 第二次在不同项目又遇到，归纳出规律 | 跨项目 | "Mac + system-scope 依赖 = 禁用 -am" |
| **L3 instinct** | 自动注入 | 第三次验证后晋升，自动注入所有新项目的规则文件 | 全局、自动 | `build.md` 规则：禁用 `mvn -am` |

### 进化路径

```
踩坑 → lesson(单次记录) → pattern(归纳规律) → instinct(自动注入规则)
         ↑ 人工确认           ↑ 人工确认            ↑ 人工确认
```

> [!important] 每级晋升都需人工确认
> ==防止错误经验扩散==。一个只在特定环境偶然触发的 lesson，不应该直接晋升为全局 instinct——必须在不同项目、不同上下文中验证后才能晋升。

## 与相关概念的对照

### vs [[记忆到技能转化]]

| 维度 | 经验三级进化 | 记忆到技能转化 |
|------|------------|--------------|
| **对象** | Harness 规则 | Agent 能力 |
| **L1** | lesson（事故记录） | L1记住事实 |
| **L2** | pattern（归纳规律） | L2总结规则 |
| **L3** | instinct（自动注入） | L3形成技能 |
| **驱动力** | `auto-learn` 规则驱动 | Agent 自我改进循环 |
| **验证** | 人工确认每级晋升 | 实践验证 |

两者结构几乎同构，但对象不同：经验三级进化管理的是==约束规则==的成熟度，记忆到技能转化管理的是==Agent能力==的成熟度。

### vs [[知识分层架构]]的三级成熟度

| 维度 | 经验三级进化 | 知识成熟度 |
|------|------------|----------|
| 级别 | lesson → pattern → instinct | draft → verified → proven |
| 机制 | 踩坑驱动晋升 | 人工验证晋升 |
| 产物 | rules/ 文件 | 知识库文档 |

> [!tip] 共性洞察
> 三级递进是知识管理的==通用模式==：单次→归纳→自动化。无论对象是规则、技能还是知识文档，核心逻辑都是"从偶然到规律到制度化"。

## 实现：`auto-learn` 规则驱动

以 `mvn -am 卡死` 为例的完整进化流程：

1. **第一次踩坑**：在项目 A 中 `mvn -am` 卡死 → 写成 `lesson`（"mvn -am 在项目 A 卡死"）
2. **第二次遇到**：在项目 B 中又遇到 → 归纳为 `pattern`（"Mac + system-scope 依赖 = 禁用 -am"）
3. **第三次验证**：在项目 C 中验证 → 晋升为 `instinct`，自动注入所有新项目的 `build.md`

触发方式：`/learn` command 或 `auto-learn` 规则自动触发。

## 与 VikingMem 的对比

VikingMem (VLDB 2026, ByteDance) 证明：更少 Token 留存 + 更智能的组织 > 全量保留（16.82% Token 留存得分 75.80，朴素 RAG 100% 留存仅 63.81）。

经验三级进化与此呼应：==不是把所有踩坑细节都保留，而是提炼出 pattern 和 instinct==——从全量保留（lesson）到精炼留存（instinct），每一步都是信息压缩和价值提纯。
