# 多智能体协作模式

## 📖 定义

多智能体协作模式是指多个 Agent 之间组织协作的==架构范式==。不同模式解决不同问题——就像人类团队协作也不止一种方式（老板分配、流水线、头脑风暴、部门分治、考试阅卷、导师带徒弟）。

## 🧠 七种核心模式

### 模式一：主从委托（Orchestrator-Worker）

**类比**：项目经理分配任务给不同开发者，各自完成后经理汇总交付。

- Orchestrator 负责理解意图 → 任务拆解 → 分配 → 等待 → 汇总结果
- Worker 独立上下文、独立工具、独立执行
- **优势**：并行执行快、上下文隔离、Worker 可各专精
- **劣势**：Orchestrator 是单点瓶颈、Worker 间无法直接通信
- **代表产品**：Claude Code（subagent）、AutoGen、DelegateManager
- **典型场景**：竞品调研——"调研 A、B、C 三家的定价策略"，Orchestrator 拆为三个 Worker 并行搜索，汇总后给出定价建议
- **适用**：可明确拆解为独立子任务的场景

> [!example] SubAgent 实现细节（主从委托的生产级实践）
> 子Agent本质上是一个工具 `delegate_task`，背后创建独立 Agent 实例执行。
>
> **delegate_task 工具定义**：
> | 参数 | 说明 |
> |------|------|
> | `goal` | 任务描述——子Agent要完成什么 |
> | `context` | 上下文——主Agent传递的背景信息 |
> | `role` | `leaf`（叶子，不能再委托）或 `orchestrator`（可以继续委托子Agent） |
>
> **子Agent 身份提示词构建**：由五块拼接而成——
> 1. 任务目标（来自 `goal`）
> 2. 上下文（来自 `context`）
> 3. 执行纪律（`DELEGATE_EXECUTION_DISCIPLINE`）
> 4. 工作边界（`DELEGATE_WORK_BOUNDARIES`）
> 5. 结果格式要求（`DELEGATE_RESULT_FORMAT`）
> 6. 技术约束：角色类型、最大迭代次数、禁止工具列表

> [!important] 权限隔离设计（leaf 子Agent）
> leaf 子Agent被严格限制权限，防止不可控行为：
>
> | 权限 | 允许/禁止 | 原因 |
> |------|----------|------|
> | `delegate_task` | **禁止** | 防止递归委托（子Agent再委托子Agent） |
> | `memory` | **禁止** | 防止并行写入冲突——多个子Agent同时修改全局持久状态会产生数据竞争 |
> | `clarify` | **禁止** | 子Agent在线程池中运行，无法与用户交互（会阻塞等待用户回复，而用户看不到） |
> | `load_skill` | **允许** | 通过全局 registry 继承，无冲突风险 |
> | MCP 工具 | **允许** | 通过 `enabled_toolsets` 继承，只读类工具天然安全 |
> | `todo` | **允许** | 独立 `TodoStore` 实例，不共享主 agent 计划 |
> | 内置工具 | **允许** | 搜索、读文件等能力类工具，只读无副作用 |

> [!warning] 四维资源控制
> 生产级系统必须从四个维度同时控制子Agent资源消耗：
>
> | 维度 | 机制 | 参数 | 说明 |
> |------|------|------|------|
> | **并发控制** | `ThreadPoolExecutor(max_workers=3)` + `_active_count` 计数器 | 最多 3 个子Agent | 防止资源耗尽 |
> | **迭代预算** | 子Agent独立 `IterationBudget` 实例 | 50 次 | 不消耗主Agent配额，子Agent有自己的"油耗表" |
> | **超时保护** | `future.result(timeout=600)` | 600s | 超时强制取消，防止卡死 |
> | **深度限制** | `current_depth >= max_depth` 时强制 `role="leaf"` | 最多 1 层委托 | 防止委托链无限嵌套 |

> [!example] 结构化返回结果 DelegateResult
> 子Agent执行完毕后，返回结构化结果供主Agent判断质量：
>
> | 字段 | 说明 |
> |------|------|
> | `goal` | 原始任务描述 |
> | `success` | 子Agent自称是否成功 |
> | `final_answer` | 子Agent给出的最终答案 |
> | `tool_calls_count` | 调用了多少次工具 |
> | `iterations_used` | 消耗了多少次迭代 |
> | `tokens_used` | 消耗了多少 Token |
> | `duration_seconds` | 执行耗时 |
>
> **关键洞察**：主Agent需要据此判断子任务质量——如果子Agent用了 50 次迭代（预算耗尽）却声称成功，应怀疑结果完整性。这些指标也会展示在结果摘要中，帮助用户了解执行效率。

### 模式二：接力传递（Sequential / Pipeline）

**类比**：工厂流水线——每道工序加工后传给下一道。

- 每个 Agent 的输出是下一个 Agent 的输入
- **优势**：流程清晰可控、每步可审计、前序为后续提供结构化上下文
- **劣势**：串行执行慢、一步出错后面全受影响、反馈循环困难
- **代表产品**：GitHub Copilot Workspace（需求→设计→编码→验证）、传统 CI/CD pipeline
- **典型场景**：用户注册功能——需求分析Agent出需求文档 → 架构设计Agent出接口和数据模型 → 代码实现Agent出源代码 → 测试验证Agent出测试报告
- **适用**：有明确先后顺序依赖的工作流

### 模式三：对等讨论（Peer Discussion / Debate）

**类比**：圆桌会议——多位专家轮流发言、互相质疑补充，最终达成共识。

- 多 Agent 多轮对话辩论 → Moderator 判断共识
- **优势**：多角度思考避免盲点、激发创造性方案
- **劣势**：Token 消耗极高、收敛时间不确定
- **代表产品**：ChatDev、CAMEL、MetaGPT
- **典型场景**：技术选型——"登录系统用 Session 还是 JWT？"，安全专家主张 Session（可即时撤销），架构师主张 JWT（适合微服务分布式），产品经理提供业务约束，多轮辩论后 Moderator 判断共识达成
- **适用**：决策类问题（技术选型、方案评审）

### 模式四：层级分治（Hierarchical）

**类比**：公司组织架构——CEO管VP，VP管总监，总监管员工。

- 上级只和直属下级通信，不跨级
- **优势**：适合超大型任务、自然模拟团队协作
- **劣势**：层级增加延迟和通信成本、容易"传话走样"
- **代表产品**：MetaGPT（CEO/CTO/PM/Engineer角色）、企业级 Agent 编排平台
- **典型场景**：电商后台管理系统——CEO Agent拆为前端/后端/测试三大模块，各经理 Agent 再分配 Worker 实现具体页面、API、测试
- **适用**：大型工程项目（多模块协作、多角色分工）

### 模式五：竞争选优（Competition / Voting）

**类比**：考试阅卷——多位老师分别打分，取平均分或投票选出最佳。

- 同一问题分发给多个Agent → Judge Agent 评估投票
- **优势**：结果质量高（多方验证）、鲁棒性强
- **劣势**：Token 成本是单Agent的N倍
- **代表产品**：AlphaCode、self-consistency、LLM-as-Judge 框架
- **典型场景**：代码 bug 检测——同一代码分发给多个Agent，各自用不同策略（逐行分析/执行追踪/模式匹配），Judge Agent 投票选出最佳诊断
- **适用**：高可靠性要求（代码bug检测、安全审计）

### 模式六：评估反馈（Generator-Critic）

**类比**：作者写初稿 → 编辑审稿批注 → 作者修改 → 直到编辑满意。

- Generator 生成 → Critic 评估 → 反馈修改 → 循环直到达标
- **优势**：输出质量持续提升、可用不同模型（快模型生成+强模型评估）
- **劣势**：多轮迭代增加延迟、Critic 可能不可靠
- **代表产品**：Reflexion、Self-Refine、Constitutional AI、ReAct 循环中的验证步骤
- **典型场景**：写 RAG 技术博客——Generator 生成初稿，Critic 指出"缺少代码示例、评估部分太浅"，Generator 修改后再评估，循环直到 Critic 满意
- **适用**：需要高质量输出的创作任务

### 模式七：动态路由（Router / Dispatch）

**类比**：医院分诊台——根据症状分配到不同科室。

- Router 意图识别 → 路由到专家 Agent
- 每个专家 Agent 有专属 system prompt + 专属工具集
- **优势**：响应快、专家 Agent 高度专注、容易水平扩展
- **劣势**：Router 可能分错类、跨领域问题需转交
- **代表产品**：Claude tool_use routing、Skill advertise→load 路由机制、各大厂智能客服
- **典型场景**：线上故障排查——"报了空指针错误查日志"，Router 路由到 DevOps Agent（加载日志查询工具定位错误）；接着"帮我修代码"，Router 再路由到 Coding Agent（加载编辑器工具修改并测试）
- **适用**：意图明确可分类的场景

## 🔑 综合对比

| 模式 | 并行性 | 通信开销 | 实现复杂度 | Token成本 | 最佳场景 |
|------|--------|---------|-----------|----------|---------|
| 主从委托 | 高 | 低 | 中 | 中 | 可拆解的独立子任务 |
| 接力传递 | 无 | 中 | 低 | 低 | 有顺序依赖的流水线 |
| 对等讨论 | 低 | 高 | 高 | 极高 | 决策/方案选择 |
| 层级分治 | 高 | 高 | 极高 | 高 | 超大型工程项目 |
| 竞争选优 | 高 | 低 | 中 | 高(N倍) | 高可靠性要求 |
| 评估反馈 | 无 | 中 | 中 | 中高 | 创作/生成类任务 |
| 动态路由 | 无 | 极低 | 低 | 低 | 意图明确可分类 |

> [!important] 生产中的组合使用
> 成熟的系统很少只用单一模式。以本项目（个人助手 Agent）为例，三层模式叠加：
>
> **层次1：动态路由** — Skill advertise 阶段匹配能力，Router 根据用户意图选择对应 Skill
> **层次2：主从委托** — DelegateManager 作为 Orchestrator，复杂任务拆解后并行子 Agent 执行再汇总
> **层次3：评估反馈（隐式）** — ReAct 循环中 LLM 自评估工具执行结果，判断是否满意并决定继续/结束

> [!tip] Claude Code 的 Multi-Agent 组合
> Claude Code 同样组合了多种模式：
> - **Router**：根据任务类型选择是否需要子 Agent
> - **Orchestrator-Worker**：主 Agent spawn 子 Agent 并行搜索/编辑
> - **Generator-Critic**：编码后运行测试 = 生成 + 验证循环
> - **Pipeline**：plan → implement → test → commit 的阶段流

## 🔑 核心收益

| 收益 | 原理 | 效果 |
|------|------|------|
| 上下文隔离 | 子agent独立上下文窗口 | 主agent只看任务+结果 |
| 并行执行 | 独立子任务同时运行 | 3个20s任务→并行只需20s |
| 失败隔离 | 子agent失败不影响其他 | 基于部分成功继续 |
| 专家化 | 不同system prompt | 各司其职 |
| 质量提升 | 多Agent交叉验证 | 减少幻觉和遗漏 |

## 🔑 关键设计挑战

| 挑战 | 解决思路 |
|------|---------|
| 任务拆解粒度 | 依赖LLM判断 + 经验阈值（3-5个子任务为宜） |
| 结果聚合 | 主agent综合推理 / 结构化合并模板 |
| 资源控制 | 并发数限制+超时+深度限制+工具权限收窄 |
| 状态一致性 | 子agent间需共享信息时，通过主agent转发/共享工具注册表/共享内存区 |
| 错误传播 | 每步输出验证+错误时回退重试+置信度传递 |
| 成本控制 | 按需使用（简单任务不拆分）+弱模型做Router/Worker |
