# 多智能体形态分类

> 2026年上半年，Agent 圈最显著的变化是单 Agent 开始过时了。不是 AI 不够强，是任务变复杂了。多 Agent 协作从论文走到了产品，名词也跟着分裂出来一堆——Sub-agent、Agent Team、Multi-Agent、Agent OS，听起来近，差别大。

## 四种核心形态

### 1. Sub-agent：一次性、隔离、向上汇报

主 Agent 派一个或一组子 Agent 出去干活，子 Agent 有自己的上下文窗口，干完只把结果汇总回来，期间互不通讯。

- 本质是**上下文隔离 + 并行加速**
- Claude Code 文档里描述为 fire-and-forget worker
- VS Code 1.109（2026年2月）把 Sub-agent 做成 IDE 一等公民，直接支持多个子 Agent 并发跑、可视化看进度
- **体验**：token 翻倍但任务质量明显跳一档，特别是在长程研究和编程场景

### 2. Agent Team：长跑、共享任务、互相通讯

多个 Sub-Agent + 一个 Team Lead，队员之间能直接对话、能争论、能在发现问题时彼此预警。

- 有 mailbox 互相通讯机制
- 共享任务列表
- **代价**：token 成本起飞
- **问题**：经常出现"两个队员都觉得对方应该先动"的死锁

### 3. Dynamic Workflows：动态编排

Anthropic 5月28日跟 Opus 4.8 一起发布，中文名"动态工作流"。

- 不是手动派子 Agent，而是让 Claude 自己写一份 orchestration 脚本
- 动态决定拉几十甚至上百个子 Agent
- 并行跑、独立验证、再交叉收敛
- 官方演示场景：大型代码库迁移、企业级 bug 跨服务排查
- **效果**：真有效果，但成本是真高

### 4. Agent OS：个人级 AI 操作系统

没有官方定义，开发者社区里折腾的形态：

- 一个 CEO Agent 做规划
- 一个 COO Agent 做路由
- 一个研究员做长程任务
- 一个个人助理采集屏幕和麦克风做上下文
- 本质是用多个开源 Agent 拼一个个人级 AI 操作系统

### Multi-Agent：最大伞概念

上面所有的形态都属于 Multi-Agent。代表性产品/实践：

| 产品 | 多智能体实现 |
|------|-------------|
| Cursor v3 | 内置最多 8 个并发 Agent |
| Google Antigravity 2.0 | Agent Teams |
| GitHub Copilot | multi-agent workspace |
| amux | 工具不可知 orchestrator（tmux + SQLite 任务板 + git worktree） |

## 爆发的三条线索

### 线索1：上下文窗口不再是瓶颈，工程能力反而成了瓶颈

1M token 已是 Claude、Gemini 的标配，但真把 1M 塞满，模型注意力切得稀碎。与其往一个上下文里硬塞，不如分裂成多个隔离上下文跑，每个干净、各自负责一块。这是 Sub-agent 流行的根本原因。

### 线索2：Agentic 任务的真实形态就是分布式系统

Hermes Agent v0.13 的 Kanban 多 Agent 看板引入了心跳、重认领、僵尸检测、retry，这些机制全是从分布式系统直接搬来的。本质是把模型当 worker，把 orchestration 当成分布式调度问题。范式一旦确立，多 Agent 协作就只是工程实现问题了。

### 线索3：企业用户真正进场

- OpenAI 4月22日发的 Workspace Agents
- 6月4日上线的 ChatGPT Workspace Agents
- 全是冲着团队协作来，目标用户从开发者扩到销售、市场、客服
- 这是从"session-based chat"到"fleet-based agent work"的范式迁移

## 演进回顾

> 2024 年我们说 Agent 是工具调用，2025 年说 Agent 是工作流，2026 年终于说到了"Agent 团队"这个层级。一个跑得好的 Agent 团队，不再像一个工具，更像一个真正的部门。
