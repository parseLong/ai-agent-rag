# 技能自创建闭环与渐进式披露

> [!quote] 核心定义
> Hermes 最独特的能力——Agent 自主从经验中创建和管理技能，形成持续的自我改进闭环。渐进式披露是闭环运转的底层基础设施——把技能加载成本从 O(N) 降到 O(被实际用到的)。

---

## 自我改进闭环

```
经验积累 → 技能Nudge触发 → review agent评估
    → 创建/更新技能 → 安全扫描 → 保存
    → 后续任务加载技能 → 发现不足 → patch更新
    → 持续优化
```

这是 Hermes 与 OpenClaw 的根本差异：**Hermes 让 Agent 自己造工具，OpenClaw 让人类写工具、Agent 使用**。

---

## 渐进式披露 — 三级访问

### 要解决的根本矛盾

~/.hermes/skills/ 可能有数百到上千个技能，总规模几十 MB。如果全量注入 system prompt → context 直接爆。

### Tier 1: `skills_list` — 只看"目录"

```
skills_list(category="mlops")
```

返回 `name + description + category`（严格控制，不返回 tags/content/linked_files）。

### Tier 2: `skill_view(name)` — 看"完整说明书"

返回完整 SKILL.md + linked_files 路径清单（不返回内容）+ readiness_status + missing_required_environment_variables。

> [!important] readiness 在 tier 2 入口就告诉 Agent
> 缺凭证 → 直接报告"我需要 HF_TOKEN"。**早失败比晚失败便宜得多**——不用浪费一次工具调用发现 "no token"。

### Tier 3: `skill_view(name, file_path)` — 按需拉支撑文件

只四个子目录：`references/` / `templates/` / `assets/` / `scripts/`。

**文件找不到时返回完整文件树**（而不是只回404）——让Agent立即知道能选哪些，不用再多调一次工具。

## 元数据约束 — 协议契约而非建议

```
MAX_NAME_LENGTH = 64
MAX_DESCRIPTION_LENGTH = 1024   // ← 创建时硬校验
```

保证 tier 1 成本**不随技能数量退化**。如果某作者写 10KB description → 单这一条就让 tier 1 退化。

> **作者的自律不可靠，把约束做成代码**

## Token 成本对比

```
一次真实任务链路：
  Tier 1: skills_list → ~25K tokens（200个技能的元数据）
  Tier 2: skill_view("axolotl") → ~5K tokens
  Tier 3: skill_view + template → ~2K tokens
  总计 ~32K tokens

  如果没有渐进式披露，全量加载200个技能：
  ~1M tokens → 直接超出大部分模型context窗口

  节省比例：30+ 倍
```

## 路径遍历防护（双重）

```python
# 语法层防护
if has_traversal_component(file_path):
    return error("Path traversal ('..') is not allowed.")

# 解析后防护 — 终极兜底
target_file = skill_dir / file_path
traversal_error = validate_within_dir(target_file, skill_dir)
```

> 即使 Agent 被 prompt injection 诱导调 skill_view，也读不到 ~/.ssh/id_rsa。

## 技能安全 — 4级信任 + 6类威胁

| 信任级别 | 来源 | safe发现 | caution发现 | dangerous发现 |
|----------|------|----------|------------|--------------|
| **builtin** | 随Hermes发行 | ✅允许 | ✅允许 | ✅允许 |
| **trusted** | openai/anthropic | ✅允许 | ✅允许 | ❌阻止 |
| **community** | 其他来源 | ✅允许 | ❌阻止 | ❌阻止 |
| **agent-created** | Agent自创建 | ✅允许 | ✅允许 | ⚠️询问 |

6大类威胁：**exfiltration** / **injection** / **destructive** / **persistence** / **network** / **obfuscation**

---

## 技能自创建限制与安全

- 名称：仅小写字母、数字、连字符，最长64字符
- 内容：最大100,000字符（~36k tokens）
- 支撑文件：最大1 MiB
- 允许子目录：`references/`, `templates/`, `scripts/`, `assets/`
- Agent创建的技能经过与社区Hub安装相同的 `scan_skill()` 扫描

## 与 OpenClaw 技能系统对比

| 维度 | Hermes | OpenClaw |
|------|--------|----------|
| 加载时机 | LLM主动调skills_list/skill_view按需加载 | Agent启动时按activation字段选择性预加载 |
| 元数据约束 | **hardcode name≤64, description≤1024** | 无字符级约束 |
| 完整内容 | 总是按需 | always-on技能启动时就进system prompt |
| 自创建 | ✅ Agent自主创建/编辑/patch | ❌ 人工编写 |
| 成本模型 | **O(被实际用到的)** | O(always-on数量) |

> **根本差异**：Hermes 把"用什么技能"完全交给 LLM 决策；OpenClaw 把"用什么技能"部分交给配置。Hermes 更通用（任何规模都能 scale）；OpenClaw 更可控（保证关键技能总在线）。
