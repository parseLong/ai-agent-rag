# Mermaid任务画布

> Mermaid 无限画布不是把历史变成一段话，而是把历史变成一张可以继续执行的任务地图。它让 Agent 不只是回忆"发生了什么"，还可以判断"下一步应该做什么"——从线性扫描历史，变成结构化理解任务。

## 核心思想

普通 summary 表达的是"有哪些信息"，Mermaid 任务图表达的是"这些信息之间是什么关系"。线性列表像日志，适合记录发生过什么；**图结构像地图**，更适合告诉 Agent 现在在哪里、从哪里来、接下来往哪里走。

```
线性摘要（日志模式）:
- 搜索了港大学费
- 搜索了港中文学费  
- 搜索了港科大学费
- 确认了三所学校的精确学费
- 生成了三校对比表

Mermaid 画布（地图模式）:
flowchart TD
    A["搜索港大学费status: done"] --> D["精确学费确认status: done"]
    B["搜索港中文学费status: done"] --> D
    C["搜索港科大学费status: done"] --> D
    D --> E["汇总对比表status: doing"]
```

画布同时表达三层信息：

| 信息类型 | 图中如何体现 |
|---------|-------------|
| **任务结构** | 先搜三所学校，再精确确认，最后汇总 |
| **当前进度** | 前四步 done，正在汇总 doing |
| **符号索引** | 每个节点关联 tool summary，可追溯到 offload.jsonl |

## Mermaid 作为符号化压缩载体的六大特点

| 特点 | 说明 |
|------|------|
| **文本化** | 本质是代码，不是图片，LLM 可以直接读写 |
| **结构化** | 可以表达节点、边、状态、依赖关系 |
| **可视化** | 人类可以渲染成图来检查任务过程 |
| **可更新** | Agent 可以持续添加、修改、合并节点 |
| **可寻址** | 节点可以绑定 node_id、summary、result_ref |
| **通用性强** | Mermaid 是公开格式，不是自创密文 |

## 符号化压缩的最优载体

Mermaid 不是唯一的压缩载体，但它是目前最符合 [[符号化压缩]] 三条设计原则的选择：

### 原则一：符号必须是通用知识

Mermaid 语法在 GitHub、技术文档、教程中广泛存在，所有主流大模型都能准确读写它。它不是自创密文，而是模型预训练阶段大量接触过的通用格式。

### 原则二：符号的生成不能过于复杂或孤立

Mermaid 的语法极其简单（节点 + 箭头 + 标签），生成和理解的认知负担几乎一样。语义由**图的拓扑结构**而非**文本约定**决定——一个 LLM 生成的 Mermaid 图，另一个 LLM 都能准确理解。

### 原则三：表达要足够自由

Mermaid Flowchart 允许任意形状的节点、任意方向的连线、任意长度的标签、任意的嵌套和分支。模型可以自主决定：
- 将连续 5 个相似操作合并为一个宏观节点
- 为关键发现单独建立高亮节点
- 用虚线箭头表示弱关联，实线表示强因果
- 用不同节点形状区分不同类型的操作

> [!important] 三条原则是通用的，不是 Mermaid 专属的
> 这些原则来自 [[符号化压缩]] 的理论框架，适用于任何为 LLM 设计压缩符号的场景。Mermaid 恰好同时满足三条原则，因此成为最优载体。但理论上，任何满足这三条原则的格式都可以作为压缩载体。详见 [[符号化压缩]]。

## Flowchart vs StateDiagram

实验结论：**Flowchart 作为压缩载体的效果比 StateDiagram 好 ~15%**

| 维度 | Flowchart（流程图） | StateDiagram（状态图） |
|------|---------------------|----------------------|
| **核心表达** | 过程和结构 | 状态的单向变迁 |
| **拓扑自由度** | 极高：任意分支、合并、循环、双向箭头 | 有限：状态之间只有单向转移 |
| **节点语义** | 万能：可以是动作、决策、数据、结论 | 固定：每个节点就是一个"状态" |
| **类比** | 万能流程画布 | 红绿灯状态机 |
| **适合场景** | Agent 高度自由的探索式执行过程 | 有严格生命周期的对象（订单状态、审批流、TCP 连接） |

> [!important] 为什么 Flowchart 更适合 Agent？
> Agent 的真实执行过程不是严格状态机，而是开放空间的探索：并行搜索、多源汇总、失败回退、交叉引用和临时分支。Flowchart 更像一张万能白板——可以表达并行分支、汇聚节点、回退路径、参考关系和多步依赖。

## 从图到画布：任务状态的外部化

Mermaid 画布不只是任务结束后生成的总结图，而是**任务执行过程中持续生长的工作白板**。

### 画布生长过程

```
初始 → 只有简单节点
  N1["整理香港高校学费status: doing"]

探索 → 逐渐长出分支
  N1["搜索港大status: done"] --> N4["精确学费确认status: done"]
  N2["搜索港中文status: done"] --> N4
  N3["搜索港科大status: done"] --> N4
  N4 --> N5["汇总对比表status: doing"]
```

### 折叠与恢复

画布可以从展开态变成折叠态，只保留轻量任务轮廓（类似 skill 的 metadata）。**折叠不是丢弃**——而是从"正在执行的展开态"转换为"可以恢复的压缩态"。

```
展开态（Active MMD）：完整的节点状态、摘要和时间戳
  ↓ 上下文压力升高时
折叠态（Metadata）：taskGoal、status、updatedTime、mmdFilePath
```

> [!tip] 折叠的本质
> 折叠只是把任务从"正在执行的展开态"转换为"可以恢复的压缩态"。它不再占用大量上下文，但保留了足够的结构，使 Agent 未来可以重新理解这个任务——而不是从零开始。

## 不是无限上下文，而是无限可达空间

"无限画布"不是说上下文窗口真的变成了无限，而是：**Agent 的工作空间扩展成了一个可以持续创建、折叠、重新展开的外部认知空间**。

> **"无限"的重点不是"无限读取"，而是"无限可达"。**
> Agent 不需要把所有历史细节放进工作记忆里，只需要保留足够清晰的结构入口和路径：知道曾经做过什么、这些任务大概是什么状态、需要时应该从哪里重新展开。

> [!important] 与 [[上下文卸载]] 的互补关系
> 卸载解决"信息太长"（原文移到外部文件），画布解决"结构丢失"（离散摘要变成任务拓扑）。
> 单独卸载像"档案柜"——材料在但缺目录；单独画布像"任务地图"——结构在但缺原文支撑。
> 组合使用：**细节可恢复（卸载），结构不丢失（画布）**。

## MMD 节点 = 任务卡片

一个 MMD 节点不是只有标题的空图，而是包含名称、状态、摘要和时间戳的"任务卡片"：

```
003-N4["timeseries-module-structure
  status: done
  summary: 列出 timeseries 目录，发现 core.py/sampled.py/binned.py，
           core.py 含 required column 检查逻辑
  Timestamp: 2026-04-16T22:19:53.895+08:00"]
```

MMD 节点保存的是**任务步骤级摘要**："这个子任务推进到了哪里？得到什么阶段性结论？"

与 [[上下文卸载]] 中 JSONL 的**工具调用级摘要**形成层级：

```
MMD 节点：002-N3 搜索复试分数线（任务步骤级摘要）
  ├─ JSONL 记录 1：搜索自划线院校复试分数线（工具调用级摘要）
  ├─ JSONL 记录 2：打开北大公告页面
  ├─ JSONL 记录 3：读取清华招生 PDF
  └─ JSONL 记录 4：核对人大新闻传播专业代码
```
