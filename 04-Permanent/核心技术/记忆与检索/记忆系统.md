# 记忆系统

> Agent的记忆模块分为短时记忆和长时记忆，是Agent区别于纯语言模型的关键能力之一。短时记忆是"工作台"，长时记忆是"仓库"——推理时从长时记忆中检索相关信息到短时记忆。

## 核心要点

- **短期记忆**：存当前对话上下文，通常用滑动窗口维护，超出窗口的旧消息被丢弃
- **长期记忆**：存跨会话的知识，通常存入向量数据库，查询时用当前问题做向量检索找回最相关的记忆片段
- **记忆更新机制**：Agent通过文件系统工具更新记忆文件，当Session接近上下文上限时自动触发记忆写入，然后压缩Session

> [!tip] LangChain的Memory实现
> - ConversationBufferMemory：完整保存全部历史
> - ConversationBufferWindowMemory：只保留最近K轮
> - ConversationSummaryMemory：用LLM定期总结历史
> - VectorStoreRetrieverMemory：存到向量库按需检索

## 短时记忆

短时记忆（Short-term Memory）是智能体在当前任务执行过程中临时存储信息的机制。类比人类认知科学中的"工作记忆"，它容量有限、持续时间短，但访问速度极快。

### 核心特点

1. **容量有限**：通常只能存储少量信息（类比人类 7±2 个项目）
2. **持续时间短**：任务结束或超时后自动清除
3. **访问速度快**：直接读取，无需检索
4. **内容动态**：随着任务推进不断更新

### HelloAgents WorkingMemory 实现

```python
class WorkingMemory:
    """工作记忆实现
    - 容量有限（默认50条）+ TTL自动清理
    - 纯内存存储，访问速度极快
    - 混合检索：TF-IDF向量化 + 关键词匹配
    """
    
    def __init__(self, config: MemoryConfig):
        self.max_capacity = config.working_memory_capacity or 50
        self.max_age_minutes = config.working_memory_ttl or 60
        self.memories = []
    
    def retrieve(self, query: str, limit: int = 5, **kwargs) -> List[MemoryItem]:
        """混合检索：TF-IDF向量化 + 关键词匹配"""
        vector_scores = self._try_tfidf_search(query)
        scored_memories = []
        for memory in self.memories:
            vector_score = vector_scores.get(memory.id, 0.0)
            keyword_score = self._calculate_keyword_score(query, memory.content)
            base_relevance = vector_score * 0.7 + keyword_score * 0.3 if vector_score > 0 else keyword_score
            time_decay = self._calculate_time_decay(memory.timestamp)
            importance_weight = 0.8 + (memory.importance * 0.4)
            final_score = base_relevance * time_decay * importance_weight
            if final_score > 0:
                scored_memories.append((final_score, memory))
        scored_memories.sort(key=lambda x: x[0], reverse=True)
        return [memory for _, memory in scored_memories[:limit]]
```

评分公式：`(相似度 × 时间衰减) × (0.8 + 重要性 × 0.4)`

### 短时记忆与上下文窗口

- **上下文窗口**：LLM 能看到的所有信息
- **短时记忆**：智能体主动选择放入上下文窗口的信息
- **关系**：短时记忆是上下文窗口的"策展人"

## 长时记忆

长时记忆（Long-term Memory）是智能体长期存储知识和经验的机制，可以跨越多个会话、多个任务持久保留，是智能体实现持续学习和个性化的基础。

### 核心特点

1. **容量大**：可以存储海量的知识和经验
2. **持久性强**：数据不会因为任务结束而丢失
3. **结构化**：通常有组织的索引和分类
4. **可检索**：支持高效的查询和检索

### 长时记忆系统架构

```
长时记忆系统
├── EpisodicMemory - 情景记忆（存储具体事件和交互历史，SQLite + Qdrant）
├── SemanticMemory - 语义记忆（存储抽象知识和概念关系，Qdrant + Neo4j）
└── PerceptualMemory - 感知记忆（存储多模态数据，SQLite + Qdrant）
```

| 记忆类型 | 存储内容 | 存储介质 | 检索方式 |
|---------|---------|---------|---------|
| 情景记忆 | 具体事件 | SQLite + Qdrant | 时间/语义检索 |
| 语义记忆 | 抽象知识 | Qdrant + Neo4j | 图谱查询 |
| 感知记忆 | 多模态数据 | SQLite + Qdrant | 特征匹配 |

### 短时记忆与长时记忆对比

| 特性 | 短时记忆 | 长时记忆 |
|------|---------|---------|
| 容量 | 有限（MB级别） | 几乎无限（GB级别） |
| 持续时间 | 任务级（秒-分钟） | 持久（天-年） |
| 访问速度 | 极快（内存读取） | 较慢（磁盘/网络IO） |
| 存储介质 | 内存（RAM） | 磁盘/数据库 |
| 遗忘机制 | TTL自动过期 | 手动管理/无过期 |

### 记忆交互流程

```
用户输入 → 短时记忆（处理中） → 长时记忆（持久化）
              ↓
         上下文窗口
              ↓
         LLM 推理
```

> [!tip] 记忆预取注入的实现
> 每轮对话前的预处理流程：
> 1. 移除上一轮的围栏消息（避免记忆上下文累积膨胀）
> 2. 提取用户最新 query
> 3. 调用 `prefetch_all` 预取相关记忆
> 4. 通过 `build_memory_context_block` 将记忆包装在 `<memory-context>` 标签中注入
>
> 围栏的关键设计：`<memory-context>` 标签明确告诉 LLM "这是记忆上下文而非用户消息"，标记 `_is_memory_fence=True`，压缩时会被优先移除——既保证推理时有记忆可用，又避免记忆占用压缩后的宝贵空间。

> [!important] 流式清洗器 StreamingContextScrubber
> 处理跨 delta 边界的 `<memory-context>` 标签过滤，防止记忆内容泄漏到用户可见的输出中。
>
> 问题本质：流式传输中每个 delta 可能只有几个字符，标签 `<memory-context>`（16字符）完全可能被拆分在多个 delta 中。例如 delta1 包含 `"...some text <memory"`，delta2 包含 `"‑context>secret data</memory‑context> visible text"`——单纯按单个 delta 匹配标签会漏过。
>
> 解决方案——有状态设计：
> - `_in_span` 标记是否当前处于围栏内部
> - `_buf` 缓冲区收集跨 delta 的部分标签片段，待完整标签匹配后再决定丢弃或保留
> - `flush()` 时未关闭的围栏内容全部丢弃——宁可丢失也不泄漏

## 短期记忆管理策略

当上下文窗口有上限（4K → 128K → 1M），聊多了装不下。业界应对策略形成由简到复杂的梯度：

| 策略 | 原理 | 信息损失 | 适用场景 |
|------|------|----------|---------|
| **滑动窗口** | 只保留最近N条消息 | 高（早期信息全丢） | 简单场景、token预算紧张 |
| **摘要压缩** | LLM对历史消息生成摘要 | 中（丢失细节保留要点） | 长对话场景 |
| **分层保护** | 保留首尾消息，压缩中间 | 中低（关键信息受保护） | 任务型对话 |
| **结构化摘要** | 按模板字段压缩 | 低（保留结构化要素） | 复杂agent场景 |
| **虚拟分页** | 模仿OS内存管理，按需换入换出 | 极低（可随时检索回来） | 超长对话 |

> [!important] 生产级压缩实践
> 触发时机：prompt token 达到 context_length × 75% 时触发。为什么是75%？因为要给LLM回复留空间，加上压缩本身也调用LLM也需要token。
> 三步压缩：①划分保护区(system+首轮+最近3轮) → ②修剪中间区工具输出 → ③LLM驱动结构化摘要
> 三级降级：①减少尾部保护条数 → ②删除最早10条工具结果 → ③只保留system+最近3轮

## 市场主流记忆方案对比

| 方案 | 存储方式 | 检索方式 | 自动化程度 | 适用场景 |
|------|---------|---------|-----------|---------|
| **Claude Code** | 分类Markdown文件(user/feedback/project) | 全量注入system prompt | Agent主动写入 | 开发者工具 |
| **ChatGPT Memory** | 服务端事实陈述句列表 | 语义相关性选择注入 | 全自动提取 | C端消费者 |
| **MemGPT/Letta** | Core+Recall+Archival分层 | Agent自主搜索 | Agent自管理 | 超长对话/复杂任务 |

### MemGPT/Letta 的三层记忆架构

MemGPT/Letta 按操作系统内存层次建模，是 CoALA 框架中 4 层分类法的具体生产化实现：

| 层级 | CoALA 对应 | 类比 | 容量 | 访问方式 |
|------|-----------|------|------|----------|
| Main Context | Working Memory | RAM | 模型上下文窗口大小 | 始终在 prompt 里 |
| Recall Memory | Episodic Memory | 交换分区 | 完整对话历史 | `conversation_search` |
| Archival Memory | Semantic Memory | 磁盘 | 无限（向量存储） | `archival_memory_search` |

关键区别：换入换出由 **Agent 自己决定**（通过函数调用），不是被动截断——这是与 [[上下文压缩]] 的根本分歧。压缩是系统被动决定"删什么"，MemGPT 是 Agent 主动决定"换入什么、换出什么"。

代价：架构复杂度高、需外部向量存储、有检索延迟。适合跨会话长期记忆，对单会话压缩有点重。Letta v0.16.7（2026年3月），GitHub 22.5k star，还在积极维护。

> [!important] MemGPT/Letta 与压缩哲学光谱
> MemGPT/Letta 代表"OS-level 内存调度"立场——不靠摘要续命，而是让 Agent 自主管理记忆层次。这天然避免了 [[递归摘要衰减与语义漂移|递归摘要衰减]]——因为换入换出的数据是原始的而非摘要的。详见 [[上下文压缩]] 的压缩哲学光谱。
| **Mem0** | 向量DB+图DB | 向量+图检索 | 全自动提取+去重 | 通用AI应用 |
| **Zep** | 向量+实体图+时序 | 混合检索 | 全自动 | 企业级应用 |
| **TencentDB Agent Memory** | 文件系统（refs+JSONL+MMD+Metadata四级折叠） | 结构化索引逐级找回 | Agent自动压缩+用户一键启用 | 长任务+多任务场景 |

> [!tip] 反直觉结论
> 个人助手场景中，几千字符的==有界文件记忆==往往比复杂的向量检索系统更实用。关键在于：分类要清晰、安全要前置、演进要有梯度。

## 记忆系统核心挑战

| 挑战 | 问题描述 | 业界解决方案 |
|------|---------|------------|
| 上下文占用 | 记忆注入占prompt空间 | 渐进式加载、选择性注入、摘要压缩 |
| 相关性检索 | 如何找到当前最相关的记忆 | Embedding语义检索+Reranking+元数据过滤 |
| 记忆过时 | 用户偏好会变 | 时间戳衰减、冲突检测更新、周期性验证 |
| 矛盾处理 | 新旧信息冲突 | Last-write-wins / LLM仲裁 / 保留两者标记冲突 |
| 安全威胁 | 记忆注入攻击 | 写入前安全扫描、敏感信息脱敏、审计日志 |

> [!important] 安全前置原则——写入时扫描而非读取时扫描
> 写入 memory 时先执行 `scan_context_threats(content)` 扫描威胁模式（API Key、Token、恶意指令注入）。发现威胁则拦截，返回 "Blocked: content matches threat pattern"。
>
> 为什么在写入时而非读取时扫描？因为写入时拦截是"源头治理"——恶意内容一旦写入就会在后续每次读取时都被注入到 prompt 中，逐次防御成本远高于一次拦截。读取时扫描虽然也能发现威胁，但每次读取都要扫描，且恶意内容已存在于记忆库中，属于被动防御。

## 记忆到技能的演进

记忆系统的终极目标是实现 ==记忆→技能的转化==——从被动回忆到主动执行：

- **L1记住事实**：写入feedback/user记忆
- **L2总结规则**：记忆聚合→行为约束注入system prompt
- **L3形成技能**：生成SKILL.md（含workflow+tools+trigger）

## 记忆系统的范式演进趋势

> 来源：[[Agent技术范式演变（2023-2026）]]

记忆系统正在经历从"纯向量检索"向"文件系统化沉淀+向量检索混合管理"的根本性转变：

### 短期记忆：从"存储"到"管理与压缩"

核心挑战已从单纯的"存储"转向**管理与压缩**——在有限 [[上下文窗口]] 内高效利用 token：
- **阈值控制**：基于固定 token 数或动态语义密度阈值触发压缩
- **结构化摘要**：中间过程对话提炼 Summary，保留头尾关键指令和最终结论
- **重点提取**：从冗长对话流中提取关键事实或状态变化，剔除无关噪音

### 长期记忆：从"向量数据库主导"到"文件系统主导"

长期记忆细分为两个方向，各有不同演进路径：

> [!note] 术语对照
> 文章中使用的"事项型记忆/知识型记忆"与认知科学中的"情景记忆/语义记忆"是一一对应的：
> - ==事项型记忆（Episodic）== = 情景记忆 — 存储具体事件和交互历史
> - ==知识型记忆（Semantic）== = 语义记忆 — 存储抽象知识和概念关系

| 子类型 | 早期方式 | 当前趋势 | 代表实践 |
|--------|---------|---------|---------|
| **事项型记忆**（Episodic） | 向量检索 | 文件系统记录（MEMORY.md / 日志文件） | [[本地Agent（Claw）]]、Hermes |
| **知识型记忆**（Semantic） | 纯 [[RAG]] 向量检索 | 本地文件系统 + Obsidian + 轻量向量检索（QMD/SQLite） | LLM-Wiki |

**事项型记忆的文件系统化**：用结构化 Markdown（如 MEMORY.md）记录用户偏好、历史行为、待办事项——比向量检索更可控、更易读，便于 Agent 直接读取时间序列上的状态变化。[[本地Agent（Claw）]]和 Hermes Agent 等框架越来越倾向于使用文件系统记录事项型记忆。

> [!example] 三模块持久化设计（Claude Code 实战）
> 将长期记忆拆分为三个有界文件，各司其职：
> - **MEMORY.md**（约2200字符）— 环境事实、项目约定、关键决策，由 Agent 主动写入
> - **USER.md**（约1375字符）— 用户画像、偏好习惯，由 Agent 主动写入
> - **db**（不限字数）— 会话历史，系统自动存储
>
> Agent 通过 memory 工具操作前两个文件，schema 包含四个字段：
> - `action`：add / replace / remove
> - `target`：memory / user（对应 MEMORY.md 或 USER.md）
> - `content`：要写入的新内容
> - `old_text`：replace / remove 时定位目标条目
>
> schema 的 description 明确了"主动保存"原则——不需要等用户要求才保存，遇到以下情形即主动写入：用户纠正了之前的行为、表达了偏好或习惯、提供了环境事实、确立了约定或规范。

> [!example] 用户画像写入的具体触发逻辑
> 三种操作对通知的行为不同：
> - **add**：新增画像条目，写入后触发 `_notify_memory_write` 通知
> - **replace**：用 `old_text` 定位目标条目替换，写入后也触发通知
> - **remove**：删除画像条目，但不触发 `on_memory_write`——删除是"减法操作"，不需要通知下游
>
> 设计意图：写入和替换是"增量行为"，可能影响后续推理策略，需要通知；删除是"清理行为"，不影响推理方向，无需通知。

**知识型记忆的混合架构**：企业级场景下，纯文件系统 + grep 类检索不够精准，需要搭配 QMD 或 SQLite 等轻量向量化检索机制，甚至更加高度复杂的场景还需要企业级的向量检索——==保留 RAG 的海量知识优势，同时获得目录结构、标签、链接的显式组织能力==，使得知识的召回更加精准和可解释。

### LLM-Wiki 与本地知识库理念

> [!note] LLM-Wiki
> **LLM-Wiki** 是 Karpathy 等提出的本地化知识库理念，核心思路：Agent 直接访问组织良好的 Markdown 知识库（如 Obsidian vault），通过目录结构、标签、链接来显式组织知识——比纯向量检索更可控、更可读、更易组织。
>
> **GBrain** 是类似方向的本地知识库项目。
>
> 这代表了 Agent 知识型记忆的一个关键趋势：从"纯向量检索"走向"文件系统+轻量检索"的混合架构。个人知识库场景非常适合这种模式；企业级海量知识场景仍需搭配向量检索。

> [!tip] 反直觉结论
> 个人助手场景中，几千字符的==有界文件记忆==往往比复杂的向量检索系统更实用。关键在于：分类要清晰、安全要前置、演进要有梯度。
