# 符号化压缩

> 真正可用的压缩，不应该依赖模型"记住了什么符号"，而应该依赖模型"能够从符号中推理出什么结构"。[[Mermaid任务画布]] 是符号化压缩的一个最优载体，但三条设计原则本身是通用的——适用于任何为 LLM 设计压缩符号的场景。

## 压缩的本质：从稀疏到稠密

对大模型而言，"压缩"只有一个定义——**用更少的 token 表达等量（或近似等量）的语义**。本质上就是把稀疏信息提炼成稠密信息，减少上下文中的字数，同时尽量保持语义不变。

一次搜索返回 2000 tokens 的原文，其中真正有信息量的可能只有 80 tokens 的核心结论。剩下的 1920 tokens 是搜索引擎的 UI 碎片、广告、重复内容、格式标记——都是"稀疏"的部分。去掉它们只留核心结论，就是最朴素的压缩。

但这还只是**文本层面的压缩**。更极致的压缩，是**改变信息的表示形式本身**。

## 两种压缩路线：记忆驱动 vs 结构驱动

### 记忆驱动压缩（不稳定）

利用模型已经"记住"的符号来做压缩。一个符号之所以短但信息量大，是因为模型在训练数据中大量见过它，建立了"符号→一整套语义"的映射。

经典例子是 **MBTI 人格类型**——"INTJ"仅仅四个字母，大模型就能立刻理解内向、直觉、思考、判断等含义。这是一种"记忆驱动"的压缩。

**但这种压缩并不稳定**，有三个致命弱点：

| 弱点 | 说明 |
|------|------|
| **模型依赖性** | 不同 LLM 无法理解模型未见过的自定义符号 |
| **上下文依赖性** | 同一个 LLM 在不同上下文中对符号的理解可能不一致 |
| **适配性差** | 自定义编码（如 T1:HKU:FIN:462K 表示"港大金融硕士 46.2 万"）虽然极其紧凑，但跨模型、跨时间复用困难 |

就像人与人之间的"黑话"：在熟人之间可以高效沟通，但一旦脱离语境，就可能完全失效。放到 Agent 场景中，如果压缩依赖模型未见过的符号，想跨时间、跨 agent 的沟通就需要额外维护一套"符号对照表"，增加系统复杂度。

> [!important] 短表达 ≠ 有效压缩
> **并不是所有"短表达"都是有效压缩。**"这场饭局像一场大型狼人杀"之所以成立，是因为听者能从已有知识推理出结构——而不是因为"狼人杀"本身包含所有信息。详见 [[语言压缩哲学]]。

### 结构驱动压缩（稳定）

**结构化表示**（如流程、关系、状态）具有更强的稳定性。它们不是把语义压缩进一个"标签"，而是把信息重组为一种可以被直接解析和推理的形式。

| 维度 | 记忆驱动（标签式） | 结构驱动（拓扑式） |
|------|-------------------|-------------------|
| 稳定性 | 依赖模型训练数据，不稳定 | 依赖拓扑结构可推理，稳定 |
| 跨模型 | 模型未见过的符号理解退化 | 结构化图语言可被任意模型解析 |
| 跨上下文 | 同一符号不同语境可能偏差 | 图拓扑的语义由结构而非约定决定 |
| 维护成本 | 需要维护符号对照表 | 不需要额外约定 |
| 典型例子 | MBTI、团队缩写、项目黑话 | [[Mermaid任务画布]]、JSON Schema、YAML |

大模型往往能比自然语言更高效、稳定地理解结构化表达——因为结构化表达的语义由**拓扑关系**而非**文本约定**决定。

> [!quote] 核心判断
> **如果我们希望对 Agent 的执行历史进行有效压缩，关键不在于发明更短的"词"，而在于利用一种既通用（包含在世界知识内）、又可推理的结构化符号系统。**

## 三条设计原则

经过多次实验，符号化压缩载体需要满足三条原则：

### 原则一：符号必须是通用知识

选择的符号载体必须是**所有主流 LLM 都能在预训练阶段大量接触过的通用格式**。

如果我们发明一套自定义压缩编码（如 T1:HKU:FIN:462K），虽然极其紧凑，但：
- 不同的 LLM 无法理解它（训练数据中没见过）
- 同一个 LLM 在不同上下文中理解不一致
- 可适配性和灵活性很差

[[Mermaid任务画布]] 满足这条原则——Mermaid 语法在 GitHub、技术文档、教程中广泛存在，所有主流大模型都能准确读写。

### 原则二：符号的生成不能过于复杂或孤立

如果压缩涉及复杂的编码规则（如多层嵌套缩写体系），生成端和理解端之间很容易产生**语义偏移**——生成符号的 LLM 和后续理解符号的 LLM 对同一个符号的解读不一致。

[[Mermaid任务画布]] 满足这条原则——语法极其简单（节点 + 箭头 + 标签），生成和理解的认知负担几乎一样。语义由图的拓扑结构而非文本约定决定。

### 原则三：表达要足够自由，让模型充分发挥能力

如果符号系统规则过于死板（如"每个节点必须严格包含 5 个字段"），模型在生成时会被格式束缚，无法根据实际情况灵活调整表达。

[[Mermaid任务画布]] 满足这条原则——Flowchart 允许任意形状的节点、任意方向的连线、任意长度的标签、任意的嵌套和分支。模型可以自主决定：
- 将连续 5 个相似操作合并为一个宏观节点
- 为关键发现单独建立高亮节点
- 用虚线箭头表示弱关联，实线表示强因果
- 用不同节点形状区分不同类型的操作

## 符号化压缩与 [[上下文卸载]] 的配合

符号化压缩解决的是**"如何压缩单条信息"**——让每条信息变成更短、更稠密、更容易被模型理解的符号。

但单条信息的压缩还不够。在真实长任务里，Agent 面对 50 条压缩后的符号时，还需要理解它们之间的关系——这正是 [[Mermaid任务画布]] 解决的**"如何组织任务结构"**问题。

```
符号化压缩（单条信息） → Mermaid节点摘要
组织任务结构（多条信息） → Mermaid画布拓扑图
完整信息保留（可恢复） → 上下文卸载 refs/*.md
```

三者结合形成 [[TencentDB Agent Memory]] 的四级折叠架构。

## 符号化压缩的哲学根基

符号化压缩的理论根基来自 [[语言压缩哲学]]：

> 语言压缩并不依赖字面信息的完整保留，而依赖语义结构的有效保留。好的压缩不是信息更少，而是信息密度更高——它删除了不影响理解的枝节，却保留了能够支撑后续推理的核心语义。

符号化压缩正是这条哲学的工程实现——用结构化符号代替原始文本，让信息密度更高但语义结构被完整保留。
