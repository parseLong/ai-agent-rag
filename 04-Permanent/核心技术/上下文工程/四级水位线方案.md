# 四级水位线方案

> 想象一台电脑的内存压力监视器——四个 Tier 不是互斥而是**累积**：Tier 3 触发时会先做完 Tier 1 和 Tier 2 再做摘要。即使最坏情况，需要送给 LLM 的内容量也已被前两步免费砍掉一大块。

![[笔记同步助手/images/b8dd98729118a958fc7719b587aa8bd7_MD5.png]]

## Tier 0：什么都不做（< 60%）

上下文宽裕，模型注意力没散，最好的优化是**不优化**。

## Tier 1：Snip——便宜的整理（60-80%）

到了 60% 开始**预防性维护**，没有 LLM 调用，纯字符串处理：

- 截短老的工具输出：一次 grep 返回 5000 token → 保留前几行 + 工具名 + "还有 X 条结果被省略"
- 截短用户消息里的代码块：200 行代码 → 保留文件名注释 + 前几行 + 总行数标注

几个细节：
- **保护区**内的工具输出和代码块不动
- **PROTECTED_TOOLS**（Skill、Task）享有豁免——输出有状态意义，删了 Agent 立刻糊涂
- 用户的**纯文本指令**永远不动，只压缩 markdown 代码块

> 成本是零，但能挡住相当一部分增长。

## Tier 2：Prune——更狠的释压（80-95%）

预防性维护不够了：

- Tier 1 已经截短的工具输出进一步替换成占位符 `[Content compacted to save space]`
- 裁掉 assistant 旧文本——保留前两句 + `[truncated]`
- 截断阈值整体下调，能压的全压

依然零 LLM 成本，依然不动保护区，依然不动用户纯文本。

> Snip 保留更多元信息（"我之前 grep 过这个文件"）；Prune 是彻底擦除。能用前者解决的就别上后者。

## Tier 3：Summarize——兜底（≥ 95%）

只有 Tier 1 + Tier 2 都救不回来时才触发 LLM 摘要。做的是**增量摘要**：

```
1. 找出"上次摘要之后 → 保护区之前"的消息作为 delta
2. LLM 输入：上次摘要 + delta → 生成合并摘要
3. 替换旧摘要，删除 delta 消息
4. 保护区不动
```

第一次触发时"上次摘要"为空，相当于普通摘要；之后每次都是追加合并，避免反复重写历史导致语义漂移。

摘要 prompt 用结构化输出，四段：**进展 / 文件 / 待办 / 上下文**（用户偏好、错误、约束）。

## 阈值选择：为什么是 60 / 80 / 95？

| 阈值 | 逻辑 | 说明 |
|------|------|------|
| **60%** | 预防性维护的甜点 | 再低频繁触发没意义，再高模型注意力已开始下降 |
| **80%** | 危险线 | 离溢出还有缓冲但已不能等 |
| **95%** | 最后防线 | 再不调 LLM 就要爆了 |

三个值都是可配置的常量，接入远程配置后能热更新。出问题时把任意一个调到 1.0 就能禁用对应 Tier。

## 增量摘要 vs 全量摘要

| 方式 | 比喻 | 特点 |
|------|------|------|
| **全量摘要** | 每周把过去三个月的工作重新写一份周报 | 每次重写全部历史，贵且不准 |
| **增量摘要** | 维护一份持续更新的项目状态，每周只追加和修订变化的部分 | 更短更便宜更准；同一文件多次提及时最新状态覆盖旧状态 |

增量摘要的隐藏好处：同一个文件被多次提及时，最新状态覆盖旧状态；全量摘要里这个文件可能被描述好几次而且互相矛盾。

## 红线：什么东西任何 Tier 都不能动

| 内容 | 原因 |
|------|------|
| 保护区内的所有消息 | 模型短期连贯性的命脉 |
| 用户消息的纯文本部分 | 用户意图就是任务来源 |
| PROTECTED_TOOLS 的输出（Skill / Task） | 高度结构化的关键信息或会话级状态 |
| MICRO_COMPACTION_EXEMPT 的工具（Task / AskUserQuestion） | 保住对话流的关键节点 |
| 带 compactionProtected 标记的 Part | 业务侧明确指定的"必须保留" |

> [!important] 不存在"为了救场破例"
> 如果保护区都救不下来，那就让上下文真的爆出来——比错误压缩造成模型行为漂移要好。

## 可观测性

压缩发生在背后，没有可观测性调起来全靠玄学。SSE 事件暴露的信息：

- 当前触发的 tier（0 / 1 / 2 / 3）
- 当时的 token 使用率（来自 LLM 真实 usage）
- Snip 截了几个 part、Prune 替换了几个 part
- Summarize 是否真的调用了
- 预估节省了多少 token
- 命中 ReplacementCache 的 part 数（衡量跨轮一致性收益）

前端据此渲染压缩面板，接入 trace 系统后可基于真实生产数据调水位线，不再拍脑袋。

## 暂未做的取舍

- **主动 cache-aware 调度**（Claude Code 的做法）：目前用 ReplacementCache 实现被动一致性，还没做按 cache 边界主动选择压谁
- **可逆隐藏**（OpenCode 的做法）：用时间戳标记而非真删，等用户需要"回滚到某次压缩之前"时再上
- **回放最后一条用户消息**（OpenCode 的做法）：Tier 3 触发频率本来就低，优先级不高
- **用户消息一字不动**（Codex 的做法）：已不动纯文本但仍截断代码块，是否进一步走极端看生产数据
- **分层长期记忆**（MemGPT / Letta / Mem0）：跨会话记忆是另一个产品方向，不是这一轮要解决的问题
