# 递归摘要衰减与语义漂移

> 每次对摘要再做摘要，语义偏差逐轮累积——像"周报的周报的周报"，越缩越失真。这不是某个模型的缺陷，而是**信息压缩的数学特性**：每次有损压缩都会丢失信息，多次叠加后偏差不可逆。

## 问题：全量摘要的递归衰减

全量摘要的做法是每次压缩时**从头到尾重新摘要全部历史**。看起来每次都生成"新鲜"的摘要，但有一个隐蔽的致命缺陷：

```
第1次压缩：完整历史 → 摘要₁（丢失5%细节）
第2次压缩：摘要₁ + 新增 → 摘要₂（摘要₁的偏差被继承，再丢失5%）
第3次压缩：摘要₂ + 新增 → 摘要₃（摘要₂的偏差被继承，再丢失5%）
...
第N次压缩：摘要ₙ ≈ 原始历史的模糊影子
```

每一轮摘要都站在上一轮摘要的基础上——上轮丢掉的细节永远回不来，上轮引入的偏差被固化放大。这就是**递归摘要衰减**。

### 具体表现

| 衰减类型 | 说明 | 例子 |
|---------|------|------|
| **事实丢失** | 关键细节逐轮消失 | 变量名从摘要₁里消失 → 永远不会在摘要₂里再现 |
| **语义偏移** | 含义逐轮被扭曲 | "用户要求修复登录页" → 摘要₃变成"用户要求改UI" |
| **矛盾累积** | 同一实体被多次描述且互相矛盾 | 文件 A 在摘要₁里描述为"已改"，摘要₃里描述为"待改"——哪个是对的？ |
| **粒度退化** | 信息颗粒度逐轮变粗 | "修复了第42行的空指针异常" → "修复了一些bug" → "做了一些修改" |

> [!important] 这是信息论规律而非工程问题
> Shannon 的信息论告诉我们：有损压缩每次都丢信息，多次压缩的累积损失不可逆。无论 prompt 写得多好、模型多强，摘要的摘要必然比原始摘要更不精确。这不是可以通过"更好的 prompt"解决的——它是数学规律。

## Amp 的立场：递归摘要导致性能逐步衰减

Amp 引用了 OpenAI 的一份内部研究来佐证：递归摘要会导致性能逐步衰减。所以他们干脆**不做压缩**，替代方案是 `/handoff`——开新线程携带要点。

> [!quote] Amp 的设计哲学
> "一系列有焦点的短步骤，比一个逐渐退化的长对话好。"——本质上是**绕过衰减问题而非对抗它**。如果衰减不可逆，那就不要让对话长到需要衰减。

## 解决方案一：增量摘要

增量摘要的核心思路是**避免递归**——不是每次从头重写全部历史，而是维护一份活的摘要，每次只合并新增部分：

```
第1次压缩：完整历史 → 摘要₁（丢失5%）
第2次压缩：摘要₁ + delta₂ → 合并摘要₂（只处理新增的 delta₂）
第3次压缩：摘要₂ + delta₃ → 合并摘要₃（只处理新增的 delta₃）
...
```

**关键区别**：摘要₂ 里的旧内容来自摘要₁（已被压缩），但新增内容 delta₂ 是**原始的、未被压缩过的**。合并时模型可以：
- 主动取舍："这个文件改过了，旧描述更新一下"
- 补偿偏差：发现摘要₁描述有误时修正

> [!tip] 比喻
> - **全量摘要** = 每周把过去三个月的工作重新写一份周报 → 越写越模糊
> - **增量摘要** = 维护一份持续更新的项目状态文档，每周只追加和修订变化的部分 → 状态始终反映最新事实

### 增量摘要的隐藏好处

同一个文件被多次提及时，最新状态自然覆盖旧状态——全量摘要里这个文件可能被描述好几次而且互相矛盾，增量摘要里只有最新的一条。

## 解决方案二：不压缩，换线程

Amp 的 `/handoff` 方案本质上是**零递归策略**：既然递归摘要必然衰减，那就干脆不压缩。每次需要续命时，开新线程携带一份精心策划的要点——要点是**一次性的、不递归的**。

代价是丢失对话连续性——新线程里的 Agent 没有"我之前做了什么"的完整语境，只有交接要点。但至少要点是准确的、不失真的。

## 解决方案三：结构化摘要 + 保护区

[[四级水位线方案|四级水位线]]和 Claude Code 的 autoCompact 都采用结构化摘要——固定字段（进展/文件/待办/上下文）强制模型保留关键结构信息。加上保护区（最近几轮不压缩）确保短期连贯性不被摘要侵蚀。

这不能消除递归衰减，但**显著减缓**它——结构化约束让模型在摘要时保留最重要的骨架，保护区保证最近的决策不被遗忘。

## 三种方案的取舍

| 方案 | 递归风险 | 连续性 | 成本 | 适用场景 |
|------|---------|--------|------|---------|
| **增量摘要** | 低（delta 是原始的） | 中高 | 中（每次只合并 delta） | 大多数长对话场景 |
| **换线程 /handoff** | 零（不递归） | 低（新线程无完整语境） | 低（一次交接） | 任务明确、阶段分明 |
| **结构化摘要 + 保护区** | 中（递归但减缓） | 高 | 高（每次 LLM 调用） | 需要最大连续性的场景 |

> [!important] 核心判断
> 如果衰减不可逆，那么所有压缩方案都在**管理衰减的速度**而非消除它。增量摘要管理得最好（每次只引入新的原始信息），换线程完全绕过它，结构化摘要减缓它但没有解决它。
