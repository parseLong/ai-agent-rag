# 上下文卸载

> 不是所有信息都必须一直放在模型眼前——暂时不需要的内容先搬到上下文窗口之外（文件系统、数据库等），上下文里只保留摘要、路径或索引；需要时再通过索引找回。上下文卸载的目标是**让 Agent 的工作记忆变轻**：眼前只放当前要用的信息，细节放在外部，靠索引随时找回。

## 核心思想

Agent 在长任务中，最容易被撑爆的不是用户问题，而是**过程信息**：一次网页搜索可能返回几千字，一次文件读取可能返回上万字，一次代码调试可能产生大量日志。如果这些全部留在上下文里，Agent 就像一张铺满资料的办公桌——东西都还在，但真正要用的那一页反而越来越难找到。

上下文卸载的核心逻辑：

```
完整 tool result（几千 token）
  ↓ 卸载到外部
refs/*.md → 保存原文（完整信息仍在，只是不在上下文里）
  ↓ 保留索引
上下文中只留：摘要 + result_ref 路径
```

> [!important] 与 [[上下文压缩]] 的根本区别
> **压缩**是把信息变短（摘要替代原文），信息密度提高但信息量减少。
> **卸载**是把信息移走（原文到外部文件），上下文变轻但原文可恢复。
> 卸载更像"文件搬到档案柜"，压缩更像"文件缩成一句话"。
> 两者经常组合使用：先卸载完整信息到外部，再压缩上下文中的摘要。

## 行业实践

| 产品 | 卸载策略 | 说明 |
|------|---------|------|
| **Manus** | 文件系统即上下文 | 网页、PDF 等大内容保留 URL 或文件路径而非完整内容，符合"可恢复信息不长期占上下文"原则 |
| **Claude Skills** | 渐进式披露 | 技能文件夹不是一开始全部塞进上下文，而是"需要时再展开"，避免窗口过载 |
| **Claude Code** | 工具输出清理 | 保留架构决策、未解决 bug 和实现细节，清理冗余工具输出 |
| **TencentDB Agent Memory** | 四级折叠卸载 | Raw→JSONL→MMD→Metadata 逐级变轻，原文存在 refs/*.md，靠索引随时找回。详见 [[TencentDB Agent Memory]] |

> [!tip] 共同原则
> 这些方案共享一个核心思想：**可恢复的信息不应该长期占据上下文窗口**。文件路径、URL、索引远比完整内容便宜，而且完整内容可以按需取回。

## 卸载解决什么问题？

上下文卸载解决的是**"信息太长"**的问题，但不一定能解决**"结构丢失"**的问题。

如果只是把工具返回都保存到外部、在上下文里留下摘要和路径，Agent 不会被大段内容撑爆，但随着任务变长，它看到的可能变成一串线性记录：

```
搜索了港大学费，原文见 file_001
搜索了港中文学费，原文见 file_002
搜索了港科大学费，原文见 file_003
```

这些记录每条都不长，但**仍然是线性的**。Agent 需要自己判断哪些信息属于同一子任务、哪些是前置步骤——如果任务有几十次工具调用、多个分支，单纯摘要列表会越来越难用。

> [!important] 卸载 + 结构化 = 完整方案
> 单独卸载像"档案柜"——完整材料在，但缺少目录和项目看板。
> 单独 [[Mermaid任务画布]] 像"任务地图"——结构在，但没有底层原文支撑。
> 真正适合长任务的方案是**两者结合**：细节可恢复（卸载），结构不丢失（画布）。
> 详见 [[TencentDB Agent Memory]] 的四级折叠架构。

## Anthropic 的实验发现

Anthropic 的实验显示，当上下文长度超过 max window 的 **80%** 时，会发生"上下文腐烂"——模型注意力涣散，推理能力随之下降。

这正是上下文卸载要解决的核心问题：**不是省 token，而是保护模型的注意力**。
