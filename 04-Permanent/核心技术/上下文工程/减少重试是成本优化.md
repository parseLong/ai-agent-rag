# 减少重试是成本优化

> 真正烧钱的，常常不是第一次调用太长，而是第一次不合格——于是又来一轮。减少重试本身就是一种非常硬的成本优化手段。

## 🧠 核心洞察

很多人"看起来只补了一句话"的修正，实际含义是：

```
错误的第一次调用
+
第二、第三次修正调用
=
整包上下文被反复付款
```

重试不是"多花了一次输出"，而是**多花了一整轮的完整上下文**——重新描述问题、重新附带历史、重新拉代码或工具结果、再跑一轮模型。

## 🔑 重试成本的完整链条

```
第一次没答对
→ 重新描述问题（新一轮输入 Token）
→ 重新附带历史（新一轮会话 Token）
→ 重新拉代码或工具结果（新一轮检索 + 工具往返 Token）
→ 再跑一轮模型（新一轮推理 + 输出 Token）
```

每一环都是成本，而且**整包上下文在这条链上被反复付款**。一次不合格输出触发的重试，成本是首次调用的 2-3 倍甚至更多。

## 🔑 为什么重试特别隐蔽

| 原因 | 说明 |
|------|------|
| **单次修正看起来很轻** | "只补了一句话"——但这句话触发的是一轮全新请求 |
| **不会出现在单次指标里** | 单次 API 调用费用正常，但累计 3 次 = 首次成本的 3 倍 |
| **容易归因到"模型不行"** | 其实很多重试的根源是提示词不清晰、格式约束缺失、上下文有噪音 |

## 🔑 减少重试的三条杠杆

### 杠杆一：格式约束更清楚

指定输出格式（diff、步骤、结论、表格、JSON），不只是让输出更短，更重要的是让输出**更可用** → 重试更少。

> 这类约束值钱，因为它同时省两类：输出更短 + 结果更可用 → 重试更少。

[[Caveman]] 和 `/compact` 指令就是这个杠杆的具体实现。

### 杠杆二：任务边界更明确

把意图一次说完，而不是聊天式拆碎。越具体、越完整的指令，浪费的轮次越少。

> 一次对话，Agent 一次性跑完整个任务，省去了所有中间确认的来回成本。

Orchestrator-Worker 模式中每个 Worker 只做一件事，也是任务边界明确的体现。

### 杠杆三：上下文给得更准

引用文件用 `@路径` 而非只写文件名——免去搜索链路，直接定位。减少无关上下文噪音，模型不容易分神走偏。

代码图谱（[[Graphify]] / [[CodeGraph]]）也是这个杠杆——让 AI 在动手读文件之前就知道该读哪里，减少"搜索→发现漏了→再搜索"的重试循环。

## 🔑 与"单价优化"的区别

> [!important] 只看单价不看总成本
> 便宜模型如果导致更多重试、更多搜索、更多上下文回填，总成本未必更低。真正该看的是**完成一次任务的总成本**。

| 优化方向 | 看什么 | 陷阱 |
|----------|--------|------|
| **单价优化** | 单次 API 调用费用 | 便宜模型 → 更多重试 → 总成本更高 |
| **重试优化** | 完成一次任务的总成本 | 从失败链路入手，减少整条链路的重复付款 |
