# 上下文腐蚀与注意力衰减

> 上下文腐蚀（Context Rot）——随着上下文窗口中的 tokens 增加，模型从上下文中准确回忆信息的能力反而下降。不同模型的退化曲线或许更平滑，但这一特征几乎在所有模型上都会出现。

## Context Rot：上下文规模与注意力集中度的张力

针堆找针（needle-in-a-haystack）类基准揭示了一个现象：上下文中的 tokens 越多，模型从中准确提取信息的概率越低。这不是偶然，而是源自 LLM 的架构约束：

1. **注意力稀释**：Transformer 让每个 token 与上下文中所有 token 建立关联，形成 n² 级别的两两注意力关系。随着上下文增长，模型对这些关系的建模能力被"拉薄"
2. **训练分布偏倚**：短序列比长序列更常见，模型对"全上下文依赖"的经验更少
3. **位置编码牺牲**：位置编码插值等技术让模型适配比训练期更长的序列，但牺牲了对 token 位置的精确理解

> [!important] 性能梯度而非悬崖
> 这些因素共同形成一个**性能梯度**而非"悬崖式"崩溃：模型在长上下文下依旧强大，但相较短上下文，在信息检索与长程推理上的精度有所下降。

## Lost in the Middle：中间遗忘效应

当上下文过长时，模型对开头和结尾的内容记忆较好，对中间大段内容的关注度大幅下降。这就是 **"Lost in the Middle"（中间遗忘）** 效应。

这一效应直接影响 [[上下文注入]] 的位置策略——关键信息不应放在中间位置：

```
上下文窗口中的注意力分布：

┌─────────────────────────────────────────────┐
│ ▓▓▓▓▓▓▓  ░░░░░░░░░░░░░░░░░░░░░  ▓▓▓▓▓▓▓▓▓ │
│  HEAD区     MIDDLE区（易遗忘）        TAIL区  │
│  (高关注)    (低关注)                (高关注)  │
└─────────────────────────────────────────────┘
```

> [!tip] 注入位置策略
> 与 [[上下文压缩]] 的保护区划分策略一致：HEAD 区和 TAIL 区不压缩，MIDDLE 区可压缩。重要信息（系统指令、当前任务）放在头部和尾部，检索文档放在中间——因为中间位置的信息最容易被遗忘。

## 上下文焦虑症：长周期任务失败的首要原因

Anthropic 在 Harness Engineering 实践中发现，长周期任务失败的首要原因不是模型能力不足，而是 **上下文焦虑症（Context Anxiety）**。当模型在单次会话中连续运行数小时，处理包含大量代码、设计和状态信息的上下文时，会逐渐接近其注意力窗口的容量极限，产生三种典型症状：

| 症状 | 表现 | 后果 |
|------|------|------|
| **信息丢失** | 早期对话中的关键设计决策和约束条件被逐渐遗忘或边缘化 | 设计一致性丧失 |
| **注意力崩溃** | 模型在处理新信息时无法有效关联先前的上下文 | 前后逻辑矛盾 |
| **提前收工倾向** | 模型感知到上下文窗口接近极限时，下意识试图提前结束任务 | 任务远未完成就停止 |

> [!quote] 核心发现
> **不是"能力不够"而是"上下文焦虑"**——这是 Anthropic Harness Engineering 的核心洞察。模型在最长 6 小时以上的运行中，上下文焦虑是导致任务失败的主因，而非模型本身的推理能力。

### Claude Code 的焦虑对抗策略

Claude Code 通过多层机制对抗上下文焦虑：

1. **五层压缩策略**：不等焦虑发作就主动压缩（详见 [[上下文压缩]]）
2. **子代理架构**：主代理保持高层理解，子代理在干净窗口深挖后只回传摘要（详见 [[上下文重置与交接]]）
3. **记忆提取**：extractMemories 子代理持续将重要信息沉淀到外部存储
4. **错误记忆保持**：Compact 压缩中专门保留 "Errors and fixes" 段

### OpenClaw 的焦虑对抗策略

OpenClaw 通过预算机制和 Dreaming 对抗焦虑：

1. **三级 Compaction**：pre-request 主动压缩、timeout-triggered 紧急压缩、overflow 降级
2. **Dreaming 三阶段**：Light → REM → Deep，持续整理记忆防止退化
3. **Bootstrap Budget**：启动上下文也有预算，防止初始就膨胀
4. **Prompt Warning**：文件被截断时主动告知 Agent，让它用工具补读而非基于不完整信息推断

## 上下文腐蚀的工程应对总览

| 应对策略 | 原理 | 对应笔记 |
|---------|------|---------|
| **主动压缩** | 不等窗口满才压缩，预防性缩减历史 | [[上下文压缩]] |
| **上下文重置** | 彻底重启新 Agent，通过结构化 Handoff 续命 | [[上下文重置与交接]] |
| **注入位置优化** | 重要信息放 HEAD/TAIL，检索放 MIDDLE | [[上下文注入]] |
| **预算塑形** | Token Budget 告知模型额度，塑形推理投入 | [[上下文预算与资源分配]] |
| **子代理隔离** | 庞杂搜索上下文留在子代理内部，主代理保持聚焦 | [[上下文重置与交接]] |
| **记忆沉淀** | 重要信息写入上下文外持久化存储 | [[记忆系统]] |

> [!important] 即使模型能力持续提升，"在长交互中维持连贯性与聚焦"仍是构建强健智能体的核心挑战。谨慎而系统的上下文工程将长期保持其关键价值。
