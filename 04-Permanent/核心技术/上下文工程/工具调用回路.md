# 工具调用回路

> 模型先读一大包上下文，再调用工具；工具返回一段结果；结果又被塞回上下文，再进入下一轮。一次任务不是一次计费，而是一连串"请求—返回—再请求"的链条。

## 🧠 核心洞察

很多人直觉上把一次工具调用当成"一个动作"——"让它查个文件"，不就是一个操作吗？

但实际上模型经历的是**四段上下文交换**：

```
理解工具定义 → 生成调用参数 → 接收返回值 → 再结合返回值继续思考
```

每一段都有 Token 消耗，而且**工具返回值本身又成为下一轮的输入**——形成回路。

## 🔑 回路的三层放大效应

### 第一层：工具定义的常驻成本

一个 Tool / MCP 不只是"多一个按钮"，而是多一段要让模型理解的说明、参数模式和调用约束。工具越多 → 常驻定义越重 → 每轮都带着这些定义。

### 第二层：工具返回值的膨胀

工具返回的结果直接进入上下文，成为下一轮的输入。一次 Playwright 快照 56KB，20 条 GitHub issue 59KB——这些返回值比原问题长得多。

### 第三层：回路与重试的叠加

如果第一次调用结果不理想（格式不对、找错文件、推理过度），模型需要再发一轮请求来修正。于是：

```
错误的第一次调用
+
第二、第三次修正调用
=
整包上下文被反复付款
```

> [!warning] 回路 + 重试 = 成本灾难
> 回路本身已经让一次任务变成多段计费。加上重试，每修一次都可能把整包上下文再付一遍。

## 🔑 为什么工具越多越贵

很多人装了 GitHub、Notion、Jira、Slack、Browser、Filesystem、Kubernetes、Docker 等十几个 MCP，看起来很强，实际高频用的就两三个。但**每一个都在每轮请求中携带定义**，而且：

- **选择空间变大 → 决策变慢** → 更多推理 Token
- **错误调用概率变高** → 更多重试 → 回路放大
- **每轮携带定义更重** → 更多输入 Token

工具治理和依赖治理同理：关键不是能不能装，是有没有必要常驻。

## 🔑 对抗回路的三条路径

| 路径 | 原理 | 具体手段 |
|------|------|----------|
| **减少工具数量** | 少装低频 MCP，常驻只留高频 | 工具治理、[[模型路由]]中 CLI 优先原则 |
| **压缩工具返回值** | 工具输出进上下文前先压 | [[RTK]]（命令输出）、[[context-mode]]（MCP 返回值）、[[headroom]]（所有上下文） |
| **沙箱化工具结果** | 不直接进上下文，按需取回 | [[context-mode]] 的 Sandbox 机制、[[headroom]] 的 CCR 可逆压缩 |

> [!tip] CLI 优先原则
> `gh pr create`、`kubectl get pods`、`docker ps`、`git diff`——这些成熟 CLI 不需要额外的工具定义。AI 直接走命令更轻，不触发回路。详见 [[AI Coding Agent Token 成本控制]]。
