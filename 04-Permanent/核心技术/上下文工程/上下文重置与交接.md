# 上下文重置与交接

> 不是所有内存泄漏都能靠"清理缓存"解决，有时候得重启进程。在长周期任务中，当一个 Agent 完成一个冲刺后，系统会启动一个新的 Agent，只传递必要的状态信息——从而彻底解决上下文窗口溢出问题。

## Context Reset vs Context Compaction

针对 [[上下文腐蚀与注意力衰减|上下文焦虑症]]，有两种应对策略：

| 策略 | 做法 | 适用场景 | 信息保留 | 原理 |
|------|------|----------|---------|------|
| **Context Compaction** | 在原会话中压缩历史，保留关键信息 | 上下文焦虑不严重的短-中期任务 | 有损但连续 | 类似"清理缓存" |
| **Context Reset** | 彻底起一个新 Agent，通过结构化工件进行工作交接 | 任务极长或模型焦虑明显的情况 | 选择性但完整重启 | 类似"重启进程" |

> [!important] 关键选择标准
> Compaction 是保守策略——压缩后仍在同一 session 内继续，有损但保持连续性。Reset 是激进策略——彻底重启获得干净的注意力窗口，但需要精心设计交接机制确保不丢失关键信息。**不是所有场景都需要 Reset，但长时程复杂任务必须有这个选项。**

## 结构化 Handoff：Context Reset 的具体实现

Anthropic 的 `claude-progress.txt` 模式是 Context Reset 的工程化落地——session 切换时不是简单"摘要前文"，而是写入 **5 层结构**：

```
结构化 Handoff 文件（claude-progress.txt）
├── 1. 状态快照 ── 代码仓库的当前状态、已完成的部分、未完成的 TODO
├── 2. 叙事上下文 ── "我们为什么做这个决定"的故事线
├── 3. 决策日志 ── 关键技术决策及其理由
├── 4. 优先队列 ── 下一步应该做什么的排序
└── 5. 警告与陷阱 ── 已知的坑、必须避免的路径
```

新 session warm start 时读取这个文件，立刻知道"上次做到哪、下一步该干什么"。这就是 Context Reset 的具体实现形态——不是让新 Agent 从零开始，而是通过结构化工件实现**选择性但完整的重启**。

## 两阶段 Session 轮换

前沿做法是预防性的两阶段轮换——不等焦虑症状出现就主动换新 session：

```
上下文使用率:
0% ──────────────── 60% ────── 80% ────── 100%
     │                │          │          │
     │                │          │          │
     │         阶段1: 记忆同步    │     危险区
     │         将重要状态写入      │
     │         外部存储           │
     │                          │
     │              阶段2: 优雅 Handoff
     │              切换到新 session
     │              读取 Handoff 文件暖启动
```

> [!important] 为什么不能同时触发
> 两阶段之间的间隙给同步留出完成时间——同时触发会造成竞态。60% 阶段写入外部存储，80% 阶段才切换 session，中间的 20% 是安全缓冲。

**现有框架的现状**：OpenClaw 和 Hermes 目前都是"等 overflow 才压缩"，没有预防性的两阶段轮换。这正是 Anthropic "上下文重置"思想的工程化落地——**不等焦虑症状出现就主动换新 session**。

## Claude Code 的重置与交接实践

Claude Code 通过**子代理架构**实现隐式的 Context Reset：

1. **子代理干净上下文**：主代理 spawn 子代理时，子代理只获得任务描述（不含完整对话历史），拥有干净的注意力窗口
2. **Fork 隔离机制**：基于 Git worktree 实现文件系统隔离，用字节相同的占位文本替换 tool_result 保持 prompt cache 键一致
3. **Don't peek 原则**：主代理不应 Read Fork 的 output_file，中间工具噪音不回流主上下文
4. **凝练摘要回传**：子代理探索后只回传 1,000–2,000 tokens 的摘要，庞杂搜索上下文留在子代理内部

```
主代理（高层规划与综合）
    │
    ├─→ 子代理 1（干净窗口深挖代码搜索）
    │    └─→ 凝练摘要回传（1-2K tokens）
    │
    ├─→ 子代理 2（干净窗口做代码审查）
    │    └─→ 凝练摘要回传（1-2K tokens）
    │
    └─→ 子代理 3（对抗性验证）
         └─→ Check 格式输出
```

> [!quote] Never delegate understanding 原则
> 给子代理写提示词像给刚走进房间的聪明同事交代任务——它没看过这个对话。命令式简短提示词产出浅薄、通用的工作。主代理必须始终保持对任务的**高层理解与控制**，子代理只是执行已明确的任务片段。

### 子代理类型与上下文隔离

| 类型 | 上下文策略 | 说明 |
|------|----------|------|
| **Explore** | `omitClaudeMd: true` + 只读 | 避免 CLAUDE.md 膨胀和角色冲突 |
| **Plan** | `omitClaudeMd: true` + 只读 | 同上，输出结构化计划 |
| **Verification** | 只读 + 可写临时目录 | 后台运行，结果不阻塞主流程 |

## OpenClaw 的重置与交接现状

OpenClaw 当前**没有内置的 session 级 reset 机制**：

- Compaction 是主要手段，Context Engine 的 `compact()` 接口支持可插拔的压缩策略
- 但 Context Engine 的 `assemble()` 接口已经为 Reset 预留了扩展点
- Dreaming 有类似的"整理"能力，但没有和 session 切换联动

> [!note] 演进方向
> 把 Reset 作为 Compaction 失败后的**兜底策略**纳入 Context Engine 契约——当压缩后的 context 仍超过有效注意力范围（而非 token 上限），触发 reset：写入结构化 Handoff 文件 → 关闭当前 session → 启动新 Agent 实例 → 读取 Handoff 文件暖启动。

## 面向长时程任务的三种策略取舍

| 策略 | 适用场景 | 核心机制 | 注意力代价 |
|------|---------|---------|-----------|
| **Compaction** | 需要长对话连续性的任务 | 压缩历史为摘要，保留关键决策 | 低——仍在同一窗口 |
| **结构化笔记** | 有里程碑/阶段性成果的迭代开发 | 将关键信息写入上下文外持久化存储 | 极低——只有笔记在窗口中 |
| **子代理架构** | 需并行探索的复杂研究/分析 | 多个专长子代理在干净窗口深挖 | 最低——主代理只看摘要 |

> [!tip] 经验法则
> - 压缩整合：适合需要长对话连续性的任务，强调上下文的"接力"
> - 结构化笔记：适合有里程碑/阶段性成果的迭代式开发与研究
> - 子代理架构：适合复杂研究与分析，能从并行探索中获益
