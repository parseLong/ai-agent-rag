# 上下文窗口管理

> 模型能处理的最大 Token 数决定了上下文窗口，长对话和复杂任务需要策略性管理这个有限空间。窗口管理不是简单的"满了就截"，而是预防性的、分层的、弹性恢复的系统工程。

## 通用管理方案

### 长对话处理三种方案

| 方案                | 做法                | 优点       | 缺点        |
| ----------------- | ----------------- | -------- | --------- |
| **滑动窗口**          | 只保留最近 N 轮         | 简单高效     | 丢失早期关键信息  |
| **对话摘要压缩**        | 用 LLM 定期总结历史      | 信息损失少    | 增加推理成本和时间 |
| **关键信息结构化提取**（推荐） | 提取关键实体和槽位存入"用户档案" | 核心信息永不丢失 | 需要设计提取策略  |

> [!tip] 实际组合使用
> **最近 5 轮完整保留 + 更早内容做摘要 + 关键信息结构化存档** ——这是最实用的组合策略。短时记忆保连续性，长时记忆保关键事实。

### 1M 上下文的意义

1M Token 的上下文窗口不只是"能写更长的 prompt"，而是让 Agent 的：
- 多轮任务真正可落地（不需要频繁压缩）
- 整库代码理解成为可能（一次读取整个项目）
- 长文档推理更可靠（减少 [[上下文腐蚀与注意力衰减|Context Rot]]）

但更大的窗口≠更好的结果——[[上下文腐蚀与注意力衰减]] 效应意味着窗口越长，注意力越稀薄。

## Claude Code 的弹性状态机管理

Claude Code 的查询循环不是简单的 while loop，而是**7 个 Continue 站点组成的弹性状态机**。每个站点都是一个可恢复的位置，专门处理上下文窗口的不同压力：

```
站点 1: collapse_drain_retry — PTL 错误 → 排空上下文 → 重试
站点 2: reactive_compact_retry — 排空不足 → 压缩历史 → 重试
站点 3: max_output_tokens_escalate — 输出超限 → 升级 max_tokens → 重试
站点 4: max_output_tokens_recovery — 注入恢复消息 → 从断点继续
站点 5: stop_hook_blocking — Stop Hook 拦截 → 反馈给模型 → 继续
站点 6: token_budget_continuation — Budget 模式 → 自动续费 → 继续
站点 7: (normal) — 正常工具执行 → tool_result 反馈 → 继续
```

| 站点类别 | 处理的问题 | 恢复策略 |
|---------|-----------|---------|
| **站点 1-2** | 上下文空间不足 | 排空或压缩后重试 |
| **站点 3-4** | 输出空间不足 | 升级限额或断点续传 |
| **站点 5** | 外部干预 | 反馈后继续 |
| **站点 6** | 长时程任务 | 自动续费 |
| **站点 7** | 正常反馈 | 继续循环 |

> [!important] 弹性恢复的核心设计
> 每个站点都是可恢复的——模型不会因为单次失败就终止会话。站点 1-2 处理**输入侧**压力（上下文太大），站点 3-4 处理**输出侧**压力（生成被截断），站点 6 处理**时间侧**压力（任务太长）。三侧压力都有对应的恢复机制。

### System Prompt 的缓存边界

Claude Code 将 System Prompt 分为静态段和动态段，以优化缓存：

```
[7 个静态段] ── cacheScope: 'global'（跨会话缓存）
    ↓
__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__ ── 缓存分界线
    ↓
[10+ 个动态段/条件段] ── cacheScope: null（每次生成）
```

这种分区设计让静态内容可以跨用户/组织共享缓存，动态内容每次单独注入——在窗口管理和成本管理之间取得平衡。

## OpenClaw 的预算贯穿 Runtime 管理

OpenClaw 把上下文窗口管理融入更广泛的**预算体系**——`contextTokenBudget`（按模型解析）贯穿 Context Engine 的 `assemble`、`compact`、`afterTurn` 三个核心接口：

```
预算贯穿的数据流：

contextTokenBudget ──→ assemble(budget) ── 组装 prompt
                      → compact()          ── 压缩
                      → afterTurn(ctx)     ── 后处理评估

三级 Compaction 对预算的响应：
L1: 预算 > 70% 使用率 → 预防性压缩
L2: timeout + > 65% → 紧急压缩
L3: overflow → 降级处理
```

### Bootstrap Budget：启动上下文也要管理

OpenClaw 特别设计了 Bootstrap Budget——每次会话启动时加载的上下文（SOUL.md、USER.md、MEMORY.md 等）也有预算：

| 参数 | 说明 |
|------|------|
| `maxChars` | 单文件最大字符数 |
| `totalMaxChars` | 所有启动文件总字符上限 |
| `nearLimitRatio = 0.85` | 达到总预算的 85% 就开始警告 |

> [!note] 告警是给 LLM 看的
> 文件被截断时，系统会发 prompt warning 告知 Agent "有文件被截了"——LLM 知道自己被截断后，会主动用工具补读完整文件，而不是基于不完整信息瞎猜。

## 两阶段 Session 轮换：前沿实践

现有框架（Claude Code 和 OpenClaw）都是"接近上限才压缩"的模式。前沿做法是预防性的**两阶段 Session 轮换**：

```
0% ──────── 60% ──────── 80% ──────── 100%
            │            │
    阶段1: 记忆同步      │
    写入外部存储         │
                        │
            阶段2: 优雅 Handoff
            切换到新 session
            读取 Handoff 文件暖启动
```

60% 使用率时触发记忆同步（将重要状态写入外部存储），80% 触发优雅 Handoff（切换到新 session）。两阶段之间的间隙给同步留出完成时间——同时触发会造成竞态。

> [!important] 不等焦虑症状出现就主动换新 session
> 这正是 Anthropic "上下文重置"思想的工程化落地。详见 [[上下文重置与交接]]。

## 上下文窗口的层级结构

一个完整的 LLM 上下文窗口，包含以下几层信息在争夺有限空间：

| 层级 | 内容 | 说明 | 优先级 |
|------|------|------|--------|
| **System Prompt** | 系统级指令、角色定义、行为边界 | 定义"它是谁" | 最高 |
| **工具定义** | 工具 schema、描述、权限信息 | 定义"它能做什么" | 高 |
| **检索文档** | 从知识库动态注入的相关资料 | RAG 核心输入 | 中 |
| **工具输出** | API 调用结果、数据库查询等 | 动态获取的外部信息 | 中 |
| **对话历史** | 当前会话的所有对话记录 | 提供连续性 | 低 |
| **用户输入** | 当前轮次的用户消息 | 触发本次推理 | 最高（本轮） |

每层都至关重要，却又都在争夺有限的 [[Token]] 空间。窗口管理要解决的就是**信息注意力分配**的问题。
