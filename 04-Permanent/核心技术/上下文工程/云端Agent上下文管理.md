# 云端 Agent 上下文管理

> 前面四级水位线解决的是"上下文里压什么、压多狠"。但云端多用户 Agent 还得处理几件 CLI 工具可以无视的事——用户关掉浏览器再回来压缩状态不能丢、Pod 重启后决策必须一致、工具日志要支持审计、sandbox 输出动辄几十 MB。这四层是**CLI 工具都不必处理的**——它们是单机工具，重启意味着开新会话。

## 6.1 存储分离：完整日志落盘，对话里只留截断版

每次工具调用（bash、read、grep）可能吐出几万 token。直接塞进对话历史不行，全丢掉又损失调试和审计能力。

**做法**：完整输出落盘，对话里只保留截断版 + 一条回取路径。

```
engine 层：
  persistTruncatedOutput()
    → 完整内容写到沙箱 _internal/truncated-outputs/{callId}.log
    → 沙箱写失败降级到 COS 直传

截断版 metadata 带 fullLogPath：
  模型看到："前几行内容 + [截断] + 完整日志路径"
  → 知道完整内容在哪，但没花 token 去读

前端展示：
  按 sandbox-file API 现取现读，完全绕开 context 约束
```

> 本质上是把**"模型的工作记忆"**和**"用户的审计需求"**解耦了。CLI 工具不用管这个——它们没有前端，工具输出要么留在 context 里，要么就没了。

## 6.2 工具差异化：不是所有工具都该同等对待

一段 grep 输出和一次 Skill 调用的信息密度不在一个量级，用统一阈值处理就是粗暴。四梯度分类：

| 梯度 | 工具 | 说明 | 压缩策略 |
|------|------|------|----------|
| **完全保护** | `PROTECTED_TOOLS = ['Skill', 'Task']` | Skill 关联知识库绑定，Task 是会话级元任务。输出有状态意义，删了 Agent 立刻糊涂 | 任何 Tier 都不动 |
| **微压缩豁免** | Task / AskUserQuestion | AskUserQuestion 的输出是用户的回答，删了等于把用户回话抹掉 | 被动小幅压缩也跳过 |
| **白名单可压** | bash / read / grep / websearch … | 无状态读取类工具，压缩的主力 | 正常按 Tier 处理 |
| **差异化存储预算** | 按 tool 区分落盘上限 | Read 30KB、Bash 50KB、WebSearch 15KB。Bash 经常吐大段构建日志，Read 相对可控 | 单次输出上限按工具区分 |

## 6.3 跨轮缓存：让压缩决策在重启后还算数

场景：用户聊到第 20 轮，第 8 轮和第 14 轮做过 snip。然后实例重启或请求路由到另一个 Pod。第 21 轮触发新一轮压缩——如果什么都不记得，新进程会从头重新判断，结果可能跟之前不一样。

两个严重问题：
- **Prompt Cache 全废**：消息序列前缀变了，缓存命中率掉到零，每一轮都按新 prompt 计费
- **模型困惑**：同一段历史在不同轮次里"长得不一样"，模型可能重新触发已处理过的工具调用，甚至鬼打墙

**解法：ReplacementCache**——把每轮截断决策按 part ID 存进 Redis（key 形如 `msgOptCache:{sessionId}`，TTL 30 分钟）：

```
下一轮无论同进程还是另一个 Pod：
  → 先查缓存
  → 已经决定过怎么压的 part 直接复用之前的结果
  → 没决定过的才走新策略
```

效果：
- 同一个 part 在整个会话里始终长一样 → 消息前缀稳定 → Prompt Cache 友好
- 跨实例、跨重启无感
- 从缓存读出的数据先过 `isValidReplacementEntry()` 校验，损坏的直接丢弃重算

> [!important] 与 [[上下文压缩|滑窗式 stub 陷阱]]的关联
> ReplacementCache 是 stub 单调推进的具体实现——按 part ID 持久化，下次直接复用不重算。这保证了"一个 part 一旦被标成 stub，后续所有 turn、所有 step 里都保持 stub 不变"。

## 6.4 多用户隔离

最朴素但绝对不能省的一层。所有压缩状态按 `(userId, sessionId)` 二元组隔离：

- 数据库写入强制带双条件 `WHERE userId = ? AND sessionId = ?`，杜绝越权读写
- COS 路径形如 `user_sessions/{userId}/sessions/{sessionId}/_internal/...`，天然按用户分区
- SSE 压缩事件按 sessionId 严格过滤订阅，一个用户收不到另一个用户的压缩通知

> 多租户场景下，**"压错给谁"比"压错什么"严重得多**。这一层没有为了性能而省的空间。

## 为什么 CLI 工具不用这些

| 特化层 | CLI 工具为什么不需要 |
|--------|---------------------|
| 存储分离 | CLI 没有前端，工具输出要么留在 context 里要么就没了 |
| 工具差异化 | CLI 通常只有几种工具，不需要复杂分类 |
| 跨轮缓存 | CLI 重启 = 开新会话，压缩状态丢了就丢了 |
| 多用户隔离 | CLI 是单用户单进程 |

> 云端 Agent 不一样：用户合上电脑去吃饭、明天回来，期待的是接着上次继续，而不是"对不起请重新介绍一下需求"。
