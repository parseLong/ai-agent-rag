# 可逆压缩

> 压缩后能否恢复原始信息？这看似小问题，实则决定了 Agent 是否能"回溯"到压缩前的状态——对调试、审计、错误恢复都有深远影响。

## 二元设计选择

压缩本质上是删除信息以节省空间。关键选择是：**删除后能否恢复？**

| 策略 | 做法 | 比喻 | 优势 | 代价 |
|------|------|------|------|------|
| **可逆压缩** | 用标记/占位符替代原始内容，数据仍在数据库中 | 软删除（数据库 `deleted_at` 标记） | 理论上可回溯；支持审计和调试；不丢信息 | 需额外存储空间；占位符本身也占 token |
| **不可逆压缩** | 物理删除原始内容，用摘要或占位符替代 | 硬删除（数据库 `DELETE FROM`） | 释放空间最彻底；操作最简单 | 信息永久丢失；无法回溯；调试和审计困难 |

> [!tip] 数据库设计的类比
> 这和数据库的软删除 vs 硬删除是同一设计模式——软删除给记录打 `deleted_at` 标记而非物理删除，硬删除直接 `DELETE FROM`。软删除支持审计和恢复但占空间，硬删除简洁彻底但不可逆。Agent 压缩面临的是同一取舍。

## 各家做法

| 产品 | 可逆性 | 实现方式 | 说明 |
|------|--------|---------|------|
| **OpenCode** | ✅ 可逆 | 时间戳标记 + 数据库保留 | 数据还在数据库里，模型看到占位符 `[Old tool result content cleared]`，理论上可恢复 |
| **Cursor** | ✅ 可回溯 | Dynamic Context Discovery | 压缩后聊天历史变成可搜索文件，Agent 能回头检索原始细节 |
| **MUR AI** | ✅ 可回溯（部分） | 存储分离——完整日志落盘 | 完整输出写到 `_internal/truncated-outputs/{callId}.log`，对话里只保留截断版 + `fullLogPath` |
| **Claude Code** | ❌ 不可逆 | 物理删除 + 结构化摘要替代 | microCompact 用 `[Old tool result content cleared]` 占位，autoCompact 用结构化摘要替代，原始内容不复存在 |
| **Codex CLI** | ❌ 不可逆 | 物理删除所有 assistant 回复和工具结果 | 由 handoff 摘要替代，原始内容不复存在 |
| **Amp** | ❌ 不压缩 | `/handoff` 开新线程 | 不存在"压缩后恢复"的问题——根本不压缩旧线程 |
| **headroom** | ✅ 可逆 | CCR 可逆压缩 + 本地数据保留 | 原始数据全部保留在本地，模型只收到压缩版；需要细节时通过 `headroom_retrieve` 按需取回 |

## OpenCode：可逆隐藏的设计

OpenCode 的 Prune 步骤（轻量、无 LLM 调用）：
- 往回遍历消息，跳过最近 2 轮用户对话
- 保护最近 40k token 工具输出不动
- 更老的工具输出用**时间戳标记**为"已压缩"
- 数据**仍在数据库里**——只是模型看到的变成占位符

```
原始消息：[tool_result: 5000 token grep 输出]
压缩后模型看到：[Old tool result content cleared — compressed at 2026-06-08T14:30]
实际数据库中：原始 5000 token 内容仍在，只是标记为 compressed
```

> [!important] OpenCode 没有做"给模型看的详细版"和"给用户看的精简版"的分离
> 只有一份摘要同时服务于模型和界面展示。如果做了分离，可逆性更强——模型看精简版，前端可按需显示完整版。

## Cursor：压缩 + 可回溯的新范式

Cursor 2026 年新增 Dynamic Context Discovery：把聊天历史变成**可搜索的文件**——即使压缩后 Agent 也能回头检索原始细节。A/B 测试减少了 46.9% 的总 token 消耗。

这创造了一种新范式：**压缩是不可逆的，但历史是可搜索的**。压缩把上下文窗口里的内容精简了，但完整历史作为外部数据源保留下来，Agent 可以用搜索工具按需回溯。

> 已知问题：压缩后模型有时"忘掉"刚做的编辑——说明可搜索≠可自动恢复，模型不会主动去搜索被压缩掉的内容。

## MUR AI：存储分离——模型工作记忆 vs 用户审计需求

MUR AI 的做法把"可逆性"拆成了两个维度：

| 维度 | 对谁 | 可逆性 | 机制 |
|------|------|--------|------|
| **模型的工作记忆** | Agent | ❌ 不可逆 | 对话里只保留截断版 + 占位符 |
| **用户的审计需求** | 人类 | ✅ 可回溯 | 完整输出落盘到 `_internal/truncated-outputs/{callId}.log` |

模型看到的是"前几行内容 + [截断] + 完整日志路径"——它知道完整内容在哪，但没花 token 去读。前端展示时按需调 sandbox-file API 现取现读。

> 本质上是**把"模型的工作记忆"和"用户的审计需求"解耦了**。模型不需要完整的 5000 行 grep 输出，但事后审计可能需要。两个需求用不同的机制满足。

## 可逆性的工程代价

| 代价 | 说明 |
|------|------|
| **存储空间** | 原始数据需要持久化到数据库或文件系统，不能只是内存中的占位符 |
| **占位符 token** | `[Old tool result content cleared]` 本身也占 token——如果标记太多，占位符的总 token 开销可能不小 |
| **搜索延迟** | Dynamic Context Discovery 模式下，回溯需要额外搜索步骤 |
| **数据一致性** | 可逆意味着数据库里有"压缩前的版本"和"压缩后的版本"，需要维护一致性 |
| **云端额外成本** | 存储分离需要额外的 COS/S3 存储，沙箱写失败的降级路径 |

## 什么时候需要可逆性？

| 场景 | 是否需要可逆性 | 原因 |
|------|----------------|------|
| **审计合规** | ✅ 需要 | 需要事后追溯 Agent 的每一步操作和原始输出 |
| **调试排错** | ✅ 需要 | Agent 行为异常时需要回溯"它当时看到了什么" |
| **用户回溯** | ✅ 需要 | 用户想看"刚才那个工具输出的完整结果" |
| **纯代码 Agent** | ❌ 可不做 | CLI 工具没有前端，压缩后信息要么在 context 里要么没了 |
| **短对话** | ❌ 可不做 | 压缩频率低，可逆性收益不大 |
