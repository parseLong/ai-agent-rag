# 上下文工程

> 上下文工程（Context Engineering）关注的是"在每一次模型调用前，如何以可复用、可度量、可演进的方式，拼装并优化输入上下文"，从而提升正确性、鲁棒性与效率。上下文窗口不是"能装多少"的问题，而是"装什么、何时装、何时清"的系统设计问题。

## 从提示工程到上下文工程

提示工程关注如何编写和组织 LLM 的指令以获得更优结果（系统提示的写法与结构化策略）；上下文工程则是在推理阶段，策划与维护"最优的信息集合（tokens）"——不仅包含提示本身，还包含工具定义、检索文档、对话历史、外部数据等一切会进入上下文窗口的信息。

| 维度 | 提示工程 | 上下文工程 |
|------|----------|------------|
| **关注范围** | 输入指令本身 | 进入上下文窗口的所有信息 |
| **核心问题** | "说什么" | "装什么、何时装、何时清" |
| **适用场景** | 单轮/短对话 | 多轮/长时程任务 |
| **管理对象** | System Prompt | 系统指令 + 工具 + 检索 + 历史 + 用户输入 |

> [!quote] 金鱼助理隐喻
> 大语言模型的本质就是一位"金鱼助理"——每次对话，它能看到的信息被严格限制在 [[上下文窗口]] 之内。窗口外的一切，它一无所知。准备简报的过程，就是上下文工程。

## 上下文工程的子领域

上下文工程是一个完整的工程学科，包含多个相互关联的子领域：

```
上下文工程
├── 上下文注入  ── 将外部信息注入到 Prompt 中
│   ├── 注入管道设计（Claude Code 三管道 / OpenClaw Context Engine 契约）
│   ├── 注入位置策略（避免 Lost in the Middle）
│   └── 注入格式设计（<system-reminder> 标签 / 结构化分区）
│
├── 上下文压缩  ── 压缩历史以适应窗口限制
│   ├── Claude Code 五层压缩策略
│   ├── OpenClaw 三级 Compaction
│   └── 结构化笔记替代历史膨胀
│
├── 上下文窗口管理  ── 管理有限空间的分配
│   ├── 滑动窗口 / 摘要压缩 / 结构化提取
│   ├── 预算分配与优先级排序
│   └── JIT 上下文检索 vs 预加载
│
├── 上下文腐蚀对抗  ── 应对注意力衰减
│   ├── Context Rot / Lost in the Middle
│   ├── 上下文焦虑症
│   └── 有效上下文"解剖学"
│
├── 上下文重置与交接  ── 长时程任务的上下文续命
│   ├── Context Reset vs Compaction
│   ├── 结构化 Handoff
│   └── 两阶段 Session 轮换
│
└── 上下文预算  ── 稀缺资源的显式量化与降级
    ├── Token 预算作为行为塑形器
    └── 多种稀缺资源的预算体系
```

## 有效上下文的"解剖学"

在"有限注意力预算"约束下，优秀的上下文工程目标是：**用尽可能少、但高信号密度的 tokens，最大化获得期望结果的概率**。落实到实践中，围绕以下组件开展工程化建设：

- **System Prompt**：信息层级"刚刚好"——既不过度硬编码（脆弱 if-else），也不过于空泛（缺具体信号）。分区组织，追求"最小必要信息集"
- **Tools**：职责单一、相互低重叠、接口语义清晰。"臃肿工具集"是常见失败模式——如果人类都说不准用哪个工具，别指望智能体做得更好
- **Few-shot Examples**：精挑细选一组多样且典型的示例，直接画像"期望行为"。好的示例胜过千言万语

> [!important] 总指导思想：信息充分但紧致
> 每新增一个 token 都消耗注意力预算的一部分。上下文必须被视作一种有限资源，且具有边际收益递减。

## Claude Code 的上下文工程实践

Claude Code 是 Anthropic 的官方 CLI 工具，在生产级 Agent 系统中完整实践了上下文工程：

| 上下文工程维度 | Claude Code 实现 |
|---------------|-----------------|
| **注入** | 三管道注入（System Context + User Context + CLAUDE.md Memory），优先级合并 + `<system-reminder>` 标签格式化 |
| **压缩** | 五层递进压缩（applyToolResultBudget → snipCompact → microCompact → contextCollapse → autoCompact） |
| **窗口管理** | 7 个 Continue 站点实现弹性状态机，Budget 注入塑形行为 |
| **腐蚀对抗** | 分层安全防御 + 对抗性验证 + Compact 压缩保留错误记忆 |
| **重置** | 子代理架构（干净上下文窗口探索后只回传摘要） |
| **预算** | Token Budget 告知模型可用额度，塑造推理投入 |

详见 [[Claude Code 源码架构]] 中各维度。

## OpenClaw 的上下文工程实践

OpenClaw 是开源 Agent 平台，用微内核架构把上下文管理做成可插拔的插件系统：

| 上下文工程维度 | OpenClaw 实现 |
|---------------|--------------|
| **注入** | Context Engine 契约接口（bootstrap → ingest → afterTurn → assemble → compact → maintain），可插拔策略 |
| **压缩** | 三级 Compaction（L1 pre-request → L2 timeout-triggered → L3 overflow） + identifier-preservation 验证 |
| **窗口管理** | 预算贯穿 Runtime——10 种稀缺资源显式量化并配降级路径 |
| **腐蚀对抗** | Dreaming 三阶段（Light → REM → Deep）整理记忆防退化 |
| **重置** | 无内置 session 级 reset，但有扩展点预留 |
| **预算** | Bootstrap Budget + contextTokenBudget + 工具输出硬限（16K 字符） |

详见 [[OpenClaw架构深度解析]] 和 [[OpenClaw与Hermes：源码里的 AI Agent 架构知识大复盘]]。

## 两者对比的关键洞察

| 对比维度 | Claude Code | OpenClaw |
|---------|-------------|----------|
| **架构风格** | 整体集成式——上下文管理嵌入查询引擎 | 微内核插件式——Context Engine 是独立契约 |
| **压缩策略** | 五层递进（更细粒度） | 三级响应（更粗但可插拔） |
| **注入设计** | 固定三管道 + `<system-reminder>` | 可插拔 assemble() 接口 |
| **预算哲学** | Token Budget 作为行为塑形器 | 10 种稀缺资源全部显式量化 |
| **记忆沉淀** | extractMemories 子代理 + SessionMemory | Dreaming 三阶段加权晋升 |
| **重置能力** | 子代理干净上下文 + Fork 隔离 | 无内置 reset，但契约预留扩展点 |

> [!quote] 共同判断
> 两个框架都把记忆和上下文当成长期投入的主战场，而不是可选功能。工具和通道可以靠生态补齐，但"Agent 对上下文的持续管理能力"必须框架自己深入做——这是长时程任务中 Agent 能否稳定交付的关键变量。

## 三次进化：从 Prompt 到 Context 到 Harness

上下文工程是工程演进的第二阶段，其局限催生了第三阶段：

```
Prompt Engineering ──→ Context Engineering ──→ Harness Engineering
    "说什么"               "给什么信息"            "系统级约束+验证+反馈"
```

Prompt Engineering 和 Context Engineering 的**共同盲区**：都管不到执行过程中的行为约束、结果验证、错误纠正。完成这三件事的，是 [[驾驭工程（Harness Engineering）]]。

> [!important] 上下文工程的定位
> 上下文工程解决的是"信息供给"问题——在有限的窗口内，为模型提供最精准、最高信号密度的上下文。但它**不解决执行约束和结果验证**——这正是 Harness Engineering 的领域。上下文工程是 Harness 的**导航仪**，而非**方向盘**。
