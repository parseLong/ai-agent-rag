# 上下文注入

> 将检索到的外部信息、工具返回结果、历史记忆等注入到Prompt中，为LLM提供决策所需的上下文。注入不是"越多越好"，而是"精准投放"。

## 核心要点

- **注入来源**：RAG检索结果、工具调用返回、对话历史、系统提示词、用户偏好档案
- **注入策略**：直接拼接（最简单）、摘要后拼接（节省Token）、结构化标记（用特殊标签区分不同来源）
- **关键约束**：注入总量不能超过[[上下文窗口]]限制，需要优先注入最相关信息

> [!note] 上下文工程
> 狭义的上下文工程特指Prompt的工程实现（如Rules、CLAUDE.md、AGENTS.md等），广义上也包含LLM使用外部工具（如Skills，是工具与提示词结合的典范）。

## Claude Code 的上下文注入管道

上下文注入管道是驾驭层的**导航仪**——自动将环境信息、项目指令、记忆内容注入到 LLM 的上下文窗口中，让模型"知道"当前在什么环境、做什么项目、有什么规则。Claude Code 的注入管道不是简单拼接，而是**优先级合并 + 格式化 + 缓存感知**的精密系统。

### 三条注入管道

#### 1. System Context（系统上下文）

```
来源：src/context.ts → getSystemContext()

注入内容：
  ├─ git status (分支、状态、最近提交)
  ├─ git config (用户名)
  ├─ system prompt injection (反缓存注入)

特点：
  ├─ 会话级缓存（对话开始时获取，后续不变）
  ├─ 截断到 2000 字符（防止溢出）
  ├─ CCR/resume 模式下跳过
```

#### 2. User Context（用户上下文）

```
来源：src/context.ts → getUserContext()

注入内容：
  ├─ CLAUDE.md 内容（多层合并）
  ├─ 当前日期
  ├─ 其他环境信息

特点：
  ├─ 每次消息都注入（动态）
  ├─ 尊重 --bare 模式（不注入）
  ├─ 尊重 CLAUDE_CODE_DISABLE_CLAUDE_MDS（禁用 CLAUDE.md）
```

#### 3. CLAUDE.md Memory Loading（记忆加载）

```
来源：src/utils/claudemd.ts → getClaudeMds()

加载优先级（后加载优先级更高 → 模型更关注）：
  1. Managed memory  (/etc/claude-code/CLAUDE.md)  ← 组织级，最低优先
  2. User memory     (~/.claude/CLAUDE.md)         ← 个人全局
  3. Project memory  (CLAUDE.md, .claude/rules/*)  ← 项目级
  4. Local memory    (CLAUDE.local.md)             ← 本地项目，最高优先

发现策略：
  ├─ 从当前目录向上遍历到根目录
  ├─ 距离 cwd 更近的文件优先级更高
  ├─ 支持 @include 指令（最大深度 5）
  ├─ 支持 frontmatter paths 做 glob 范围限定

处理：
  ├─ 剥离 HTML 注释（保留代码块中的）
  ├─ MEMORY.md 入口截断（行数和字节上限）
  ├─ contentDiffersFromDisk 追踪（缓存失效检测）
```

### 注入格式：system-reminder 标签

所有注入内容被包裹在 `<system-reminder>` 标签中：

```xml
<system-reminder>
As you answer the user's questions, you can use the following context:
# gitStatus
Current branch: main
Main branch: main
...

# claudeMd
Project instructions from CLAUDE.md...

IMPORTANT: this context may or may not be relevant to your tasks.
You should not respond to this context unless it is relevant.
</system-reminder>
```

**关键设计**：`"IMPORTANT: this context may or may not be relevant"` —— 这不是废话，而是**防止模型对注入内容过度反应**。

### CLAUDE.md 的 @include 机制

```
支持四种引用：
  @path     → 项目内路径引用
  @./path   → 相对路径引用
  @~/path   → 用户目录引用
  @/path    → 绝对路径引用

最大嵌套深度：5 层（防止循环引用）
```

### 上下文注入与缓存的关系

```
注入管道设计考虑了 API prompt caching：

  静态注入（缓存）：
    ├─ getSystemContext() → 会话级缓存
    ├─ CLAUDE.md → 变化时失效，否则缓存

  动态注入（不缓存）：
    ├─ 当前日期 → 每次变化
    ├─ git status → 会话级缓存
    ├─ MEMORY.md → 动态截断

  系统提示词的缓存边界：
    ├─ 静态段 → cacheScope: 'global'（跨会话缓存）
    ├─ 动态段 → cacheScope: null（每次生成）
```

## 规约规则与注入边界

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

上下文管理本质上是一系列**规约规则（Reduction Rules）**的集合——Harness 必须定义清晰规则，决定在 Token 预算紧张时，哪些信息应该被牺牲、哪些被保留。

**注入边界（Injection Boundary）**定义了外部信息（如 RAG 结果）应在 Prompt 的哪个位置（前、中、后）注入，以达到最佳效果，避免出现"Lost in the Middle"问题。

> [!important] 注入位置的影响
> 注入位置不是随意的——研究表明，LLM 对开头和结尾的内容关注度最高，中间大段内容容易被忽略。因此关键信息的注入位置直接影响模型决策质量。这与 [[上下文压缩]] 的保护区划分策略一致：HEAD 区和 TAIL 区不压缩，MIDDLE 区可压缩。

## Token 转化流水线

> 来源：[[万字干货：理解 Harness Engineering，看这一篇就够了]]

在三层记忆之上，Harness 还需要一条 **Token 转化流水线（Token Transformation Pipeline）**，在每轮调用前把多源信息规约成一个可控的 Prompt：

| 步骤 | 说明 |
|------|------|
| **① 信息源收集** | 聚合用户问题、短期记忆、长期知识检索结果等 |
| **② 相关性排序** | 基于时间、语义相似度等指标，对候选信息打分 |
| **③ 压缩与摘要** | 对冗长低密度内容做摘要或结构化提炼 |
| **④ 预算分配** | 按照预设 Token 预算，为不同信息类别分配额度 |
| **⑤ 模板组装** | 使用结构化模板（显式标注 `[user_request]`、`[tool_output]` 等）拼装最终 Prompt |

> [!important] 核心思想：把注意力管理变成外部工程问题
> 与其指望模型"自己想清楚该关注什么"，不如通过 Token 转化机制主动构建上下文，把**有限的窗口留给真正重要的信息**。这与 [[上下文窗口管理]] 的"关键信息结构化提取"策略和 [[上下文压缩]] 的保护区划分策略一脉相承。

## OpenClaw 的 Context Engine 契约对比

OpenClaw 把上下文注入抽象为**可插拔的 Context Engine 契约接口**，与 Claude Code 的固定三管道形成对比：

```typescript
interface ContextEngine {
  bootstrap?(ctx)          // 会话初始化（对应 Claude Code 的 System Context）
  ingest(msg)              // 吸收一条 message
  ingestBatch?(batch)      // 吸收一轮 turn
  afterTurn?(ctx)          // 一轮结束后做后处理
  assemble(budget)         // 按 tokenBudget 组装 prompt（对应 Claude Code 的消息组装）
  compact()                // 压缩（对应 Claude Code 的 autoCompact）
  maintain?()              // 分支重写
  prepareSubagentSpawn?()  // 子 agent 派生前
  onSubagentEnded?()       // 子 agent 结束后
}
```

| 对比维度 | Claude Code | OpenClaw |
|---------|-------------|----------|
| **架构风格** | 固定三管道注入 | Context Engine 契约可插拔 |
| **注入来源** | git status + CLAUDE.md + currentDate | 可自定义 assemble() 实现 |
| **注入格式** | `<system-reminder>` 标签 | 由 Context Engine 插件决定 |
| **优先级** | cwd 向上遍历加载 CLAUDE.md | 由 Context Engine 插件决定 |
| **缓存感知** | 静态段/动态段分离 + 缓存分界线 | `ingest` → `assemble(budget)` 显式预算 |
| **可扩展性** | 需改源码 | 换一个 Context Engine 插件即可 |

> [!note] 核心差异
> Claude Code 的注入是**集成式**——三条管道、固定格式、固定优先级，与查询引擎深度耦合。OpenClaw 的注入是**可插拔式**——Context Engine 只定义契约接口，具体实现由插件填充。前者更稳定但难定制，后者更灵活但需要更多设计工作。

> [!important] 工作记忆 vs 审计需求：注入的两层服务对象
> 注入到上下文窗口的信息服务于**模型的工作记忆**——模型需要的是精简的、高信号密度的上下文。但同一个工具输出可能还有另一个读者——**人类的审计需求**：调试排错、合规审计、事后回溯。
> 
> 这两个需求是**正交的**：模型不需要完整的 5000 行 grep 输出，但事后审计可能需要。优秀的设计会把两者解耦——对话里只保留截断版（给模型），完整输出落盘到外部存储（给人类）。详见 [[云端Agent上下文管理]] 的存储分离设计。
