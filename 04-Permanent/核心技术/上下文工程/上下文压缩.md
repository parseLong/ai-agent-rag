# 上下文压缩

> 检索结果过长导致超出模型上下文限制或干扰注意力时，需要压缩只保留最相关部分。压缩不是等到窗口满了才做——而是**预防性的、分层的、渐进的**。

## 通用压缩方法

| 方法 | 说明 | 优缺点 |
|------|------|--------|
| **Map-Reduce** | 把长文档分段处理，每段分别与问题做相关性判断 | 并行速度快，但可能丢失跨段信息 |
| **Refine** | 先用部分文档生成初步答案，再逐步引入其他文档完善 | 能利用上下文但速度较慢 |
| **相关性过滤** | 用轻量级模型计算每个片段与问题的相关性分数 | 简单高效，但依赖相关性算法质量 |
| **LLM压缩** | 用模型提取关键信息生成摘要 | 最高质量但最高成本 |
| **滚动摘要** | 定期将旧对话压缩为摘要，只保留精华 | 信息损失可控但增加推理成本 |

> [!tip] 实践建议
> 通常组合使用：先用轻量级方法粗筛，再用 LLM 精筛，既保证效果又控制成本。

## Claude Code 的五层递进压缩策略

Claude Code 的上下文压缩不是单一触发点，而是**五层递进**——体现 [[分层安全防御]] 的思想，不是等到完全耗尽才做最重的压缩，而是逐级应对：

```
原始消息
→ applyToolResultBudget()    — ① 工具结果大小限制（每次消息）
→ snipCompact()              — ② 片段压缩（功能开关控制）
→ microCompact()             — ③ 微压缩（缓存旧 tool_result）
→ contextCollapse()          — ④ 上下文折叠（PTL 错误时）
→ autoCompact()              — ⑤ 自动压缩（Token 缓冲不足时）
```

| 层级 | 策略 | 触发条件 | 代价 | 说明 |
|------|------|----------|------|------|
| **①** | `applyToolResultBudget` | 每次消息 | 无 | 工具结果的大小上限（不压缩，只截断过大输出） |
| **②** | `snipCompact` | 功能开关 `SNIP_COMPACT` 控制 | 极低 | 精简工具结果中的冗余内容 |
| **③** | `microCompact` | 旧 tool_result 替换 | 低 | 将旧工具结果替换为占位文本 `[Old tool result content cleared]`，不调用模型 |
| **④** | `contextCollapse` | PTL 错误（上下文超模型限制） | 中 | 排空积累的上下文后重试，或调用 `compact()` 将历史消息总结为压缩摘要 |
| **⑤** | `autoCompact` | Token 缓冲量低于 13,000 | 高 | 调用 LLM 生成结构化压缩摘要 |

### autoCompact 的结构化压缩

最重的压缩是 `autoCompact()`，它将长对话历史压缩为**结构化摘要**，使用两阶段设计：

1. **`<analysis>` 分析块**：按时间顺序分析每条消息，识别请求意图、方法策略、关键决策、文件名和代码片段、错误及修复方式
2. **`<summary>` 总结块**：九个结构化段落——Primary Request / Key Technical Concepts / Files and Code Sections / Errors and fixes / Problem Solving / All user messages / Pending Tasks / Current Work / Optional Next Step

**压缩后恢复消息**的关键指令：不确认摘要、不复述背景、不用过渡语，直接继续工作。

**自动压缩触发参数**：

| 参数 | 值 | 含义 |
|------|-----|------|
| `AUTOCOMPACT_BUFFER_TOKENS` | 13,000 | 低于此值自动触发压缩 |
| `WARNING_THRESHOLD_BUFFER_TOKENS` | 20,000 | 低于此值发出警告 |
| `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES` | 3 | 连续压缩失败熔断器 |

> [!important] CACHED_MICROCOMPACT 精简策略
> 当 `CACHED_MICROCOMPACT` 功能标志启用时，会话特定指南和工具结果总结指令被缩减——**认知经济学**：上下文紧张时不向模型展示冗余指令，只保留行动所需的最小规则集。

## OpenClaw 的三级 Compaction 策略

OpenClaw 的上下文压缩同样不是单一触发点，而是分**三级响应**：

| 级别 | 触发条件 | 位置 | 动作 |
|------|----------|------|------|
| **L1: Pre-request** | 每次调 LLM 前主动检查 | Context Engine `assemble()` | 预防性压缩，保持 token 预算 |
| **L2: Timeout-triggered** | LLM 首 token 超时**且** prompt > 65% context | `run.ts` 主循环 | 紧急压缩，用缩短的 prompt 立即重试 |
| **L3: Context overflow** | API 返回 `context_length_exceeded` | `run.ts` 主循环 | 三级降级：auto-compact → 截断 oversized tool result → 抛错 |

> [!note] L2 是容易忽视但很巧妙的一层
> LLM 首 token 慢不一定是服务端慢，很可能是 context 太大导致 prefill 耗时过长。用"超时 + 大 context"双条件判断主动压缩后重试，比盲目切换 profile 更经济。

### Compaction 的精细化实现

OpenClaw 的 Compaction 比 Hermes 更精细一层：

| 策略 | 说明 |
|------|------|
| `identifier-policy` | 压缩时保留 symbol identifiers（函数名/文件路径）的出现频次 |
| `identifier-preservation` | 压缩后验证关键 identifier 没丢 |
| `tool-result-details` | 专门处理 tool 调用输出的摘要格式 |
| `retry` | 压缩本身的重试（压缩调用失败时回退到原始 messages） |

> [!important] 压缩后验证
> OpenClaw 的 `identifier-preservation` 是一个关键的工程洞见——**压缩的质量直接决定 Agent 是否还能"记住"之前做了什么**。如果压缩丢掉了关键函数名或文件路径，Agent 会在后续轮次中找不到正确的代码位置，导致任务失败。

### 超预算的 4 路决策

检查到 overflow 后不是只有一个动作，而是分情境路由：

```
overflow 检测后决策：
├── fits ──────── 不需压缩
├── compact_only ─ 没工具结果可裁 → 只能 compact
├── truncate_only ─ 工具结果可裁足够 → 截断工具结果就够
└── compact+truncate ─ 两者都做
```

## 两种策略的对比

| 维度 | Claude Code（五层） | OpenClaw（三级） |
|------|---------------------|-----------------|
| **粒度** | 更细——从每次消息的 Budget 到紧急的 autoCompact | 更粗——pre-request / timeout / overflow |
| **最轻压缩** | microCompact（占位文本） | identifier-policy + truncation |
| **最重压缩** | autoCompact（LLM 生成结构化摘要） | L3 compact（LLM 压缩 + identifier 保留验证） |
| **可插拔性** | 紧集成于查询引擎 | 通过 Context Engine 契约可替换 |
| **恢复策略** | 压缩后恢复消息"直接继续" | Prompt warning 告知截断 → Agent 用工具补读 |
| **验证机制** | 错误记忆保持（Compact 保留 Errors and fixes） | identifier-preservation 验证关键标识符未丢失 |

## 压缩的保护区划分策略

受 [[上下文腐蚀与注意力衰减|Lost in the Middle]] 效应影响，压缩应遵循保护区划分：

```
上下文窗口中的保护区：

┌─────────────────────────────────────────────┐
│ ▓▓▓▓▓▓▓  ░░░░░░░░░░░░░░░░░░░░░  ▓▓▓▓▓▓▓▓▓ │
│  HEAD区     MIDDLE区（可压缩）        TAIL区  │
│  (不压缩)   (可压缩/可摘要)          (不压缩) │
└─────────────────────────────────────────────┘
```

- **HEAD 区**（系统指令、当前任务）：永不压缩
- **MIDDLE 区**（历史对话、检索文档）：可压缩、可摘要
- **TAIL 区**（最近几轮对话、当前用户输入）：不压缩
