# 上下文压缩

> 检索结果过长导致超出模型上下文限制或干扰注意力时，需要压缩只保留最相关部分。压缩不是等到窗口满了才做——而是**预防性的、分层的、渐进的**。上下文压缩的目标从来不是省 token——省钱是顺带的，它要解决的问题是**保护模型的注意力**。

## 第一代：撑不住才动手

第一代压缩的逻辑一句话能讲完：上下文满了 → 全量摘要替换。直观好实现，但有五个根本痛点：

| 痛点 | 说明 | 后果 |
|------|------|------|
| **悬崖式触发** | 不溢出时一动不动，溢出后一口气全量摘要 | 触发时模型已经信息过载走神，紧接着又被剥夺大半上下文 |
| **全量摘要丢细节** | 几十轮压成几百字 | 变量名、函数签名、错误堆栈、用户措辞全部丢失 |
| **Token 估算粗糙** | `text.length / 3` 估算 | 中英混合误差 30-50%，以为安全实际已溢出 |
| **不区分信息价值** | 5000 行 grep 输出和关键诊断同等对待 | 前者裁了不影响任务，后者裁了 Agent 立刻不会干活 |
| **用户内容一刀切** | 用户贴的代码和工具输出等同处理 | "用户贴的代码被压没了，模型忘了要改什么" |

> 根本问题：它把压缩当突发事件处理，而不是一种**持续维护的能力**。

## 通用压缩方法

| 方法 | 说明 | 优缺点 |
|------|------|--------|
| **Map-Reduce** | 把长文档分段处理，每段分别与问题做相关性判断 | 并行速度快，但可能丢失跨段信息 |
| **Refine** | 先用部分文档生成初步答案，再逐步引入其他文档完善 | 能利用上下文但速度较慢 |
| **相关性过滤** | 用轻量级模型计算每个片段与问题的相关性分数 | 简单高效，但依赖相关性算法质量 |
| **LLM压缩** | 用模型提取关键信息生成摘要 | 最高质量但最高成本 |
| **滚动摘要** | 定期将旧对话压缩为摘要，只保留精华 | 信息损失可控但增加推理成本 |

> [!tip] 实践建议
> 通常组合使用：先用轻量级方法粗筛，再用 LLM 精筛，既保证效果又控制成本。

## 第二代：六大产品的横向对比

过去一年，主流 Agent 不约而同走向"分层 + 渐进"，但味道各不相同——六家产品，六种哲学，说明这件事没有显而易见的最优解：

| 产品 | 核心策略 | 一句话概括 |
|------|----------|------------|
| **Claude Code** | 五段流水线，按成本递增排列 | 便宜的本地操作先上，LLM 摘要兜底 |
| **Codex CLI** | 保留近期用户消息原文，其余替换为 handoff 摘要 | 用户说的话最准确，模型说的可以重写 |
| **OpenCode** | 时间戳标记隐藏 + 结构化摘要 + 回放最后一条用户消息 | 不真删，理论上可恢复 |
| **Cline** | `/smol`（`/compact`）生成摘要后在同一任务内接续 | 自动 + 手动双模式 |
| **Cursor** | 自动摘要 + 提示开新对话 + 历史可搜索 | 压缩后仍能回溯原始历史 |
| **Amp** | 不做递归压缩，用 `/handoff` 开新线程携带要点 | 长对话本身就是问题，换线程比压缩好 |
| **MemGPT / Letta** | 上下文 = RAM，历史 = 磁盘，Agent 自主换入换出 | 操作系统级的内存调度 |

### 压缩哲学光谱：三种根本立场

六家产品可以归到三个哲学位置上——这不是细节差异，而是**根本的取舍**：

| 哲学立场 | 代表产品 | 核心判断 | 对 [[递归摘要衰减与语义漂移|递归衰减]]的态度 |
|---------|---------|---------|--------------------------|
| **Session-内压缩** | Claude Code、Cline、Cursor、OpenCode | 长对话可以靠分层压缩维持——衰减可以管理 | 接受衰减但减缓它（增量摘要、保护区、结构化输出） |
| **线程/会话切换** | Amp、Codex CLI | 长对话本身就是问题——与其让对话退化不如开新线程 | 绕过衰减（不压缩，用 handoff 要点替代） |
| **OS-level 内存调度** | MemGPT / Letta | Agent 应自主管理记忆层次——不是被动截断而是主动换入换出 | 转移衰减到外部存储（原始数据在 Archival Memory 中可检索，不靠摘要） |

> [!important] 光谱不是优劣排列
> 三种立场各有适用场景：Session-内压缩适合需要对话连续性的长任务；线程切换适合阶段分明的迭代开发；OS-level 调度适合跨会话长期记忆。一个成熟的 Agent 产品通常需要**组合使用**——MUR AI 的四级水位线是 Session-内压缩，但 `/handoff` 是线程切换，存储分离借用了 OS-level 的思路。

### Codex CLI：近期用户消息优先保护

Codex 在约 95% 容量时触发，生成一份 handoff 摘要替换旧历史。重建后上下文只剩三部分：handoff 摘要、近期约 20k token 内的用户消息原文、系统指令。所有 assistant 回复和工具结果物理删除，由摘要替代。设计哲学是把压缩当作**"同事间的工作交接"**。

### OpenCode：可逆隐藏 + 回放最后一条指令

两步走：
- **Prune（轻量，无 LLM）**：每次成功响应后自动触发，往回遍历消息，跳过最近 2 轮用户对话，保护最近 40k token 工具输出，更老的用时间戳标记为"已压缩"。数据还在数据库里，模型看到占位符 `[Old tool result content cleared]`。
- **Summary（重量，调 LLM）**：超模型输入上限时触发，生成五段式结构化摘要（目标/指令/发现/已完成/相关文件），自动回放用户最后一条消息——模型从最近指令继续而非从摘要继续。

### Cline：自动 + 手动双模式

从 v3.25 开始两种压缩：`/smol`（别名 `/compact`）手动触发，Auto-Compact 接近上限时自动触发，行为一致。如果开启 Focus Chain，待办列表穿越压缩存活，作为进度锚点。

### Cursor：压缩 + 可回溯

上下文超出时自动压缩 + 提示开新对话。2026 年新增 Dynamic Context Discovery——把聊天历史变成可搜索文件，即使压缩后 Agent 也能检索原始细节，A/B 测试减少 46.9% 总 token 消耗。已知问题：压缩后模型有时"忘掉"刚做的编辑（高优 bug）。

### Amp：不压缩，换线程

立场鲜明：递归摘要导致性能逐步衰减，干脆不做。替代方案是 `/handoff`——打包要点到新线程，用户可审查和编辑带过去的内容。线程是一等公民：`@@` 引用其他线程、`threads: map` 可视化关系。理念是**"一系列有焦点的短步骤，比一个逐渐退化的长对话好"**。但 2026 Neo CLI 更新也加了 90% 窗口自动上下文管理——对纯手动路线的妥协。

### MemGPT / Letta：上下文当 RAM

学术派代表，按操作系统内存层次建模：

| 层级 | 类比 | 容量 | 访问方式 |
|------|------|------|----------|
| Main Context | RAM | 模型上下文窗口大小 | 始终在 prompt 里 |
| Recall Memory | 交换分区 | 完整对话历史 | `conversation_search` |
| Archival Memory | 磁盘 | 无限（向量存储） | `archival_memory_search` |

关键区别：换入换出由 **Agent 自己决定**（函数调用），不是被动截断。代价是架构复杂度高、需外部向量存储、有检索延迟。适合跨会话长期记忆，对单会话压缩有点重。

## 第二代共识：七条原则

方案放在一起看，细节差异很大，但有七条原则几乎人人认同：

1. **分层渐进，不一刀切**：定义多个水位线，越接近上限手段越激进。系统永远在小幅维护，避免悬崖式塌方。
2. **成本严格递增**：便宜的先做（字符串截断、placeholder 替换），贵的最后做（LLM 摘要）。能用零成本释放的空间，不花钱买。
3. **增量摘要优于全量摘要**：保留一份活的摘要，每次只把新增部分合并进去。好处：单次输入更短更便宜也更准；避免"摘要的摘要"导致[[递归摘要衰减与语义漂移|语义漂移]]；合并时模型可主动取舍。
4. **用真实 token，别估算**：LLM API 每次返回 `usage.totalTokens`，免费、精确、唯一可信。`text.length / 3` 只在内部排序时凑合用，触发判断必须用真实值。
5. **用户消息有特权**：用户的指令、问题、代码——任务来源。Codex 一字不动，OpenCode 压缩后回放最后一条，其他方案至少保证用户纯文本不裁。
6. **保护近端**：无论怎么压，最近几轮不能动。定义保护区——如最近 8000 token 内的所有消息，任何级别都不参与压缩。
7. **单调边界，绝不滑窗**：stub 决策一旦做出，对应位置的字节必须从此固定。下一 turn、下一 step 再看到同一位置永远是同一个 stub。实现方式：按 part ID 持久化（Redis 或内存映射）直接复用；或交给服务端 `cache_edits` / `context_management` API，客户端字节零变化。

### 跨压缩生存策略：进度锚点与动态回溯

压缩会丢信息，但某些信息**必须穿越压缩存活**。两种具体策略：

**进度锚点（Cline Focus Chain）**：待办列表穿越压缩存活下来，作为 Agent 的进度锚点。即使对话历史被压缩成几百字的摘要，任务列表仍然在——Agent 知道"还剩什么要做"。这是把任务元数据从"可压缩的对话层"提升到"不可压缩的结构层"。

**动态上下文发现（Cursor Dynamic Context Discovery）**：把聊天历史变成可搜索文件，即使压缩后 Agent 也能回头检索原始细节。A/B 测试减少 46.9% 总 token 消耗。这是把"压缩"和"回溯"解耦——压缩是不可逆的，但历史是可搜索的。详见 [[可逆压缩]]。

> [!tip] 进度锚点的设计启示
> 压缩丢的是"过程细节"（具体 grep 输出、中间推理步骤），但不应该丢的是"任务状态"（还剩什么要做、关键决策是什么）。Focus Chain 的思路是：把任务状态从对话层抽出来，变成独立的结构化数据——任何 Tier 都不动它。

### 实施陷阱：滑窗式 stub 替换 = 每步缓存失效

> [!warning] 隐蔽的坑
> 第二代"分层渐进"如果实施得不对，会掉进另一个隐蔽的坑。设规则"保留最近 N 条 tool 结果，更老的替换成 stub"，如果判断在 step-loop 里每一步都重算——每完成一个 step（新增 2 条消息），就有 1 条旧 tool 结果滑出窗口被替换成 stub。它的字节变了，从这个位置往后的整段 prompt 前缀对 Prompt Cache 就失效了。
>
> 真实案例：4 轮、177 step、59 分钟的会话烧了 $77.3，其中 **83%（$64.8）全是 cache_write**。从 step 7 起前缀缓存就死了，剩下 50 多个 step 每一步都在为"窗口又挪了一格"买单。
>
> **stub 决策必须单调推进**——只能大跳，不能滑窗。一个 part 一旦被标成 stub，后续所有 turn、所有 step 里都保持 stub 不变。

## Claude Code 的五层递进压缩策略

Claude Code 的上下文压缩不是单一触发点，而是**五层递进**——体现 [[分层安全防御]] 的思想，不是等到完全耗尽才做最重的压缩，而是逐级应对：

```
原始消息
→ applyToolResultBudget()    — ① 工具结果大小限制（每次消息）
→ snipCompact()              — ② 片段压缩（功能开关控制）
→ microCompact()             — ③ 微压缩（缓存旧 tool_result）
→ contextCollapse()          — ④ 上下文折叠（PTL 错误时）
→ autoCompact()              — ⑤ 自动压缩（Token 缓冲不足时）
```

| 层级 | 策略 | 触发条件 | 代价 | 说明 |
|------|------|----------|------|------|
| **①** | `applyToolResultBudget` | 每次消息 | 无 | 工具结果的大小上限（不压缩，只截断过大输出） |
| **②** | `snipCompact` | 功能开关 `SNIP_COMPACT` 控制 | 极低 | 精简工具结果中的冗余内容 |
| **③** | `microCompact` | 旧 tool_result 替换 | 低 | 将旧工具结果替换为占位文本 `[Old tool result content cleared]`，不调用模型 |
| **④** | `contextCollapse` | PTL 错误（上下文超模型限制） | 中 | 排空积累的上下文后重试，或调用 `compact()` 将历史消息总结为压缩摘要 |
| **⑤** | `autoCompact` | Token 缓冲量低于 13,000 | 高 | 调用 LLM 生成结构化压缩摘要 |

### autoCompact 的结构化压缩

最重的压缩是 `autoCompact()`，它将长对话历史压缩为**结构化摘要**，使用两阶段设计：

1. **`<analysis>` 分析块**：按时间顺序分析每条消息，识别请求意图、方法策略、关键决策、文件名和代码片段、错误及修复方式
2. **`<summary>` 总结块**：九个结构化段落——Primary Request / Key Technical Concepts / Files and Code Sections / Errors and fixes / Problem Solving / All user messages / Pending Tasks / Current Work / Optional Next Step

**压缩后恢复消息**的关键指令：不确认摘要、不复述背景、不用过渡语，直接继续工作。

> [!important] 压缩时刻意保持消息序列前缀稳定
> 让 [[KV 缓存|Prompt Cache]] 命中率不会因为压缩而掉下来。

### 服务端路径：cache_edits 与 context_management API

Claude Code 内部还有两条更激进的路径——"把脏活交给服务端，客户端一个字节都别改"：

- **cached_microcompact**：把"删掉旧 tool 结果"包装成 API 层的 cache_edits 指令。客户端发出的 prompt 原封不动，服务端在已缓存的前缀上直接抠掉指定内容——字节没变，缓存不失效。
- **apiMicrocompact**：更彻底，直接调 Anthropic 的 context_management API（beta context-management-2025-06-27，Vertex / Bedrock 也支持），让服务端按 input_tokens 阈值自动裁剪旧的工具调用。

> 前面的本地五段流水线在很多场景下已经退居二线了——能让服务端做的就让服务端做，本地操作永远是兜底。

**自动压缩触发参数**：

| 参数 | 值 | 含义 |
|------|-----|------|
| `AUTOCOMPACT_BUFFER_TOKENS` | 13,000 | 低于此值自动触发压缩 |
| `WARNING_THRESHOLD_BUFFER_TOKENS` | 20,000 | 低于此值发出警告 |
| `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES` | 3 | 连续压缩失败熔断器 |

> [!important] CACHED_MICROCOMPACT 精简策略
> 当 `CACHED_MICROCOMPACT` 功能标志启用时，会话特定指南和工具结果总结指令被缩减——**认知经济学**：上下文紧张时不向模型展示冗余指令，只保留行动所需的最小规则集。

## OpenClaw 的三级 Compaction 策略

OpenClaw 的上下文压缩同样不是单一触发点，而是分**三级响应**：

| 级别 | 触发条件 | 位置 | 动作 |
|------|----------|------|------|
| **L1: Pre-request** | 每次调 LLM 前主动检查 | Context Engine `assemble()` | 预防性压缩，保持 token 预算 |
| **L2: Timeout-triggered** | LLM 首 token 超时**且** prompt > 65% context | `run.ts` 主循环 | 紧急压缩，用缩短的 prompt 立即重试 |
| **L3: Context overflow** | API 返回 `context_length_exceeded` | `run.ts` 主循环 | 三级降级：auto-compact → 截断 oversized tool result → 抛错 |

> [!note] L2 是容易忽视但很巧妙的一层
> LLM 首 token 慢不一定是服务端慢，很可能是 context 太大导致 prefill 耗时过长。用"超时 + 大 context"双条件判断主动压缩后重试，比盲目切换 profile 更经济。

### Compaction 的精细化实现

OpenClaw 的 Compaction 比 Hermes 更精细一层：

| 策略 | 说明 |
|------|------|
| `identifier-policy` | 压缩时保留 symbol identifiers（函数名/文件路径）的出现频次 |
| `identifier-preservation` | 压缩后验证关键 identifier 没丢 |
| `tool-result-details` | 专门处理 tool 调用输出的摘要格式 |
| `retry` | 压缩本身的重试（压缩调用失败时回退到原始 messages） |

> [!important] 压缩后验证
> OpenClaw 的 `identifier-preservation` 是一个关键的工程洞见——**压缩的质量直接决定 Agent 是否还能"记住"之前做了什么**。如果压缩丢掉了关键函数名或文件路径，Agent 会在后续轮次中找不到正确的代码位置，导致任务失败。

### 超预算的 4 路决策

检查到 overflow 后不是只有一个动作，而是分情境路由：

```
overflow 检测后决策：
├── fits ──────── 不需压缩
├── compact_only ─ 没工具结果可裁 → 只能 compact
├── truncate_only ─ 工具结果可裁足够 → 截断工具结果就够
└── compact+truncate ─ 两者都做
```

## 两种策略的对比

| 维度 | Claude Code（五层） | OpenClaw（三级） |
|------|---------------------|-----------------|
| **粒度** | 更细——从每次消息的 Budget 到紧急的 autoCompact | 更粗——pre-request / timeout / overflow |
| **最轻压缩** | microCompact（占位文本） | identifier-policy + truncation |
| **最重压缩** | autoCompact（LLM 生成结构化摘要） | L3 compact（LLM 压缩 + identifier 保留验证） |
| **可插拔性** | 紧集成于查询引擎 | 通过 Context Engine 契约可替换 |
| **恢复策略** | 压缩后恢复消息"直接继续" | Prompt warning 告知截断 → Agent 用工具补读 |
| **验证机制** | 错误记忆保持（Compact 保留 Errors and fixes） | identifier-preservation 验证关键标识符未丢失 |

## 压缩的保护区划分策略

受 [[上下文腐蚀与注意力衰减|Lost in the Middle]] 效应影响，压缩应遵循保护区划分：

```
上下文窗口中的保护区：

┌─────────────────────────────────────────────┐
│ ▓▓▓▓▓▓▓  ░░░░░░░░░░░░░░░░░░░░░  ▓▓▓▓▓▓▓▓▓ │
│  HEAD区     MIDDLE区（可压缩）        TAIL区  │
│  (不压缩)   (可压缩/可摘要)          (不压缩) │
└─────────────────────────────────────────────┘
```

- **HEAD 区**（系统指令、当前任务）：永不压缩
- **MIDDLE 区**（历史对话、检索文档）：可压缩、可摘要
- **TAIL 区**（最近几轮对话、当前用户输入）：不压缩
