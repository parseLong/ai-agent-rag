# 上下文预算与资源分配

> Token Budget 不只是财务工具——它是一个**行为塑形器**。通过告诉模型"你有 X tokens 可以花"，系统鼓励模型在复杂任务上投入更多推理步骤。更进一步的工程洞见：上下文窗口不是唯一的稀缺资源——runtime 中的每一种稀缺资源都应该显式量化，并配一条超限后的降级路径。

## Token Budget：从成本追踪到行为塑形

### Claude Code 的 Budget 告知策略

Claude Code 的 `TOKEN_BUDGET` 功能标志启用时，会将 Token 预算信息注入到系统提示词中——告诉模型"你有多少 tokens 可以花费"。这不只是财务约束，而是**行为塑形**：

```
没有 Budget 告知时：
  模型可能在简单任务上过度推理，在复杂任务上草率收工
  → "自我评估偏差"：模型以为自己完成了但其实没完成

有 Budget 告知时：
  模型知道有充足额度 → 在复杂任务上投入更多推理步骤
  模型知道额度紧张 → 更精简高效地行动
  → 类似给 ReAct 循环赋予了"时间资源维度"
```

### Budget 的 Continue 站点

在 Claude Code 的查询循环中，Token Budget 有专门的 Continue 站点：

```
站点 6: token_budget_continuation

当 Token Budget 模式启用时：
  ├─ 模型接近目标 → 自动续费继续
  ├─ 不需要在每个里程碑手动确认是否继续
  └─ 让长时程任务可以"自驱动"运行
```

## OpenClaw：10 种稀缺资源的显式量化

OpenClaw 对"预算"的理解不是"token 限额"那么窄，而是把 runtime 里的每一种稀缺资源都显式量化，并配一条超限后的降级路径：

| 稀缺资源 | 预算形式 | 超预算后的降级 |
|---------|---------|---------------|
| **上下文窗口** | `contextTokenBudget`（按模型解析，Context Engine assemble/compact/afterTurn 都传入） | 三级 Compaction 降级（L1 → L2 → L3） |
| **单次工具输出** | `MAX_TOOL_RESULT_CONTEXT_SHARE = 0.3`（30% 软限）+ 16K 字符硬限 | head + tail 截断，最小保留 2000 字符 |
| **启动上下文** | `bootstrap-budget.ts` 的 `maxChars` + `totalMaxChars` + `nearLimitRatio = 0.85` | 按优先级裁剪单文件 + 发 prompt warning |
| **循环迭代次数** | `MAX_RUN_LOOP_ITERATIONS`（按 profile 数动态算） | 抛 FailoverError，切模型 |
| **Overflow 压缩尝试** | `MAX_OVERFLOW_COMPACTION_ATTEMPTS = 3` | 放弃压缩，截断最大 tool result |
| **Timeout 压缩尝试** | `MAX_TIMEOUT_COMPACTION_ATTEMPTS`（独立计数） | 放弃压缩尝试，走 profile 轮换 |
| **凭证可用性** | Cooldown 时间预算（30s → 1min → 5min 分级递增） | 轮换到下一个 profile；冷却后允许探针式重试 |
| **子 Agent 递归** | `spawnDepth` + `subagentRole: orchestrator/leaf` | 达到深度后强制 leaf（禁止再 spawn） |
| **Lane 并发** | Cron / Subagent / Nested / Default 四车道独立队列 | 超过 lane 并发上限进队列等待 |
| **Steer 速率** | `STEER_RATE_LIMIT_MS`（父对子发送 steer 的节流） | 超过间隔内的 steer 被丢弃 |

> [!important] 核心洞见
> **预算不只是上限，而是降级路径**——每种资源都有"超了怎么办"的明确策略。这和 Hermes 用单一 `IterationBudget` 计数器的风格形成鲜明对比。OpenClaw 把"不确定性"从 runtime 中消灭：每种失败都有对应的处理，不会出现"不知道该怎么办"的悬空状态。

## 超预算的 4 路决策

OpenClaw 在检查到上下文 overflow 后不是只有一个动作，而是分情境路由：

```
let route: PreemptiveCompactionRoute = "fits";

if (overflowTokens > 0) {
  if (toolResultReducibleChars <= 0) {
    route = "compact_only";      // 没工具结果可裁 → 只能 compact
  } else if (toolResultReducibleChars >= truncateOnlyThresholdChars) {
    route = "truncate_only";     // 工具结果可裁够多 → 只截断不 compact
  } else {
    route = "compact_and_truncate"; // 都不够 → 同时 compact + 截断
  }
}
```

这 4 路决策不是"全靠压缩"的单一路径——而是根据具体情况选择最经济的降级方式。

## JIT 上下文检索 vs 预加载

上下文预算的另一个重要维度是**何时注入信息**：

| 策略 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **预加载（Pre-loading）** | 推理前一次性检索所有相关数据 | 速度快 | 占用大量窗口、信息可能过时 |
| **JIT 检索（Just-in-time）** | 维护轻量化引用，运行时通过工具动态加载 | 窗口干净、信息新鲜 | 需要更多工具调用 |

> [!quote] JIT 的认知模式
> JIT 的认知模式更贴近人类——我们不会死记硬背全部信息，而是用文件系统、收件箱、书签等外部索引按需提取。允许智能体自主导航与检索还能实现**渐进式披露**：每一步交互产生新上下文，反过来指导下一步决策。

### Claude Code 的 JIT 实践

Claude Code 的工具系统天然支持 JIT：
- `GrepTool` / `GlobTool`：按需搜索而非预加载整个代码库
- `ReadTool`：按需读取文件而非全量注入
- 子代理 Explore：专门做 JIT 的搜索探索

### OpenClaw 的 JIT 实践

OpenClaw 的 Context Engine `assemble()` 接口支持可插拔策略，包括检索型：
- 不预先加载所有相关数据
- 维护轻量化引用（文件路径、存储查询、URL 等）
- 运行时通过工具动态加载所需数据

## 混合策略：最实用的方案

在不少场景中，**混合策略**更有效：前置加载少量"高价值"上下文以保证速度，然后允许智能体按需继续自主探索。

```
混合策略示例：
├── 预加载：项目约定说明（README/指南） → 确保基本理解
├── JIT：glob/grep 原语 → 即时检索具体文件
└── 边界选择取决于任务动态性与时效要求
```

> [!tip] 引用的元数据也是信号
> `tests/test_utils.py` 与 `src/core/test_utils.py` 的命名暗示就不同。目录层级、命名约定、时间戳等都在隐含传达"目的与时效"。JIT 模式下这些元数据本身就能帮助精化行为。

## 预算塑形与上下文工程的协同

```
预算塑形 + 上下文工程的协同循环：

预算分配 → 决定哪些信息进入窗口
  ↓
上下文腐蚀 → 窗口越长腐蚀越严重
  ↓
压缩触发 → 预算紧张时主动压缩
  ↓
重置决策 → 压缩不够时触发 Reset
  ↓
新 session → 新的预算分配开始
```
