# SDD（Spec-Driven Development）

> 在 Agent 场景里，SDD 的核心作用不是"先写文档再开发"，而是**把一件事从模糊想法逐步转成可执行单元，并且把这个过程完整留下来**。

---

## 核心流程

```
模糊需求 → spec.md → plan.md → tasks.md → 执行 → 留痕
```

| 文档         | 职责               | 转化                                          |
| ---------- | ---------------- | ------------------------------------------- |
| `spec.md`  | 规格说明——目标、验收标准    | "分页有问题" → "切换页码后列表数据未刷新，原因是查询参数未传递 page 参数" |
| `plan.md`  | 技术方案——涉及文件、实现步骤  | 问题 → 方案                                     |
| `tasks.md` | 任务清单——每个任务的描述和状态 | 方案 → 可执行步骤，每一步有明确的输入、输出和完成标准                |

## 文档结构

```

│   ├── plan.md      # 技术方案：涉及文件、实现步骤
│   └── tasks.md     # 任务清单：每个任务的描述和状态
├── tasks.json       # 任务执行状态（供程序读取）
└── debug/
    ├── prompts/     # 每一步的 prompt
    └── agent.log    # 执行日志
```

## SDD 的价值

如果一次执行没有留下足够上下文，就回答不了四个关键问题：

1. 它当时看到了什么输入？
2. 为什么做出这个判断？
3. Prompt 在哪个环节失效了？
4. 哪些动作已经足够稳定，可以固化成 Skill？

没有这些记录，系统只能不断"重来一次"；有了这些记录，系统才可能真正"学会下一次做得更好"。

> **留痕不是为了 debug，而是为了进化。**

## SDD 与 Vibe Coding 的对比

| 维度 | [[Vibe Coding]] | SDD |
|------|-----------------|-----|
| 路径 | 先易后难 | 先难后易 |
| 前期 | 快速产出，成就感爆棚 | 写 spec 很慢，做设计很枯燥 |
| 后期 | 打地鼠，10倍 debug 时间还回来 | 执行、验证、迭代全有据可循 |
| 结果 | 越写越多，AI 上下文越来越混乱 | 每次执行可复盘 → 可重复 → 可规模化 |

> **大道如夷，而民好径。**

## SDD 是 Agent 自举的基础

Agent 能"自己修自己的 bug"，前提是三样东西：

1. **清晰的设计文档** —— AI 知道每个模块该做什么、不该做什么
2. **SDD 流程** —— spec → plan → tasks 的标准路径，AI 按照同样方式处理所有需求
3. **constitution.md** —— 架构约束文件，定义代码组织规范、命名规则、模块边界

没有这些基础设施，AI"自己修自己"就只是碰运气。有了这些，它才能在框架内工作，而不是自由发挥。
