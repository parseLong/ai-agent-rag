# Harness不是目的，知识才是护城河

> 腾讯 AI Team 的工程交付实践：==工作流是手段，知识是目的==。当 Harness Engineering 成为 2026 年最热门话题时，他们发现了一个被低估的事实——构建 Harness 工作流不是最终目的，私域和团队知识的沉淀才是真正的技术护城河。

---

## 核心论点 → [[知识沉淀护城河]]

> ==工作流只是管道，知识才是流过管道的活水。==

---

## Harness 三支柱与知识的位置

| 支柱 | 知识相关能力 |
|------|-------------|
| **上下文工程** | ==知识检索注入==、渐进式披露 |
| **架构约束** | 降级策略、安全边界 |
| **持续治理** | ==知识生命周期==、==自动衰减== |

> 知识管理本身就是 Harness Engineering 的==核心能力==，不是附属品。详见 [[驾驭工程]] 和 [[Harness Engineering深度解析]]。

---

## 知识分层架构 → [[知识分层架构]]

五层存储 × 五种类型 × 三级成熟度。关键设计：**知识可以"向上提升"**——Layer 3 的项目知识，被判定为跨项目通用时，自动提升到 Layer 1 或 Layer 2。

---

## 知识成熟度与衰减 → [[知识成熟度与自动衰减]]

> ==过时知识比缺失知识更危险==——因为它会误导 Agent。

---

## 团队知识库共建共享 → [[团队知识库共建共享]]

独立 Git 仓库作为"单一事实来源"，三种角色 + 区块链思想贡献模式 + 自动冲突解决。

---

## 工作流知识沉淀闭环 → [[工作流知识沉淀闭环]]

INIT 注入 → 各阶段按需查询 → ARCHIVE 自动提取。冷启动：`/flow-import` 通过 3 个 Agent 管道导入历史项目知识。

---

## 知识按需消费 → [[三级渐进式索引]]

从"推送"到"主动查询"的范式转变。Layer A(~50行) → Layer B(~300行) → Layer C(按需)。==上下文效率提升一个数量级==。

---

## 人机交互瓶颈 → [[人机交互瓶颈与远程操控]]

> ==好的 Harness 工程不仅要设计"Agent 怎么跑"，还要设计"人怎么随时参与"。==

---

## 落地经验

### 冷启动挑战
最大挑战不是设计架构，而是==让已有项目的隐性知识显性化==。做法：多源收集→渐进导入→断点恢复。

### Big Model vs Big Harness
详见 → [[知识沉淀护城河#Big Model vs Big Harness 的务实立场]]

### 文件系统即状态机
AI Team 的设计哲学：所有知识是 Markdown 文件——==可见性、可版本化、可迁移性、IDE 原生==。这不是技术妥协，而是刻意选择。这与 [[5条驭驭工程原则]] 中 System of Record 和 [[运行时环境（Runtime）]] 中"从无状态到有状态隔离运行时"的理念一致。

---

## 展望

| 方向 | 说明 |
|------|------|
| **语义检索增强** | 三级索引 + 向量检索实现语义级知识发现 |
| **跨团队知识联邦** | 不同团队仓库安全共享 Layer 1 技术知识，保护 Layer 2 业务知识边界 |
| **知识质量自动评估** | 模型评估知识条目的质量和时效性 |
| **全异步工作流** | Agent 自主执行非关键路径，仅关键决策点异步通知人类审批 |
